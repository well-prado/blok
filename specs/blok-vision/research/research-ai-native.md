# AI-Native Developer Platforms: Research Brief for the Blok Vision

## How they do it (concrete)

**MCP exposes a self-describing catalog of actions over JSON-RPC.** The whole protocol is three primitives: `tools/list` (discovery ā€” returns each tool's `name`, `title`, `description`, `inputSchema`, optional `outputSchema`, and `annotations`), `tools/call` (invocation by name + `arguments`), and `notifications/tools/list_changed` (the catalog is dynamic; the server pushes a signal and the client re-lists). Tools are **model-controlled** ā€” the LLM decides what to call from the descriptions alone, with no config file or manual wiring. Schemas are plain JSON Schema. Results carry `content[]` (text/image/resource_link/embedded resource) plus optional `structuredContent` validated against `outputSchema`. Errors split two ways: protocol errors (unknown tool, bad args) vs. tool-execution errors returned as a normal result with `isError: true` and a human-readable message the model can read and recover from. `annotations` (e.g. read-only vs. destructive) let clients gate dangerous calls behind human confirmation. ([MCP tools spec](https://modelcontextprotocol.io/specification/2025-06-18/server/tools))

**Install/distribution is converging on package managers + a registry.** MCP servers ship as npm/PyPI/Docker packages; the dominant install pattern is zero-install `npx -y @scope/server` wired into the client's config (`"command": "npx", "args": ["-y", "@pkg"]`). There's now an official registry at `registry.modelcontextprotocol.io` with `server.json` metadata + versioning + checksums, plus community directories (Smithery, Glama, mcp.so) and one-shot CLIs like `npx add-mcp <keyword>` that search a registry and write the config for you. ([packaging guide](https://chatforest.com/guides/mcp-server-packaging-distribution/), [one-click npx](https://dev.to/airbyte/one-click-mcp-installs-with-npx-189j), [add-mcp](https://github.com/neondatabase/add-mcp))

**Claude Code Skills are markdown-packaged procedures with progressive disclosure.** A skill is a directory: `SKILL.md` (required) + optional `template.md`, `examples/`, `scripts/`. Frontmatter is the trigger surface ā€” `description` (+ `when_to_use`, capped at 1,536 chars in the listing) tells the model when to auto-load; the directory name becomes the `/slash-command`. Key properties: the **body loads only when invoked** ("long reference material costs almost nothing until you need it"), `allowed-tools` pre-authorizes tools, `disable-model-invocation: true` makes it manual-only, `context: fork` runs it in a subagent, and `` !`cmd` `` injects live shell output (e.g. `git diff HEAD`) into the prompt before the model reads it. Skills can bundle scripts the model executes and a `plugin.json` to also ship MCP servers/hooks. Custom commands and skills have **merged** ā€” same `/name` interface. ([Claude Code Skills docs](https://code.claude.com/docs/en/skills))

**Anthropic's own tool-writing guidance** (the authoritative source on what works): consolidate related ops into one high-level tool (`schedule_event`, not `list_users`+`list_events`+`create_event`); namespace with prefixes (`asana_projects_search`); return **semantic** context not opaque IDs (names/URLs, not UUIDs ā€” resolving UUIDs to language "significantly improves precision"); cap responses (~25k tokens) with pagination/filtering defaults; write **steering error messages** that show the correct format and what to fix; and drive the whole thing with eval suites measuring tool-call patterns and error rates. ([Anthropic ā€” Writing tools for agents](https://www.anthropic.com/engineering/writing-tools-for-agents))

**Empirical backing for schema-first.** A controlled study found strict JSON-Schema tool specs cut **invalid (format) calls** from a mean of 5.39 ā†’ 3.72 vs. prose docs ā€” but schemas do **not** fix *semantic* misuse (schema-valid-but-pointless calls actually rose, 0.93 ā†’ 3.03), and field-level validation diagnostics did **not** beat a plain schema on recovery in their pilot. Schemas solve syntax; planning quality is a separate problem. (Caveat: this pilot had zero end-task success, so treat the recovery findings as weak.) ([Schema-First Tool APIs, arXiv](https://arxiv.org/html/2603.13404v1))

## Patterns worth stealing for Blok

- **Treat the node/workflow catalog as a `tools/list` surface.** A Blok MCP server should expose discovery (`blok_search_nodes`, `blok_search_workflows`), inspection (`blok_node_schema` ā†’ returns the node's Zod-derived JSON Schema), scaffolding (`blok_create_node`, `blok_create_workflow`), and install (`blok_install_node`/`blok_install_workflow` from the registry) ā€” each with a tight JSON Schema. The `listChanged` notification maps perfectly onto "you just installed a node, here are the new authoring actions."
- **Schema-as-contract is already half-built.** Blok nodes use Zod input/output schemas. Emit those as JSON Schema for both MCP tool definitions *and* for an "expected inputs" lookup the AI consults before writing a workflow step. This is the single highest-leverage AI-native move ā€” the LLM authors `inputs: {...}` correctly because it can fetch the exact schema first.
- **Steering error messages over error codes.** Blok's `MapperResolutionError` and Zod failures should return *what to write instead* ā€” e.g. "step `respond` references `$.state.fetch` but no step has id `fetch`; available ids: [...]". This is the recovery loop the AI needs.
- **Skills as the scaffolding layer.** Ship `blokctl`-backed skills: a `/blok-create-node`, `/blok-create-workflow`, `/blok-add-trigger` skill whose `SKILL.md` bundles the canonical templates (`examples/`, `template.ts`) and a `scripts/validate.sh` that runs `blok lint`. Progressive disclosure means the full authoring rules cost zero tokens until invoked. Inject live context with `` !`blokctl list-nodes` `` so the AI scaffolds against *installed* nodes, not hallucinated ones.
- **One-command install, registry-backed.** Mirror `npx add-mcp`: `blokctl add node <name>` / `blokctl add workflow <name>` resolving against the Blok registry with versioning + checksums. Make the MCP `install` tool a thin wrapper over the same CLI so AI-via-MCP and human-via-CLI hit identical code paths.
- **Consolidate authoring into high-level tools.** Don't expose one MCP tool per CLI flag. `blok_scaffold_workflow(intent, available_nodes)` that returns a complete, lint-passing workflow beats five granular tools the model has to orchestrate.

## Pitfalls / criticisms to avoid

- **Schemas fix syntax, not reasoning.** Don't expect JSON Schema alone to make the AI build *correct* workflows ā€” semantic misuse rises. Pair schemas with worked examples and an eval suite (Blok already has `WorkflowTestRunner` ā€” wire it into the AI loop so the model self-verifies). ([arXiv study](https://arxiv.org/html/2603.13404v1))
- **Deep nesting / optional-heavy schemas are footguns.** "Flexible nested dictionaries, complex configuration objects, optional fields ā€¦ frequently lead to missing keys, incorrect structure, or invalid values." Blok's persistence knobs (`ephemeral`/`spread`/`as`/`idempotencyKey`/`retry`/`maxDuration`/`subworkflow`ā€¦) are a lot of optional surface per step ā€” the AI will mis-combine mutually-exclusive ones (`as`+`spread`). Mitigate with explicit schema constraints and a validator that emits the exact "pick one" message. ([LLM-friendly API design](https://www.agentic-patterns.com/patterns/llm-friendly-api-design/))
- **Tool-list bloat.** Claude Code truncates skill descriptions past a char budget and the MCP `tools/list` competes for context. Don't expose 50 granular Blok tools; consolidate and namespace. ([skills doc](https://code.claude.com/docs/en/skills), [Anthropic tools](https://www.anthropic.com/engineering/writing-tools-for-agents))
- **Opaque identifiers.** Returning runtime IDs/UUIDs (run ids, node hashes) without semantic names degrades precision. Return node *names* and human-readable workflow summaries.
- **Trust/safety on install + destructive actions.** MCP spec mandates a human-in-the-loop for tool invocation and treats annotations from untrusted servers as suspect. Installing a node = running third-party code; the registry needs checksums/signing and the MCP `install` tool should be annotated destructive so clients gate it. ([MCP tools spec](https://modelcontextprotocol.io/specification/2025-06-18/server/tools))
- **Version drift with `npx`.** Unpinned `npx` "may get unexpected updates." Pin node/workflow versions in the registry resolution. ([one-click npx](https://dev.to/airbyte/one-click-mcp-installs-with-npx-189j))

## Specific lessons for the Blok vision ("an AI builds a Blok backend in a day")

1. **Ship a Blok MCP server as the AI's primary surface** ā€” three layers: discover (search registry + list installed), inspect (return Zodā†’JSON Schema for any node + a workflow grammar reference), act (scaffold + install + run-test). This directly serves vision items #3 (registry), #4 (versioned node packages), #6 (MCP install).
2. **The schema is the API.** Because Blok nodes already declare Zod input/output, you get LLM-correct authoring nearly for free by surfacing those schemas at author time. This is your moat vs. n8n (whose node params are loosely typed JSON). Lean into it for vision #7 (expression ergonomics): an MCP `blok_resolve_expression(expr, ctx_shape)` tool that validates `$.state.x` references against the actual workflow's step ids before the AI commits them.
3. **CLI + MCP + Skills share one engine.** `blokctl` is the kernel; MCP tools and Skills are thin presentation layers over it. Same install, same scaffold, same validate ā€” so human and AI paths can't diverge. (Matches your already-shipped `blokctl observability add` pattern ā†’ extend to `blokctl add node/workflow/trigger`, vision #5.)
4. **Steering errors close the loop.** Every load-time/run-time error (`set_var` removed, mutually-exclusive `as`+`spread`, unresolved `$.state.<id>`, branch `when` raw-ctx footgun) should already emit the fix ā€” Blok does some of this. Audit them to ensure every message names the available alternatives, because the AI's recovery rate is bounded by error-message quality.
5. **Bundle a small eval/test step into the AI workflow.** "Build a backend in a day" only works if the AI can self-correct. `WorkflowTestRunner` + `NodeTestHarness` exposed as an MCP `blok_test` tool lets the model verify before declaring done ā€” turning your test harness into an AI-native feedback signal.
6. **Re. vision #8 (is `.ts` the right format?):** for *AI authoring*, the JSON workflow shape (which already mirrors the TS DSL one-for-one per CLAUDE.md) is the more LLM-friendly target ā€” it's pure schema-validatable data with no execution semantics to reason about, exactly what the schema-first literature favors. Keep TS for humans who want types; have the AI emit/validate JSON and offer a JSONā†’TS codegen. (Flag: this is my inference from the design literature, not a benchmarked result for Blok specifically.)

## Uncertainty flags

- The arXiv schema-first study is an explicitly-labeled **pilot with zero end-task success** ā€” its directional finding (schemas cut format errors, not semantic errors) is plausible and matches Anthropic's guidance, but the recovery/diagnostics conclusions are weakly supported. Don't over-index on "field-level diagnostics don't help."
- The MCP **official registry** is young (v0.1) and evolving; treat versioning/checksum/signing specifics as a moving target before committing a Blok registry design to mirror it exactly.
- Lesson #6 (JSON over TS for AI authoring) is a reasoned inference, not measured on Blok. Worth an actual A/B in your later spec work.
