# Workflow Definition Formats & Expression Languages — Research Brief for Blok

*Flagged uncertainty up front: most "criticism" sources are vendor blogs and Medium posts, not neutral benchmarks. I weighted primary specs (CEL, JSONata, n8n docs) and direct vendor positioning over third-party listicles. The n8n official expressions doc 404'd; n8n expression specifics below come from the search-result corpus, not the primary page — treat them as directionally correct, not verbatim.*

## How they do it (concrete)

**Code-as-workflow (Temporal, Trigger.dev v3).** Workflows are plain functions in a real language (TS/Go/Python). You get the host language's loops, conditionals, types, npm packages, IDE autocomplete, unit tests, and — critically — *normal git diffs and PR review*. Trigger.dev's explicit pitch: "you write regular code… you get full IDE support… standard git workflows (readable diffs, pull requests, branches), and any npm package works as an integration." Temporal adds deterministic replay (event-sourced state machine) so crashes resume exactly where they died. Cost: no visual artifact, and the runtime must constrain the language (Temporal's determinism rules) — you can't just write arbitrary code.

**JSON (n8n).** Workflow is a JSON document of nodes + connections; the canvas is the primary authoring surface and JSON is the serialization. Expressions are `{{ ... }}` JavaScript snippets embedded in node parameters, referencing data via `$json`, `$node`, `$items`, env vars. Strength: instant value preview in the editor, 400+ prebuilt nodes, click-to-connect. Documented weakness: **JSON is a terrible git citizen** — exports carry node positions, internal IDs, timestamps, so "two exports of the same unchanged workflow look different to Git"; merge conflicts are "a JSON diff that looks like it was generated by a stressed-out compiler"; multi-writer edits cause *silent overwrites*. Teams are reaching for CRDTs and custom diff tools to cope.

**YAML (GitHub Actions, Argo Workflows).** Declarative documents; Argo models DAGs as Kubernetes CRDs (each step a container). Strengths: readable, version-controllable, "describe *what* not *how*." Documented weaknesses: "YAML authoring is painful — you have to look up payload schemas for every service call"; parallelism and control flow are constrained by what the DSL exposes; and the widely-cited line — **"JSON/YAML are awful languages for writing imperative programs—they add no value but introduce immense complexity."** YAML pushes you to re-invent loops/conditionals/expressions inside string fields once logic gets real.

**Visual-as-source-of-truth.** The round-trip-engineering literature is blunt: "the reversed model differs from the original unless aided by laborious annotations," and generated code "rapidly differs from the model." LangChain's *Not Another Workflow Builder* argues the visual middle tier is "squeezed from both directions" — simple cases go to no-code agents, complex cases "quickly get too complicated to manage in a visual builder," so code wins the hard end. Notably it does **not** discuss code↔canvas bidirectional sync at all — a tell that nobody has solved making visual the source of truth.

**Expression languages:**
- **CEL** — non-Turing-complete, mutation-free, linear-time, type-checked, compile-once/eval-many. The headline safety claim: *"While it's dangerous to blindly call eval() on a user's Python code, you can safely execute a user's CEL code."* Evaluates in nanoseconds–microseconds. Built for IAM policies, routing, access control — *boolean/value evaluation, not data reshaping*.
- **JSONata** — XPath-3.1-inspired query+transform; concise, JS-like function syntax, lenient array handling ("does the right thing" on arrays vs scalars). Powerful for reshaping. Criticisms: steep learning curve for non-FP users, smaller ecosystem than JSONPath/JMESPath, AST-interpreted so **slow on large data** (Nearform's "JSONata performance dilemma"), and a *silent-failure footgun* — "if any field along the path is undefined, there will be no errors… no data."
- **jq** — UNIX-pipe paradigm, C-based, fast, ubiquitous CLI. Less approachable: foreign paradigm, requires explicit array iteration or it errors.
- **Liquid** — Shopify template language; simple, safe, but fundamentally a *templating* tool (string output), not a query/transform or predicate language.
- **n8n JS expressions** — max power (full JS), instant preview, but it's `eval`-class: unsandboxed-feeling, no type checking, hard for static analysis/AI to verify.

## Patterns worth stealing for Blok

- **Code as source of truth, visual as a generated/synced view** — the consensus winner for AI-native + complex workflows. Blok's `.ts` workflows already sit here; this is the right side of the fence.
- **Compile-once / evaluate-many** (CEL) — Blok already resolves `$.` and `js/` before node execution; formalize the resolved form as a cached, type-checked plan.
- **Static type-checking of expressions** (CEL's mandatory check phase) — Blok's `$` proxy gives TS types at author time; lean harder into making *every* expression typed and statically validated, not a string that fails at runtime.
- **Two-tier expression strategy**: a small *safe* sandboxed language for the 90% case (predicates, field references, simple transforms) + a typed code escape hatch for the 10%. CEL/JSONata for the former; TS for the latter.
- **n8n's instant value preview** — the single best authoring-ergonomics feature in the space. Steal it for Studio regardless of format choice.

## Pitfalls / criticisms to avoid

- **Do not make JSON the source of truth.** The diffing/merge/silent-overwrite pain is real and well-documented; Blok already avoids this by keeping `.ts` canonical. Keep JSON workflows as an *import/interop* format, not the primary one.
- **Do not push imperative logic into YAML/JSON string fields.** That's how you reinvent a bad programming language. Branch/switch/loops belong in typed primitives (Blok already has `branch()`/`switch`) — not stringly-typed config.
- **JSONata's silent-undefined failure** directly contradicts Blok's recent move to `BLOK_MAPPER_MODE=strict` (fail-fast on bad expressions). If Blok adopts JSONata, it must wrap it to throw on undefined-path, not inherit the lenient default.
- **JSONata performance** — AST interpretation is slow on big payloads; don't put it on hot data-pipeline paths without measuring.
- **`js/` unsandboxed expressions** are the AI-friendliness and safety liability: an LLM can emit `js/ctx.state.x.y` that typechecks-clean but throws, and a marketplace can't safely run arbitrary downloaded JS expressions. This is the strongest argument to *augment* `js/` with a constrained language.
- **Visual round-trip is a trap if visual is canonical** — generated code drifts from the model. One direction must own truth.

## Specific lessons for the Blok vision

1. **Keep `.ts` as the canonical workflow format — but answer "is .ts best?" honestly: yes for humans and AI, no for the marketplace/visual layer.** TS gives types, diffs, PR review, npm, and is what LLMs author most reliably *as code*. But structured-output research shows LLMs are *most reliable* emitting schema-constrained JSON, and the visual canvas needs a serializable graph. Resolution: **TS is source-of-truth; a typed JSON IR is the compile target** that the canvas and registry consume. Blok already has the JSON mirror — promote it to a real intermediate representation with a published JSON Schema (gives IDE validation + LLM constrained decoding for free).

2. **The canvas should generate/round-trip the IR, never own it.** Edit on canvas → emit IR → regenerate/patch `.ts` (or store IR alongside). Don't try to make the canvas the source of truth — the literature says that drifts.

3. **Replace/augment `$.`+`js/` with a two-tier expression model:**
   - **Tier 1 (default, safe):** a constrained expression language for references, predicates (`branch` `when`), and light transforms. **CEL is the strongest candidate** for *predicates* (safe, fast, type-checked, AI-verifiable, sandboxable for marketplace nodes) — it directly fixes the `branch when` raw-ctx footgun in your memory by being statically checkable. **JSONata is the candidate for *data reshaping*** where CEL is too weak. Caveat: two languages = two things to learn; consider whether `$`-proxy-typed-TS already covers Tier 1 well enough.
   - **Tier 2 (escape hatch):** keep typed TS expressions via the `$` proxy for power users. But **rename/contain raw `js/`** so untyped, unsandboxed code is an explicit opt-in, not the default — important before any marketplace ships downloadable workflows.

4. **AI-native angle:** publish the JSON IR schema → LLMs author via constrained decoding (high reliability) → compile to `.ts`/canvas. Pair with the `$`-proxy types so the AI gets the same autocomplete contract a human does. This is the bridge between "AI assembles a backend in a day" and "humans review it as code."

5. **For Studio:** steal n8n's live expression preview; steal CEL's "type-check at config time" so a bad expression lights up red on the canvas before run.

**Net recommendation:** *Format* — keep TS canonical, formalize a JSON IR (with published schema) as the compile target that powers canvas + registry + AI authoring; never make JSON or visual the source of truth. *Expressions* — keep typed `$`-proxy TS as the power tier; add **CEL as a safe, statically-checkable predicate/condition language** (fixes the `branch when` footgun and enables sandboxed marketplace execution); evaluate **JSONata only if/when authors need real data-reshaping** the proxy can't express, and if adopted, force strict-undefined semantics to match `BLOK_MAPPER_MODE=strict`. Demote raw `js/` to an explicit opt-in escape hatch.
