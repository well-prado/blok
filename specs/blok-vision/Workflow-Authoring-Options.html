Blok · authoring rethink

How should a Blok workflow read?

Four directions for writing workflows, nodes, and triggers — each killing the $ / js/ syntax and moving layout to a separate file. The same "order intake" workflow is shown in every one, so you can compare readability head-to-head. Grounded in a source-level study of n8n and Trigger.dev.

The same workflow, five ways

Flip between today and each option. Order intake: validate the body → check stock via an HTTP node → branch → create the order & respond 201, or respond 409.

import { workflow, branch } from "@blokjs/helper";

export default workflow({
  name: "Order intake",
  version: "1.0.0",
  trigger: { http: { method: "POST", path: "/orders" } },   // trigger tangled into the object
  steps: [
    { id: "validate", use: "validate-order",
      inputs: { body: $.req.body } },                        // $ proxy -> "js/ctx.request.body"
    { id: "checkStock", use: "@blokjs/api-call",
      inputs: { url: "js/`https://inv.api/stock/${ctx.state.validate.productId}`" } }, // raw js/ string
    branch({
      id: "route",
      when: 'ctx.state.checkStock.inStock === true',         // raw ctx string - NOT $, the silent-500 footgun
      then: [
        { id: "createOrder", use: "db-create-order",
          inputs: { productId: $.state.validate.productId,
                    qty: $.state.validate.qty,
                    stock: $.state.checkStock } },
        { id: "ok", use: "@blokjs/respond",
          inputs: { status: 201, body: $.state.createOrder }, ephemeral: true },
      ],
      else: [
        { id: "no", use: "@blokjs/respond",
          inputs: { status: 409, body: { error: "out of stock" } }, ephemeral: true },
      ],
    }),
  ],
});
import { workflow, branch, http } from "@blokjs/helper";
import { validateOrder } from "../nodes/validate-order";
import { httpGet, createOrder, respond } from "@blokjs/helpers";

export default workflow("order-intake", (s) => {
  const validate   = s.step(validateOrder, { body: s.trigger.body });
  const checkStock = s.step(httpGet, {
    url: s.tpl`https://inventory.internal/stock/${validate.productId}`,  // typed interpolation
  });

  return branch(checkStock.inStock, {                         // boolean handle - no string
    then: (s) => {
      const order = s.step(createOrder, { productId: validate.productId, qty: validate.qty, stock: checkStock });
      return s.step(respond, { status: 201, body: order });
    },
    else: (s) => s.step(respond, { status: 409, body: { error: "out of stock" } }),
  });
}, { trigger: http.post("/orders") });

    use: blok/http

      url: https://inventory.internal/stock/${{ steps.validate.productId }}

  - id: route
    when: ${{ steps.checkStock.inStock }}            # branch on a boolean
    then:
      - id: createOrder             # custom DB node
        use: db-create-order
        with: { productId: ${{ steps.validate.productId }}, qty: ${{ steps.validate.qty }}, stock: ${{ steps.checkStock }} }
      - { use: blok/respond, with: { status: 201, body: ${{ steps.createOrder }} } }
    else:
      - { use: blok/respond, with: { status: 409, body: { error: out of stock } } }
import { defineWorkflow, http } from "@blok/sdk";
import { validateOrder } from "../nodes/validate-order";
import { createOrder } from "../nodes/create-order";

export default defineWorkflow(
  { id: "order-intake", trigger: http({ method: "POST", path: "/orders" }) },
  async (req, { run, respond }) => {
    const body = await run(validateOrder, { body: req.body });        // custom node

    const stock = await run(http.get, {                               // built-in node
      url: `https://inv.internal/stock/${body.productId}`,
    });

    if (!stock.inStock) {
      return respond(409, { error: "out of stock", productId: body.productId });
    }

    const order = await run(createOrder, {                            // custom DB node
      productId: body.productId, qty: body.qty, reserved: stock.reservationId,
    });
    return respond(201, order);
  },
);
import { flow, http, branch } from "@blokjs/flow";
import { validateOrder } from "../nodes/validate-order";
import { createOrder } from "../nodes/create-order";

export default flow("order-intake", (s) => {
  const req = http.post("/orders");                       // trigger -> typed handle

  const v     = s.step("validate", validateOrder, { body: req.body });
  const stock = s.http("checkStock", {                    // built-in HTTP node
    url: "https://inv.api/stock", query: { productId: v.out.productId },
  });

  return branch(stock.out.inStock, {
    then: (s) => {
      const order = s.step("createOrder", createOrder, {
        productId: v.out.productId, qty: v.out.qty, stock: stock.out,
      });
      return s.respond(201, order.out);
    },
    else: (s) => s.respond(409, { error: "out of stock" }),
  });
});

What's wrong with today

The baseline every option must beat

  • Three dialects for one idea. $.state.x (a Proxy), "js/ctx.state.x" (a raw string), and bare ctx.* in when: — same concept, three incompatible forms.
  • The $ proxy is invisible magic — it looks like a value but stringifies to "js/ctx.state.x" and gets eval'd at run time. Readers can't tell a reference from a literal.
  • Typos crash at 2am, not at compile. A wrong id is a runtime MapperResolutionError, never a tsc error.
  • The branch.when footgun is live — it must be a raw ctx.* string; a $ there silently mis-routes (the documented 500).
  • The data graph is invisible — reading steps[] top-to-bottom doesn't show who feeds whom; you grep $.state.X.

What's already good (keep it)

  • defineNode() is clean — Zod in/out, pure execute(). The pain is the wiring layer, not the node.
  • Ordered steps[] list — reads top-to-bottom; n8n reconstructs order from a name-keyed graph. Keep it.
  • Layout doesn't exist yet — moving it to a sidecar file is greenfield, zero migration.

What we learned from the competition

n8n — visual, but fragile

Wires steps by mutable display name ("connections": { "Webhook": … }) and stores position:[x,y] inside the workflow. Expressions are name-keyed strings: ={{ $('Webhook').json.x }}.

Take: never key on names; never inline layout — exactly your instinct.

Trigger.dev — code-first clarity

A workflow is a function. Steps are const x = await task.triggerAndWait(...); data flow is just variables, tsc-checked. No graph, no expression DSL — but no visual canvas either.

Take: the variable is the edge — kill the string DSL.

GitHub Actions — readable refs

${{ steps.build.outputs.url }} reads as English: step → its output → the field. One marker for "this is a reference," used by 100M+ people.

Take: id-based steps.<id>.field is the familiar, readable shape.

The options in detail

A

Refined Typed TS DSL

Best version of today
Reads like Temporal with GitHub-Actions clarity — type-checked & rename-safe, not a Proxy-to-eval round-trip.

Keep TypeScript as the source of truth, delete the $ proxy and every js/ string. Each step returns a typed handle; later steps read prior outputs as plain, autocompleted TS — checkStock.inStock, checked by tsc, not stringified and eval'd.

Devs first · read-only legible to non-coders · AI-friendly (typed feedback)
Data flow — replacing $
// today: opaque string, typo = runtime crash, three dialects
inputs: { url: $.state.checkStock.inStock }      // proxy -> "js/ctx.state..."
when:   'ctx.state.checkStock.inStock === true'  // raw-ctx string, the footgun

// Option A: one form, autocompleted, tsc-checked, rename-safe
s.step(createOrder, { stock: checkStock })       // whole output
branch(checkStock.inStock, { ... })              // boolean handle, same as any ref
Layout — a separate file
// workflows/orders/intake.layout.json   (separate, Studio-owned)
{ "version": 1,
  "nodes": {
    "validate":   { "x": 0,   "y": 0 },
    "checkStock": { "x": 220, "y": 0 },
    "order":      { "x": 440, "y": -80 }
  },
  "viewport": { "zoom": 1, "x": 0, "y": 0 } }
Pros
  • Zero new dialect — it's just TypeScript
  • tsc is the linter: typo'd ref → compile error
  • Full autocomplete, go-to-def, rename-refactor
  • Rename-safe; references key off the handle, not a name
Cons / risks
  • Build-time codegen to derive step id from variable name
  • Sound typed Proxy→binding is fiddly (today's $ is the warning)
  • Graph→code round-trip is lossy-ish (nested callbacks)
  • Developers only — non-coders read but don't author
B

Clean Declarative YAML

Readable by anyone
The most universally readable of the four — anyone who's seen a GitHub Actions file reads it cold.

A workflow you read like a recipe, top to bottom. References are ${{ steps.checkStock.inStock }} — the exact syntax 100M+ GitHub Actions users know. ${{ }} is the only expression marker, so literal vs. reference is unambiguous; pixels live in a sidecar.

Non-coders / ops / AI agents (the sweet spot) · devs lose compile-time ref safety
Data flow — replacing $
today  ->  { url: $.state.checkStock }   "js/ctx.state.checkStock"   when: 'ctx.state.x'
this   ->  { url: ${{ steps.checkStock.url }} }              when: ${{ steps.checkStock.inStock }}

trigger.body  replaces  $.req.body
steps.<id>    replaces  $.state.<id>      -- one grammar, three contexts (trigger, steps, env)
Layout — a separate file
// workflows/order-intake.canvas.json   (Studio-owned, runner ignores)
{ "for": "order-intake.blok.yaml",
  "nodes": {
    "validate":   { "x": 80,  "y": 40 },
    "checkStock": { "x": 80,  "y": 180 },
    "createOrder":{ "x": 280, "y": 320 }
  },
  "viewport": { "zoom": 1, "x": 0, "y": 0 } }
Pros
  • Reads top-to-bottom like a recipe — steps array is the order
  • One familiar grammar ${{ steps.x.y }}, zero $/js/ctx triad
  • Literal vs. reference visually obvious (the wrapper)
  • Loader-time id checking — typo fails at load
Cons / risks
  • YAML is stringly — not IDE-checked like a typed handle (needs schema LSP)
  • Inside ${{ }} you'll want some power — keep it path + vetted helpers, not full JS
  • New file format: YAML loader + Studio round-trip
  • A third surface unless YAML replaces JSON
C

Code-First Imperative

Cleanest for engineers
The most readable option for engineers, bar none — but trades away the visual-authoring audience unless paired with a declarative form.

A workflow is a plain async function. A step runs, returns a value, you assign it to a const, the next step uses that variable. The dependency graph is the variable bindings — stock.inStock instead of a string that secretly points at a step id. No $, no js/.

Devs: ideal · Non-coders: poorly (no visual authoring) · AI: excellent
Data flow — replacing $
// before (today)
inputs: { reserved: $.state.checkStock.reservationId }   // proxy -> "js/..." -> eval at runtime
when:   'ctx.request.query.x === "true"'                 // raw-ctx, a third dialect

// after (Option C) - one dialect: TypeScript
const stock = await run(http.get, { ... });
run(createOrder, { reserved: stock.reservationId });     // plain access, typed, rename-safe
if (req.query.x === "true") { ... }                      // it's just JS
Layout — a separate file
// orders/intake.layout.json - Studio derives the graph by tracing run() calls
{ "workflow": "order-intake",
  "nodes": {
    "validateOrder": { "x": 80,  "y": 120 },
    "http.get":      { "x": 320, "y": 120 },
    "createOrder":   { "x": 560, "y": 40  }
  },
  "viewport": { "zoom": 1, "x": 0, "y": 0 } }
Pros
  • Zero reference syntax — data flow is variables & return values
  • Compile-time safety + autocomplete + rename-refactor
  • One mental model: if/await/Promise.all/try-catch
  • Layout fully out-of-band, optional, auto-derivable
Cons / risks
  • Non-coders can't author or read it — the hard tradeoff
  • Static graph extraction is real work; dynamic flow → 'traced at runtime'
  • Graph→code round-trip is a code-generation problem
  • Imperative escape hatches (raw fetch, shared state) need lint guardrails
D

Fluent Pipeline / Hybrid

The sweet spot
Reads like a function (Trigger.dev) but stays a renderable list (n8n minus the name-keyed graph) — typed-A meets readable-B, with the $/js/ seam deleted.

An ordered list of named steps where each step's id becomes a variable the next steps read by name — v.out.productId, not a magic string. Reads top-to-bottom like a pipeline, stays 100% declarative so a canvas renders it without executing code, and the reference is a typed handle the compiler checks.

Devs: primary win · Non-coders: via the canvas (lossless round-trip) · AI: very high
Data flow — replacing $
// today:
inputs: { productId: $.state.validate.productId }   // proxy -> "js/ctx.state..." -> eval'd
when:   'ctx.request.query.x === "true"'            // 3rd dialect, raw ctx, footgun

// Option D - ONE dialect, same handle in inputs AND conditions:
inputs: { productId: v.out.productId }              // typed handle, no $, no js/, no ctx
branch(stock.out.inStock, ...)                      // compiles to { from: "validate", path: "productId" }
Layout — a separate file
// orders/intake.layout.json - logic file has ZERO coordinates (Windmill OpenFlow)
{ "flow": "order-intake",
  "viewport": { "x": 0, "y": 0, "zoom": 1 },
  "nodes": {
    "validate":    { "x": 120, "y": 80 },
    "checkStock":  { "x": 120, "y": 220 },
    "createOrder": { "x": 40,  "y": 360 }
  } }
Pros
  • One dialect everywhere — same handle in inputs AND branch() conditions
  • Typo = compile error (handles typed from node's Zod output)
  • Reads top-to-bottom yet stays declarative → canvas renders without running code
  • Rename-safe + layout-free logic; defineNode() untouched
Cons / risks
  • The builder is real engine surface (records edges → JSON), not just sugar
  • JSON wire form is wordier: { from, path } vs v.out.x
  • Branch-scoped handles can't escape their arm (correct, but a new rule)
  • Migration: existing $/js/ workflows need a codemod

My read — where I'd point

Option D (Fluent Pipeline) is the sweet spot for Blok specifically: it deletes the $/js/ seam and gives one typed, rename-safe dialect — while staying a declarative ordered list, so the Studio canvas can still render and round-trip it without executing code. That last property is the thing C gives up and the thing your whole visual-Studio vision depends on.

Pair it with Option B (YAML) as the declarative wire / non-coder surface. D's builder compiles to exactly the kind of id-based, edge-structured JSON that B's ${{ steps.x.y }} reads from — so devs write typed TS (D), non-coders and AI read/generate YAML (B), and both project to one workflow IR (this is spec S1). The canvas edits the same IR.

A is the safe "best version of today" if you want minimal change. C is the cleanest for engineers but loses the visual round-trip. This isn't final — it's the direction I'd validate first.
