# Blok Framework Fixes — Runtime Adapter Integration

This document details all fixes applied to integrate the Blok framework's HTTP trigger and runtime adapter system with PHP and Python SDK nodes. These issues affect any project using `runtime.php` or `runtime.python3` step types.

---

## 1. HTTP Trigger Response Handling (Critical)

**File:** `src/triggers/http/runner/HttpTrigger.ts` (line ~370)

**Problem:**
The HTTP trigger reads `ctx.response.data` to send the HTTP response body. However, for runtime adapter nodes (PHP, Python, Go, etc.), after `RunnerSteps.runSteps()` finishes, `ctx.response` is set to the **raw data** from the last step — not wrapped in the `{ data, contentType, success, error }` structure that module (JS) nodes use.

**Root cause:**
In `RunnerSteps.runSteps()`:
```javascript
const model = await step.process(ctx, step);
ctx.response = model.data;  // model.data = raw node return value for runtime nodes
```

For module nodes, `model.data` is `{ data: {...}, contentType: "...", success: true }`, so `ctx.response.data` works.

For runtime adapter nodes, `RuntimeAdapterNode.run()` returns:
```javascript
{ success: true, data: rawPhpOrPythonResult, error: null }
```
So `model.data` = `rawPhpOrPythonResult` (e.g., `{id, title, ...}`) and `ctx.response.data` is `undefined`, resulting in empty HTTP response bodies.

**Fix:**
```typescript
// Before (only works with module nodes):
const data = ctx.response.data;
c.header("Content-Type", ctx.response.contentType);

// After (works with both module and runtime nodes):
const hasWrapper =
    ctx.response &&
    typeof ctx.response === "object" &&
    "data" in ctx.response &&
    "contentType" in ctx.response;
const data = hasWrapper ? ctx.response.data : ctx.response;
const contentType = hasWrapper ? ctx.response.contentType : "application/json";
c.header("Content-Type", contentType);
```

**Impact:** Without this fix, ALL runtime adapter workflows return `200 OK` with an empty body, even though the workflow executed successfully. The data is processed internally but never sent to the client.

## 2. PDO Parameter Placeholder Syntax (Critical — PHP SDK)

**Files:** All PHP nodes using PostgreSQL queries (`runtimes/php/nodes/*.php`)

**Problem:**
PHP nodes generated by the project scaffolding used PostgreSQL-native `$1, $2, $3...` parameter placeholders in SQL queries:
```php
$stmt = $pdo->prepare('INSERT INTO tutorials (...) VALUES ($1, $2, $3, ...)');
$stmt->execute([$value1, $value2, $value3]);
```

PDO's `pdo_pgsql` driver does **NOT** support `$1` positional placeholders. PDO uses `?` (positional) or `:name` (named). The `$1, $2...` syntax is for PHP's native `pg_prepare()`/`pg_query_params()` functions, not PDO.

**Behavior:** The `$1` tokens are NOT treated as bound parameters. PDO prepares the statement but the `->execute([...])` values are never bound. PostgreSQL inserts `NULL` for all columns, even though the PHP code has valid data.

**Diagnosis challenge:** The error appears as a database NOT NULL constraint violation with ALL values being null — not as a "wrong parameter syntax" error. This makes it extremely difficult to debug because the PHP node's validation checks pass (the data IS correct in `$config`), but the SQL binding silently fails.

**Fix:**
Replace all `$1, $2, ...` with `?` in every SQL query:
```php
// Before (broken):
$stmt = $pdo->prepare('INSERT INTO tutorials (...) VALUES ($1, $2, $3)');

// After (correct):
$stmt = $pdo->prepare('INSERT INTO tutorials (...) VALUES (?, ?, ?)');
```

For dynamically built SQL (search/update with variable WHERE clauses), replace the `sprintf('$%d', $paramIndex)` pattern:
```php
// Before (broken):
$conditions[] = sprintf('search_vector @@ plainto_tsquery(\'english\', $%d)', $paramIndex);
$paramIndex++;

// After (correct):
$conditions[] = "search_vector @@ plainto_tsquery('english', ?)";
```

**Note on type casting:** For PostgreSQL-specific types like `text[]` and `jsonb`, use explicit casts with `?`:
```sql
-- Array type:
tags && ?::text[]
INSERT INTO ... (tags) VALUES (?::text[])

-- JSONB type:
messages || ?::jsonb
```

**Affected files (9 total):**
- `TutorialStoreNode.php` — GET, SEARCH, CREATE, UPDATE, DELETE queries
- `SessionManagerNode.php` — CREATE, GET, APPEND_MESSAGE, UPDATE_CONTEXT, CLOSE queries
- `ActionValidatorNode.php` — SELECT query
- `AuditLoggerNode.php` — INSERT query
- `WorkflowRegistryNode.php` — 2x SELECT queries

## 3. Runtime Adapter Config Wrapping (Critical — PHP & Python SDKs)

**Files:** All PHP/Python node `execute()` methods

**Problem:**
The Blok runner's `HttpRuntimeAdapter.createExecutionRequest()` sends node config as:
```json
{
  "node": {
    "name": "tutorial-store",
    "type": "runtime.php",
    "config": {
      "inputs": {
        "operation": "create",
        "title": "resolved value",
        "steps": [...]
      }
    }
  }
}
```

The PHP SDK's `NodeRegistry::execute()` passes `$req->node->config` as the `$config` parameter to nodes. The Python SDK does the same with `execution_request.node.config`.

So `$config` (PHP) / `config` (Python) is:
```
{ "inputs": { "operation": "create", "title": "resolved value", ... } }
```

But nodes access inputs directly:
```php
$config['operation'];  // NULL — it's at $config['inputs']['operation']
```
```python
config.get("message")  # None — it's at config["inputs"]["message"]
```

**Fix (PHP):**
Add at the start of every node's `execute()` method:
```php
public function execute(Context $ctx, array $config): mixed
{
    $config = $config['inputs'] ?? $config;
    // ... rest of method
}
```

**Fix (Python):**
Add at the start of every node's `execute()` method:
```python
def execute(self, ctx: Context, config: Dict[str, Any]) -> Any:
    config = config.get('inputs', config)
    # ... rest of method
```

The `?? $config` / `config` fallback ensures backward compatibility if the runner changes this behavior.

**Root cause in framework:** The `@blokjs/helper` `StepNode.addStep()` stores inputs as:
```javascript
this._config.nodes[config.name] = { inputs: config.inputs };
```

This `{ inputs: {...} }` wrapper persists through `Configuration.getNodes()` → `ctx.config` → `NodeBase.blueprintMapper()` → `HttpRuntimeAdapter.createExecutionRequest()` → `node.config`.

**Recommended framework fix:** Either:
1. Have `HttpRuntimeAdapter.createExecutionRequest()` unwrap inputs: `config: nodeConfig?.inputs || nodeConfig || {}`
2. Or have the PHP/Python SDKs unwrap `config.inputs` automatically in `NodeRegistry.execute()` / `node_registry.execute()`

## 4. Search Query Parameter Handling with PDO `?` Placeholders

**File:** `runtimes/php/nodes/TutorialStoreNode.php` (search method)

**Problem:**
When converting from `$1, $2` to `?` placeholders, the search query method has a subtlety: the `ts_rank()` function in the SELECT clause uses the same query parameter as the `search_vector @@` condition in the WHERE clause. With `$1`, you could reference the same parameter multiple times. With `?`, each `?` is a separate positional parameter.

Additionally, the count query (SELECT COUNT) doesn't include the `ts_rank()` select extra, so it needs fewer parameters than the data query.

**Fix:**
Maintain separate parameter arrays for the count and data queries:
```php
$conditionParams = [];  // params for WHERE clause only
$rankParams = [];       // params for SELECT clause (ts_rank)

if (is_string($query) && $query !== '') {
    $conditions[] = "search_vector @@ plainto_tsquery('english', ?)";
    $selectExtra = ", ts_rank(search_vector, plainto_tsquery('english', ?)) AS rank";
    $conditionParams[] = $query;
    $rankParams[] = $query;  // same value, separate param
}

// Count query — only WHERE params
$countStmt->execute($conditionParams);

// Data query — rank params (in SELECT) come before WHERE params
$dataStmt->execute([...$rankParams, ...$conditionParams]);
```

## 5. PHP SDK ReactPHP Body Buffer Limit (Critical)

**File:** `.blok/runtimes/php/src/Server/Server.php`

**Problem:**
The PHP SDK's `Server.php` creates a `ReactPHP\Http\HttpServer` with no explicit middleware configuration:
```php
$server = new HttpServer(function (ServerRequestInterface $request): HttpResponse {
    return $this->handleRequest($request);
});
```

Without explicit middleware, ReactPHP wraps the handler with default `RequestBodyBufferMiddleware`. The default buffer size is derived from PHP's `post_max_size` CLI value — but ReactPHP's **socket read buffer** is limited to **65,536 bytes (64KB)**.

When the Blok runner sends `/execute` requests to the PHP SDK, the request body includes the full `context.vars` object. As the workflow progresses, `ctx.vars` accumulates data from all previous steps — including embedding vectors (768 floats ≈ 15KB), retrieval results, AI responses, session data, etc. By the later steps in a pipeline, the request body easily exceeds 64KB.

**Behavior:** ReactPHP receives the request but the body is silently truncated to 0 bytes. The PHP handler sees an empty string, `json_decode` fails with "Syntax error", and the SDK returns HTTP 400. The Blok runner reports `HTTP runtime 'php' returned HTTP 400: Bad Request` with no further detail.

**Diagnosis challenge:** The error appears as a generic "invalid JSON" from the PHP SDK. The body is empty despite the `Content-Length` header showing the correct size (e.g., 89054 bytes). This only happens for later steps in the pipeline where `ctx.vars` has grown large — earlier PHP steps in the same workflow execute fine.

**Fix:**
Add explicit streaming + buffer middleware with a larger limit:
```php
use React\Http\Middleware\RequestBodyBufferMiddleware;
use React\Http\Middleware\RequestBodyParserMiddleware;
use React\Http\Middleware\StreamingRequestMiddleware;

$server = new HttpServer(
    new StreamingRequestMiddleware(),
    new RequestBodyBufferMiddleware(16 * 1024 * 1024), // 16 MB
    new RequestBodyParserMiddleware(),
    function (ServerRequestInterface $request): HttpResponse {
        return $this->handleRequest($request);
    }
);
```

**Impact:** Without this fix, any workflow with >64KB accumulated context (common in AI agent pipelines) will fail at later PHP steps with a cryptic 400 error.

## 7. Runtime Error Observability — The Biggest Problem (Critical — Architecture)

**Files:**
- `@blokjs/runner/dist/adapters/HttpRuntimeAdapter.js` (runner side)
- `@blokjs/runner/dist/RuntimeAdapterNode.js` (runner side)
- `@blokjs/runner/dist/RunnerSteps.js` (runner side)
- `.blok/runtimes/php/src/Server/Server.php` (PHP SDK)
- `.blok/runtimes/python3/bin/serve.py` (Python SDK)

**Problem:**
The runtime adapter architecture has **zero useful error propagation**. When a runtime node fails, the developer gets a generic one-line error with no indication of:
- **Which step** in the workflow failed
- **Which node** was being executed
- **What the SDK actually returned** (the response body is discarded on non-2xx)
- **What the node's input was** (the resolved config that was sent)
- **What the SDK-side error was** (stack trace, exception type, validation message)

This made every single fix in this document take 5-10x longer to diagnose than it should have.

### What actually happens today

**1. `HttpRuntimeAdapter.execute()` throws away the error response body:**
```javascript
// Current code in HttpRuntimeAdapter.js:
if (!response.ok) {
    throw new Error(
        `HTTP runtime '${this.kind}' at ${this.baseUrl} returned HTTP ${response.status}: ${response.statusText}`
    );
}
```
When the PHP/Python SDK returns a 400 or 500, the response **body** contains the actual error details (e.g., `{"errors": {"message": "invalid JSON: Syntax error"}}`). But this body is **never read**. The adapter throws immediately with only the HTTP status code.

**2. `RunnerSteps.runSteps()` doesn't log which step failed:**
```javascript
// Current behavior — no step name in the error:
const model = await step.process(ctx, step);
```
When a step fails, the error propagates up as a generic error. The runner's log output shows `"message": "HTTP runtime 'php' at localhost:9005 returned HTTP 400"` — but there's no mention of which of the 10 steps caused it. In a 10-step pipeline with 5 PHP steps, you have to guess.

**3. PHP SDK returns structured errors but nobody reads them:**
The PHP SDK correctly returns detailed error info:
```json
{
    "success": false,
    "data": null,
    "errors": {"message": "invalid JSON: Syntax error"}
}
```
But because the HTTP adapter throws on the status code alone (see #1 above), this detail is lost.

**4. Python SDK `ConnectionResetError` noise hides real errors:**
After every Python node execution, the Python SDK logs a `ConnectionResetError: [Errno 104] Connection reset by peer` traceback. This happens because the Blok runner closes the HTTP connection after reading the response, but the Python `http.server` logs the socket close as an error. These noisy tracebacks fill the logs and bury the actual error messages.

### Real debugging sessions caused by this

**Debugging the 64KB body buffer issue (Fix #5):**
- Error shown: `HTTP runtime 'php' at http://localhost:9005 returned HTTP 400: Bad Request`
- No indication it was the `format-response` step (step 8 of 10)
- No indication the body was empty (Content-Length: 89054, actual body: 0 bytes)
- Had to manually patch the PHP SDK Server.php with `error_log()` inside a running Docker container to discover the body was empty
- Had to write a Python script to binary-search the exact byte threshold (65,536 bytes)
- Total debug time: ~45 minutes for what should have been a 2-minute diagnosis

**Debugging the PDO placeholder issue (Fix #2):**
- Error shown: `SQLSTATE[23502]: Not null violation: null value in column "app_id"`
- No indication which SQL query failed or what parameters were (not) bound
- Had to add `error_log(json_encode($config))` to the PHP node inside a running container to verify the data WAS correct
- The actual cause (PDO ignoring `$1` placeholders) was invisible — no PHP warning, no PDO exception, just silent NULL insertion
- Total debug time: ~30 minutes

**Debugging the config wrapping issue (Fix #3):**
- Error shown: `Validation failed: 'operation' is required`
- Looked like the workflow wasn't passing the input — but it was
- Had to trace through 8 framework files (`StepNode.js` → `Configuration.js` → `NodeBase.js` → `Mapper.js` → `HttpRuntimeAdapter.js` → PHP SDK `Server.php` → `ExecutionRequest.php` → `NodeRegistry.php`) to understand the `{inputs: {...}}` wrapping
- Total debug time: ~40 minutes

### Recommended Fixes

**1. `HttpRuntimeAdapter.execute()` — Read error response body:**
```javascript
if (!response.ok) {
    let errorDetail = '';
    try {
        const body = await response.json();
        errorDetail = body?.errors?.message || JSON.stringify(body?.errors) || '';
    } catch { /* body not JSON */ }

    throw new Error(
        `HTTP runtime '${this.kind}' at ${this.baseUrl} returned HTTP ${response.status}: ${response.statusText}` +
        (errorDetail ? ` — ${errorDetail}` : '')
    );
}
```

**2. `RunnerSteps.runSteps()` — Include step name and index in errors:**
```javascript
try {
    const model = await step.process(ctx, step);
    ctx.response = model.data;
} catch (error) {
    const enrichedError = new Error(
        `Step ${index + 1}/${steps.length} "${step.name}" (node: ${step.node}, type: ${step.type}) failed: ${error.message}`
    );
    enrichedError.cause = error;
    throw enrichedError;
}
```

**3. `RuntimeAdapterNode.run()` — Log the request payload size:**
```javascript
const body = JSON.stringify(request);
logger.debug(`Sending ${body.length} bytes to ${this.adapter.kind} for node "${this.targetNode.node}"`);
```

**4. PHP SDK `Server.php` — Return Content-Length mismatch in error:**
```php
$body = (string) $request->getBody();
$expectedLength = $request->getHeaderLine('Content-Length');
$actualLength = strlen($body);

if ($actualLength === 0 && $expectedLength > 0) {
    return $this->jsonResponse(400, [
        'success' => false,
        'errors' => [
            'message' => sprintf(
                'Empty body received (Content-Length: %s, actual: 0). This usually means the request exceeded the body buffer limit.',
                $expectedLength
            ),
        ],
    ], $headers);
}
```

**5. Python SDK — Suppress `ConnectionResetError` noise:**
```python
# In the HTTP handler, catch and silence expected connection resets:
import logging
logging.getLogger('http.server').setLevel(logging.WARNING)
```
Or override `handle_one_request()` to not log `ConnectionResetError`.

**6. Add structured logging to the runner for every step:**
```
[step 1/10] validate-auth (runtime.php) → started
[step 1/10] validate-auth (runtime.php) → completed (23ms)
[step 2/10] init-session (runtime.php) → started
[step 2/10] init-session (runtime.php) → completed (45ms, payload: 1.2KB)
...
[step 8/10] format-response (runtime.php) → started (sending 89KB)
[step 8/10] format-response (runtime.php) → FAILED: HTTP 400 — "invalid JSON: Syntax error"
```

This single improvement would have reduced the total debugging time for all 5 fixes from ~3 hours to ~15 minutes.

## Summary of Framework-Level Fixes Needed

| # | Component | Fix | Severity |
|---|-----------|-----|----------|
| 1 | `HttpTrigger.ts` response handling | Support both wrapped and raw `ctx.response` | Critical — empty responses |
| 2 | PHP SDK code generation | Use `?` not `$1` for PDO placeholders | Critical — silent data loss |
| 3 | Runtime adapter config unwrapping | Unwrap `{inputs: {...}}` in adapter or SDK | Critical — all inputs null |
| 4 | PDO search with `?` placeholders | Separate param arrays for count vs data queries | Moderate — search breaks |
| 5 | PHP SDK ReactPHP body buffer | Add explicit 16MB `RequestBodyBufferMiddleware` | Critical — large payloads fail |
| 6 | Runtime error observability | Read error bodies, log step names, structured logging | **Critical — DX** |

### Recommended Framework Changes

1. **In `HttpRuntimeAdapter.execute()`:** Read the response body on non-2xx before throwing, so the SDK's error message is included in the error.

2. **In `RunnerSteps.runSteps()`:** Wrap each step execution in a try/catch that enriches errors with the step name, index, node name, and runtime type.

3. **In `HttpRuntimeAdapter.createExecutionRequest()`:** Unwrap the inputs before sending to SDK:
   ```javascript
   config: nodeConfig?.inputs || nodeConfig || {}
   ```
   This would eliminate the need for `$config = $config['inputs'] ?? $config` in every PHP/Python node.

4. **In `HttpTrigger` response handler:** Detect whether `ctx.response` has a `data` wrapper or is raw data, and handle both cases.

5. **In PHP SDK scaffold/documentation:** Use `?` PDO placeholders instead of `$1` PostgreSQL-native syntax. Update any code generators or templates.

6. **In PHP SDK `Server.php`:** Add explicit `StreamingRequestMiddleware` + `RequestBodyBufferMiddleware(16 * 1024 * 1024)` + `RequestBodyParserMiddleware` to the `HttpServer` constructor. The default 64KB socket buffer is insufficient for multi-step workflows where `ctx.vars` accumulates data from previous steps.

7. **In PHP SDK `Server.php`:** Detect Content-Length vs actual body mismatch and return a descriptive error instead of "invalid JSON".

8. **In Python SDK `serve.py`:** Suppress `ConnectionResetError` tracebacks from `http.server` that flood logs after every node execution.

9. **In the runner:** Add structured per-step logging with step index, name, node, runtime type, payload size, duration, and outcome (success/failure with error detail).
