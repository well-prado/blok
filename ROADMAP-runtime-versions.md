# Blok Runtime Version Management — Roadmap

> First-class runtime version detection, pinning, and enforcement for the Blok multi-runtime workflow engine.

---

## Motivation

Blok supports 8 language runtimes (Node.js, Python3, Go, Java, Rust, PHP, C#, Ruby) as sidecar containers, each communicating via `POST /execute` and `GET /health` HTTP endpoints. Today, runtime versions are detected at project creation time and shown as hints, but there is no mechanism to:

1. **Pin** a runtime version to a project (so the team uses the same version)
2. **Enforce** minimum version requirements at startup
3. **Declare** per-node runtime version constraints (e.g., "this node requires Python >= 3.11")
4. **Validate** that the developer's environment satisfies all constraints before running

This feature adds all four capabilities with the best possible developer experience.

---

## Design Principles

Drawn from deep research into industry-leading version management tools:

| Principle | Inspiration | Application in Blok |
|-----------|-------------|---------------------|
| **Transparent pinning** | Volta (JS toolchain manager) | Versions auto-detected and pinned at project creation, stored in committed config |
| **Strict-by-default** | pnpm engines, Go 1.21+ | Version mismatches are errors, not warnings. `--skip-version-check` to override |
| **Zero-config happy path** | mise (universal version manager) | If versions match, developer sees green checkmarks and nothing else |
| **Structured error messages** | Go toolchain, Rust MSRV | Every error: expected vs found, exact fix command, constraint source |
| **Per-package constraints** | Rust `rust-version`, Python `requires-python` | Nodes declare `runtimeRequirements` alongside their definition |
| **Familiar syntax** | npm semver ranges | `>=3.10.0`, `^1.22.0`, `~1.22.0` — syntax JS/TS developers already know |

## Feature Specification

### 1. Project-Level Version Pinning

**What:** When a Blok project is created, the detected runtime versions are recorded in `.blok/config.json` alongside a semver constraint.

**Schema extension** (additive, backward compatible):

```json
{
  "runtimes": {
    "python3": {
      "kind": "python3",
      "label": "Python 3",
      "port": 9007,
      "startCmd": "python3 bin/serve.py",
      "cwd": ".blok/runtimes/python3",
      "version": "3.12.0",
      "requiredVersion": ">=3.12.0"
    }
  }
}
```

- `version` — exact version detected at project creation (informational)
- `requiredVersion` — semver constraint enforced at `blokctl dev` startup

**Default constraint logic:** Detected version `X.Y.Z` becomes `>=X.Y.0` (allows patch updates within the same minor version).

### 2. Startup Version Enforcement

**What:** `blokctl dev` validates all runtime versions before spawning processes.

- **Pass:** Green checkmarks, dev server starts normally
- **Fail:** Structured error with expected/found/fix, exit code 1
- **Override:** `blokctl dev --skip-version-check` downgrades errors to warnings

### 3. Per-Node Runtime Requirements

**What:** Node authors can declare minimum runtime versions in `defineNode()`.

```typescript
export default defineNode({
  name: "ml-inference",
  description: "ML model inference",
  runtimeRequirements: {
    python3: ">=3.11.0",
  },
  input: z.object({ model: z.string(), data: z.unknown() }),
  output: z.object({ prediction: z.unknown(), confidence: z.number() }),
  async execute(ctx, input) { /* ... */ },
});
```

Enforced at workflow load time in `Configuration.runtimeResolver()`. When both project and node constraints exist, the stricter one wins.

### 4. `blokctl check` Command

**What:** A standalone validation command for CI/CD integration.

```
$ blokctl check

  Blok Runtime Version Check
  ──────────────────────────

  Project runtimes:
    ✓ Python 3  3.12.0 (requires >=3.10.0)
    ✓ Go        1.22.5 (requires >=1.21.0)

  Node requirements:
    ✓ ml-inference     python3 >=3.11.0 ✓
    ✓ data-transform   go >=1.21.0 ✓
    ✓ @blok/api-call   no requirements

  All checks passed.
```

Exit code 0 (pass) or 1 (fail).

### 5. Runtime Health Version Reporting

**What:** SDK containers report their runtime version via the `/health` endpoint. The runner extracts and stores this in `RuntimeRegistry` for runtime validation of node requirements.

### 6. Marketplace Integration

**What:** Marketplace packages declare runtime version requirements in their manifest. `>=` operator support added to `RuntimeDiscovery.resolveVersion()`.

## Semver Constraint Syntax

| Syntax | Meaning | Example |
|--------|---------|---------|
| `>=3.10.0` | Minimum version (default) | Python 3.10+ |
| `^1.22.0` | Compatible with 1.x (major locked) | Go 1.22+ but <2.0 |
| `~1.22.0` | Patch-level only (minor locked) | Go 1.22.x |
| `3.12.0` | Exact pin | Exactly Python 3.12.0 |

The `>=` operator is the default generated by the CLI. Advanced users can edit `.blok/config.json` to use caret, tilde, or exact pins.

## Implementation Phases

### Phase 0: Documentation
Create this roadmap and progress tracker.

### Phase 1: Semver Foundation
Shared semver utility with `parseSemver`, `compareSemver`, `satisfiesConstraint`, `computeDefaultConstraint`, `formatVersionMismatch`. Independently testable, no user-facing changes.

### Phase 2: Config Schema Extension
Add `version` + `requiredVersion` to `RuntimeConfig`. Update `setupRuntime()` and `writeProjectConfig()`. Add `validateProjectRuntimes()` function. Backward compatible.

### Phase 3: CLI Version Enforcement
Wire validation into `blokctl dev` with `--skip-version-check`. Update project creation hints. Create `blokctl check` command.

### Phase 4: Node-Level Runtime Requirements
Add `runtimeRequirements` to `FnNodeDefinition` + `FunctionNode`. Create `RuntimeVersionValidator`. Add version tracking to `RuntimeRegistry`. Enforce in `Configuration.runtimeResolver()`.

### Phase 5: Runtime Health Version Reporting
Enhance `HttpRuntimeAdapter.checkHealth()` to extract and store runtime version from SDK `/health` response.

### Phase 6: Marketplace Integration
Add `>=` operator to `RuntimeDiscovery.resolveVersion()`. Add `runtimeVersions` to `RuntimePackageManifest`.

## Industry Research Context

This design was informed by analysis of:

- **asdf / mise** — Universal version managers with `.tool-versions` / `mise.toml` config format
- **Volta** — JS toolchain manager with transparent `package.json`-based pinning
- **npm/pnpm engines field** — Declarative runtime constraints in `package.json`
- **Python `requires-python`** — PEP 621 minimum version in `pyproject.toml`
- **Rust `rust-version` (MSRV)** — Cargo.toml minimum supported Rust version, MSRV-aware resolver
- **Go `go.mod` directive** — Hard minimum Go version requirement since Go 1.21, automatic toolchain switching
- **Docker image pinning** — Version control via specific image tags and digests
- **Deno / Bun** — Notable for lacking built-in version constraint mechanisms (open issues)

Key takeaway: the best DX comes from **strict enforcement with escape hatches**, **zero-config on the happy path**, and **structured error messages with exact fix commands**.
