This example builds a simple workflow that fetches data from an
external API using the built-in `@blokjs/api-call` node and returns
the response.

## Prerequisites

- A Blok project with the HTTP trigger
- Node.js v22 or later

If you haven't created a project yet:

```bash
npx blokctl@latest create project
```

When prompted:
- Pick a project name
- Select **HTTP** as the trigger
- Select **NodeJS** as the runtime
- Choose **YES** when asked to install examples

## Workflow

Save this as `workflows/json/countries.json`:

```json
{
  "name": "countries.list",
  "description": "Fetches the list of countries with capitals",
  "version": "1.0.0",
  "trigger": {
    "http": { "method": "GET", "path": "/countries" }
  },
  "steps": [
    {
      "id": "fetch",
      "use": "@blokjs/api-call",
      "inputs": {
        "url": "https://countriesnow.space/api/v0.1/countries/capital",
        "method": "GET",
        "headers": { "Content-Type": "application/json" },
        "responseType": "application/json"
      }
    },
    {
      "id": "respond",
      "use": "@blokjs/respond",
      "inputs": { "body": "$.state.fetch" }
    }
  ]
}
```

The `name` follows the dotted `domain.action` convention so the
[typed client](/d/introduction/workflows#naming-convention) exposes it
as `blok.countries.list(...)`. Names must be unique across the
project — the registry rejects duplicates at boot.

This workflow:

<Steps>
  <Step title="Triggered by GET /countries">
    The HTTP trigger maps the route to this workflow file.
  </Step>
  <Step title="Calls the external API">
    The `fetch` step uses `@blokjs/api-call` to GET the country list.
    Its output auto-stores at `ctx.state.fetch`.
  </Step>
  <Step title="Returns the response">
    The `respond` step reads `$.state.fetch` and returns it.
  </Step>
</Steps>

## Run it

<Steps>
  <Step title="Start the project">
    ```bash
    bun run dev
    ```
  </Step>
  <Step title="Hit the endpoint">
    ```bash
    curl http://localhost:4000/countries
    ```
    You'll get a JSON response with the list of countries and their
    capitals.
  </Step>
</Steps>

## Customize

### Change the endpoint

Modify the `url` to fetch data from a different API:

```json
{
  "id": "fetch",
  "use": "@blokjs/api-call",
  "inputs": {
    "url": "https://your-api-endpoint.com/data",
    "method": "GET"
  }
}
```

### Add query parameters

```json
{
  "id": "fetch",
  "use": "@blokjs/api-call",
  "inputs": {
    "url": "https://countriesnow.space/api/v0.1/countries/capital?filter=region&value=Europe",
    "method": "GET"
  }
}
```

### Add authentication

```json
{
  "id": "fetch",
  "use": "@blokjs/api-call",
  "inputs": {
    "url": "https://api.example.com/data",
    "method": "GET",
    "headers": {
      "Content-Type": "application/json",
      "Authorization": "Bearer your-api-key"
    }
  }
}
```

### Cache the result

Add `idempotencyKey` to make repeat requests free:

```json
{
  "id": "fetch",
  "use": "@blokjs/api-call",
  "inputs": { "url": "https://countriesnow.space/api/v0.1/countries/capital" },
  "idempotencyKey": "countries-list",
  "idempotencyKeyTTL": 86400000
}
```

The first request hits the upstream API. Every subsequent request
within 24 hours returns the cached response without an HTTP call. See
[idempotent step caching](/d/reliability/idempotency).

### Transform the data

Add another step before `respond`:

```json
{
  "id": "filter",
  "use": "@blokjs/mapper",
  "inputs": {
    "mapper": {
      "europeOnly": "$.state.fetch.data.filter(c => c.region === 'Europe')"
    }
  }
}
```

Then point `respond` at `$.state.filter`.
