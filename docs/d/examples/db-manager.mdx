The Database Smart Query (DB Manager) example demonstrates how to create an AI-powered database query interface using Blok . This advanced example showcases how to build a workflow that allows users to query a PostgreSQL database using natural language prompts, which are then converted to SQL queries using AI.

## Features

- Interactive UI for database exploration
- Natural language to SQL conversion using AI
- Dynamic database schema introspection
- Real-time query execution
- Results visualization

## Prerequisites

Before running this example, ensure you have:

- A Blok  project set up with the HTTP trigger
- Node.js (v22 or later) and npm installed
- Docker and Docker Compose installed
- An OpenAI API key for the AI query generation

## Environment Setup

### 1. Set Up Your Blok  Project

If you haven't created a project yet, you can do so with:

```bash
npx blokctl@latest create project
```

Follow the prompts:
- Provide a name for your project
- Select "HTTP" as the trigger
- Select "NodeJS" as the runtime
- Choose "YES" when asked to install examples

### 2. Configure Environment Variables

Create a `.env.local` file in your project root with your OpenAI API key:

```
OPENAI_API_KEY=your_openai_api_key
```

Replace `your_openai_api_key` with your actual OpenAI API key.

### 3. Set Up the Required Infrastructure

The DB Manager example requires two infrastructure components that are already included in your project:

#### PostgreSQL Database

Navigate to the PostgreSQL infrastructure directory and start the containers:

```bash
# First, create the shared network if it doesn't exist
docker network create shared-network

# Navigate to the PostgreSQL directory
cd infra/postgresql

# Start the PostgreSQL containers
docker-compose up -d
```

This will start:
- A PostgreSQL database with the sample `dvdrental` database
- Adminer, a database management tool accessible at http://localhost:8080

#### Metrics Infrastructure (Optional but Recommended)

For monitoring your application, you can also start the metrics infrastructure:

```bash

cd infra/metrics

# Start the metrics containers
docker-compose up -d
```

This will start:
- Prometheus for metrics collection
- Grafana for metrics visualization (accessible at http://localhost:3000)
- Loki for log aggregation

## Workflow Structure

The DB Manager workflow lives in `src/workflows/db-manager.ts`. One
workflow serves the HTML UI **and** the JSON API by routing on
`(method, function-param)`. The four routes — UI, list tables, run a
prompt, and a fallback — collapse into a single `switchOn` on a
computed route key, with each handler's output flowing through typed
handles:

```ts src/workflows/db-manager.ts
import { workflow, step, switchOn, tpl, http } from "@blokjs/core";
import { ExprNode } from "@blokjs/helpers";
import databaseUi from "../nodes/database-ui";
import postgresQuery from "../nodes/postgres-query";
import queryGenerator from "../nodes/query-generator";
import errorNode from "../nodes/error";

const PG = {
  host: "localhost",
  port: 5432,
  database: "dvdrental",
  user: "postgres",
  password: "example",
};

export default workflow(
  "db-manager",
  { version: "1.0.0", trigger: http.any("/db-manager/:function?") },
  (req) => {
    // Collapse (method, function) into one route key the switch can read.
    const route = step("route-key", ExprNode, {
      expression:
        "`${ctx.request.method.toLowerCase()}:${ctx.request.params.function || 'index'}`",
    });

    switchOn(
      route,
      {
        cases: [
          // GET /db-manager — serve the HTML UI.
          { when: "get:index", do: () => { step("database-ui", databaseUi, {}); } },

          // GET /db-manager/tables — list public tables.
          {
            when: "get:tables",
            do: () => {
              step("get-tables", postgresQuery, {
                ...PG,
                query:
                  "SELECT table_name FROM information_schema.tables WHERE table_schema = 'public'",
              });
            },
          },

          // POST /db-manager/execute-prompt — schema → AI → run → shape.
          {
            when: "post:execute-prompt",
            do: () => {
              const columns = step("get-table-columns", postgresQuery, {
                ...PG,
                query: tpl`SELECT column_name, udt_name AS data_type FROM information_schema.columns WHERE table_schema = 'public' AND table_name = ${req.body.table_name}`,
              });

              const generated = step("query-generator", queryGenerator, {
                table_name: req.body.table_name,
                columns: columns.data,
                prompt: req.body.prompt,
              });

              const executed = step("execute-query", postgresQuery, {
                ...PG,
                query: generated.query,
              });

              // Shape the response — ExprNode reads the handles via ctx.state.
              step("respond", ExprNode, {
                expression:
                  "({ total: ctx.state['execute-query'].total, results: ctx.state['execute-query'].data, query: ctx.state['query-generator'].query })",
              });
            },
          },
        ],
        default: () => { step("method-not-allowed", errorNode, {}); },
      },
      { id: "route" },
    );
  },
);
```

A few v2 things to notice:

- **One `switchOn` instead of nested `@blokjs/if-else` + `conditions[]`.**
  A single `ExprNode` computes a `method:function` route key; the
  switch routes on it — far flatter than three levels of nested
  branches.
- **Data flows through typed handles.** `columns.data` feeds the AI
  generator, `generated.query` feeds the executor — no `$.state.<id>`
  strings. The final `respond` step reads `ctx.state[...]` inside an
  expression because it does object shaping the handle layer leaves to
  the node.
- **`http.any(...)` (the "ANY" method).** The handle DSL's wildcard
  HTTP route — never `"*"`.

## UI rendering with a route switch

A single workflow can serve both an HTML UI and JSON API responses by
routing on `(method, path)`. The DB Manager example computes a route
key and uses one `switchOn` to do exactly that.

### How UI rendering works

In traditional web apps you'd have separate routes for HTML and JSON.
In Blok, one workflow handles both:

<Steps>
  <Step title="Compute a route key">
    An `ExprNode` step collapses `(method, function-param)` into one
    string like `"get:tables"` or `"post:execute-prompt"`.
  </Step>
  <Step title="Switch dispatch">
    `switchOn(route, …)` matches the key against its `cases` and runs
    the matching arm's callback.
  </Step>
  <Step title="Response">
    The leaf step generates the response (HTML for UI, JSON for API).
  </Step>
</Steps>

### Route-key structure

In v2, N-way routing on one value uses `switchOn` over a computed
discriminant — far flatter than nesting `branch` arms:

```ts
const route = step("route-key", ExprNode, {
  expression: "`${ctx.request.method.toLowerCase()}:${ctx.request.params.function || 'index'}`",
});

switchOn(route, {
  cases: [
    { when: "get:index",  do: () => { step("database-ui", databaseUi, {}); } },
    { when: "get:tables", do: () => { /* … */ } },
    // …
  ],
  default: () => { step("method-not-allowed", errorNode, {}); },
}, { id: "route" });
```

Each case is a `{ when, do }` pair: `when` is a **literal** value (or
array of values) matched against the discriminant; `do` is a callback
where `step()` calls register into that arm. The `default` arm runs
when nothing matches. For two-way decisions reach for
[`branch`](/d/primitives/branch); for 3+ cases on one value,
`switchOn` keeps the intent flat.

### The Database-UI Node

The `database-ui` node is responsible for serving the HTML, CSS, and JavaScript for the UI. When a GET request is made to the root path (`/db-manager`), the route key is `"get:index"` and the `switchOn` routes the request to this node.

The node is implemented as a custom node that returns HTML content. The HTML includes:

1. **HTML Structure**: The basic structure of the page
2. **CSS Styling**: Styles for the UI components
3. **React Components**: JavaScript code for the interactive UI
4. **API Integration**: Code to communicate with the workflow's API endpoints

Here's a simplified example of how the `database-ui` node might be implemented using `defineNode()`:

```typescript
import { defineNode } from "@blokjs/core";
import { z } from "zod";
import fs from 'node:fs';
import path from 'node:path';

export default defineNode({
  name: "database-ui",
  description: "Serves the database UI HTML",
  contentType: "text/html",
  
  input: z.object({}),
  output: z.string(),
  
  async execute(ctx, inputs) {
    try {
      // Read the HTML file
      const htmlPath = path.resolve(__dirname, './ui/index.html');
      const htmlContent = fs.readFileSync(htmlPath, 'utf8');
      return htmlContent;
    } catch (error: unknown) {
      throw new Error((error as Error).message);
    }
  },
});
```

### The Complete UI Rendering Flow

When a user accesses the DB Manager at `http://localhost:4000/db-manager`, the following happens:

1. The HTTP trigger receives the GET request
2. The `route-key` step computes the discriminant — `"get:index"` for a bare GET
3. `switchOn` matches the `"get:index"` case, so the `database-ui` node is executed
4. The `database-ui` node returns HTML content with the appropriate Content-Type header
5. The browser renders the HTML, which includes the React application
6. The React application makes API requests to the other endpoints defined in the workflow

### API Endpoints in the Same Workflow

The same workflow also handles API requests through different route keys:

1. **GET /db-manager/tables**: Lists available database tables — route key `"get:tables"`
2. **POST /db-manager/execute-prompt**: Generates and executes an SQL query — route key `"post:execute-prompt"`

### Benefits of This Approach

Using a `switchOn` route key for UI rendering in Blok offers several advantages:

1. **Unified Workflow**: Both UI and API logic are contained in a single workflow
2. **Simplified Deployment**: No need for separate frontend and backend deployments
3. **Shared Context**: UI and API can share the same context and configuration
4. **Flat Routing**: One `switchOn` over a computed key replaces deeply nested branches
5. **Content Negotiation**: Different content types can be served based on request parameters

### Implementing Your Own UI Rendering

To implement UI rendering in your own Blok workflows:

1. **Create a UI Node**: Implement a node with `defineNode()` that returns HTML content
2. **Compute a Route Key**: Collapse `(method, path-param)` into one string and `switchOn` it
3. **Set Content-Type**: Set `contentType: "text/html"` in the node definition
4. **Include Client-Side Code**: Add JavaScript for client-side interactivity
5. **Define API Endpoints**: Add more `cases` to the same `switchOn` for API endpoints

## The Postgres-Query Node

The DB Manager example uses the `postgres-query` node to interact with the PostgreSQL database. This node is a custom implementation that allows executing SQL queries against the database.

Here's a simplified version of the `postgres-query` node using `defineNode()`:

```typescript
import { defineNode } from "@blokjs/core";
import { z } from "zod";
import pg from "pg";

export default defineNode({
  name: "postgres-query",
  description: "Executes SQL queries against a PostgreSQL database",
  
  input: z.object({
    user: z.string(),
    password: z.string(),
    host: z.string(),
    port: z.number().optional().default(5432),
    database: z.string(),
    query: z.string(),
  }),
  
  output: z.object({
    total: z.number(),
    data: z.array(z.record(z.unknown())),
  }),
  
  async execute(ctx, inputs) {
    const { Client } = pg;
    const client = new Client({
      user: inputs.user,
      password: inputs.password,
      host: inputs.host,
      port: inputs.port,
      database: inputs.database,
    });

    await client.connect();
    const result = await client.query(inputs.query);
    await client.end();

    return {
      total: result.rowCount || 0,
      data: result.rows,
    };
  },
});
```

The `postgres-query` node is used in three places in the workflow:

1. **get-tables**: To fetch the list of available tables from the database
2. **get-table-columns**: To fetch the columns of a specific table
3. **execute-query**: To execute the SQL query generated by the AI

## Custom Node Implementations

The DB Manager example uses several custom nodes:

### 1. QueryGeneratorNode

This node uses OpenAI to generate an SQL query based on a natural language prompt and table schema:

```typescript
import { defineNode } from "@blokjs/core";
import { z } from "zod";
import { createOpenAI } from "@ai-sdk/openai";
import { generateText } from "ai";

const columnSchema = z.object({
  column_name: z.string(),
  data_type: z.string(),
  primary_key: z.string().optional(),
});

export default defineNode({
  name: "query-generator",
  description: "Generates SQL queries from natural language prompts using AI",
  
  input: z.object({
    table_name: z.string(),
    columns: z.array(columnSchema),
    prompt: z.string(),
  }),
  
  output: z.object({
    query: z.string(),
  }),
  
  async execute(ctx, inputs) {
    const { table_name: tableName, columns, prompt } = inputs;

    // Format column information
    const tableSchema = columns
      .map(
        (col) => `${col.column_name} (${col.data_type}${col.column_name === col.primary_key ? ", PRIMARY KEY" : ""})`,
      )
      .join(", ");

    // Generate SQL query using AI
    const openai = createOpenAI({
      compatibility: "strict",
      apiKey: process.env.OPENAI_API_KEY,
    });

    const ai_prompt = `Table: ${tableName}
Schema: ${tableSchema}

Generate a SQL query for the following request: ${prompt}

Return ONLY the SQL query with no explanations, additional text or markdown code group.

Double check the query to not include markdown code blocks or any other text that is not a valid SQL query.`;

    const { text: sqlQuery } = await generateText({
      model: openai("gpt-4o"),
      system: `You are a SQL expert. Generate only valid SQL queries without any explanations or markdown. The query should be executable directly against a PostgreSQL database.`,
      prompt: ai_prompt,
    });

    return {
      query: sqlQuery,
    };
  },
});
```

### 2. Response shaping with `ExprNode`

The final `respond` step shapes the executor's output into the JSON
the UI expects. No custom node needed — the built-in `ExprNode`
(`@blokjs/expr`) evaluates an expression against `ctx` and returns it:

```ts
import { ExprNode } from "@blokjs/helpers";

step("respond", ExprNode, {
  expression:
    "({ total: ctx.state['execute-query'].total, results: ctx.state['execute-query'].data, query: ctx.state['query-generator'].query })",
});
```

## Troubleshooting

If you encounter issues:

1. **Database Connection Issues**:
   - Ensure the PostgreSQL container is running: `docker ps | grep postgres`
   - Check the connection parameters in the workflow (the `PG` config object)
   - Verify that the `dvdrental` database is properly initialized

2. **AI Query Generation Issues**:
   - Verify your OpenAI API key in the `.env.local` file
   - Check the OpenAI API usage limits and quotas
   - Examine the error messages in the console

3. **UI Rendering Issues**:
   - Check the browser console for JavaScript errors
   - Verify that the `database-ui` node is correctly returning HTML content
   - Ensure the Content-Type header is set to `text/html`

## Conclusion

The DB Manager example demonstrates the power and flexibility of Blok  for building complex applications. By combining UI rendering, API endpoints, database access, and AI integration in a single workflow, you can create sophisticated applications with minimal code and configuration.

Key takeaways:

1. **Unified Workflow**: Both UI and API logic are contained in a single workflow
2. **Conditional Logic**: A `switchOn` over a computed route key enables routing and content negotiation
3. **Custom Nodes**: Specialized nodes like `postgres-query` and `query-generator` encapsulate specific functionality
4. **AI Integration**: Natural language processing enhances the user experience
5. **Database Access**: Direct database interaction without additional middleware

This example serves as a template for building your own database-driven applications with Blok .
