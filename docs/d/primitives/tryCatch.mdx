`tryCatch` wraps a step group with error recovery. If any step in `try`
throws, control jumps to `catch` with a typed **error handle** populated.
Otherwise `try` runs to completion and `catch` is skipped. Each arm is a
**callback** where `step()` calls register into that arm.

The saga + workflow-compensation pattern â€” transactional multi-step writes
with rollback.

## Quickstart

```ts
import { workflow, step, tryCatch, branch, eq, http } from "@blokjs/core";
import stripeCharge from "../nodes/stripe-charge";
import stripeRefund from "../nodes/stripe-refund";
import insertOrder from "../nodes/insert-order";

export default workflow("orders.save", { version: "1.0.0", trigger: http.post("/orders") }, (req) => {
  tryCatch("save-with-rollback", {
    try: () => {
      step("charge", stripeCharge, { amount: req.body.amount });
      step("record", insertOrder, { order: req.body });
    },
    catch: (err) => {
      branch("compensate", eq(err.stepId, "record"), {
        then: () => { step("refund", stripeRefund, { reason: err.message }); },
      });
    },
  });
});
```

If `record` throws (a DB constraint violation, say), the `catch` arm fires the
refund. The `err` handle tells it *which* step failed.

## The signature

`tryCatch(id, { try, catch, finally? })`:

- **`try`** â€” a callback. The first step that throws jumps to `catch`.
- **`catch`** â€” a callback that receives the **error handle** (`err` below).
- **`finally`** â€” an optional callback that runs after `try`/`catch`,
  succeed or fail.

## The error handle

The `catch` callback receives a typed handle modeling the runtime error
envelope. Read its fields like any other handle:

| Field | Source |
|---|---|
| `err.message` | The deepest error in the cause chain. Strips the framework's `[step N/M] X failed: â€¦` wrap. |
| `err.name` | `Error.name` of the deepest error. |
| `err.stack` | `Error.stack` when present. |
| `err.code` | First `GlobalError.code` walking the cause chain. For `@blokjs/throw` errors this is the explicit code; for Zod validation errors it's `400`. **Optional** â€” `undefined` for non-`GlobalError` throws. |
| `err.stepId` | The id of the step that threw. **Optional.** Use it to route the recovery. |

`stepId` is the headline. Real catch arms typically route on it:

```ts
catch: (err) => {
  switchOn(err.stepId, {
    cases: [
      { when: "charge", do: () => { step("notifyBilling", notifyBilling, { reason: err.message }); } },
      { when: "record", do: () => { step("refund", stripeRefund, { reason: err.message }); } },
      { when: "ship", do: () => { step("releaseInventory", releaseInventory, {}); } },
    ],
  }, { id: "route-by-failure" });
},
```

<Warning>
The `err` handle is **scoped to the `catch` arm** â€” reading it in `try`,
`finally`, or after the `tryCatch` resolves to `undefined`. Default
`err.code` / `err.stepId` before branching on them; both are optional.
</Warning>

## Reading state inside catch

A step that throws **does not persist state** (the v0.5.1 fix). So inside
`catch`, routing on `err.stepId` is a *truthful* "this step did not succeed"
check â€” exactly what selective compensation needs. Branch on which step threw:

```ts
catch: (err) => {
  // the charge step threw â†’ nothing to undo; anything later â†’ refund the charge
  branch("maybe-refund", ne(err.stepId, "charge"), {
    then: () => { step("refund", stripeRefund, { reason: err.message }); },
  });
},
```

`catch` only fires for **step errors**. It does **not** fire for `wait`
deferrals or run cancellation (`ctx.signal`) â€” those are control signals, not
failures. Don't put cleanup that must run on cancel inside a `catch`.

## Common patterns

<AccordionGroup>
  <Accordion title="Multi-step saga with selective rollback" icon="rotate-left">
    Roll back only the steps that actually succeeded, gated on their handles:

    ```ts
    tryCatch("signup-saga", {
      try: () => {
        const account = step("createAccount", createAccount, { email: req.body.email });
        step("sendWelcome", sendWelcome, { to: account.id });
        step("scheduleOnboarding", scheduleOnboarding, { user: account.id });
      },
      catch: (err) => {
        branch("rollback", ne(err.stepId, "createAccount"), {
          then: () => { step("deleteAccount", deleteAccount, { reason: err.message }); },
        });
      },
    });
    ```
  </Accordion>
  <Accordion title="Try a sub-workflow, recover" icon="git-branch">
    A sub-workflow's failure surfaces as a single error in the catch arm â€”
    re-queue for later instead of failing the parent:

    ```ts
    tryCatch("process", {
      try: () => { step("run", null, { payload: req.body }, { subworkflow: "expensive-pipeline" }); },
      catch: () => { step("requeue", workerPublish, { queue: "retry-queue", payload: req.body }); },
    });
    ```
  </Accordion>
  <Accordion title="Compose with retry" icon="rotate-cw">
    Step-level [`retry`](/d/reliability/retry) handles transient failures;
    `tryCatch` handles "if it *ultimately* fails, do something different." A
    `try` step with `retry: { maxAttempts: 3 }` runs three attempts before the
    catch arm fires:

    ```ts
    try: () => {
      step("flaky", apiCall, { url: req.body.url }, { retry: { maxAttempts: 3 } });
    },
    ```
  </Accordion>
</AccordionGroup>

## Footguns

- **The `err` handle is catch-scoped** â€” it reads `undefined` outside `catch`.
- **`err.code` / `err.stepId` are optional** â€” default them before branching.
- **Unique step ids across arms** â€” `try`, `catch`, and `finally` share the
  flat per-workflow id namespace. Reusing one throws at load time.
