title: "switch — n-way routing"

`switchOn` dispatches to the first case whose `when` matches a discriminant
handle. Cleaner than nested `branch` when you have three or more arms on the
same value, and the array `when` syntax groups related cases without
duplicating logic. Each case's `do` is a **callback** where `step()` calls
register into that arm.

Used in the [webhook router example](/d/examples/webhook-router) to route
GitHub's many event types (push / pull_request / issues / …) by header value.

## Quickstart

```ts
import { workflow, step, switchOn, http } from "@blokjs/core";
import handlePush from "../nodes/handle-push";
import handlePr from "../nodes/handle-pr";
import handleIssues from "../nodes/handle-issues";
import { LogNode } from "@blokjs/helpers";

export default workflow("github.webhook", { version: "1.0.0", trigger: http.post("/webhook") }, (req) => {
  switchOn(req.headers["x-github-event"], {
    cases: [
      { when: "push", do: () => { step("push", handlePush, { event: req.body }); } },
      {
        when: ["pull_request", "pull_request_review", "pull_request_review_comment"],
        do: () => { step("pr", handlePr, { event: req.body }); },
      },
      { when: ["issues", "issue_comment"], do: () => { step("issues", handleIssues, { event: req.body }); } },
    ],
    default: () => { step("unknown", LogNode, { level: "warn", message: "unknown event" }); },
  }, { id: "route" });
});
```

Three cases, four event types — `pull_request*` collapses into one group via
the array `when`.

## The signature

`switchOn(discriminant, { cases, default? }, { id })`:

- **`discriminant`** — a **handle** (a step output field or a trigger field).
  It lowers to the switch's `on` expression and is resolved against the live
  `ctx` at run time.
- **`cases`** — an ordered list of `{ when, do }`. The first `when` match wins.
- **`default`** — an optional fallback callback when no case matches.

<Warning>
Each `when` must be a **static literal** — a string, number, boolean, or an
array of those. A handle passed as `when` is rejected at author time: the
switch engine matches `when` by literal `===` with no mapper resolution, so a
handle would never match. Read the discriminant into the discriminant handle,
and write each case label as a plain literal.
</Warning>

## Matching semantics

The discriminant is compared **by string equality** to each `when` entry (or
any element when `when` is an array). For non-string discriminants both sides
are `String()`-coerced before comparing.

Need range or numeric comparisons? Use [`branch`](./branch.mdx) with a `gt` /
`lt` comparator instead — `switchOn` is for exact-match routing.

## Using a result downstream

Like `branch`, handles minted **inside** a case arm don't escape it. When two
arms produce a value you need later, write them to one shared key with `as:`
(unique step ids, same `as` target):

```ts
switchOn(req.body.tier, {
  cases: [
    { when: "enterprise", do: () => { step("scoreEnt", score, { v: 100 }, { as: "score" }); } },
    { when: "free", do: () => { step("scoreFree", score, { v: 1 }, { as: "score" }); } },
  ],
}, { id: "route" });

// both arms wrote ctx.state.score — readable by id "score" downstream
step("respond", RespondNode, { body: { score: req.body.id } });
```

This is footgun #1 — see [Handles & footguns](/d/primitives/handles-and-footguns).

## Footguns

- **Static `when` only** — a `when` is matched literally with no mapper pass;
  a handle is rejected at author time.
- **Arm-scoped handles** don't escape their case — use a shared `as:` key.
- **Unique step ids** — every step id is flat per workflow, including across
  cases and `default`. Reusing one throws at load time.

Each case's `do` callback can contain any primitive — nested `branch`,
`forEach`, `tryCatch`, or sub-workflow steps. Reliability knobs
(`idempotencyKey`, `retry`) go on the regular steps inside a case, never on
the `switchOn` itself.
