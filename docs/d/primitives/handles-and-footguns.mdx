The control-flow primitives — [`branch`](./branch.mdx),
[`switchOn`](./switch.mdx), [`forEach`](./forEach.mdx),
[`tryCatch`](./tryCatch.mdx) — all mint and read **typed handles**. A handle is
a reference to a step's output: `step("fetch", …)` returns one, and so does the
trigger entry (`req`, `job`, …). A step inside an arm is a step — its output
lands on `ctx.state[<id>]` (its handle) on success, exactly like a top-level
step.

That single shared namespace is what makes the primitives compose, and it's
also where every "my control flow misbehaves" report comes from. This page is
the contract — what a handle is readable from, and the four footguns it
surfaces. Two of them (duplicate step id, forEach state-key collision) are
**rejected at workflow load time** so they fail loudly; the arm-scope reads are
guarded by the step builder's `canRead` check at author time.

## The handle contract

A *handle* is a typed reference to a step's output, minted by `step()`,
`forEach()` (the results array), or the trigger entry argument. Handles passed
into a later `step()`'s `inputs` are lowered automatically — you never write
`$.state.<id>` or `js/ctx…` by hand.

| You can read a handle… | Outcome |
|---|---|
| produced **before** the control-flow step | readable inside either arm and after the step |
| produced **inside an arm**, read in the **same arm** (or a descendant) | readable |
| produced inside an arm, read in a **sibling arm** | not readable — a `then` handle isn't visible in `else`, case A not in case B, `try` not in `catch` |
| produced inside an arm, read **after** the flow step | not readable — arm-local handles don't survive the arm |
| a `forEach` item / index handle, read **after** the loop | not readable — only the loop's returned results array survives |
| any handle on an `ephemeral: true` step | not readable — `ephemeral` skips persistence, so no state slot exists |

The rule in one line: **a handle is readable from its producing scope or a
descendant of it — never from a sibling scope or after the scope exits.**

### Getting a value *out* of an arm

When a downstream step needs a value one arm computed, you have two sanctioned
moves:

1. **Return it from the control-flow step** and read the flow step's own handle
   (`branch`/`forEach`/`switchOn` return one).
2. **Write both arms to the same `as:` key** and read that key after the flow
   step — see footgun #1 below.

```ts
branch("route", gt(order.total, 100), {
  then: () => { step("premium", score, { v: 1 }, { as: "tier" }); },
  else: () => { step("standard", score, { v: 0 }, { as: "tier" }); },
});

// both arms wrote ctx.state.tier — the next step reads it uniformly by id "tier"
step("respond", RespondNode, { body: req.body });
```

### Typed conditions, never strings

`branch` and `switchOn` conditions are built from handles, not hand-written
strings:

| Condition input | Built from |
|---|---|
| `eq(order.status, "done")` | a comparator + handle + literal |
| `gt(order.count, limit)` | a comparator + two handles |
| `order.inStock` (boolean field) | a truthiness check on a handle |

Use the typed comparators from `@blokjs/core` — `eq`, `ne`, `gt`, `gte`, `lt`,
`lte`, `not`. A raw `"ctx.state.x > 10"` or a `js/…` string bypasses the typed
lowering and throws at runtime — the comparators emit the correct form for you.
A `switchOn` `when` is the opposite: it must be a **static literal**, never a
handle (the switch engine matches by literal `===` with no resolution).

## The four footguns

### 1 — Arm-scoped handles don't escape their arm

A handle minted **inside** a `then` / `else` / case / `try` / `catch` arm is
scoped to that arm. Reading it from a sibling arm, or after the flow step,
won't find it — the step builder's `canRead` guard rejects the read at author
time, and at runtime there's no state slot.

The fix is the [escape moves above](#getting-a-value-out-of-an-arm): return the
value from the flow step, or write both arms to one `as:` key.

<CodeGroup>
```ts Wrong
branch("route", gt(order.qty, 10), {
  then: () => { step("bulk", route, { lane: "bulk" }); },
  else: () => { step("standard", route, { lane: "standard" }); },
});
// `bulk` / `standard` handles are gone here — reading either is undefined
step("notify", notify, { lane: /* ??? — neither handle survives */ });
```
```ts Right
branch("route", gt(order.qty, 10), {
  then: () => { step("bulk", route, { lane: "bulk" }, { as: "lane" }); },
  else: () => { step("standard", route, { lane: "standard" }, { as: "lane" }); },
});
// both arms wrote ctx.state.lane — readable by id "lane"
step("notify", notify, { lane: req.body.email });
```
</CodeGroup>

### 2 — Ephemeral handles read undefined

`ephemeral: true` skips state persistence entirely — there is **no**
`ctx.state[<id>]` slot for that step. The value is available only via the
*immediately* next step (through `ctx.prev` under the hood); its handle reads
`undefined` anywhere else.

Use `ephemeral` for side-effect-only steps (logging, metrics, audit) whose
output nobody reads downstream. The moment a later step needs the value, drop
`ephemeral`.

```ts
// Wrong — `audit` is ephemeral, so its handle is undefined downstream
step("audit", LogNode, { event: "viewed" }, { ephemeral: true });
step("respond", RespondNode, { body: /* reading the audit handle → undefined */ });

// Right — ephemeral steps end the chain; read a real handle instead
const user = step("load", loadUser, { id: req.params.id });
step("audit", LogNode, { event: "viewed", id: user.id }, { ephemeral: true });
step("respond", RespondNode, { body: user });
```

### 3 — Never reuse a step id (throws at load)

Every step id is a key in **one flat per-workflow config map** — including ids
inside mutually-exclusive `branch` / `switchOn` / `tryCatch` arms. Two steps
with the same id collide: last definition wins, so the matched arm silently
runs with the *other* arm's inputs.

The builder **throws at load time** before that can happen,
naming both colliding arm paths.

<CodeGroup>
```ts Wrong
switchOn(req.body.kind, {
  cases: [
    { when: "a", do: () => { step("h", score, { v: 1 }); } },
    { when: "b", do: () => { step("h", score, { v: 2 }); } }, // ← duplicate "h"
  ],
}, { id: "route" });
```
```ts Right
switchOn(req.body.kind, {
  cases: [
    { when: "a", do: () => { step("scoreA", score, { v: 1 }, { as: "score" }); } },
    { when: "b", do: () => { step("scoreB", score, { v: 2 }, { as: "score" }); } },
  ],
}, { id: "route" });
// unique ids; both write ctx.state.score via `as:` for a uniform downstream read
```
</CodeGroup>

If two arms must feed the same downstream key, use `as:` with **distinct** ids
(`runA` / `runB` → `as: "run"`).

### 4 — Never name a forEach `as:` / `asIndex` after an existing key (throws at load)

A `forEach` writes its current item to `ctx.state[<as>]` and the index to
`ctx.state[<as>Index]` on every iteration. Those keys live in the **same flat
`ctx.state`** as step outputs, so naming `as:` after an existing step id (or
another `forEach`'s `as`) silently clobbers one with the other.

The builder **throws at load time** for both collision shapes — including
collisions with an `as:`-renamed step's key, and nested loops that reuse an
enclosing loop's `as`.

<CodeGroup>
```ts Wrong
const item = step("item", load, {});            // step id "item"
forEach(item.rows, (row) => {                    // default `as` derives "item" → collides
  step("proc", process, { row });
}, { as: "item" });
```
```ts Right
const rows = step("rows", load, {});
forEach(rows.data, (row) => {
  step("proc", process, { row });
}, { as: "row" });
```
</CodeGroup>

## Quick reference

```
HANDLE:   step()/forEach()/trigger-entry mint typed refs — pass them into inputs directly
SCOPE:    readable in its own scope or a descendant; never a sibling/after
ESCAPE:   return from the flow step, OR write both arms to one `as:` key
CONDITION: build with eq/ne/gt/gte/lt/lte/not — never a raw string; switchOn `when` is a static literal
GUARDS (throw at load time):
  duplicate step id across arms        → names both arm paths
  forEach `as`/`asIndex` ↔ existing key
SCOPE GUARD (author-time canRead):
  arm-local / post-arm / ephemeral handle reads → rejected
NEVER:    reuse a step id · name `as:` after a step id · read an ephemeral/cross-arm handle
```
