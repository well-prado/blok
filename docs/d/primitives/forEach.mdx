---
title: "forEach — iterate"
description: "Sequential or parallel iteration over an array. The iteration body is a nested pipeline with full access to the live ctx."
icon: "list-checks"
---

`forEach` runs a nested pipeline once per item in an array. Iterations
run sequentially by default; flip to parallel by setting
`mode: "parallel"` and a `concurrency` budget.

Used in the [worker fan-out example](/d/examples/worker-fanout) to
parallel-enqueue jobs, and in the
[chat-memory example](/d/examples/chat-memory) implicitly via the
LLM agent loop.

## Quickstart

### Sequential

```json
{
  "id": "process-each",
  "forEach": {
    "in": "js/ctx.request.body.items",
    "as": "item",
    "do": [
      {
        "id": "process",
        "use": "@blokjs/api-call",
        "inputs": {
          "url": "js/`https://internal/process/${ctx.state.item.id}`",
          "method": "POST",
          "body": "js/ctx.state.item"
        }
      }
    ]
  }
}
```

### Parallel with concurrency cap

```json
{
  "id": "fan-out",
  "forEach": {
    "in": "js/ctx.request.body.items",
    "as": "item",
    "mode": "parallel",
    "concurrency": 5,
    "do": [
      { "id": "enqueue", "use": "@blokjs/worker-publish", "inputs": { "queue": "jobs", "payload": "js/ctx.state.item" } }
    ]
  }
}
```

1000 items don't serialize 1000 RPCs — five run concurrently, the next
batch starts as one finishes.

## Config fields

| Field | Type | Default | Notes |
|---|---|---|---|
| `in` | `string` (JS expression) | required | Resolves to an array. Non-array values become `[]` (no iterations). |
| `as` | `string` | required | Variable name written to `ctx.state[<as>]` for each iteration. |
| `do` | `Step[]` | required | Steps to run per item. |
| `mode` | `"sequential"` \| `"parallel"` | `"sequential"` | Iteration order. |
| `concurrency` | `number` | `1` | Max in-flight iterations when `mode: "parallel"`. Ignored in sequential mode. |

## Iteration ctx variables

Inside the `do` pipeline:

| Variable | Value |
|---|---|
| `ctx.state[<as>]` | The current item. |
| `ctx.state[<as>Index]` (sequential) | 0-based index. |
| `ctx.state.itemIndex` (parallel) | 0-based index in original array. |

The outer workflow's `ctx.state` is visible too — iterations inherit
the full context.

## ctx.state result

After the forEach completes, `ctx.state[<step-id>]` holds an array of
iteration results, one per item, in original order:

```json
"steps": [
  {
    "id": "fan-out",
    "forEach": { /* ... */ }
  },
  {
    "id": "summary",
    "use": "@blokjs/expr",
    "inputs": {
      "expression": "({ count: ctx.state['fan-out'].length, items: ctx.state['fan-out'] })"
    }
  }
]
```

Each element is the **last sub-step's response envelope** —
`{data, error, success, ...}`. To extract just the data:

```javascript
ctx.state['fan-out'].map(r => r && r.data && r.data.jobId).filter(Boolean)
```

See the [worker fan-out example](/d/examples/worker-fanout)'s
`respond` step for the canonical extraction pattern.

## Common patterns

### Parallel API fan-out

```json
{
  "id": "enrich",
  "forEach": {
    "in": "js/ctx.state.users.data",
    "as": "user",
    "mode": "parallel",
    "concurrency": 10,
    "do": [
      {
        "id": "details",
        "use": "@blokjs/api-call",
        "idempotencyKey": "js/`user-details:${ctx.state.user.id}`",
        "inputs": {
          "url": "js/`https://api.example.com/users/${ctx.state.user.id}`",
          "method": "GET"
        }
      }
    ]
  }
}
```

`idempotencyKey` on the iteration body caches per-user — re-runs are
free.

### Per-row retry

Each iteration is independent; a per-step retry inside the body
re-runs only that iteration, not the whole forEach:

```json
{
  "id": "process",
  "use": "@blokjs/api-call",
  "retry": { "maxAttempts": 3 },
  "inputs": { "url": "..." }
}
```

If iteration 17 of 100 fails three times, it errors — the forEach
**continues** the other 99 if running in parallel, but propagates the
error in sequential mode. Wrap the body in [`tryCatch`](./tryCatch.mdx)
to keep going.

### Wait inside iteration (Phase 2/3 runtime)

```json
{
  "id": "throttled",
  "forEach": {
    "in": "js/ctx.request.body.items",
    "as": "item",
    "mode": "parallel",
    "concurrency": 5,
    "do": [
      { "id": "record", "use": "@blokjs/expr", "inputs": { "expression": "..." } },
      { "id": "throttle", "wait": { "for": "100ms" } }
    ]
  }
}
```

When the first iteration's `wait` fires, peer iterations cancel and
the runner persists a parallel-mode cursor. On dispatch re-entry,
ForEachNode pre-populates `results[]` from `completedResults`,
re-launches cancelled iterations from inner step 0, and resumes the
wait-firing iteration at its `innerStepIndex`. See
[wait-inside-primitives](./wait.mdx#wait-inside-primitives) for the
full design + the `idempotencyKey` warning for side-effecting inner
steps.

## Composition

- `forEach` inside `forEach` — two-level nesting works
  (Phase 4 runtime supports nested cursor persistence)
- `forEach` inside `switch` / `branch` arms — common
- Sub-workflow per item — `{ id: "x", subworkflow: "process-item",
  inputs: { item: "js/ctx.state.item" } }`
- Step-level `idempotencyKey` / `retry` / `maxDuration` apply inside
  iteration bodies
