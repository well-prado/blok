`forEach` runs a callback once per item in an array handle. Iterations run
sequentially by default; flip to parallel with `mode: "parallel"` and a
`concurrency` budget. The body receives a per-item handle and an index handle,
both scoped to that iteration.

Used in the [worker fan-out example](/d/examples/worker-fanout) to
parallel-enqueue jobs.

## Quickstart

```ts
import { workflow, step, forEach, http } from "@blokjs/core";
import processItem from "../nodes/process-item";

export default workflow("items.process", { version: "1.0.0", trigger: http.post("/items") }, (req) => {
  forEach(req.body.items, (item) => {
    step("process", processItem, { id: item.id, payload: item });
  });
});
```

`req.body.items` is a handle to an array. `item` is a per-iteration handle —
read its fields (`item.id`, `item.payload`) like any other handle.

## The signature

`forEach(iterable, (item, index) => {…}, opts?)`:

- **`iterable`** — a **handle to an array** (a step output field or a trigger
  field). It lowers to the loop's `in` expression.
- **`item` / `index`** — per-iteration handles, **arm-scoped** to the body.
  `item` is the current element; `index` is the 0-based position.
- **`opts`** — `{ as?, mode?, concurrency? }`. `as` names the iteration
  variable's state key (defaults to a name derived from the iterable);
  `mode: "parallel"` runs iterations concurrently; `concurrency` caps in-flight
  iterations (default 10).
- **Returns** — a handle to the **results array** (one entry per item, in
  original order), readable after the loop.

### Parallel with a concurrency cap

```ts
const items = step("validate", validate, { raw: req.body }).items;

const results = forEach(items, (item) => {
  step("enqueue", workerPublish, { queue: "jobs", payload: item });
}, { mode: "parallel", concurrency: 5 });
```

1000 items don't serialize 1000 RPCs — five run concurrently, the next starts
as one finishes.

## Reading the results

The returned handle is the results array — readable after the loop, one entry
per item in original order:

```ts
const results = forEach(req.body.users, (user) => {
  step("enrich", apiCall, { url: tpl`https://api.example.com/users/${user.id}` });
});

step("summary", summarize, { count: results.length, items: results });
```

Each element is the **last sub-step's response envelope** (`{ data, error,
success, … }`). The item/index handles do **not** survive the loop — only the
returned results handle does (footgun: arm-scoped handles, see
[Handles & footguns](/d/primitives/handles-and-footguns)).

## Common patterns

<AccordionGroup>
  <Accordion title="Parallel API fan-out with per-item cache" icon="bolt">
    Cache each iteration on its own key so re-runs skip completed work:

    ```ts
    forEach(users, (user) => {
      step("details", apiCall, {
        url: tpl`https://api.example.com/users/${user.id}`,
        method: "GET",
      }, { idempotencyKey: tpl`user-details:${user.id}` });
    }, { mode: "parallel", concurrency: 10 });
    ```
  </Accordion>
  <Accordion title="Per-row retry" icon="rotate-cw">
    A per-step `retry` inside the body re-runs only that iteration, not the
    whole loop. In parallel mode a permanently-failing item errors but peers
    continue; in sequential mode the error propagates. Wrap the body step in
    [`tryCatch`](./tryCatch.mdx) to keep going:

    ```ts
    forEach(rows, (row) => {
      step("process", processRow, { row }, { retry: { maxAttempts: 3 } });
    });
    ```
  </Accordion>
  <Accordion title="Sub-workflow per item" icon="git-branch">
    Dispatch a named child workflow once per element — the per-item handle
    feeds the child's inputs:

    ```ts
    forEach(req.body.items, (item) => {
      subworkflow("handle", "process-item", { item });
    });
    ```
  </Accordion>
</AccordionGroup>

## Footguns

- **Item / index handles are arm-scoped** — they read `undefined` after the
  loop. Only the returned results handle survives.
- **Never name `as:` after an existing step id** (or another loop's `as`) —
  iteration variables share the flat `ctx.state` namespace with step outputs;
  a collision throws at load time. This is footgun #4.
- `forEach` nests freely — inside a `branch` / `switchOn` arm, inside another
  `forEach`, or wrapping a `tryCatch`. Reliability knobs (`idempotencyKey`,
  `retry`, `maxDuration`) go on the regular steps inside the body.
