The WebSocket trigger handles WebSocket upgrades on the HTTP server's
port and runs a workflow per inbound message. Same-port operation
with [HTTP](./http.mdx), [SSE](./sse.mdx), and
[Webhook](./webhook.mdx) — Hono's `upgradeWebSocket` handler is wired
into the existing server before the catch-all route.

## Quickstart

The build callback receives the connection's entry handle (`conn`) — read the inbound message
at `conn.body`, path params at `conn.params`.

```ts src/workflows/ws-echo.ts
import { workflow, step } from "@blokjs/core";
import { WsReplyNode } from "@blokjs/helpers";

export default workflow("ws-echo", {
  version: "1.0.0",
  trigger: { websocket: { path: "/ws/echo", events: ["message", "open", "close"] } },
}, (conn) => {
  step("reply", WsReplyNode, { message: { echo: conn.body, at: Date.now() } });
});
```

Browser side:

```javascript
const ws = new WebSocket("ws://localhost:4000/ws/echo");
ws.onmessage = (ev) => console.log(JSON.parse(ev.data));
ws.onopen = () => ws.send(JSON.stringify({ hello: "world" }));
```

## Config fields

| Field | Type | Default | Notes |
|---|---|---|---|
| `path` | `string` | required | URL path. Supports dynamic params (e.g. `/ws/room/:roomId`). |
| `events` | `string[]` | `["message"]` | Which WS lifecycle events trigger the workflow. Supported: `"open"`, `"message"`, `"close"`, `"error"`. |
| `concurrencyKey` | `string` \| `$.path` | — | Per-key cap on concurrent connections. |
| `concurrencyLimit` | `number` | `1` | Slot count per resolved key. |

## The `conn` handle

| Read | Source |
|---|---|
| `conn.body` | Parsed message body (JSON when possible, raw string otherwise) |
| `conn.headers` | Headers from the WS upgrade request |
| `conn.params` | Path params from `:name` in the WS path |
| `ctx.vars._ws_connection` | Connection metadata (id, opened-at, remote address) |
| `ctx.vars._ws_event` | The lifecycle event name (`"open"` / `"message"` / `"close"` / `"error"`) |
| `ctx.signal` | Cancelled when the client disconnects |

## Helper nodes

Three helpers wrap the WS surface:

- [`@blokjs/ws-reply`](/d/reference/helpers/ws-reply) — reply only to
  the connection that triggered the workflow
- [`@blokjs/ws-broadcast`](/d/reference/helpers/ws-broadcast) — fan
  out a message to every connection on a path / channel
- [`@blokjs/ws-close`](/d/reference/helpers/ws-close) — close the
  connection with a code + reason

## Cross-process broadcast (backplane)

By default, WS connections live in the Node process that accepted the
upgrade. `@blokjs/ws-broadcast` reaches only connections on the same
process. For multi-process deployments (HPA, load-balanced), set:

```bash
BLOK_WS_BACKPLANE=redis
BLOK_WS_BACKPLANE_REDIS_URL=redis://...
```

The backplane fans broadcasts through Redis pub/sub so every process
delivers to its local connections. Per-connection reply
(`ws-reply`) doesn't need the backplane — replies always go to the
connection that triggered the current workflow run, which lives on the
current process.

## Common patterns

### Echo / RPC

The simplest pattern — every `message` event spawns a workflow run
that processes the request and replies via `@blokjs/ws-reply`.

### Chat room (per-room broadcast)

```ts
import { workflow, step, tpl } from "@blokjs/core";
import { WsBroadcastNode } from "@blokjs/helpers";

export default workflow("chat-room", {
  version: "1.0.0",
  trigger: { websocket: { path: "/ws/room/:roomId", events: ["message"] } },
}, (conn) => {
  step("broadcast", WsBroadcastNode, {
    channel: tpl`room-${conn.params.roomId}`,
    message: conn.body,
  });
});
```

Every WS connection on `/ws/room/<roomId>` lands in the same broadcast
group automatically based on the path pattern.

### Server-pushed updates

WS triggers fire on `events: ["open"]` too — send the connecting client
an initial state snapshot on open, then handle `message` events for
incremental updates. The lifecycle event lives at `ctx.vars._ws_event` (a ctx var, not on the
entry handle) so the `branch` condition is a raw `ctx.*` string:

```ts
import { workflow, step, branch } from "@blokjs/core";
import { WsReplyNode } from "@blokjs/helpers";

export default workflow("ws-live", {
  version: "1.0.0",
  trigger: { websocket: { path: "/ws/live", events: ["open", "message"] } },
}, () => {
  branch("init-or-update", "ctx.vars._ws_event === 'open'", {
    then: () => { step("init-snapshot", WsReplyNode, { message: "..." }); },
    else: () => { step("handle-update", WsReplyNode, { message: "..." }); },
  });
});
```

## Composition

WebSocket triggers compose with [concurrency keys](/d/reliability/concurrency-keys)
for per-tenant connection caps. Other reliability knobs
(idempotency, retry, replay) live on the **step** level inside the
workflow body — same as any other trigger.
