The cron trigger runs a workflow on a schedule expressed in standard
cron syntax. Each workflow registers one schedule; the trigger process
fires the workflow at every match.

## Quickstart

The build callback's argument is the **entry handle** — for cron it's the tick, named `tick`.
A tick has **no body** (cron carries no payload); read schedule metadata from `tick.params`.

```ts src/workflows/daily-cleanup.ts
import { workflow, step } from "@blokjs/core";
import apiCall from "@blokjs/api-call";

export default workflow("daily-cleanup", {
  version: "1.0.0",
  trigger: { cron: { schedule: "0 2 * * *", timezone: "America/New_York" } },
}, () => {
  step("purge-stale", apiCall, { url: "https://api.example.com/cleanup", method: "POST" });
});
```

This fires at 2:00 AM every day in `America/New_York`.

## Config fields

| Field | Type | Default | Notes |
|---|---|---|---|
| `schedule` | `string` | required | Standard cron expression (`m h dom mon dow`). |
| `timezone` | `string` | `"UTC"` | IANA timezone — `"America/New_York"`, `"Europe/London"`, etc. |
| `concurrencyKey` | `string` \| `$.path` | — | Useful when one schedule fans out per-tenant; concurrencyKey prevents overlap. |
| `middleware` | `string[]` | `[]` | Per-trigger middleware. |

## Cron expression reference

```
 ┌───── minute  (0 - 59)
 │ ┌─── hour    (0 - 23)
 │ │ ┌─ day of month  (1 - 31)
 │ │ │ ┌─ month  (1 - 12 or JAN-DEC)
 │ │ │ │ ┌─ day of week  (0 - 6 or SUN-SAT)
 │ │ │ │ │
 0 2 * * *     # 02:00 every day
*/5 * * * *    # every 5 minutes
 0 0 * * 1     # midnight every Monday
 0 9 1 * *     # 09:00 on the 1st of each month
```

## The `tick` handle

A tick carries no inbound payload, so the entry handle named `tick`
exposes only schedule metadata — **no `tick.body`** (reading it is a type
error; there is no slot for it). Like every trigger, `tick` roots at
`ctx.request`.

| Read | Source |
|---|---|
| `tick.params.jobId` | The cron job id for this schedule |
| `tick.params.schedule` | The configured cron expression |
| `tick.headers["x-cron-schedule"]` | The schedule, also surfaced as a header |
| `tick.headers["x-cron-timezone"]` | The configured timezone |
| `tick.headers["x-cron-job-id"]` | The cron job id |
| `ctx.vars._cron_context` | `{ jobId, scheduledTime, executionTime, schedule, timezone, manual }` — the side-channel (see the Date-vs-ISO note below) |

<Note>
**Date vs ISO string.** The runner builds a `CronExecutionContext` whose
`scheduledTime` / `executionTime` are JavaScript **`Date` objects** and
puts it on `ctx.request.body`. The `tick` handle deliberately omits
`.body` (a tick is "no payload"), so reach for the side-channel
`ctx.vars._cron_context` instead — there the two times are **ISO
strings** (`.toISOString()`), the JSON-safe form. Don't assume the same
representation on both: `ctx.request.body.scheduledTime` is a `Date`,
`ctx.vars._cron_context.scheduledTime` is a string.
</Note>

Build per-schedule context in a first step and read it downstream via its handle:

```ts
const cfg = step("build-ctx", ExprNode, { expression: "({ batchSize: 100, dryRun: false })" });

step("run", purgeBatch, { batchSize: cfg.batchSize, dryRun: cfg.dryRun });
```

## Process model

Cron triggers run in a dedicated Node process — `blokctl dev` spawns
one when any workflow has a `trigger.cron` config. The process polls
the schedule on a sub-second tick and dispatches when a match fires.

The cron process installs the same crash / orphan / shutdown / janitor
handlers as HTTP and Worker, and runs trigger-level `middleware` on each
fire — identical chain resolution (process-global → workflow → trigger)
to the other trigger kinds.

Schedule durability is broker-agnostic — the trigger doesn't persist
"missed" executions across restarts by default. For exactly-once
historical replay, layer a [delay/TTL](/d/scheduling/delay-and-ttl) on
the workflow steps or use a job queue with a [cron-style scheduler
extension](https://docs.bullmq.io/guide/jobs/repeatable) like
BullMQ-Repeatable.

## Common patterns

### Daily report → Slack

```ts
import { workflow, step } from "@blokjs/core";
import apiCall from "@blokjs/api-call";
import { LlmStreamNode } from "@blokjs/helpers";

export default workflow("daily-metrics-digest", {
  version: "1.0.0",
  trigger: { cron: { schedule: "0 9 * * 1-5", timezone: "America/Los_Angeles" } },
}, () => {
  step("fetch-metrics", apiCall, { url: "https://internal/metrics" });
  const summary = step("summarize", LlmStreamNode, {
    model: "gpt-4o-mini", messages: "...", channel: "internal-summary",
  });
  step("post-slack", apiCall, {
    url: "js/process.env.SLACK_WEBHOOK_URL",
    method: "POST",
    body: { text: summary.fullText },
  });
});
```

09:00 every weekday, summarize yesterday's metrics with an LLM, post
to Slack.

### Per-tenant nightly job (sub-workflow fan-out)

```ts
import { workflow, step, forEach, subworkflow } from "@blokjs/core";
import apiCall from "@blokjs/api-call";

export default workflow("nightly-per-tenant-backup", {
  version: "1.0.0",
  trigger: { cron: { schedule: "0 0 * * *" } },
}, () => {
  const tenants = step("list-tenants", apiCall, { url: "https://internal/tenants" });

  forEach(tenants.data, (tenant) => {
    subworkflow("backup", "backup-tenant", { tenantId: tenant.id });
  }, { mode: "parallel", concurrency: 5 });
});
```

The cron fires once; the `forEach` fans out to one
[sub-workflow](/d/composition/sub-workflows) per tenant, 5 in parallel.

### Skipping overlapping runs

If a cron run takes longer than the schedule interval, the next tick
fires while the previous is still running. Use `concurrencyKey` to
serialize:

```ts
trigger: { cron: { schedule: "*/5 * * * *", concurrencyKey: "self", concurrencyLimit: 1 } }
```

A literal `"self"` key means "only one run of this workflow at a time
across all processes." Excess ticks NACK with retry — or add
`onLimit: "queue"` to defer.

## Composition

Cron triggers compose with:

- [`concurrencyKey`](/d/reliability/concurrency-keys) — prevent
  overlapping runs
- Sub-workflows + `forEach` — fan-out per item / tenant
- Step-level [`idempotencyKey`](/d/reliability/idempotency) — make
  re-runs cheap if the schedule fires twice (e.g. DST transitions)
- Step-level [`retry`](/d/reliability/retry) + `maxDuration` — for
  transient failures and runaway runs
