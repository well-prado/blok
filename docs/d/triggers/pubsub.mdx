The Pub/Sub trigger consumes messages from a cloud pub/sub broker and
runs a workflow per delivery. Three providers ship in the box. Runs
in its own Node process; cross-process state goes through the broker.

For an in-process event bus (single-process pub/sub for HTTP + SSE
chains), use the [SSE trigger](./sse.mdx) instead — Pub/Sub is the
multi-process / multi-cloud sibling.

## Quickstart

The build callback's argument is the **entry handle** — for pub/sub it's the message, named
`msg` (`msg.body` is the payload, `msg.headers` are the provider attributes). The provider
message id lives at `msg.params.messageId` — the natural idempotency key.

```ts src/workflows/on-order-placed.ts
import { workflow, step } from "@blokjs/core";
import apiCall from "@blokjs/api-call";

export default workflow("on-order-placed", {
  version: "1.0.0",
  trigger: { pubsub: { provider: "gcp", topic: "orders.placed", subscription: "fulfillment-svc" } },
}, (msg) => {
  step("fulfill", apiCall, {
    url: "https://fulfillment.internal/api/orders",
    method: "POST",
    body: msg.body,
  }, { idempotencyKey: msg.params.messageId });
});
```

## Providers + config

### GCP Pub/Sub

```ts
trigger: { pubsub: {
  provider: "gcp",
  topic: "orders.placed",
  subscription: "fulfillment-svc",
  maxMessages: 10,
} }
```

Env vars:

```bash
GOOGLE_APPLICATION_CREDENTIALS=/path/to/key.json
PUBSUB_PROJECT_ID=my-gcp-project
# or for the emulator:
PUBSUB_EMULATOR_HOST=localhost:8085
```

### AWS SNS + SQS

```ts
trigger: { pubsub: {
  provider: "aws",
  topic: "arn:aws:sns:us-east-1:123:orders-placed",
  subscription: "https://sqs.us-east-1.amazonaws.com/123/fulfillment-queue",
} }
```

`topic:` is the SNS ARN; `subscription:` is the SQS queue URL that
SNS fans out to. Env vars: standard AWS credentials chain
(`AWS_REGION`, `AWS_ACCESS_KEY_ID`, etc.).

### Azure Service Bus

```ts
trigger: { pubsub: { provider: "azure", topic: "orders", subscription: "fulfillment-sub" } }
```

Env var: `AZURE_SERVICEBUS_CONNECTION_STRING`.

## Config fields

| Field | Type | Notes |
|---|---|---|
| `provider` | `"gcp"` \| `"aws"` \| `"azure"` | Required. |
| `topic` | `string` | Provider-specific identifier (topic name / SNS ARN / Service Bus topic). |
| `subscription` | `string` | Provider-specific subscription (GCP subscription / SQS URL / Azure subscription). |
| `maxMessages` | `number` | Max messages pulled per batch (GCP). |
| `concurrencyKey` | `string` \| `$.path` | Per-key processing cap. |
| `concurrencyLimit` | `number` | Slot count per resolved key. |
| `middleware` | `string[]` | Per-trigger middleware. |

## The `msg` handle

| Read | Source |
|---|---|
| `msg.body` | Message payload (JSON-parsed when possible, raw string otherwise) |
| `msg.headers` | Provider-specific message attributes (lowercased) |
| `msg.params.topic` | Topic name |
| `msg.params.subscription` | Subscription identifier |
| `msg.params.messageId` | Provider message id — use as `idempotencyKey` for replay safety |
| `msg.params.publishTime` | When the provider received the publish |
| `ctx.vars._pubsub_message` | Full message metadata |

## Acknowledgment semantics

The trigger ACKs successful runs and NACKs on workflow exceptions —
the broker redelivers per its own retry policy:

- **GCP** — based on the subscription's `ackDeadlineSeconds` and
  exponential backoff settings
- **AWS SQS** — visibility timeout on the queue
- **Azure** — max delivery count + dead-letter queue

The trigger doesn't implement its own retry budget — broker semantics
own that. Configure DLQs on the provider side.

## Common patterns

### Cross-service event handling

Service A publishes to `orders.placed` → Service B subscribes via this
trigger → fulfills the order. Decoupled, async, durable.

### Idempotency on every message

```ts
step("process", apiCall, { url: "https://internal/process", body: msg.body },
  { idempotencyKey: msg.params.messageId });
```

Brokers retry on NACK — `idempotencyKey` makes the processing step
safe to re-run.

### Fan-out within the trigger process

```ts
import { forEach, step } from "@blokjs/core";

forEach(msg.body.orderLines, (line) => {
  step("ship", null, { line }, { subworkflow: "ship-line-item" });
});
```

One pub/sub message becomes one parent run + N sub-workflow runs. The
parent ACKs only when all sub-runs complete.

## Composition

- `idempotencyKey` on steps — recommended for any side-effect step
- `concurrencyKey` on the trigger — per-tenant / per-resource caps
- Sub-workflows + `forEach` — fan-out per item in the message
- `maxDuration` on steps — bound runaway processing independent of
  the broker's visibility timeout
