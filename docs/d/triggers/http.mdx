The HTTP trigger turns a Blok workflow into an HTTP endpoint. It owns
the listening Node server and exposes a Hono app that
[SSE](./sse.mdx) / [WebSocket](./websocket.mdx) /
[Webhook](./webhook.mdx) triggers can mount on for shared-port
operation.

## Quickstart

The build callback's argument is the **entry handle** ‚Äî for HTTP it's the request, named `req`
(`req.body`, `req.params.id`, `req.query.q`, `req.headers["x-‚Ä¶"]`).

```ts src/workflows/hello.ts
import { workflow, step, http } from "@blokjs/core";
import { ExprNode } from "@blokjs/helpers";

export default workflow("hello", { version: "1.0.0", trigger: http.get("/hello") }, () => {
  step("respond", ExprNode, { expression: "({ message: 'hello world' })" });
});
```

```bash
curl http://localhost:4000/hello
# ‚Üí {"message":"hello world"}
```

## Config fields

The `http.*` helper sets `method` + `path`; the rest go in its options object ‚Äî
`http.post("/api", { accept, headers, concurrencyKey, ... })`.

| Field | Type | Default | Notes |
|---|---|---|---|
| method | `http.get` / `.post` / `.put` / `.patch` / `.delete` / `.any` | required | One helper per verb. `http.any()` is the wildcard (`"ANY"`). |
| path | first arg to the helper | derived | URL path with `:name` for dynamic params (e.g. `http.get("/users/:id")`). When omitted, the path is derived from the workflow's file path ‚Äî see [file-based routing](#file-based-routing). |
| `accept` | `string` | `"application/json"` | Response content-type. Use `"text/html"` for UI nodes. |
| `headers` | `Record<string, string>` | ‚Äî | Required headers validated at trigger entry. A request missing any declared header is rejected before any step runs. |
| `middleware` | `string[]` | `[]` | Per-trigger middleware chain. Resolved by name from the middleware registry. |
| `concurrencyKey` | `string` \| `$.path` | ‚Äî | Per-key in-flight cap. See [concurrency keys](/d/reliability/concurrency-keys). |
| `concurrencyLimit` | `number` | `1` | Slot count per resolved key. |
| `onLimit` | `"throw"` \| `"queue"` | `"throw"` | What happens when the limit is hit. |
| `delay` | `string` \| `number` | ‚Äî | Defer the run. See [delay & TTL](/d/scheduling/delay-and-ttl). |
| `ttl` | `string` \| `number` | ‚Äî | Expire if not started in time. |
| `debounce` | `{ key, mode, delay, maxDelay? }` | ‚Äî | Collapse rapid same-key requests. See [debounce](/d/scheduling/debounce). |

## File-based routing

`BLOK_FILE_BASED_ROUTING=true` (default since v0.6) maps workflow file
paths to URLs:

| File | URL |
|---|---|
| `workflows/json/health.json` | `/health` |
| `workflows/json/users/list.json` | `/users/list` |
| `workflows/json/users/index.json` | `/users` |
| `workflows/json/users/[id].json` | `/users/:id` |
| `workflows/json/users/[id]/orders.json` | `/users/:id/orders` |

Files / folders starting with `_` or `.` are skipped (drafts, utility
files). If `trigger.http.path` is set explicitly, it overrides the
derived path.

When `BLOK_FILE_BASED_ROUTING=false`, the legacy `/<workflow-name>`
catch-all dispatch handles requests.

## The `req` handle

The build callback's `req` argument reads the incoming request:

| Read | Source |
|---|---|
| `req.method` | HTTP verb (lowercase) |
| `req.body` | Parsed JSON / form / raw text depending on `Content-Type` |
| `req.headers["x-‚Ä¶"]` | Lowercased header map |
| `req.params.id` | Path params (from `:id` in `path`) |
| `req.query.q` | Parsed query string |
| `ctx.signal` | An `AbortSignal` cancelled when the client disconnects or `POST /__blok/runs/:runId/cancel` fires |

The raw body is captured before parsing ‚Äî webhook trigger HMAC
verifiers need the wire bytes, not the JSON-parsed object. See
[webhook trigger](./webhook.mdx) for that flow.

## Controlling the response

By default a workflow's final step output becomes the response body:
an object ‚Üí JSON, a string ‚Üí written verbatim, both at **status 200**
with the `Content-Type` from `accept` (or the node's `contentType`).
That covers most JSON APIs with zero ceremony.

When you need a **custom status, headers, `Set-Cookie`, a redirect, or a
binary body**, end the workflow with the [`RespondNode`](/d/reference/helpers/respond)
helper ‚Äî it returns a branded envelope the trigger unpacks. It's purely additive:
workflows that don't use it are unchanged.

| Capability | Input |
|---|---|
| Custom status | `status` (e.g. `201`, `404`, `206`) |
| Arbitrary headers | `headers` (e.g. `{ "Content-Disposition": "attachment; filename=\"x.pdf\"" }`) |
| Redirect | `status: 302` + `headers: { Location: "/next" }` |
| Cookies | `cookies` ‚Äî an **array** of raw `Set-Cookie` strings (the header repeats, so you can set several) |
| Content-Type | `contentType` (overrides `accept`) |
| Binary body | `body` set to a `Buffer` / `Uint8Array` / `ArrayBuffer` ‚Äî sent as raw bytes, not JSON |

```ts
import { workflow, step, http } from "@blokjs/core";
import { RespondNode } from "@blokjs/helpers";
```

### Redirect (302)

```ts
step("go", RespondNode, { status: 302, headers: { Location: "/dashboard" } }, { ephemeral: true });
```

### Set a session cookie

```ts
step("login", RespondNode, {
  body: { ok: true },
  cookies: ["session=abc123; Path=/; HttpOnly; SameSite=Lax"],
}, { ephemeral: true });
```

Pass multiple entries in `cookies` to emit multiple `Set-Cookie`
headers in one response.

### Binary file download

```ts
const file = step("render", renderPdf, { id: req.params.id });

step("download", RespondNode, {
  body: file.bytes,
  contentType: "application/pdf",
  headers: { "Content-Disposition": 'attachment; filename="report.pdf"' },
}, { ephemeral: true });
```

<Tip>
A node that simply **returns** a `Buffer` / `Uint8Array` / `ArrayBuffer`
(with a `contentType`) is also sent as raw bytes ‚Äî you only need
`@blokjs/respond` when you also want a custom status, headers, or cookies.
</Tip>

<Note>
Status/headers/cookies are honored for **in-process** (`module` / `local`)
nodes ‚Äî the same surface as `contentType`. Returning binary or a respond
envelope directly from a non-NodeJS `runtime.*` node is not yet wired.
</Note>

## Common patterns

### Path params

```ts
import { workflow, step, tpl, http } from "@blokjs/core";
import query from "../nodes/query";

export default workflow("orders.get", { version: "1.0.0",
  trigger: http.get("/users/:userId/orders/:orderId") }, (req) => {
  step("query", query, {
    sql: tpl`SELECT * FROM orders WHERE user = ${req.params.userId} AND id = ${req.params.orderId}`,
  });
});
```

`req.params.userId` / `req.params.orderId` are typed handles; `tpl` embeds them in the string.

### Wildcard method

`http.any()` matches every verb. Route per-method via a `branch` step on `req.method`
(use the typed comparator `eq(req.method, "post")`). The [DB Manager example](/d/examples/db-manager)
uses exactly this shape.

### HTML UI from one workflow

Set `accept: "text/html"` and return a string from a node. UI nodes
typically read an HTML file template and run EJS:

```ts
trigger: http.get("/admin", { accept: "text/html" })
```

See the [chat demo](/d/examples/chat) for a complete UI-node example.

### Stream tokens with SSE

Pair an HTTP POST handler with an SSE subscriber ‚Äî they share the
in-process bus when both triggers run in one process. The
[streaming chat example](/d/examples/chat) walks the full pattern.

### Required headers

List headers that every request must carry under `headers`. A request
missing any of them is rejected with `400` at trigger entry, before any
step runs. An empty declared value asserts presence only; a non-empty
value also requires an exact match (handy for pinning an API version):

```ts
trigger: http.post("/api", { headers: { "x-api-key": "", "x-api-version": "2026-01" } })
```

## Composition

Drop any reliability or scheduling knob into the trigger options:

```ts
trigger: http.post("/render", {
  concurrencyKey: "ctx.request.body.userId",
  concurrencyLimit: 5,
  onLimit: "queue",
  ttl: "5m",
})
```

Per-tenant fairness (5 renders per user, anything over that defers
via `onLimit: "queue"`), bounded by a 5-minute TTL ‚Äî declarative.
