The manual trigger marks a workflow that has **no listener** â€” no HTTP
route, no queue subscription, no schedule. It runs only when something
**invokes it directly**: application code, the test harness, or another
workflow via a [sub-workflow](/d/composition/sub-workflows) step. Use it
for shared logic you want addressable by name but never reachable from
the network.

Both halves are **shipped**: the typed `args` entry handle (the
author-facing surface this page documents) and the programmatic dispatch
runtime â€” `ManualTrigger.dispatch()` from `@blokjs/runner`
([#434](https://github.com/well-prado/blok/issues/434)). You can also
invoke a manual workflow through a
[sub-workflow](/d/composition/sub-workflows) step when calling from
inside another workflow.

## Dispatching from host code

`ManualTrigger` is the push entrypoint â€” it never binds a socket. Give it
your nodes, `listen()` to register its workflows, then `dispatch(name, args)`:

```ts
import { ManualTrigger } from "@blokjs/runner";

const manual = new ManualTrigger();
manual.setNodeMap(nodeMap);        // your { nodes, workflows }
await manual.listen();             // registers the manual workflows (no port)

// args land at ctx.request.body â†’ the typed `args` entry handle.
const result = await manual.dispatch("recalc-totals", { invoiceId: "inv_42" });
```

`dispatch` runs the workflow through the full runner â€” tracing,
persistence, retries, idempotency, concurrency â€” so the run shows up in
Blok Studio like any HTTP run, and returns the final step's
`ctx.response.data`. Each call uses a fresh `Configuration`, so
overlapping dispatches are isolated. Dispatching a name no workflow
claims throws loudly (the no-listener guarantee) rather than dropping the
call silently.

## Quickstart

The build callback's argument is the **entry handle** â€” for a manual
workflow it's the dispatch arguments, named `args`. The caller-supplied
args land at `ctx.request.body`, exactly where MCP and webhook put their
payload, so `args.body` reads them.

```ts src/workflows/recalc-totals.ts
import { workflow, step } from "@blokjs/core";
import { z } from "zod";
import apiCall from "@blokjs/api-call";

export default workflow("recalc-totals", {
  version: "1.0.0",
  description: "Recompute order totals. Invoked by code or another workflow.",

  // â†’ types the `args.body` handle (see "Typed args" below)
  input: z.object({ orderId: z.string() }),

  trigger: { manual: {} },
}, (args) => {
  step("recalc", apiCall, {
    url: "https://internal/orders/recalc",
    method: "POST",
    body: { orderId: args.body.orderId },
  });
});
```

Nothing mounts. The workflow is registered by name in the
[workflow registry](/d/reference/workflow-registry) and waits to be
called.

## Config fields

| Field | Type | Notes |
|---|---|---|
| `manual` | `{}` | An empty config marks the workflow manual. No route, queue, or schedule is mounted. |
| `middleware` | `string[]` | Per-trigger middleware chain run before the workflow body. |

<Note>
`manual` carries **no validated config schema** â€” it's a marker kind, so
keep the config block empty. There's nothing per-trigger to configure:
dispatch behavior is controlled by the `dispatch()` call, not the trigger
block.
</Note>

## The `args` handle

A manual workflow's caller-supplied arguments arrive at
`ctx.request.body` â€” the same `ctx.request` funnel every trigger uses, so
the workflow body looks structurally identical to an HTTP or MCP one.

| Read | Source |
|---|---|
| `args.body` | The dispatch arguments object the caller passed |
| `args.headers` | Caller-supplied headers (empty unless the dispatcher sets them) |
| `args.params` | Caller-supplied path-style params (empty by default) |
| `args.query` | Caller-supplied query-style params (empty by default) |

## Typed args

Declare a workflow-level `input` Zod schema and `args.body` is **typed
from it**; with no `input`, `args.body` is `unknown`. This is the same
one rule the [webhook](./webhook.mdx), [gRPC](./grpc.mdx), and
[MCP](./mcp.mdx) triggers follow â€” declare the schema to get a typed
entry handle, omit it and reads are `unknown` (you narrow them yourself):

```ts
// input: z.object({ orderId: z.string() })  â†’ args.body.orderId is string
// (no input)                                â†’ args.body is unknown
```

The schema types the handle for authoring; it is **not** auto-validated
at dispatch. Validate in a step or middleware if you need a runtime
guarantee.

## How it's invoked

A manual workflow has no listener, so something else dispatches it:

- **Another workflow** â€” a [sub-workflow](/d/composition/sub-workflows)
  step (`subworkflow("recalc", "recalc-totals", { orderId })`). The
  parent step's resolved `inputs` become the child's `ctx.request.body`
  â€” i.e. the manual workflow's `args.body`. This works today.
- **The test harness** â€” `WorkflowTestRunner` (from `@blokjs/runner`) runs
  a workflow with a supplied body. `loadWorkflow()` takes a plain object (or
  JSON string) with a **top-level `steps` array** â€” it does **not** accept the
  handle-DSL `workflow()` builder result (whose steps live under `_config`).
  Pass the steps array directly, or feed it `myWorkflow.toJson()` /
  `myWorkflow._config`.
- **Application code** â€” `ManualTrigger.dispatch(name, args)` from
  `@blokjs/runner` (see [Dispatching from host code](#dispatching-from-host-code)
  above). In-process invocation by name with an args object, no network
  hop, full runner semantics. A `blokctl invoke` CLI wrapper over the same
  surface is a possible follow-up.

## Composition

Because a manual run goes through the normal runner, the full step-level
reliability surface composes:

- [`idempotencyKey`](/d/reliability/idempotency) / [`retry`](/d/reliability/retry) / [`maxDuration`](/d/reliability/max-duration) on steps
- [Sub-workflows](/d/composition/sub-workflows) for further fan-out
- Studio tracing â€” every manual run is recorded like any other

Trigger-level scheduling knobs (`delay`, `ttl`, `debounce`,
`concurrencyKey`) gate *route entry*; a manual workflow has no route, so
put those concerns on whatever invokes it instead.
