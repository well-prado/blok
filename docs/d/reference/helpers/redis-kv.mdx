Drop-in replacement for [`@blokjs/in-memory-kv`](./in-memory-kv) with
the same `get` / `set` / `delete` / `list` actions. Connects via the
`REDIS_URL` env var (default `redis://127.0.0.1:6379`) using `ioredis`.

Used in the [chat-memory](/d/examples/chat-memory) and
[LLM agent](/d/examples/llm-agent) examples for server-side
conversation history.

Registered as `@blokjs/redis-kv`. In the handle DSL, import the node
object (`RedisKvNode`) from `@blokjs/helpers` and pass it to `step()`.
Build dynamic keys from handles with `tpl`.

## Quickstart

```ts
import { workflow, step, tpl, http } from "@blokjs/core";
import { RedisKvNode } from "@blokjs/helpers";

export default workflow("chat-history", { version: "1.0.0", trigger: http.get("/chat/:sessionId") }, (req) => {
  step("load-history", RedisKvNode, {
    action: "get",
    key: tpl`chat:${req.params.sessionId}:history`,
  });
});
```

Set + TTL â€” feed an earlier step's handle straight into `value`:

```ts
const build = step("build", buildUserData, { id: req.params.sid });

step("save", RedisKvNode, {
  action: "set",
  key: tpl`session:${req.params.sid}`,
  value: build,
  ttlMs: 86400000,
});
```

## Inputs

| Field | Type | Notes |
|---|---|---|
| `action` | `"get"` \| `"set"` \| `"delete"` \| `"list"` | Operation. |
| `key` | `string` | Required for `get`/`set`/`delete`. Ignored for `list`. |
| `value` | unknown | Required for `set`. JSON-encoded internally â€” any JSON-serializable value works. |
| `ttlMs` | `number` | Optional per-key TTL in ms. Redis evicts when the TTL elapses. |
| `prefix` | `string` | For `list` only â€” `SCAN MATCH <prefix>*`. Without a prefix, `list` returns ALL keys (usually undesirable). |

## Outputs

| Action | Output shape |
|---|---|
| `get` | `{ action, key, value, exists }` â€” `value` and `exists` are undefined when the key doesn't exist. |
| `set` | `{ action, key, value }` |
| `delete` | `{ action, key, deleted }` â€” `deleted` is `true` if the key existed. |
| `list` | `{ action, entries: [{ key, value }, ...] }` |

## Connection model

One shared `ioredis` client per process, lazy-created on first use,
keyed off `REDIS_URL`. ioredis handles command queueing internally â€”
all calls funnel through the single client. The client lives until
the process exits.

`list` uses `SCAN` with `MATCH <prefix>*` to avoid the multi-second
block `KEYS` produces on large databases.

## Composition

- The [chat-memory example](/d/examples/chat-memory) chains
  `load-history â†’ llm-stream â†’ save-history` (with `ephemeral: true`
  on save) as the canonical pattern.
- For cross-process pub/sub layered on top of KV, layer Redis pub/sub
  separately â€” this helper is KV only.
- Pair with `ttlMs` for ephemeral session state (24h sessions, 1h tokens, etc.).

## Failure modes

- Missing `ioredis` install â†’ throws on first invocation with a clear
  install hint (`bun add ioredis`).
- Connection failure â†’ ioredis retries internally with exponential
  backoff. Set a `retry` on the step for higher-level recovery.
