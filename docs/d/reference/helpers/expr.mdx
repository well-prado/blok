Runs a JS expression via `new Function()` with `ctx`, `data`, `func`,
`vars` in scope (same surface as the runner's blueprint mapper). Use
when you need a small computed value ‚Äî derived field, counter, boolean
test ‚Äî inside a loop, branch, or chained step.

Registered as `@blokjs/expr`. In the handle DSL, import the node object
(`ExprNode`) from `@blokjs/helpers` and pass it to `step()`.

The cleanest "response shape" step at the end of a workflow:

```ts
import { workflow, step, http } from "@blokjs/core";
import { ExprNode } from "@blokjs/helpers";

export default workflow("list-items", { version: "1.0.0", trigger: http.get("/items") }, () => {
  // ... a "fetch" step runs earlier ...
  step("respond", ExprNode, {
    expression: "({ ok: true, count: ctx.state.fetch.data.length, items: ctx.state.fetch.data })",
  });
});
```

## Quickstart

```ts
step("derived", ExprNode, {
  expression: "ctx.state.user.firstName + ' ' + ctx.state.user.lastName",
});
```

## Inputs

| Field | Type | Notes |
|---|---|---|
| `expression` | `string` | A JS expression. Evaluated against the live ctx. Required. **A raw `ctx.*` JS string** ‚Äî it is itself mapper-resolved, so do NOT prefix it with `js/` (that double-evaluates) and do NOT pass a `$`/handle. Write plain `ctx.state.x.y`. |

## Output

The expression's return value, written to `ctx.state[<step-id>]`.

## Scope inside the expression

| Identifier | What it is |
|---|---|
| `ctx` | The full Context ‚Äî `ctx.request`, `ctx.state`, `ctx.vars`, `ctx.logger`, etc. |
| `data` | `ctx.response.data` (last step's output) OR `ctx.request.body` as fallback. |
| `vars` | `ctx.vars` ‚Äî alias of `ctx.state` for back-compat. |
| `func` | Reserved; currently empty record. |

## Common patterns

### Response shaping

```ts
step("respond", ExprNode, {
  expression: "({ ok: true, fullText: ctx.state.stream.fullText, turn: ctx.state.counter })",
});
```

The HTTP trigger reads the final state and writes the expression's
return value as the response body.

### Computed predicates inside branches

The `branch` condition already takes a comparator ‚Äî `@blokjs/expr`
is for the cases where you need to **stash** a computed value into
state for downstream steps:

```ts
step("compute-priority", ExprNode, {
  expression: "(ctx.request.body.tier === 'enterprise' || ctx.request.body.priorityFlag) ? 'high' : 'normal'",
});
```

Now `ctx.state['compute-priority']` holds `"high"` or `"normal"` for
later branches.

### Transforming list elements

```ts
step("extract-ids", ExprNode, {
  expression: "ctx.state.users.data.map(u => u.id).filter(id => id != null)",
});
```

## Security

The expression runs via `new Function()` with `"use strict"`. **Do
NOT pass untrusted strings as `expression`** ‚Äî it's full JS evaluation.
For computed expressions from user input, use a sandboxed math parser
(e.g. `mathjs`) instead.

The `expression` field is a **literal string** in the workflow source, so
under the v2 default this is safe ‚Äî workflow definitions ship with the
project and aren't accepted from user input.

## Composition

- Works inside [primitives](/d/primitives/index) ‚Äî `forEach.do`,
  `switch.cases[].do`, `tryCatch.try`, etc.
- Pair with [`ctx-publish`](./ctx-publish) when you want to publish
  the result under a name that isn't the step id
- Step-level reliability knobs (idempotency, retry) apply but rarely
  helpful ‚Äî expressions are pure compute
