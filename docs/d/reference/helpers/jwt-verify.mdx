Verifies a JWT token using the `jose` library. Supports symmetric
(HS256/384/512), asymmetric (RS, ES, EdDSA), and JWKS-fetched public
keys. Validates `iss`, `aud`, `exp` claims. Returns the decoded
payload + convenience aliases (`subject`, `issuer`, `audience`,
`expiresAt`) on success.

Registered as `@blokjs/jwt-verify`. In the handle DSL, import the node
object (`JwtVerifyNode`) from `@blokjs/helpers` and pass it to `step()`.
Strip the `Bearer ` prefix off the header with a small `ExprNode` step,
then feed its handle into the verifier.

## Quickstart

### HS256 with shared secret

```ts
import { workflow, step, http } from "@blokjs/core";
import { ExprNode, JwtVerifyNode } from "@blokjs/helpers";

export default workflow("auth", { version: "1.0.0", trigger: http.post("/secure") }, () => {
  const bearer = step("bearer", ExprNode, {
    expression: "ctx.request.headers['authorization']?.replace('Bearer ', '')",
  });

  step("auth", JwtVerifyNode, {
    token: bearer,
    secret: process.env.JWT_SECRET,
    issuer: "https://auth.example.com",
    audience: "my-api",
  });
});
```

### RS256 with JWKS endpoint

```ts
const bearer = step("bearer", ExprNode, {
  expression: "ctx.request.headers['authorization']?.slice(7)",
});

step("auth", JwtVerifyNode, {
  token: bearer,
  jwksUri: "https://auth.example.com/.well-known/jwks.json",
  issuer: "https://auth.example.com",
  audience: "my-api",
  algorithms: ["RS256"],
});
```

## Inputs

| Field | Type | Notes |
|---|---|---|
| `token` | `string` | The JWT to verify. Required. |
| `secret` | `string` | Shared HS256/384/512 secret. Exactly one of `secret`, `publicKey`, or `jwksUri` must be set. |
| `publicKey` | `string` (PEM) | RSA/EC public key. |
| `jwksUri` | `string` (URL) | JWKS endpoint. Keys are fetched + cached. |
| `issuer` | `string` \| `string[]` | Allowed `iss` value(s). |
| `audience` | `string` \| `string[]` | Allowed `aud` value(s). |
| `algorithms` | `string[]` | Algorithm allowlist. Default permissive â€” restrict in production (e.g. `["RS256"]`). |

## Outputs

| Field | Notes |
|---|---|
| `claims` | The full decoded JWT payload. |
| `subject` | `claims.sub` convenience alias. |
| `issuer` | `claims.iss` |
| `audience` | `claims.aud` |
| `expiresAt` | `claims.exp` (unix seconds) |

## Failure modes

Each throws `GlobalError` with `code: 401` and structured
`body: { reason, message }`:

| reason | Cause |
|---|---|
| `missing_token` | Empty token input. |
| `malformed_token` | Token isn't `header.payload.signature`. |
| `invalid_signature` | HMAC / signature didn't verify against the key. |
| `token_expired` | `exp` is in the past. |
| `issuer_mismatch` | `iss` not in the allowlist. |
| `audience_mismatch` | `aud` not in the allowlist. |
| `algorithm_not_allowed` | Token algorithm not in the explicit allowlist. |

## JWKS cache

When `jwksUri` is set, the helper fetches the JWKS document and caches
keys by `kid` in-process. Re-fetched lazily on cache miss. Use
`_resetJwksCacheForTests()` in test setup to clear between runs.

## Common patterns

### Bearer-token middleware

Verify, then stash the identity for downstream steps. `CtxPublishNode`
writes under a chosen `ctx.state` name:

```ts
import { CtxPublishNode, ExprNode, JwtVerifyNode } from "@blokjs/helpers";

const bearer = step("bearer", ExprNode, {
  expression: "ctx.request.headers['authorization']?.slice(7)",
});

const verify = step("verify", JwtVerifyNode, {
  token: bearer,
  secret: process.env.JWT_SECRET,
  issuer: "...",
  audience: "...",
});

step("stash-identity", CtxPublishNode, {
  name: "identity",
  value: { userId: verify.subject, claims: verify.claims },
});
```

Mounted as trigger-level middleware, downstream steps see
`ctx.state.identity.userId` once verified.

### Algorithm allowlist (defense against alg-mix attacks)

ALWAYS pin `algorithms:` in production. A token signed with a different
algorithm than expected (or `none`) can otherwise sneak through.

```ts
step("auth", JwtVerifyNode, { token: bearer, jwksUri: "...", algorithms: ["RS256"] });
```
