A simple in-process `Map`-backed key-value helper. Identical API to
[`@blokjs/redis-kv`](./redis-kv) — swap one for the other when you
need durability or cross-process state.

NOT durable: clears on process restart. NOT cross-process: each
worker process sees its own store.

Registered as `@blokjs/in-memory-kv`. In the handle DSL, import the node
object (`InMemoryKvNode`) from `@blokjs/helpers` and pass it to `step()`.

## Quickstart

```ts
import { workflow, step, http } from "@blokjs/core";
import { InMemoryKvNode } from "@blokjs/helpers";

export default workflow("save-user", { version: "1.0.0", trigger: http.post("/users") }, () => {
  step("save", InMemoryKvNode, { action: "set", key: "user:42", value: { name: "Alice" } });
});
```

## Inputs

| Field | Type | Notes |
|---|---|---|
| `action` | `"get"` \| `"set"` \| `"delete"` \| `"list"` \| `"clear"` | Operation. |
| `key` | `string` | Required for get/set/delete. |
| `value` | unknown | Required for set. |
| `prefix` | `string` | For `list` only — filter keys starting with this prefix. |

No TTL — entries live until explicitly deleted or the process exits.

## Outputs

| Action | Output |
|---|---|
| `get` | `{ value }` — `undefined` if missing |
| `set` | `{ key, value }` |
| `delete` | `{ deleted: boolean }` |
| `list` | array of `{ key, value }` |
| `clear` | `{ count }` of removed entries |

## When to use

- Unit tests of a workflow that exercises KV state
- Local-dev demos where Redis would be overkill
- Single-process state with no durability requirement (caching a
  computed value within the lifetime of one process)

## When NOT to use

- Any production deployment (use Redis-KV)
- Multi-process scaffolds (different processes don't share state)
- Anything that needs to survive a deploy
