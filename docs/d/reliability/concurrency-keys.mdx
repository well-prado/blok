import { StudioRunHeader, StudioEventBadge, StudioConcurrencyTile } from "/snippets/studio-mocks.jsx";

Set `concurrencyKey` on a trigger and the runner gates incoming runs
against an in-flight count for the resolved key. The named-mutex pattern
serializes per-tenant work; bumping `concurrencyLimit` turns it into a
fairness throttle.

Unlike the per-step reliability primitives, `concurrencyKey` lives on
the **trigger config** (`trigger.http` or `trigger.worker`). The gate
runs at run-entry, before any step executes.

## Quickstart

A multi-tenant render endpoint that allows up to five concurrent renders
per tenant:

<CodeGroup>
```ts TypeScript
import { workflow, $ } from "@blokjs/helper";

export default workflow({
  name: "render-pdf",
  version: "1.0.0",
  trigger: {
    http: {
      method: "POST",
      path: "/render",
      concurrencyKey: $.req.body.tenantId,
      concurrencyLimit: 5,
    },
  },
  steps: [
    { id: "fetch", use: "@blokjs/api-call", inputs: { url: $.req.body.sourceUrl } },
    { id: "render", use: "@blokjs/pdf-render", inputs: { html: $.state.fetch } },
    {
      id: "upload",
      use: "@blokjs/s3-upload",
      inputs: { key: $.req.body.outputKey, body: $.state.render },
    },
  ],
});
```

```json JSON
{
  "name": "render-pdf",
  "version": "1.0.0",
  "trigger": {
    "http": {
      "method": "POST",
      "path": "/render",
      "concurrencyKey": "$.req.body.tenantId",
      "concurrencyLimit": 5
    }
  },
  "steps": [
    { "id": "fetch", "use": "@blokjs/api-call", "inputs": { "url": "$.req.body.sourceUrl" } },
    { "id": "render", "use": "@blokjs/pdf-render", "inputs": { "html": "$.state.fetch" } },
    { "id": "upload", "use": "@blokjs/s3-upload", "inputs": { "key": "$.req.body.outputKey", "body": "$.state.render" } }
  ]
}
```
</CodeGroup>

POST six times in parallel with the same `tenantId`. The first five
acquire a slot and run. The sixth returns `429`:

```http
HTTP/1.1 429 Too Many Requests
Retry-After: 1
Content-Type: application/json

{
  "error": "Concurrency limit reached",
  "workflowName": "render-pdf",
  "concurrencyKey": "tenant-abc",
  "concurrencyLimit": 5,
  "currentInFlight": 5,
  "retryAfterMs": 1000,
  "runId": "run_throttled_..."
}
```

## Field reference

The three fields are defined as a reusable mixin in
[core/workflow-helper/src/types/TriggerOpts.ts:19-50](https://github.com/deskree-inc/blok/blob/main/core/workflow-helper/src/types/TriggerOpts.ts#L19-L50)
and spread into both `HttpTriggerOptsSchema` and
`WorkerTriggerOptsSchema`.

<ParamField path="concurrencyKey" type="string">
  Per-key concurrency gate. Literal string or `$.<path>` proxy
  expression evaluated against the live `ctx` at run-entry. When set,
  runs sharing the resolved key contend for at most `concurrencyLimit`
  concurrent slots. When unset, no gating applies — zero overhead.
</ParamField>

<ParamField path="concurrencyLimit" type="integer (1-10000)" default="1">
  Maximum concurrent runs for the resolved `concurrencyKey`. Default
  `1` matches Trigger.dev's named-mutex pattern (exclusive lock per
  key). Bump for throughput-oriented use cases.
</ParamField>

<ParamField path="concurrencyLeaseMs" type="integer ms (>= 1000)" default="3600000">
  Lease duration for the slot in milliseconds. Default 1h. Process-wide
  override via `BLOK_CONCURRENCY_LEASE_MS`. The lease is the crash-
  safety upper bound on slot leaks.
</ParamField>

<ParamField path="onLimit" type='"throw" | "queue"' default='"throw"'>
  Behavior when the gate denies a run. `"throw"` (default) returns
  `429 Retry-After` (HTTP) or NACKs (worker) — the existing behavior.
  `"queue"` defers the run via the in-process scheduler and re-attempts
  acquisition every second until a slot frees. HTTP returns
  `202 Accepted` with `Location`; worker ACKs without retry. New run
  state `"queued"` (transient) — see [Queue mode](#queue-mode-onlimit-queue).
</ParamField>

All four fields require `concurrencyKey`. Setting `concurrencyLimit`,
`concurrencyLeaseMs`, or `onLimit` without `concurrencyKey` is rejected
at workflow load with `` "`concurrencyLimit` requires `concurrencyKey` to be set." ``.

## Mental model — named mutex per key

The default `concurrencyLimit: 1` plus a key derived from request data
gives you a classic named mutex: at most one run for a given resource.

```ts
// Exactly one checkout at a time per cart
trigger: {
  http: {
    method: "POST",
    path: "/checkout/:cartId",
    concurrencyKey: $.req.params.cartId,
    // concurrencyLimit defaults to 1
  },
}
```

Concurrent checkouts on the same cart serialize: the second 429s with
`Retry-After: 1`, the client backs off and retries, and the runner
admits it once the first releases.

For per-tenant rate limiting, raise the limit:

```ts
// Up to five renders per tenant in flight at once
{ concurrencyKey: $.req.body.tenantId, concurrencyLimit: 5 }
```

The contract is `(workflowName, resolvedKey)`-scoped — different
workflows and different keys never contend with each other.

## Worker triggers

Worker triggers carry both a pre-existing `concurrency` field (parallel
consumers) and the new concurrency-key fields. They're orthogonal:

```ts
trigger: {
  worker: {
    queue: "render-jobs",
    concurrency: 10,                        // 10 parallel consumers (parallelism cap)
    concurrencyKey: $.req.body.tenantId,    // per-tenant fairness within those consumers
    concurrencyLimit: 2,                    // max 2 concurrent per tenant
  },
}
```

A burst of 100 jobs from `tenant-A` distributes 2 to A's slots; the
remaining 8 worker slots are free for other tenants.

When a job hits the limit, the worker NACKs with `willRetry: true`
regardless of `job.attempts` — throttling is transient resource state,
not a permanent failure, so it doesn't count against the workflow's
retry budget. The job-queue back-off (BullMQ delay, NATS redelivery)
handles spacing.

## Run lifecycle

The gate runs in [core/runner/src/TriggerBase.ts](https://github.com/deskree-inc/blok/blob/main/core/runner/src/TriggerBase.ts)
between `tracker.startRun()` and `runner.run()`:

<Steps>
  <Step title="Resolve key">
    The runner evaluates `concurrencyKey` against the live `ctx` (proxy
    expressions compile to `js/ctx....`). Failure to resolve (throw or
    `null`/`undefined`) is fail-open — the gate is skipped and the run
    proceeds.
  </Step>
  <Step title="Acquire slot">
    `runStore.acquireConcurrencySlot(workflow, key, limit, runId,
    leaseExpiresAt)` returns `{acquired, currentInFlight}`. SQLite wraps
    this in a transaction so two simultaneous acquires can't both grant
    a slot at the limit.
  </Step>
  <Step title="On grant — run">
    The lock is held until the workflow's `finally` block calls
    `releaseConcurrencySlot`. Idempotent at the store layer — a stale
    re-release after lease expiry is a no-op.
  </Step>
  <Step title="On denial — throttle">
    `tracker.markRunThrottled(...)` flips status to `"throttled"` and
    emits <StudioEventBadge type="RUN_THROTTLED" />. The runner throws
    `ConcurrencyLimitError` with structured info; trigger transports
    translate it to `429` (HTTP) or NACK (Worker).
  </Step>
</Steps>

The throttled status is distinct from `"failed"` — no step ran, nothing
threw application logic. SLA dashboards filter on status to separate
rate-limit pressure from real errors.

## Lease semantics

Slots have a hard `expires_at` timestamp. Happy path: the workflow's
`finally` block releases the slot. Crash path (process dies before
release): the next acquire to the same `(workflow, key)` bucket lazy-
purges the stale entry.

Default 1h lease; override per-trigger:

```ts
{ concurrencyKey: $.req.body.userId, concurrencyLeaseMs: 4 * 60 * 60 * 1000 }  // 4h
```

Or process-wide via env var:

```bash
BLOK_CONCURRENCY_LEASE_MS=14400000 bun run dev
```

A workflow that legitimately runs longer than the lease keeps holding
its slot — the lease only matters when the process exits abnormally.

## Queue mode (`onLimit: "queue"`)

The default `onLimit: "throw"` rejects over-limit runs immediately
with a `429`. Switch to `onLimit: "queue"` and over-limit runs defer
on the in-process scheduler instead, retrying acquisition every second
until a slot frees:

```ts
trigger: {
  http: {
    method: "POST",
    path: "/render",
    concurrencyKey: $.req.body.tenantId,
    concurrencyLimit: 5,
    onLimit: "queue",
  },
}
```

The 6th request from a tenant returns `202 Accepted` instead of `429`:

```http
HTTP/1.1 202 Accepted
Location: /__blok/runs/run_abc123
Content-Type: application/json

{
  "runId": "run_abc123",
  "workflowName": "render-pdf",
  "status": "queued",
  "scheduledAt": 1735741201000,
  "debounced": false,
  "pingCount": 1
}
```

The client polls `Location` until status flips from `queued` to
`running` to `completed | failed`. The mechanism reuses the
[scheduling deferred-dispatch plumbing](/d/scheduling/overview) — when
the timer fires, the run re-enters the gate and either acquires (then
runs) or re-defers (status flips `queued → running → queued` briefly,
scheduler timer is replaced).

<Note>
Queue mode trades immediate back-pressure for tail latency: callers
don't need to implement client-side retry, but a hot bucket can have
many queued runs all waking on the same 1s tick (a thundering herd).
The trigger-level lease (default 1h) bounds slot leaks if a holder
process dies, so queued runs eventually make progress.
</Note>

| | `onLimit: "throw"` (default) | `onLimit: "queue"` |
|---|---|---|
| HTTP response | `429 Too Many Requests` | `202 Accepted` + `Location` |
| Worker behavior | NACK with redelivery | ACK without retry |
| New run state | `throttled` (terminal) | `queued` (transient) |
| Event | <StudioEventBadge type="RUN_THROTTLED" /> | <StudioEventBadge type="RUN_QUEUED" /> |
| Re-defer cap | n/a | indefinite (lease-bounded) |
| Cancellation | terminal — operator does nothing | `POST /__blok/runs/:id/cancel` aborts the queued run |

`onLimit: "queue"` runs are cancellable while waiting — see
[cancellation](/d/reliability/cancellation).

## Composition with prior tiers

The gate sits at the run level, above the per-step primitives. It
composes cleanly with the rest of the reliability stack.

| Feature | Interaction |
|---|---|
| [Idempotency cache](/d/reliability/idempotency) | Cache check happens INSIDE step execution. A run that's all cache hits still holds a slot for the duration of the cache fetches and persistence applies (~ms). |
| [Per-step retry](/d/reliability/retry) | Retry happens within a single granted slot. The retry loop never re-enters the gate. |
| [Replay](/d/reliability/replay) | Replays go through the gate on the new run. If the workflow is over-limit at replay time, the new run throttles too — correct, since the limit is dynamic state. |
| [Sub-workflows](/d/composition/sub-workflows) | Children invoke directly via `SubworkflowNode`, NOT through `TriggerBase.run()`. Sub-workflow steps DO NOT contend for the parent's slot — only the top-level run does. |
| [Scheduling](/d/scheduling/overview) | The scheduling gate (`debounce` / `delay`) runs BEFORE the concurrency gate. A delayed run that becomes throttled at dispatch flips `delayed → throttled` cleanly. |

<Note>
Sub-workflows don't contend for their parent's slot. If you need to
gate inside a sub-workflow step, do it on the child workflow's trigger
config — the child runs through its own dispatch path when invoked
out-of-band, but in-process invocations from a parent skip the trigger
entirely.
</Note>

## Storage

The gate is durable on SQLite (default for production). Migration v7
adds a `concurrency_locks` table with a primary key on
`(workflow_name, concurrency_key, run_id)` plus indexes on `expires_at`
and `(workflow_name, concurrency_key)`. The schema is documented on
the [trace storage page](/d/reference/trace-storage#concurrency_locks-migration-v7).

| Backend | Behavior |
|---|---|
| **SQLite** (default) | Durable. Read-modify-write wrapped in a transaction so two acquires can't both grant at the limit. |
| **InMemory** (dev) | `Map<bucketKey, Map<runId, expiresAt>>`. Cleared on process exit. |
| **Postgres** | Migration v3 persists `concurrency_locks` to PG. Hybrid model — sync reads against an in-memory mirror, writes async-persist via the existing write queue. Survives restart in-process, but cross-process gating still requires the dedicated NATS KV backend below. |

`BLOK_TRACE_ENABLED=false` disables the gate (the same store backs
trace recording and the lock table).

### Cross-process backend (NATS KV)

Single-process backends cap horizontal scale at one Blok process per
unique key bucket. Switch to the NATS KV backend to coordinate the
gate across multiple processes:

```bash
BLOK_CONCURRENCY_BACKEND=nats-kv \
BLOK_CONCURRENCY_NATS_SERVERS=nats://localhost:4222 \
bun run http:dev
```

When set, the `RunTracker` delegates `acquireConcurrencySlot` and
`releaseConcurrencySlot` to a `NatsKvConcurrencyBackend` that reads
and writes one JSON document per `(workflowName, concurrencyKey)`
bucket, using NATS KV's revision-based compare-and-swap (OCC) to
serialize concurrent acquires. The trigger installs the backend in
`listen()` before serving traffic; connect failures log and fall back
to the in-process SQLite gate.

The full env var reference and connection options live on the
[env vars page](/d/operations/env-vars#concurrency-backend-nats-kv).
Source: [`core/runner/src/concurrency/NatsKvConcurrencyBackend.ts`](https://github.com/deskree-inc/blok/blob/main/core/runner/src/concurrency/NatsKvConcurrencyBackend.ts).

| Aspect | Behavior |
|---|---|
| **Acquire** | One `kv.get` + `kv.update` round-trip per attempt. Bounded retry (10) on CAS failure, then fail-closed. |
| **Lazy purge** | Expired leases are pruned inside the same `acquireSlot` call that observes them. |
| **Idempotent re-acquire** | Same `runId` refreshes its lease without double-counting. |
| **Lease leak protection** | Per-lease `expiresAt` filtered on every acquire; no separate sweeper required. |
| **High-cardinality buckets** | Single-document model has linear JSON growth; for >50 active leases per bucket, contention rises. A per-lease key model is on the roadmap. |

### Cross-process backend (Redis)

The Redis backend (Tier C #4) is the alternative cross-process gate
when NATS isn't available. Same storage shape — one JSON document per
`(workflowName, concurrencyKey)` bucket — but atomicity comes from
server-side **Lua scripts** instead of OCC retry loops. Acquire,
release, and purge each run as a single `EVAL`, so there's no
retry-on-contention.

```bash
BLOK_CONCURRENCY_BACKEND=redis \
BLOK_CONCURRENCY_REDIS_URL=redis://localhost:6379 \
bun run http:dev
```

Connection defaults: `connectTimeout: 5s`, `maxRetriesPerRequest: 0`,
`enableOfflineQueue: false`, `lazyConnect: true` — fail-fast on
broker outage instead of buffering.

| Aspect | NATS KV | Redis |
|---|---|---|
| **Concurrency control** | Revision-based CAS, bounded 10-retry loop | Lua scripts (single round-trip, no retry) |
| **Throughput per bucket** | OCC retry can amplify under contention | Linear with Redis throughput |
| **Operational dependency** | NATS server | Redis 7+ |
| **Connection pool** | NATS client (single conn per process) | ioredis (single conn per process) |

Env vars: `BLOK_CONCURRENCY_REDIS_URL` (preferred — `redis://...`), or
discrete `BLOK_CONCURRENCY_REDIS_HOST` / `_PORT` / `_USERNAME` /
`_PASSWORD` / `_DB` / `_TLS=1`. The
[env vars page](/d/operations/env-vars#concurrency-backend-redis) has
the full reference.

### Production-refusal safety gate

**Both backends refuse to start in `NODE_ENV=production` if the
bucket name / key prefix is left at the default (`blok-concurrency`).**

Two deployments sharing one broker would silently corrupt each other's
`(workflow, key)` buckets. Operators MUST set
`BLOK_CONCURRENCY_NATS_KV_BUCKET` (NATS) or
`BLOK_CONCURRENCY_REDIS_KEY_PREFIX` (Redis) per deployment — e.g.
`prod-us-east`, `prod-eu-west`, `staging`. The backend logs a fatal
error on boot with a clear "set this env var to fix" message.

### When to switch

- **Single process** — leave it default (SQLite). Zero ops overhead,
  highest throughput per key.
- **Two or more processes sharing a workflow** — pick NATS KV or Redis.
  Without one, each process tracks its own bucket and the effective
  limit is `concurrencyLimit × processCount`.
- **You already run NATS** — pick `nats-kv` for one-stack deployments.
- **You already run Redis** — pick `redis` for the same reason.
- **Latency-sensitive acquires** — Redis (single round-trip) edges out
  NATS KV (OCC retry).

## Recipes

<AccordionGroup>
  <Accordion title="Multi-tenant SaaS rate limiter">
    `concurrencyKey: $.req.body.tenantId` with `concurrencyLimit: 5`.
    Each tenant gets up to 5 concurrent renders; the 6th returns
    `429 Too Many Requests`. The client retries after `Retry-After`
    seconds and the runner admits as slots free up.
  </Accordion>

  <Accordion title="Mutex per resource (cart, document, user record)">
    `concurrencyKey: $.req.params.cartId` with the default
    `concurrencyLimit: 1`. The classic named mutex. Concurrent
    checkouts on the same cart serialize.
  </Accordion>

  <Accordion title="Worker fairness across tenants">
    Worker `concurrency: 10` with `concurrencyKey: $.req.body.tenantId`
    and `concurrencyLimit: 2`. Up to 10 jobs in parallel total, but no
    single tenant gets more than 2 worker slots — keeps a noisy tenant
    from starving the others.
  </Accordion>

  <Accordion title="Long-running ML pipeline with custom lease">
    `concurrencyLeaseMs: 14400000` (4h) for a workflow that legitimately
    runs longer than the default lease. The lease only matters in the
    crash path; happy-path release fires from the `finally` block
    regardless.
  </Accordion>

  <Accordion title="Production kill-switch">
    `BLOK_CONCURRENCY_DISABLED=1 bun run prod` — short-circuits the gate
    process-wide. Useful for emergency capacity recovery without
    redeploying code.
  </Accordion>
</AccordionGroup>

## Common pitfalls

<AccordionGroup>
  <Accordion title="Setting concurrencyLimit without concurrencyKey">
    Rejected at workflow load. The limit is meaningless without a key
    to scope it — fix the config rather than working around the
    rejection.
  </Accordion>

  <Accordion title="Expecting sub-workflows to contend for the parent's slot">
    They don't. Children invoke in-process via `SubworkflowNode` and
    bypass `TriggerBase.run()` entirely. If you need inner-loop fairness,
    add `concurrencyKey` to the child workflow's trigger config and
    invoke it out-of-band (e.g. via `wait: false` plus an HTTP
    self-call) — step-level concurrency keys are a planned follow-up.
  </Accordion>

  <Accordion title="Treating throttled as a failure">
    It isn't — no step ran. SLA dashboards should filter throttled out
    of error rate calculations. The 429 response is a back-pressure
    signal, not a bug.
  </Accordion>

  <Accordion title="Default lease too short for long workflows">
    The 1h default is conservative. If your workflow has legitimate runs
    longer than that and you've seen "phantom" admission of new runs
    while the previous is still working, raise `concurrencyLeaseMs` so
    the lease outlasts realistic worst-case execution.
  </Accordion>
</AccordionGroup>

## Not yet shipped

- **Capped exponential backoff for `onLimit: "queue"` re-defer.** v1
  uses a fixed 1s retry interval. A hot bucket can wake all queued
  runs on the same tick (thundering herd).
- **Wakeup-on-release model.** Today re-defer is timer-driven. A
  release signal from the holder would let the next queued run
  acquire immediately instead of waiting for the next tick.
- **`concurrencyQueueTimeoutMs`** — TTL on queued runs. v1 retries
  indefinitely (lease-bounded).
- **Redis backend** — same architecture as NATS KV; could use Lua for
  single round-trip atomic acquire. Pending user pull.
- **Step-level concurrency keys** — different invariant set, separate
  plan. Today the gate is at the run level only.
