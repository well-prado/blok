Passing data between steps should feel boring. In Blok, authors use **typed
handles**. A step returns data, the runner persists that data, and the handle
you got from `step()` points at the persisted slot.

## The rule

<Tip>
**Return values write state. Handles read state.**

A node returns its output. On success, the runner stores it at
`ctx.state[<step-id>]`. The authoring DSL gives you a typed handle to that
slot, so downstream steps pass `order.id`, `lookup.user.email`, or the whole
`lookup` handle directly in their inputs.
</Tip>

```ts
import { http, step, tpl, workflow } from "@blokjs/core";
import { z } from "zod";
import apiCall from "@blokjs/api-call";
import { RespondNode } from "@blokjs/helpers";

export default workflow(
  "fetch-and-respond",
  {
    version: "1.0.0",
    trigger: http.post("/demo"),
    input: z.object({ url: z.string().url() }),
  },
  (req) => {
    const fetched = step("fetch", apiCall, { url: req.body.url });
    step("respond", RespondNode, { body: tpl`result: ${fetched.data}` }, { ephemeral: true });
  },
);
```

Inside that workflow:

- `req` is the trigger **entry handle**. It reads the incoming request, job,
  event, message, or RPC payload.
- `fetched` is a step **output handle**. Field reads on it become structural
  references the runner resolves before the next node runs.
- `tpl` embeds handles into strings without turning them into plain text too
  early.
- The `respond` step is `ephemeral`, so its return value is not persisted and
  its handle must not be read later.

## Entry handles

Every callback workflow receives one entry handle. Name it by trigger kind:

| Trigger | Entry handle convention | Common reads |
|---|---|---|
| `http` | `req` | `req.body`, `req.params.id`, `req.query.q`, `req.headers.authorization` |
| `worker` | `job` | `job.body`, `job.params.queue`, `job.params.jobId`, `job.params.attempt` |
| `webhook` | `event` | `event.body`, `event.headers`, `event.params` |
| `pubsub` | `msg` | `msg.body`, `msg.headers`, `msg.params` |
| `cron` | `tick` | `tick.params`, `tick.headers` |
| `grpc` | `rpc` | `rpc.body`, `rpc.params`, `rpc.headers` |
| `sse` | `stream` | `stream.params`, `stream.query`, `stream.headers` |
| `websocket` | `conn` | `conn.body`, `conn.params`, `conn.headers` |
| `mcp` | `call` | `call.body`, `call.headers` |

All entry handles lower to the same runtime request context. The names are for
reader clarity and TypeScript inference.

## Writing data

Nodes write data by returning it. Do not assign to `ctx.state` in a node.

```ts
import { defineNode } from "@blokjs/core";
import { z } from "zod";

export default defineNode({
  name: "load-user",
  description: "Loads a user by id",
  input: z.object({ userId: z.string() }),
  output: z.object({ user: z.object({ id: z.string(), email: z.string().email() }) }),
  async execute(ctx, input) {
    ctx.logger.log(`loading ${input.userId}`);
    const user = await db.users.findById(input.userId, { signal: ctx.signal });
    return { user };
  },
});
```

The node-side `ctx` ABI is still real and kept. Use it for runtime concerns:

| Node-side field | Use |
|---|---|
| `ctx.request` | Trigger payload after the trigger builds the run context. |
| `ctx.logger` | Logs that should appear in traces and Studio. |
| `ctx.env` | Environment access. |
| `ctx.signal` | Cooperative cancellation. Pass it to `fetch` or long-running work. |
| `ctx.publish(name, value)` | Rare side-channel publication. Prefer returning normal output. |
| `ctx.connection` | WebSocket-only connection API. Usually reached through helper nodes. |
| `ctx.stream` | SSE-only stream API. Usually reached through helper nodes. |

## Reading data

Use the handle in the workflow, not runtime context strings.

```ts
const order = step("validate", validateOrder, { body: req.body });
const score = step("score", scoreOrder, { total: order.total });
step("notify", notifyUser, {
  email: order.customer.email,
  line: tpl`order ${order.id} scored ${score.value}`,
});
```

Handles are structural:

- `order` reads the whole output of `step("validate", ...)`.
- `order.total` reads a field from that output.
- `order.items[0].sku` reads an array index and nested field.
- `tpl` is required when a handle is part of a string.

If handles, `tpl`, and branch operators cannot express what you need, use the
explicit escape hatch:

```ts
import { js } from "@blokjs/core";

step("classify", classifyOrder, {
  lane: js`${order.total} > 100 ? "premium" : "standard"`,
});
```

<Warning>
`js` is intentionally opaque to static analysis and visual tooling. Use it for
non-structural expressions such as ternaries, nullish defaults, `.map`, dates,
or environment reads. Prefer handles, `tpl`, and branch operators first.
</Warning>

## Persistence knobs

The fourth argument to `step()` changes where, or whether, the output is stored.

| Knob | Effect |
|---|---|
| none | Store at `ctx.state[id]`. This is the default. |
| `{ as: "name" }` | Store at `ctx.state[name]`; the returned handle is rooted at `name`. |
| `{ spread: true }` | Shallow-merge returned object keys into `ctx.state`; field handles root at those keys. |
| `{ ephemeral: true }` | Store nothing. The returned handle is deliberately unreadable downstream. |

```ts
const user = step("load", loadUser, { id: req.params.id });
const profile = step("lookup-profile", loadProfile, { id: user.id }, { as: "profile" });
step("audit", AuditNode, { userId: user.id }, { ephemeral: true });
```

<Warning>
An ephemeral step is for side effects only: logs, metrics, response helpers, or
one-shot emissions. Do not read its returned handle. There is no persisted slot
for the runner to resolve.
</Warning>

## Control flow

`branch`, `forEach`, `switchOn`, and `tryCatch` all work with handles.

### Branch operators

`branch` conditions are built from typed operators or a boolean handle:

```ts
import { branch, gt } from "@blokjs/core";

branch("route", gt(order.total, 100), {
  then: () => {
    step("premium", routeOrder, { lane: "premium" });
  },
  else: () => {
    step("standard", routeOrder, { lane: "standard" });
  },
});

branch("stock", order.inStock, {
  then: () => {
    step("ship", shipOrder, { id: order.id });
  },
});
```

Use `eq`, `ne`, `gt`, `gte`, `lt`, `lte`, and `not`. A boolean handle such as
`order.inStock` is a truthiness check. Do not author raw condition strings for
new workflows; the helper emits the runtime expression.

### forEach

`forEach` gives the body callback an item handle and an index handle. They are
readable only inside the loop. The loop itself returns a results-array handle
for downstream steps.

```ts
const results = forEach(order.items, (item, index) => {
  step("reserve", reserveInventory, { sku: item.sku, index });
}, { as: "line" });

step("summarize", summarizeReservations, { results });
```

### Sub-workflows

`subworkflow()` calls another workflow and returns a handle to that child
response. The parent step inputs become the child workflow's request body.

```ts
const receipt = subworkflow("receipt", "send-receipt", { orderId: order.id });
step("respond", RespondNode, { body: receipt.data }, { ephemeral: true });
```

The child name must match the registered workflow name.

## The four control-flow footguns

<AccordionGroup>
<Accordion title="1. Arm-scoped handles do not escape their arm">
A handle created inside a `then`, `else`, `case`, `try`, or `catch` arm is
readable only in that arm or its descendants. To use a value after the control
flow step, use the flow primitive's returned handle when it has one, such as
`forEach` results, or have every branch/switch/tryCatch arm write to the same
`as` key with unique step ids.
</Accordion>

<Accordion title="2. Ephemeral handles are unreadable">
`ephemeral: true` skips persistence. Use it only when nobody reads the returned
handle later.
</Accordion>

<Accordion title="3. Step ids are one flat namespace">
Every step id in a workflow must be unique, including ids inside mutually
exclusive arms. If two arms need one downstream name, give the steps unique ids
and set the same `as` key.
</Accordion>

<Accordion title="4. forEach as/asIndex keys share state">
`forEach` writes the current item to `ctx.state[as]` and the index to
`ctx.state[as + "Index"]`. Those keys share the same flat namespace as step
outputs. Pick `as` names that cannot collide with step ids or outer loops.
</Accordion>
</AccordionGroup>

See [Control-flow handles & footguns](/d/primitives/handles-and-footguns) for
the full scope rules.

## Legacy mapping

The runner still understands object-style workflows and JSON workflows after
normalization. New TypeScript authoring should use the handle DSL. When reading
old code, translate it this way:

| Legacy v1 / mapper form | v2 IR meaning | Handle DSL authoring |
|---|---|---|
| `steps[].name` + `nodes[name].inputs` | `steps[].id` + inline `inputs` | `step("id", node, inputs)` |
| `steps[].node` | `steps[].use` | Pass the node object to `step()` |
| `set_var: true` | default persistence | nothing; persistence is default |
| `set_var: false` | `ephemeral: true` | `step("id", node, inputs, { ephemeral: true })` |
| previous-step reads | previous result | prefer the producing handle |
| request/body/path reads | trigger request | read the entry handle (`req.body`, `job.params.jobId`) |
| string interpolation in mapper input | template expression | `tpl` tagged template |
| raw branch comparison | branch runtime expression | `eq`/`ne`/`gt`/`gte`/`lt`/`lte`/`not` or boolean handle |

`@blokjs/expr` is the intentional exception: its `expression` input is the
JavaScript body the node evaluates. Write plain JavaScript there, without a
mapper prefix.

## Quick reference

```txt
WRITE:   return from a node; the runner persists on success
READ:    use the step handle or entry handle
STRINGS: tpl`order ${order.id}`
BRANCH:  eq/ne/gt/gte/lt/lte/not, or a boolean handle
ESCAPE:  js`...${handle}...` only for non-structural expressions
NEVER:   write ctx.state in a node, reuse step ids, read ephemeral handles
```
