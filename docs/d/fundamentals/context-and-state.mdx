Passing data between steps is the thing you do most in a workflow. Blok has
**one** model for it, and once you have it you never think about it again:
a step returns its output, the runner stores it, and `step()` hands you a
**handle** you pass into later steps.

## The golden rule

<Tip>
**Every step returns its output. The runner persists that output to
`ctx.state[<step-id>]` (on success). `step()` returns a typed _handle_ to that
slot — pass the handle into a later step and the runner resolves it before that
step runs.**

That's the entire model. You never read or write `ctx.state` yourself.
</Tip>

```
   const fetched = step("fetch", apiCall, …)   ──► runner stores output at ctx.state.fetch
                                                    `fetched` is a handle pointing there
   step("respond", RespondNode, { body: fetched })  ──► runner resolves the handle, runs the node
```

```ts
import { workflow, step, http } from "@blokjs/core";
import apiCall from "@blokjs/api-call";
import { RespondNode } from "@blokjs/helpers";

export default workflow("demo", { version: "1.0.0", trigger: http.post("/demo") }, (req) => {
  // step(id, node, inputs) runs the node and returns a TYPED handle to its output.
  const fetched = step("fetch", apiCall, { url: req.body.url });

  // Pass the handle straight in — the runner resolves it before `respond` runs.
  step("respond", RespondNode, { body: fetched }, { ephemeral: true });
});
```

- **Returning** is how you write. A node `return`s a value; the runner stores it
  at `ctx.state[<step.id>]` **on success only** — a step that throws writes
  nothing.
- **The handle** is how you read. `step()` returns it; reading a field
  (`fetched.data`, `fetched.items[0]`) records a typed reference the runner
  resolves *before the next node runs*. You never read the value yourself.

<Warning>
Do **not** write to `ctx.state` inside a node's `execute()`. Return your output
and let the runner persist it. (If you genuinely need a side-channel value, use
`ctx.publish(name, value)` — but that's rare.)
</Warning>

## Where data comes from

A step input reads from exactly four sources. In the TypeScript DSL each one is
a typed **handle**; the same reference has an **object-style / JSON** form
(`$.state.…`) that the DSL compiles to.

| Source | Handle (TypeScript DSL) | Object-style / JSON |
|---|---|---|
| An earlier step's output | the handle `step()` returned — `fetched`, `order.qty` | `$.state.<id>` |
| The trigger payload | the build-callback's entry arg — `req.body`, `req.params.id`, `req.query.x` | `$.req.body` / `$.req.params.x` |
| The immediately previous step | rarely needed — handles supersede it | `$.prev` |
| The current `forEach` item | the loop callback's `item` arg | `$.state.<as>` |

<CodeGroup>
```ts TypeScript (handles)
import { workflow, step, http } from "@blokjs/core";
import apiCall from "@blokjs/api-call";
import { RespondNode } from "@blokjs/helpers";

export default workflow("fetch-and-respond", { version: "1.0.0", trigger: http.post("/demo") }, (req) => {
  const fetched = step("fetch", apiCall, { url: req.body.url });
  step("respond", RespondNode, { body: fetched }, { ephemeral: true });
});
```

```json JSON
{
  "name": "fetch-and-respond",
  "version": "1.0.0",
  "trigger": { "http": { "method": "POST", "path": "/demo" } },
  "steps": [
    { "id": "fetch",   "use": "@blokjs/api-call", "inputs": { "url": "$.req.body.url" } },
    { "id": "respond", "use": "@blokjs/respond",  "inputs": { "body": "$.state.fetch" }, "ephemeral": true }
  ]
}
```
</CodeGroup>

<Note>
`$.state.<id>` / `$.req.*` is the **object-style and JSON** surface — the same
data flow the handle DSL compiles to. In hand-written JSON, write the string
`"$.state.fetch"` (or the explicit `"js/ctx.state.fetch"`). You only reach for
these strings when authoring JSON; in TypeScript, pass handles.
</Note>

## Writing: just return

A node's job is to **return** its output. The runner does the rest.

```ts
import { defineNode } from "@blokjs/core";
import { z } from "zod";

export default defineNode({
  name: "eval-search",
  input: z.object({ query: z.string() }),
  output: z.object({ query: z.string(), hits: z.array(z.object({ doc: z.string(), rank: z.number() })) }),
  async execute(ctx, input) {
    // `input` is the validated, already-resolved step input.
    const hits = [0, 1, 2].map((i) => ({ doc: `${input.query} :: doc-${i}`, rank: i }));
    return { query: input.query, hits }; // ← stored at ctx.state["<this step's id>"]
  },
});
```

## Persistence knobs

By default a step stores its output at `ctx.state[<id>]`. The **4th argument**
to `step()` changes that — set it on the step, never in the node.

| Knob | Effect |
|---|---|
| *(none)* | Store at `ctx.state[<id>]` (the default). |
| `{ as: "name" }` | Store at `ctx.state.name` instead. Mutually exclusive with `spread`. |
| `{ spread: true }` | Shallow-merge the returned object's keys into `ctx.state` (multi-output nodes). |
| `{ ephemeral: true }` | Skip storage — the handle is unreadable downstream; only the immediately next step sees the value via `$.prev`. |

```ts
step("load",   fetchUserAndProfile, {}, { spread: true });   // → state.user + state.profile
step("audit",  AuditLogNode, { event: "viewed" }, { ephemeral: true }); // side-effect only
const score = step("result", compute, {}, { as: "finalScore" });        // → state.finalScore
```

## Context inside `forEach`

`forEach(iterable, (item, index) => {…}, opts?)` runs its callback once per
element. The `item` handle is the current element, scoped to the loop body;
steps inside read it like any handle, and their outputs land in
`ctx.state[<inner-id>]` for that iteration. The loop **returns** a handle to the
results array (one entry per item, in order) — that's what survives after the
loop.

```ts
import { workflow, step, forEach, http } from "@blokjs/core";
import evalSearch from "../nodes/eval-search";
import evalScore from "../nodes/eval-score";
import { RespondNode } from "@blokjs/helpers";

export default workflow("score-items", { version: "1.0.0", trigger: http.post("/score") }, (req) => {
  const results = forEach(req.body.items, (item) => {
    const search = step("search", evalSearch, { query: item.query });
    step("score", evalScore, { id: item.id, hitCount: search.hits });
  }, { as: "item" });

  step("respond", RespondNode, { body: { results } }, { ephemeral: true });
});
```

<Warning>
Pick an `as` name that does **not** collide with any step `id` in the workflow —
the loop item and step outputs share one flat `ctx.state`. The builder
**throws at load time** if they collide. The `item`/`index` handles do not
survive past the loop; read the returned results handle instead. See
[Handles & footguns](/d/primitives/handles-and-footguns).
</Warning>

## Context across sub-workflows

`subworkflow()` calls another workflow like a function and returns a handle to
its response:

- The call's resolved **`inputs` become the child's `ctx.request.body`**.
- With `wait: true` (default), the **child's response lands at
  `ctx.state[<id>]`** in the parent — and is what the returned handle points to.

```ts
import { forEach, step, subworkflow } from "@blokjs/core";
import evalScore from "../nodes/eval-score";

// Parent
forEach(load.items, (item) => {
  const retrieve = subworkflow("retrieve", "eval-retrieve", { query: item.query });
  step("score", evalScore, { hits: retrieve.data.hits });
}, { as: "item" });

// Child "eval-retrieve" reads the parent's inputs as its request body:
//   step("search", evalSearch, { query: req.body.query });
```

<Note>
The child name (second argument) must match the **name the child is registered
under**. File-based routes register under the workflow's `name`; an explicit
`Workflows.ts` map registers under the **map key**. See
[Sub-workflows](/d/composition/sub-workflows).
</Note>

## The footguns

The control-flow footguns — arm-scoped handles, ephemeral reads, duplicate step
ids, `forEach` `as:` collisions — are covered on
[**Handles & footguns**](/d/primitives/handles-and-footguns). The three below
are specific to data flow.

<AccordionGroup>
<Accordion title="Putting a js/ expression as @blokjs/expr's `expression`">
`@blokjs/expr`'s `expression` input is **itself** resolved by the mapper before
the node runs. So `expression: "js/ctx.state.uid.userId"` resolves to the *value*
first, then `@blokjs/expr` tries to evaluate that value as code — usually a
confusing error.

**Fix:** write the JS directly, no `js/` prefix:
`expression: "ctx.state.uid.userId"`.
</Accordion>

<Accordion title="Expecting `$.prev` to survive more than one step">
`$.prev` (the object-style/JSON read for the previous step) is the
**immediately** previous step's output. It's overwritten on every step. In the
handle DSL you rarely need it — a handle points at a stable `ctx.state` slot.

**Fix:** for anything beyond the adjacent step, read the step's handle (or
`$.state.<id>` in JSON).
</Accordion>

<Accordion title="A literal ${...} in an authored input string">
The mapper interpolates `${...}` placeholders in **authored** input strings. A
stray `${...}` that isn't a real ctx path **throws** under the default
`BLOK_MAPPER_MODE=strict`. (Data that merely *contains* `${...}` and flows
through a handle / `$.state.x` is never re-interpolated — it's safe.)

**Fix:** to embed a handle in a string use the `tpl` tagged template
(`tpl\`order ${order.id}\``); don't hand-write `${...}` unless it's a real ctx
path.
</Accordion>
</AccordionGroup>

## Quick reference

```
WRITE:   return your output           → runner stores it at ctx.state[<id>]
READ:    const x = step(...)           → x is a handle to that output  (the default)
         req.body / .params / .query   → the trigger payload
         the forEach `item` arg        → the current loop item
         (JSON twin: $.state.<id> · $.req.* · $.prev)
KNOBS:   step(id, node, inputs, { as | spread | ephemeral })
NEVER:   write ctx.state directly · reuse a step id · prefix @blokjs/expr with js/
```
