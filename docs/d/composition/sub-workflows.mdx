import { StudioRunHeader, StudioStepRail, StudioSubRunsStrip } from "/snippets/studio-mocks.jsx";

`subworkflow()` invokes another named workflow as if calling a
function. It returns a typed handle to the child's response — use it
downstream exactly like a `step()` handle. By default the parent blocks
until the child completes and the child's `ctx.response` becomes the
handle's value; opt into fire-and-forget dispatch with `wait: false`
and the call returns immediately with `{runId, workflowName,
scheduledAt}` while the child runs asynchronously.

This is the Tier 2 composition primitive — pair it with
[idempotent caching](/d/reliability/idempotency) and
[retry](/d/reliability/retry) and a sub-workflow becomes a cacheable,
retriable, replayable unit of work. Add `wait: false` to the mix and
you get [at-most-once dispatch deduplication](#at-most-once-dispatch-with-wait-false)
matching Trigger.dev / Stripe semantics.

## Quickstart

`subworkflow(id, name, inputs, opts?)` is the handle-DSL way to call a
child. The child is just a normal workflow; its `name` is the public
identifier you pass as the second argument.

<CodeGroup>
```ts src/workflows/send-receipt-email.ts
import { workflow, http, step } from "@blokjs/core";
import { SendGridNode, ExprNode } from "@blokjs/helpers";

export default workflow(
  "send-receipt-email",
  { version: "1.0.0", trigger: http.post("/internal/send-receipt") },
  (req) => {
    const sent = step("send", SendGridNode, {
      to: req.body.email,
      body: req.body.body,
    });
    step("respond", ExprNode, {
      expression: { messageId: sent.id, sentAt: sent.timestamp },
    });
  },
);
```

```ts src/workflows/checkout.ts
import { workflow, http, step, subworkflow } from "@blokjs/core";
import { StripeChargeNode } from "@blokjs/helpers";

export default workflow(
  "checkout",
  { version: "1.0.0", trigger: http.post("/checkout") },
  (req) => {
    const charge = step("charge", StripeChargeNode, {
      amount: req.body.amount,
      card: req.body.card,
    });

    const receipt = subworkflow("send-receipt", "send-receipt-email", {
      email: req.body.email,
      body: charge.receipt,
    });

    // `receipt` is a handle to the child's response — read it downstream.
    return receipt;
  },
);
```
</CodeGroup>

The parent's `inputs` (third argument) become the child's
`ctx.request.body`. The child reads them via `req.body.<key>` exactly as
if it had been HTTP-triggered. The child's `ctx.response` lands on the
parent's `state["send-receipt"]`, and `subworkflow()` returns a handle
pointing there.

## Signature

```ts
subworkflow(
  id: string,
  name: string | Handle,        // literal workflow name OR a handle for polymorphic dispatch
  inputs: Record<string, unknown>,
  opts?: {
    wait?: boolean;             // default true — block for the child; false = fire-and-forget
    dispatch?: "in-process" | "http-self";
    allowList?: string[];       // constrain resolved names when `name` is a handle
    idempotencyKey?: string | Handle;
    idempotencyKeyTTL?: number; // default 86_400_000 (24h)
    retry?: RetryConfig;
    as?: string;                // store the response at state[as] instead of state[id]
    ephemeral?: boolean;        // skip persistence; only ctx.prev carries it forward
  },
): Handle
```

The returned handle is rooted at `state[id]` (or `state[as]` when `as`
is set) — pass it to downstream `step()`/`subworkflow()` inputs, a
`branch`/`switchOn` discriminant, or `return` it as the workflow's
response.

<Note>
**Also supported: object-style steps.** In JSON workflows (and the v1
object DSL) the same step is a plain object —
`{ id, subworkflow, inputs, ...opts }` — and works identically. The
handle-DSL `subworkflow()` emits exactly that shape under the hood, so
the two are wire-compatible. See
[v2 step shapes](/d/reference/v2-step-shapes) for the object form.

```ts
// equivalent object-style step (JSON or v1 DSL)
{ id: "send-receipt", subworkflow: "send-receipt-email", inputs: { /* ... */ } }
```
</Note>

## How dispatch works

<Steps>
  <Step title="Lookup">
    Find the child workflow by name in the
    [Workflow Registry](/d/reference/workflow-registry). The HTTP
    trigger auto-registers every scanned workflow at boot.
  </Step>
  <Step title="Materialize">
    Build a child `Configuration` and `Runner` from the registered
    definition.
  </Step>
  <Step title="Isolate">
    Build a fresh child `Context` — new `id`, fresh `state`, fresh
    `response`, fresh `error`. Share `logger` / `env` / `eventLogger`
    with the parent for log routing consistency.
  </Step>
  <Step title="Guard">
    Increment the recursion depth counter and refuse to proceed past
    `BLOK_MAX_SUBWORKFLOW_DEPTH` (default `10`).
  </Step>
  <Step title="Track">
    Start a tracked child run with `parentRunId` and `parentNodeRunId`
    pointing back at the invoking parent step.
  </Step>
  <Step title="Run">
    Execute the child to completion (or failure). The child's
    `ctx.response` becomes the parent step's output, applied through
    the parent step's `as` / `spread` / `ephemeral` rules.
  </Step>
</Steps>

If the child throws under `wait: true`, the exception propagates to
the parent step. The parent's `retry` (if set) catches it and tries
again with a fresh child run. Under `wait: false`, child errors are
caught and routed to `tracker.failRun(childRunId, err)` plus
`console.error` — they do NOT propagate to the parent (which has
already returned).

## Wait modes

The parent step's behavior depends on `wait`:

### `wait: true` (default) — synchronous

The parent step `await`s the child runner to completion. The child's
`ctx.response` becomes the parent step's `model.data` and lands on
`state[<id>]` via `applyStepOutput`. Errors propagate to the parent
step (and its `retry` loop, if configured).

```ts
const receipt = subworkflow("send-receipt", "send-receipt-email", {
  user: user,
  order: order,
}); // wait: true is the default
```

`receipt` (and `state["send-receipt"]`) carries the child's full
response shape.

### `wait: false` — fire-and-forget

The parent step returns IMMEDIATELY with dispatch metadata. The child
runs asynchronously via `setImmediate`. Errors are caught + routed to
the tracker, NOT propagated to the parent.

```ts
const queued = subworkflow("queue-email", "send-receipt-email", {
  orderId: charge.id,
  email: req.body.email,
}, { wait: false });
```

`queued` (and `state["queue-email"]`) carries:

```ts
{
  runId: "run_abc123",
  workflowName: "send-receipt-email",
  scheduledAt: 1735741200000,        // ms since epoch when dispatch was queued
}
```

Callers poll `GET /__blok/runs/<runId>` for the actual outcome. The
child still appears in Studio's Sub-runs strip with status
transitioning `running → completed | failed` independently of the
parent's lifecycle.

### Comparison

| | `wait: true` | `wait: false` |
|---|---|---|
| Parent step output | child's `ctx.response` | `{runId, workflowName, scheduledAt}` |
| Child error propagation | thrown → parent's `retry` loop | caught → `tracker.failRun` + `console.error` |
| Parent step duration | full child execution | ~ms (dispatch overhead only) |
| Studio Sub-runs strip | yes | yes |
| Lineage (`parentRunId`, `parentNodeRunId`) | preserved | preserved |
| Cross-process | no | no (single-process `setImmediate`) |

## Output shape

The returned handle (and `$.state[<id>]`) carries whatever the child
put in `ctx.response` — typically via `@blokjs/respond` as the last
step. Feed it straight into downstream node inputs:

```ts
const fulfill = subworkflow("fulfill", "ship-order", {
  orderId: charge.orderId,
});

// later in the same parent
step("log", PostgresWriteNode, {
  row: { trackingId: fulfill.trackingId },
});
```

If the child exposes multiple outputs you want flattened to the top
level of the parent's state, pass `{ spread: true }` in the opts.

## Composing with idempotency caching

`idempotencyKey` on a sub-workflow step caches the parent step's
output. The shape of the cache changes based on `wait`:

### `wait: true` — caches the whole child result

Cache HIT means the child workflow is **never invoked** — no child run
record, no child `NODE_STARTED` events, no side effects fire.

```ts
subworkflow("callPipeline", "generate-and-send-email", {
  topic: req.body.topic,
  email: user.email,
}, {
  idempotencyKey: req.body.requestId,
  retry: { maxAttempts: 3 },
});
```

<Warning>
**A cache hit on a `wait: true` sub-workflow step suppresses every
side effect inside the child.** If the child sends an email, charges
a card, or posts to Slack, an `idempotencyKey` on the parent step will
skip those side effects entirely on cache hit.

If your child has unconditional side effects under `wait: true`, do
not put `idempotencyKey` on the parent step. Instead, cache the
individual pure-compute steps inside the child (LLM calls, fetches,
transforms) so the side effects fire every invocation while expensive
computation gets reused.
</Warning>

When the child is purely deterministic under `wait: true` (LLM call →
transform → return), caching the whole sub-workflow is the right move.
When the child sends emails or makes payments, cache inside it instead.

### At-most-once dispatch with `wait: false`

`wait: false + idempotencyKey` is the **headline composition pattern**:
at-most-once dispatch deduplication, matching Trigger.dev and Stripe
semantics.

```ts
subworkflow("send-receipt", "send-receipt-email", {
  orderId: charge.id,
  email: req.body.email,
}, {
  wait: false,
  idempotencyKey: req.body.requestId,
});
```

Behavior:

- **First call**: dispatch fires; the cache stores
  `{runId, workflowName, scheduledAt}` against the key.
- **Subsequent call (same key)**: cache HIT returns the SAME `runId`
  for the lifetime of the cache entry — regardless of the original
  child's outcome (success OR failure).
- **To retry on child failure**: the caller uses a NEW idempotency
  key. The cached `runId` always points at the original (possibly
  long-completed or long-failed) child run; the caller polls
  `/__blok/runs/<runId>` to check current state.

This matches Stripe's documented behavior: "If you retry a request
with an idempotency key, Stripe returns the original result regardless
of whether it succeeded or failed." The pattern is intentional, not a
footgun.

### Cache contract summary

| | `wait: true + idempotencyKey` | `wait: false + idempotencyKey` |
|---|---|---|
| What's cached | Child's full `ctx.response` | Dispatch metadata `{runId, workflowName, scheduledAt}` |
| Cache HIT behavior | Child not invoked at all (no side effects) | Same `runId` returned; original child's outcome stands |
| Retry on failure | Re-runs the child | Use a new key |
| Risk profile | Footgun for side-effecting children | Standard at-most-once dispatch |

## Composing with retry

Retry on a sub-workflow step behaves differently per wait mode:

- **`wait: true`** — `retry` covers full child execution. Each retry
  creates a fresh child run record under the same parent, so Studio's
  sub-runs strip on the parent shows N sequential attempts. The retry
  budget includes child execution time: a 30-second child with
  `maxAttempts: 3` can run for ~90 seconds plus backoff before failing
  terminally.
- **`wait: false`** — `retry` covers only the **dispatch** (registry
  lookup, recursion guard). The async child's failure does NOT trigger
  parent retry — the parent step has already returned successfully
  with the runId. To retry on async child failure, the caller uses a
  new idempotency key.

## Composing with replay

[Replaying](/d/reliability/replay) the parent run starts a fresh
parent run with `replayOf` set. Each sub-workflow step in the replayed
parent either:

- **Hits the cache** (when `idempotencyKey` is set and the entry hasn't
  expired) — child workflow does not run again.
- **Misses the cache** — child runs fresh and gets a new `parentRunId`
  pointing at the new parent run.

Sub-run lineage on the new parent is independent of the original —
clean slate, picks up code changes everywhere caching didn't apply.

## Cancellation cascade

When a parent run is cancelled via
[`POST /__blok/runs/:id/cancel`](/d/reliability/cancellation), every
synchronous (`wait: true`) child currently in flight also receives the
abort signal. The mechanism is `createChildContext`, which builds a
fresh `AbortController` for the child and chains it to the parent's
signal:

```ts
// core/runner/src/utils/createChildContext.ts
const childController = new AbortController();
parent.signal?.addEventListener("abort", () => childController.abort());
childCtx.signal = childController.signal;
```

Cancelling the child does NOT cascade back up to the parent —
independent lifecycle preserved. Cancelling a `wait: false` child does
NOT cascade either, because the parent step has already returned by the
time the child runs; cancel async children directly via their own
`runId`.

For nodes inside a child to actually abort on cascade, they need to
consult `ctx.signal.aborted` (or pass `ctx.signal` to `fetch`) — see
[cancellation](/d/reliability/cancellation) for the cooperative
pattern.

## Recursion guard

Sub-workflows can themselves invoke sub-workflows. Depth is tracked on
the child's ctx and capped at `10` by default.

```bash
# Override when intentional:
BLOK_MAX_SUBWORKFLOW_DEPTH=20 bun run dev
```

When the cap is exceeded, the runner throws:

```
[blok] Sub-workflow recursion limit exceeded (depth 11 > 10).
Likely a cycle: workflow "alpha" called sub-workflow "beta" too deep.
Bump via BLOK_MAX_SUBWORKFLOW_DEPTH if intentional.
```

This is a hard fail — workflows form an acyclic call graph by default.

## Workflow discoverability

By default, every workflow under the HTTP trigger's scanned directory
is auto-registered when `buildFileBasedRoutes()` runs. The trigger
logs:

```
[blok] workflow registry — N workflow(s) callable as sub-workflow
```

If two workflows share the same `name:` field from different sources,
the registry throws at startup. Rename one. See the
[registry reference](/d/reference/workflow-registry) for manual
registration and HMR notes.

## Common pitfalls

<AccordionGroup>
  <Accordion title="Caching a `wait: true` sub-workflow that sends emails">
    See the warning above. Cache inside the child, not at the parent
    step. (`wait: false` is fine — the cache holds dispatch metadata,
    not the response.)
  </Accordion>
  <Accordion title="Expecting `wait: false` errors to retry">
    They don't. Once `setImmediate` fires the dispatch, the parent
    step has already returned; child errors land in `tracker.failRun`,
    not in the parent's retry loop. To retry on async child failure,
    the caller uses a new idempotency key.
  </Accordion>
  <Accordion title="Polling the cached runId after `wait: false + idempotencyKey`">
    The cached `runId` always points at the original child run. If
    that run failed, polling shows the failure forever — that's the
    at-most-once contract. Use a new idempotency key to dispatch
    fresh.
  </Accordion>
  <Accordion title="Forgetting that the child reads from req.body, not state">
    The parent's `inputs` become `request.body` for the child. Inside
    the child, read the entry handle's `req.body.foo`, not parent state
    — the parent's state isn't visible to the child.
  </Accordion>
  <Accordion title="Recursion through a shared utility workflow">
    If `validate` calls `lookup-user` which calls `validate`, you'll
    hit the depth cap. Refactor the utility into a non-sub-workflow
    node.
  </Accordion>
  <Accordion title="Workflow name collisions">
    The registry is strict. If you accidentally give two workflows the
    same `name:`, startup fails fast — fix the name rather than try to
    work around it.
  </Accordion>
</AccordionGroup>

## Polymorphic dispatch

The `name` argument accepts a literal workflow name *or* a **handle**
(`event.body.kind`, `req.body.type`). One trigger can fan out to N
handlers without a `switch` step.

```ts
subworkflow("dispatch", event.body.kind, { event: event.body }, {
  allowList: ["handler.payment", "handler.shipping"],
});
```

When `name` is a handle, the runner resolves it against the live ctx at
dispatch time and looks up the resolved name in the workflow registry.
If a parent trigger sets a namespace (today: webhook
`namespace: "stripe"`), the resolved name is prefixed before lookup —
`invoice.paid` becomes `stripe.invoice.paid`.

`allowList` is the safety guard. When set, the **final** resolved name
(post-namespace) must be in the array — otherwise the dispatch is
rejected with a structured error. **Strongly recommended** when the
handle resolves from caller-supplied data so a malicious request body
can't escalate the workflow surface accessible from a route. The
allow-list also works on literal `name` strings for defence-in-depth
audit.

Resolution failure modes:
- Expression evaluates to a non-string or empty string → throw with the
  raw expression and the resolved value, so an operator can debug the
  ctx shape.
- Resolved name (post-namespace) not in `allowList` → throw with the
  resolved name and the full allow-list.
- Resolved name not registered → standard "Sub-workflow not found"
  error, listing the registered workflows.

## Cross-process dispatch (`dispatch: "http-self"`)

`subworkflow()` accepts a `dispatch` opt that picks the execution
strategy. Default is `"in-process"`; switch to `"http-self"` to
dispatch the child as a fresh HTTP request to the deployment's own
base URL.

```ts
subworkflow("send-receipt", "send-receipt-email", {
  user: user,
  order: order,
}, { dispatch: "http-self" });
```

| `dispatch:` value | What happens |
|---|---|
| `"in-process"` (default) | Child runs in the SAME Node process. Synchronous (`wait: true`) or `setImmediate` (`wait: false`). Cheapest, no extra hops. |
| `"http-self"` | Child is dispatched as a fresh HTTP request to `BLOK_SELF_BASE_URL` (defaults to `http://localhost:$PORT`, or port 4000 when unset). Each child can land on a different process in a horizontally-scaled deployment. |

### Requirements

- Child workflow MUST have an HTTP trigger (`trigger.http.path` set).
  A runtime error throws otherwise.
- Request URL: `${BLOK_SELF_BASE_URL}${child.trigger.http.path}`.
- Request method: `child.trigger.http.method` (default POST).
- Request body: parent step's resolved `inputs` (post-mapper).
- `BLOK_SELF_BASE_URL` must point at the deployment's reachable
  hostname — for Docker / Kubernetes, set to the service DNS name
  (e.g. `http://blok-svc:4000`).

### Lineage across the HTTP boundary

Three headers carry parent → child lineage so Studio renders the
sub-run strip correctly:

| Header | Value |
|---|---|
| `X-Blok-Parent-Run-Id` | Parent's run id. |
| `X-Blok-Parent-Node-Run-Id` | The specific NodeRun (the sub-workflow step). |
| `X-Blok-Subworkflow-Depth` | Recursion depth. Bounded by `BLOK_MAX_SUBWORKFLOW_DEPTH`. |

`TriggerBase.run` reads the headers when the request lands + threads
them into `tracker.startRun(...)`.

### Wait modes with `http-self`

- **`wait: true`** awaits the HTTP response (non-2xx → throw,
  propagating to the parent step's `retry` loop if configured).
- **`wait: false`** fires-and-forgets the fetch (`.catch + console.error`).
  Parent gets back `{runId: null, workflowName, scheduledAt, dispatch:
  "http-self", url}` — the actual child runId isn't known on the
  parent side; the receiver creates the run record when the request
  lands. Studio's Sub-runs strip surfaces the child once it appears.

### Trade-offs vs. in-process

- HTTP roundtrip adds latency (~few ms locally, more across the
  network).
- The deployment must be reachable at `BLOK_SELF_BASE_URL` from the
  dispatching process.
- Idempotency caching still wraps the WHOLE dispatch — a cache hit
  short-circuits the HTTP call too.

When to use:

- **Single process** — stick with `"in-process"`. No reason to pay the
  HTTP cost.
- **Multi-process deployment + strict isolation per child** —
  `"http-self"`. Children can land on different processes; the parent's
  call stack doesn't block on child resources.
- **Async backpressure** — `wait: false` + `http-self` decouples the
  parent from child throughput entirely.

## What's not yet supported

- **Cancellation cascade to fire-and-forget children.** Cancelling
  the parent does NOT propagate to `wait: false` children (the
  parent step has already returned by the time the child runs). Cancel
  async children directly via their own `runId` —
  [`POST /__blok/runs/:childId/cancel`](/d/reliability/cancellation).
- **Cross-process latest-payload-wins for `wait: false` http-self.**
  Today the parent's resolved inputs are serialized as the body once;
  if the parent re-runs (replay), the body comes from the new run's
  inputs.
