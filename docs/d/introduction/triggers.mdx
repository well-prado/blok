A **trigger** decides *when* a workflow runs. The same workflow body — nodes, handles, control
flow — works under any trigger; only the trigger block changes. A workflow declares **exactly
one** trigger.

```ts
// The same "process this order" logic, two different triggers:
workflow("orders.process", { version: "1.0.0", trigger: http.post("/orders") }, (req) => {
  step("process", processOrder, { order: req.body });
});

workflow("orders.process", { version: "1.0.0", trigger: { worker: { queue: "orders" } } }, (job) => {
  step("process", processOrder, { order: job.body });
});
```

The only differences: the trigger block, and the entry handle's name + shape (`req` vs `job`).

## Choosing a trigger

Picking `http` reflexively is the most common mistake. Choose by intent:

| What you're building | Trigger | Why not HTTP |
|---|---|---|
| Respond to an HTTP/REST request; JSON API; HTML; file download | **`http`** | — |
| Background / queued / async job; offload slow work | **`worker`** | HTTP blocks the caller; jobs need a queue, retries, and a DLQ |
| Run on a schedule (nightly, hourly, cron) | **`cron`** | HTTP only fires on a request |
| React to cloud topic/subscription events across services | **`pubsub`** | HTTP isn't subscribed to a broker |
| One-way live push to a browser (tokens, progress, a feed) | **`sse`** | a plain HTTP response is one-shot |
| Bidirectional realtime (chat, live cursors) | **`websocket`** | HTTP is half-duplex, no server push-back |
| Receive a **signed** provider webhook (Stripe, GitHub, Slack, Shopify, Svix) | **`webhook`** | HTTP won't verify the HMAC signature |
| Expose a workflow as a tool to an AI client (Claude, Cursor) | **`mcp`** | HTTP isn't MCP — the client can't discover it |
| High-throughput typed RPC with a `.proto` contract | **`grpc`** | HTTP/REST overhead is too high |

<Note>
**There is no `queue` trigger** — use **`worker`**. Queue consumers, fan-out, async jobs: all
`worker`. (`trigger: { queue: ... }` throws at load.)
</Note>

## Configuring a trigger

HTTP uses the `http.*` helper. Every other trigger is a raw block passed as `opts.trigger`:

<CodeGroup>
```ts HTTP
trigger: http.post("/orders")           // .get / .post / .put / .delete / .patch / .any
trigger: http.get("/users/:id")         // path params
trigger: http.any()                     // catch-all (omit path → file-based routing)
```

```ts Worker
trigger: { worker: { queue: "emails", provider: "nats", retries: 3, concurrency: 5 } }
```

```ts Cron
trigger: { cron: { schedule: "0 8 * * *", timezone: "America/New_York" } }
```

```ts Webhook
trigger: { webhook: { provider: "stripe", events: ["payment_intent.succeeded"] } }
```
</CodeGroup>

## The entry handle

The build callback's argument is the trigger's **entry handle** — the incoming payload, typed
and named by convention per trigger:

| Trigger | Entry | Read |
|---|---|---|
| `http` | `req` | `req.body`, `req.params.id`, `req.query.q`, `req.headers["x-…"]` |
| `webhook` | `event` | `event.body` (the verified payload), `event.headers` |
| `worker` | `job` | `job.body` (payload), `job.params.{queue,jobId,attempt}` |
| `cron` | `tick` | `tick.params` (schedule metadata — no body) |
| `pubsub` | `msg` | `msg.body`, `msg.headers` |
| `grpc` | `rpc` | `rpc.body` |

```ts
workflow("nightly.digest", { version: "1.0.0", trigger: { cron: { schedule: "0 8 * * *" } } }, (tick) => {
  step("build", buildDigest, { day: tick.params.scheduledTime });
});
```
