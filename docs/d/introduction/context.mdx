The Context Object (`ctx`) facilitates data sharing across the [nodes](/d/introduction/nodes) of a single [workflow](/d/introduction/workflows) run. It's the shared memory space where each step's output lands and where later steps look it up.

<Tip>
**The golden rule:** a step **returns** its output, the runner **persists** it to
`ctx.state[<step-id>]`, and later steps **read** it with `$.state.<id>`. You never
write `ctx.state` yourself. For the complete model — the four reads, persistence
knobs, forEach + sub-workflow context, and the common footguns — see
[**Context & State**](/d/fundamentals/context-and-state).
</Tip>

## What is the Context Object?

When a [Trigger](./triggers.mdx) initiates a Workflow, Blok creates a unique Context Object for that specific workflow instance. This object persists throughout the execution of that instance and is accessible to every Node within it.

Key characteristics of the Context Object:

*   **Instance-Specific**: Each run of a workflow gets its own isolated Context Object. Data in one workflow instance does not interfere with another.
*   **Shared State**: Nodes can read from and write to the Context Object, allowing them to share data and state dynamically.
*   **Data Carrier**: It carries initial data from the trigger (e.g., HTTP request body, query parameters, message payload) and makes it available to all Nodes.
*   **Configuration & Services**: It can also be used to provide access to configuration values, shared services, or utility functions that Nodes might need.

## Structure and Usage

In `Blok`, the Context Object (`ctx`) has a specific structure that enables nodes to exchange data and maintain state throughout execution:

```typescript
const ctx: Context = {
    state: {},   // ✅ Every step's output is auto-stored here at ctx.state[<id>]
    request: {
        body: {},
        method: "GET",
        headers: {},
        query: {},
        params: {}
    }, // Incoming request details (read via $.req)
    response: {} // The PREVIOUS step's output (read via $.prev) — overwritten each step
};
```

### Breakdown of the Context Object

- **`ctx.state`**: The persistent store. Every step's output is automatically
  written to `ctx.state[<step-id>]` on success, and stays there for the rest of
  the run. **This is how you pass data between steps** — read it with
  `$.state.<id>`. You don't write it yourself; just return your output.

- **`ctx.request`** (alias `ctx.req`): The trigger request — `body`, `method`,
  `headers`, `query`, `params`. Read via `$.req.body`, `$.req.params.id`, etc.

- **`ctx.response`** (alias `ctx.prev`): The **immediately previous** step's
  output. Overwritten on every step — use `$.prev` for adjacent reads only.

<Note>
`ctx.vars` is a legacy v1 alias for `ctx.state` (still readable for back-compat).
New workflows use `ctx.state` / `$.state.<id>`.
</Note>

### Reading context inside a node

A node reads what it needs from its **validated `input`** (which the workflow
author wired from context via `$.state` / `$.req` expressions) and **returns** its
output. It does not touch `ctx.state`.

```typescript
import { defineNode } from "@blokjs/runner";
import { z } from "zod";

export default defineNode({
  name: "greet-user",
  input: z.object({ userId: z.string(), greeting: z.string() }),
  output: z.object({ message: z.string() }),
  async execute(ctx, input) {
    // `input` is already resolved + validated. The author mapped it from
    // context in the workflow, e.g. inputs: { userId: $.req.params.id,
    // greeting: $.state.config.greeting }.
    return { message: `${input.greeting}, ${input.userId}!` };
    // ↑ the runner stores this at ctx.state["greet-user"] automatically.
  },
});
```

<Warning>
**Never write `ctx.state` (or the legacy `ctx.vars`) inside a node.** Return your
output and let the runner persist it. If you genuinely need a side-channel value,
call `ctx.publish(name, value)` — but that's rare.
</Warning>

## Data flow and isolation

Each run gets its own isolated context — data from one workflow instance never
leaks into another. Within a run, the flow is always the same: a step **returns**
its output, the runner **persists** it to `ctx.state[<id>]`, and later steps
**read** it via `$.state.<id>`. Treat the trigger data (`ctx.request`) as
read-only input; produce new data by returning it from steps.

## Context in Workflows

Context is essential for **passing data between nodes** and making dynamic decisions in workflows.

### Example: using `ctx.request.params`

This workflow extracts `id` from the request URL and fetches a record:

```json
{
  "name": "Get Movie Details",
  "description": "Fetches a movie by ID",
  "version": "1.0.0",
  "trigger": { "http": { "method": "GET", "path": "/movies/:id" } },
  "steps": [
    {
      "id": "get-movie",
      "use": "mongo-query",
      "type": "local",
      "inputs": {
        "collection": "movies",
        "id": "$.req.params.id"
      }
    },
    {
      "id": "respond",
      "use": "@blokjs/respond",
      "inputs": { "body": "$.state.get-movie" }
    }
  ]
}
```

How it works:
- `path: "/movies/:id"` defines a dynamic route.
- The mongo node retrieves the movie using `$.req.params.id` — which
  resolves to whatever's at `:id` in the URL (e.g. `/movies/123` →
  `123`).
- The `respond` step returns the movie data because `$.state.get-movie`
  reads the previous step's output.

### Example: chaining steps via `$.state`

This workflow stores an API response and uses it in a later step:

```json
{
  "name": "Fetch and Process Data",
  "description": "Fetches and transforms an external response",
  "version": "1.0.0",
  "trigger": { "http": { "method": "GET", "path": "/fetch-data" } },
  "steps": [
    {
      "id": "fetch-data",
      "use": "@blokjs/api-call",
      "inputs": {
        "url": "https://api.publicapis.org/entries",
        "method": "GET"
      }
    },
    {
      "id": "process-data",
      "use": "@blokjs/mapper",
      "inputs": {
        "mapper": {
          "totalApis": "$.state.fetch-data.count",
          "categories": "$.state.fetch-data.categories"
        }
      }
    },
    {
      "id": "respond",
      "use": "@blokjs/respond",
      "inputs": { "body": "$.state.process-data" }
    }
  ]
}
```

How it works:
- Every step's output is automatically stored at `ctx.state[<id>]`. That's
  the v2 default. (The legacy `set_var` field was removed in v0.5.)
- `process-data` reads from `$.state.fetch-data.count` and
  `$.state.fetch-data.categories` directly.
- The `respond` step returns the transformed data.

For background on `$.state` vs `ctx.vars` (the v1 alias) and `$.req`
vs `ctx.request`, see [v2 step shapes](/d/reference/v2-step-shapes).

## Best practices

*   **Unique step ids**: every step `id` is a key in one flat per-workflow config
    map. Never reuse an id — including across mutually-exclusive `switch`/`branch`
    arms — or they collide silently. Use `as:` if two arms must write the same
    downstream state key.
*   **Return, don't write**: nodes produce data by returning it, never by mutating
    `ctx.state` / `ctx.vars`.
*   **Read the right source**: `$.state.<id>` for any earlier step (the default),
    `$.req.*` for the trigger input, `$.prev` only for the immediately previous
    step.
*   **Schema awareness**: define Zod `input`/`output` schemas — the runner
    validates the resolved inputs before your node runs.
*   **Security**: avoid storing raw sensitive data in step outputs; it lands in
    `ctx.state` and may surface in Studio traces.

## Key takeaways

- ✅ **A step returns its output; the runner stores it at `ctx.state[<id>]`.**
- ✅ **Read earlier outputs with `$.state.<id>` — the default for cross-step data.**
- ✅ **`$.req.body` / `$.req.params` / `$.req.query` read the trigger request.**
- ✅ **`$.prev` is the previous step only, and is overwritten each step.**
- ❌ **Never write `ctx.state`/`ctx.vars` inside a node, and never reuse a step id.**
