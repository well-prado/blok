# Build backends as workflows, not spaghetti

**Blok** is a backend framework where you compose small, typed, testable **nodes** into
declarative **workflows** — triggered by HTTP, queues, schedules, webhooks, and more. Every
step is observable, every node is unit-testable, and your business logic reads like a recipe.

```ts orders.ts
import { workflow, step, branch, http, gt } from "@blokjs/core";
import validateOrder from "../nodes/validate-order";
import route from "../nodes/route";

export default workflow("orders.create", { version: "1.0.0", trigger: http.post("/orders") }, (req) => {
  const order = step("validate", validateOrder, { qty: req.body.qty });

  branch("lane", gt(order.qty, 10), {
    then: () => { step("bulk", route, { lane: "bulk" }); },
    else: () => { step("standard", route, { lane: "standard" }); },
  });
});
```

That `order.qty` is a **typed handle** — a reference to the previous step's output that the
runner resolves for you. No `$`, no `js/`, no glue code. Just TypeScript that happens to be a
workflow.

## Why Blok

<Columns cols={3}>
  <Card title="Composable by design" icon="puzzle-piece">
    A node does one thing. Workflows wire nodes together. Logic stays small, named, and
    reusable — never tangled across controllers, services, and middleware.
  </Card>
  <Card title="Typed end to end" icon="shield-check">
    Zod-validated node inputs/outputs and typed handles mean a wrong field is a compile error,
    not a 2am pager. The [typed client](/d/client/overview) extends that safety to your frontend.
  </Card>
  <Card title="Observable out of the box" icon="chart-line">
    Every run is traced step-by-step in [Blok Studio](/d/studio/index) — inputs, outputs,
    errors, and timing — with zero instrumentation.
  </Card>
  <Card title="Nine triggers, one model" icon="bolt">
    HTTP, worker, cron, pub/sub, SSE, WebSocket, webhook, MCP, gRPC. The workflow body is
    identical — only the trigger block changes.
  </Card>
  <Card title="Eight runtimes" icon="layer-group">
    TypeScript runs in-process; Go, Rust, Java, C#, PHP, Ruby, and Python nodes run as gRPC
    sidecars. Mix languages in one workflow.
  </Card>
  <Card title="Reliable by default" icon="rotate">
    Idempotency, retries, timeouts, per-tenant concurrency, scheduling, and replay are
    [one field on a step](/d/reliability/overview) — not a separate platform.
  </Card>
</Columns>

## How it fits together

<Steps>
  <Step title="Write a node" icon="puzzle">
    A `defineNode` with a Zod input + output and an `execute` function. Pure, typed, testable.
  </Step>
  <Step title="Compose a workflow" icon="workflow">
    Wire nodes with `step()` and pass typed handles between them. Add control flow with
    `branch` / `forEach` / `switchOn` / `tryCatch`.
  </Step>
  <Step title="Pick a trigger" icon="bolt">
    `http.post("/orders")`, a `worker` queue, a `cron` schedule — the trigger decides *when* the
    workflow runs.
  </Step>
  <Step title="Run + observe" icon="chart-line">
    `blokctl dev` starts everything; open Blok Studio to watch each run trace through your steps.
  </Step>
</Steps>
