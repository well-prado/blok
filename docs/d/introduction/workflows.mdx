A workflow is a sequence of [nodes](/d/introduction/nodes) that
together accomplish a piece of business logic. Workflows define what
runs, in what order, and how data moves between steps. They're the
backbone of every Blok application.

## What is a workflow

Think of a workflow as a recipe:

- **Ingredients are nodes** — each node performs a small, specific
  action (fetch user data, send an email, update a record).
- **Instructions are the workflow definition** — which nodes to use,
  in what order, and how data flows.
- **The dish is the outcome** — an API response, a completed pipeline,
  whatever the workflow is meant to produce.

Workflows are kicked off by [triggers](/d/introduction/triggers): HTTP
requests, scheduled events, queue messages, and so on.

## Definition formats

Blok accepts workflow definitions in four formats. They are
structurally identical — the same shape rendered in different syntaxes.

| Format | Best for |
|---|---|
| **JSON** | Default, easiest for non-developers and tooling |
| **TypeScript** | Type safety, IDE autocomplete, dynamic composition |
| **TOML** | Teams that prefer TOML's syntax |
| **YAML** | Teams that prefer YAML's readability |

For the rest of this page, we'll show TypeScript and JSON side by side
since they're the two most common in practice.

## Create a workflow

The fastest way is the CLI:

```bash
npx blokctl@latest create workflow
```

That scaffolds a workflow file in `workflows/json/` (or your chosen
format). You can also write the file directly.

## v2 shape

Every v2 workflow has four top-level fields:

| Field | Type | Description |
|---|---|---|
| `name` | `string` | Unique identifier within the project (see [Naming convention](#naming-convention)). Other workflows reference this name when invoking it as a [sub-workflow](/d/composition/sub-workflows). |
| `version` | `string` | Semantic version (`"1.0.0"`). |
| `trigger` | `object` | What kicks this workflow off. See [triggers](/d/introduction/triggers). |
| `steps` | `array` | The ordered sequence of work. |

Each step in `steps[]` is one of three shapes — see
[v2 step shapes](/d/reference/v2-step-shapes) for the full reference.

## Example: simple HTTP workflow

<CodeGroup>
```ts TypeScript
import { workflow, $ } from "@blokjs/helper";

export default workflow({
  name: "countries.list",
  version: "1.0.0",
  description: "Fetches the list of countries with capitals",
  trigger: {
    http: { method: "GET", path: "/countries" },
  },
  steps: [
    {
      id: "fetch",
      use: "@blokjs/api-call",
      inputs: {
        url: "https://countriesnow.space/api/v0.1/countries/capital",
        method: "GET",
        headers: { "Content-Type": "application/json" },
      },
    },
    {
      id: "respond",
      use: "@blokjs/respond",
      inputs: { body: $.state.fetch },
    },
  ],
});
```

```json JSON
{
  "name": "countries.list",
  "version": "1.0.0",
  "description": "Fetches the list of countries with capitals",
  "trigger": {
    "http": { "method": "GET", "path": "/countries" }
  },
  "steps": [
    {
      "id": "fetch",
      "use": "@blokjs/api-call",
      "inputs": {
        "url": "https://countriesnow.space/api/v0.1/countries/capital",
        "method": "GET",
        "headers": { "Content-Type": "application/json" }
      }
    },
    {
      "id": "respond",
      "use": "@blokjs/respond",
      "inputs": { "body": "$.state.fetch" }
    }
  ]
}
```
</CodeGroup>

A few things to notice in the v2 shape:

- **Each step has `id` + `use` + `inputs` inline.** No separate
  `nodes: {}` map.
- **Cross-step references use `$.state.<id>`.** In TypeScript that's
  the typed `$` proxy; in JSON it's a plain string. Both compile to
  `js/ctx.state.<id>` at workflow load time.
- **Method is `"ANY"` (not `"*"`).** Use `"ANY"` for catch-all routes;
  the runner accepts `"*"` for back-compat with a deprecation warning.

## Naming convention

A workflow's `name` is its identity. Two rules matter:

- **Names must be unique** across the project. The
  [Workflow Registry](/d/reference/workflow-registry) rejects duplicate
  names at boot, so if two workflows share a name only one ever
  registers.
- **Prefer a dotted `domain.action` form** — `countries.list`,
  `users.create`, `orders.refund`. The
  [typed client](/d/client/overview) (`@blokjs/client`) and
  [`blokctl gen app-types`](/d/cli/gen) nest workflows by their dotted
  name, so a clean name surfaces as `blok.countries.list(...)` instead
  of a quoted `blok["World Countries"]` accessor.

Duplicate names also make [`blokctl gen app-types`](/d/cli/gen) report
a collision and drop one of the colliding workflows from the generated
`BlokApp` type — another reason to keep every name unique.

## Built-in nodes

Blok ships a small set of built-in nodes, registered in `src/Nodes.ts`:

```ts
import ApiCall from "@blokjs/api-call";
import IfElse from "@blokjs/if-else";
import type { NodeBase } from "@blokjs/shared";

const nodes: { [key: string]: NodeBase } = {
  "@blokjs/api-call": new ApiCall(),
  "@blokjs/if-else": new IfElse(),
};

export default nodes;
```

| Node | Use |
|---|---|
| `@blokjs/api-call` | HTTP request to an external API |
| `@blokjs/if-else` | Conditional branching (now usually written via the v2 `branch` step) |

Reference them by string in any step's `use` field.

## Workflow execution

When a trigger fires:

<Steps>
  <Step title="Identify">
    The runtime resolves the workflow that matches the trigger event.
  </Step>
  <Step title="Initialize">
    A new [Context](/d/introduction/context) is built — request data,
    environment, and an empty `state` map.
  </Step>
  <Step title="Run">
    Steps execute in order. Each step's output is auto-stored at
    `ctx.state[id]` (the v2 default-store rule).
  </Step>
  <Step title="Complete">
    The workflow finishes when it reaches the last step or hits a step
    with `stop: true`.
  </Step>
</Steps>

## File-based routing

If `BLOK_FILE_BASED_ROUTING=true`, the URL is derived from each
workflow's file path under `workflows/json/`:

| File path | URL |
|---|---|
| `workflows/json/health.json` | `/health` |
| `workflows/json/users/list.json` | `/users/list` |
| `workflows/json/users/index.json` | `/users` |
| `workflows/json/users/[id].json` | `/users/:id` |

Files starting with `_` or `.` are skipped (utilities, drafts).
Setting `trigger.http.path` explicitly overrides the file-derived URL.
