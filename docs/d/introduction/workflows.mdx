A workflow is a sequence of [nodes](/d/introduction/nodes) that together accomplish a piece
of business logic. You author workflows in **TypeScript** with the typed-handle DSL from
`@blokjs/core`: each `step()` returns a **handle** — a typed reference to that step's output —
and you wire steps together by passing handles around. No `$`, no `js/`, no raw `ctx` strings.

## Your first workflow

```ts workflows/countries.ts
import { workflow, step, http } from "@blokjs/core";
import apiCall from "@blokjs/api-call";
import { RespondNode } from "@blokjs/helpers";

export default workflow("countries.list", { version: "1.0.0", trigger: http.get("/countries") }, (req) => {
  // step(id, node, inputs) runs a node and returns a TYPED handle.
  const countries = step("fetch", apiCall, {
    url: "https://countriesnow.space/api/v0.1/countries/capital",
    method: "GET",
  });

  // Pass the handle straight into the next step — the runner resolves it.
  step("respond", RespondNode, { body: countries }, { ephemeral: true });
});
```

<Note>
`step()` takes a node **object** (not a string ref). Import published nodes directly — a
standalone package's default export (`apiCall`) or a named export from `@blokjs/helpers`
(`RespondNode`, `ExprNode`, `LogNode`, …). Your own `defineNode` nodes are imported from
`src/nodes/...`. For a cross-runtime node, build a stub with
`runtimeNode("<name>", "runtime.<lang>")`.
</Note>

Run `blokctl dev` and hit `GET /countries` — you'll get the fetched JSON back. That's a
complete workflow: a trigger, two steps, and a typed reference (`countries`) flowing between
them.

<Note>
`workflow(name, opts, build)` takes a name, an options object (`version` + `trigger`), and a
**build callback**. The callback's argument is the trigger's *entry handle* — for HTTP it's
the request, conventionally named `req` (`req.body`, `req.params.id`, `req.query`, `req.headers`).
</Note>

## How data flows: handles

`const countries = step("fetch", apiCall, { ... })` does two things:

1. **Runs the node** and auto-persists its output to `ctx.state["fetch"]` — *on success only*
   (a step that throws writes nothing).
2. **Returns a handle** shaped like the node's output. Reading a field (`countries.data`,
   `countries[0].name`) records a typed reference the runner resolves *before the next node
   runs* — you never read the value yourself.

```ts
const order = step("validate", validateOrder, { qty: req.body.qty });

order.qty          // Handle<number> → ref to ctx.state.validate.qty
order.user.email   // nested field access, fully typed
order.items[0]     // array index (but not .map / .length — do that in a node)
```

To embed a handle in a string, use the `tpl` tagged template — a bare template literal would
lose the reference:

```ts
import { tpl } from "@blokjs/core";
step("summary", summarize, { line: tpl`order of ${order.qty} item(s)` });
```

## Triggers

A workflow declares exactly **one** trigger. HTTP uses the `http.*` helper; every other
trigger is a raw block passed as `opts.trigger`:

```ts
trigger: http.post("/orders")                       // HTTP
trigger: { worker: { queue: "emails", retries: 3 } } // background job
trigger: { cron: { schedule: "0 8 * * *" } }         // scheduled
trigger: { webhook: { provider: "stripe" } }         // signed webhook
```

The entry handle is named + typed per trigger: `http → req`, `worker → job`, `cron → tick`,
`webhook → event`, `pubsub → msg`, `grpc → rpc`. See [Triggers](/d/introduction/triggers) for
the full decision table and config of all nine.

## Persistence knobs

The 4th argument to `step()` shapes how its output is stored:

| Knob | Effect |
|---|---|
| *(none)* | Store at `ctx.state[id]` — the 95% case. |
| `{ as: "name" }` | Store at `ctx.state[name]` instead. Mutually exclusive with `spread`. |
| `{ spread: true }` | Shallow-merge the output's keys into `ctx.state` (multi-output nodes). |
| `{ ephemeral: true }` | Skip storage — the handle is unreadable downstream. Use for response/log steps. |

The same 4th argument carries per-step reliability (`idempotencyKey`, `retry`, `maxDuration`)
— see [Reliability](/d/reliability/overview).

## Control flow

Branch, loop, and handle errors with the primitives — each takes callback arms where `step()`
calls register into that arm:

```ts
import { branch, gt } from "@blokjs/core";

branch("lane", gt(order.qty, 10), {
  then: () => { step("bulk", routeOrder, { lane: "bulk" }); },
  else: () => { step("standard", routeOrder, { lane: "standard" }); },
});
```

</Warning>

## Built-in & local nodes

Reference a published node by importing it (`import apiCall from "@blokjs/api-call"`). Your own
nodes live at `src/nodes/<name>/index.ts` and are **auto-discovered** by their `defineNode`
`name` — you never hand-edit a registry to add one:

```ts src/nodes/greet/index.ts
import { defineNode } from "@blokjs/core";
import { z } from "zod";

export default defineNode({
  name: "greet",
  description: "Greets a name",
  input: z.object({ name: z.string().default("world") }),
  output: z.object({ message: z.string() }),
  async execute(_ctx, input) {
    return { message: `Hello, ${input.name}!` };
  },
});
```

| Built-in node | Use |
|---|---|
| `@blokjs/api-call` | HTTP request to an external API. |
| `@blokjs/respond` | Control the HTTP response (status, headers, cookies, binary). |
| `@blokjs/if-else` | Conditional routing (usually written via the `branch` primitive instead). |

See [Nodes](/d/introduction/nodes) to write your own.

## Naming convention

A workflow's `name` is its identity:

- **Names must be unique** across the project — the
  [Workflow Registry](/d/reference/workflow-registry) rejects duplicates at boot.
- **Prefer a dotted `domain.action` form** — `countries.list`, `users.create`,
  `orders.refund`. The [typed client](/d/client/overview) and
  [`blokctl gen app-types`](/d/cli/gen) nest workflows by their dotted name, so a clean name
  surfaces as `blok.countries.list(...)` instead of a quoted accessor.

## Execution

<Steps>
  <Step title="Identify">The runtime resolves the workflow matching the trigger event.</Step>
  <Step title="Initialize">A fresh [Context](/d/fundamentals/context-and-state) is built — request data, env, and an empty `state` map.</Step>
  <Step title="Run">Steps execute in order; each output auto-stores at `ctx.state[id]` and becomes readable through its handle.</Step>
  <Step title="Complete">The workflow finishes at the last step (or a step with `stop: true`).</Step>
</Steps>

## JSON workflows

The handle DSL compiles to the same IR as **JSON workflows** — a first-class format, ideal for
config-driven or generated workflows. The shape mirrors the DSL one-for-one: cross-step
references are `$.state.<id>` strings (which compile to `js/ctx.state.<id>`).

```json
{
  "name": "countries.list",
  "version": "1.0.0",
  "trigger": { "http": { "method": "GET", "path": "/countries" } },
  "steps": [
    { "id": "fetch", "use": "@blokjs/api-call",
      "inputs": { "url": "https://countriesnow.space/api/v0.1/countries/capital", "method": "GET" } },
    { "id": "respond", "use": "@blokjs/respond", "inputs": { "body": "$.state.fetch" }, "ephemeral": true }
  ]
}
```

Use `"ANY"` for the catch-all HTTP method, never `"*"`. JSON workflows live under
`workflows/json/` and support [file-based routing](#file-based-routing).

## File-based routing

With `BLOK_FILE_BASED_ROUTING=true`, JSON workflow URLs derive from the file path under
`workflows/json/`:

| File path | URL |
|---|---|
| `workflows/json/health.json` | `/health` |
| `workflows/json/users/list.json` | `/users/list` |
| `workflows/json/users/index.json` | `/users` |
| `workflows/json/users/[id].json` | `/users/:id` |

Files starting with `_` or `.` are skipped. An explicit `trigger.http.path` overrides the
file-derived URL.
