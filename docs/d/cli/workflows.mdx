Workflows define how Blok composes nodes into business logic. The
`blokctl` CLI scaffolds new v2 workflow files for you to customize.

## `create workflow`

Generates a new workflow definition file in your project.

```bash
npx blokctl@latest create workflow
```

You'll be prompted for a workflow name. The CLI writes a v2 JSON file
under `workflows/json/` (or another supported format).

<Tip>
Names must be unique across the project. Use a dotted `domain.action`
form (`cats.facts`, `users.create`) so the typed client and
`blokctl gen app-types` expose it as `blok.cats.facts(...)`. See the
[naming convention](/d/introduction/workflows#naming-convention).
</Tip>

## v2 workflow structure

A v2 workflow has three sections:

| Section | Purpose |
|---|---|
| `trigger` | What kicks the workflow off (HTTP method/path, worker queue, etc.). |
| `steps` | Ordered list of work. Each step has `id`, `use`, and inline `inputs`. |
| `name`, `version`, `description` | Top-level metadata. |

There's no separate `nodes: {}` map — that's a v1 leftover. v2 puts
inputs directly on the step.

## Example: fetch cat facts

Save this as `workflows/json/cats.json`:

```json
{
  "name": "cats.facts",
  "description": "Fetches random cat facts from a public API.",
  "version": "1.0.0",
  "trigger": {
    "http": { "method": "GET", "path": "/cats" }
  },
  "steps": [
    {
      "id": "fetch",
      "use": "@blokjs/api-call",
      "inputs": {
        "url": "https://catfact.ninja/fact",
        "method": "GET"
      }
    },
    {
      "id": "respond",
      "use": "@blokjs/respond",
      "inputs": { "body": "$.state.fetch" }
    }
  ]
}
```

## Trigger section

```json
"trigger": {
  "http": { "method": "GET", "path": "/cats" }
}
```

| Field | Description |
|---|---|
| `method` | HTTP method (`GET`, `POST`, `PUT`, `DELETE`, `PATCH`, `HEAD`, `OPTIONS`) or `"ANY"` to match every method. |
| `path` | URL path. Supports parameters: `/:id` (required), `/:id?` (optional). Optional when file-based routing is enabled. |

<Note>
Use `"ANY"` for the wildcard, not `"*"`. The runner accepts `"*"` for
back-compat with a deprecation warning.
</Note>

## Steps section

Every step in v2 has these fields:

| Field | Required | Description |
|---|---|---|
| `id` | yes | Stable identifier. Other steps reference output as `$.state[id]`. |
| `use` | yes | Node reference (e.g. `@blokjs/api-call`, `my-custom-node`). |
| `type` | no | `"module"` or `"local"` — inferred from `use` when omitted. |
| `inputs` | no | Inputs passed to the node. May contain `$`-proxy refs. |

`module` is for nodes installed as dependencies or class-based nodes
exported by `src/Nodes.ts`. `local` is for nodes living in your local
filesystem.

## Cross-step references

Every step's output auto-stores at `ctx.state[id]`. Reference it in a
later step's inputs as `"$.state.<id>"`:

```json
{
  "id": "respond",
  "use": "@blokjs/respond",
  "inputs": { "body": "$.state.fetch" }
}
```

For the full set of fields and persistence knobs (`as`, `spread`,
`ephemeral`, `idempotencyKey`, `retry`, `subworkflow`), see
[v2 step shapes](/d/reference/v2-step-shapes).

## Run and test

<Steps>
  <Step title="Start the project">
    ```sh
    bun run dev
    ```
  </Step>
  <Step title="Hit the endpoint">
    ```sh
    curl http://localhost:4000/cats
    ```
    The response will contain a random cat fact.
  </Step>
  <Step title="Open Studio">
    ```sh
    blokctl trace
    ```
    Open the run detail page to see each step's inputs, outputs, and
    timing.
  </Step>
</Steps>
