Blok runs your workflow steps across **eight language runtimes**. Node/TypeScript runs **inвЂ‘process** (it's always there, nothing to install). The other seven вЂ” **Go, Rust, Java, C#, PHP, Ruby, and Python 3** вЂ” run as lightweight **gRPC sidecars** that the runner talks to.

You choose which runtimes you want when you scaffold a project (`create project --runtimes go,python3`). But you don't have to decide everything up front: the **`blokctl runtime`** commands let you **add or remove a language at any time**, in place, on a project that already exists.

<Note>
**Node / TypeScript is always available.** It runs in the same process as your triggers, so there's nothing to add or remove. The `runtime` commands manage the seven *sidecar* languages: `go`, `rust`, `java`, `csharp`, `php`, `ruby`, `python3`.
</Note>

## The commands at a glance

| Command | What it does |
| --- | --- |
| `blokctl runtime add <lang>` | Install a language sidecar into this project |
| `blokctl runtime remove <lang>` | Remove a language sidecar from this project |
| `blokctl runtime list` | Show what's installed and what you can add |

Run them from your project's root (or point at it with `-d <path>`). `remove` has the alias `rm`, and `list` has `ls`.

<Tip>
Not sure where to start? Run `blokctl runtime list` to see what's installed and which languages your machine already has the toolchain for вЂ” then `blokctl runtime add` (with no argument) to pick one from an interactive menu.
</Tip>

## `runtime add` вЂ” add a language

This is the command you'll reach for when you scaffolded a project with, say, just Node and later realize you want to write a step in Python or Go.

```bash
# Add a specific runtime
blokctl runtime add python3

# вЂ¦or omit the name to pick one from an interactive menu
blokctl runtime add
```

<Steps>
  <Step title="Checks the toolchain">
    Confirms the language is installed (e.g. `python3`, or `java` **and** `mvn`). If it isn't, you get the install link вЂ” or pass `--skip-toolchain-check` to scaffold anyway.
  </Step>
  <Step title="Fetches a versionвЂ‘matched SDK">
    Downloads the SDK source for **your project's Blok version**, so the gRPC contract always matches your `@blokjs/runner`. (Cached under `~/.blok`, so the next add is instant. Use `--local <path>` to point at a checkout instead.)
  </Step>
  <Step title="Builds it">
    Runs the language's install/build (`pip install`, `go mod download`, `cargo build`, вЂ¦). Nothing is written to your project until this succeeds, so a failed build never leaves you halfвЂ‘configured.
  </Step>
  <Step title="Wires it in">
    Merges the runtime into `.blok/config.json`, `.env.local`, and `supervisord.conf`, and creates `runtimes/<lang>/nodes/` for your code. Your triggers and any other runtimes are left untouched.
  </Step>
</Steps>

After a successful add you'll see exactly what changed:


<Note>
**No application code changes are needed.** The runner already knows how to dispatch to all seven languages вЂ” adding one is purely configuration plus the SDK files. You never touch `@blokjs/runner` or your trigger code.
</Note>

### Use it in a workflow

A step runs on a sidecar when you reference a node by name and tag it with its
runtime. Write your node under `runtimes/<lang>/nodes/`, then declare the step:

```json JSON workflow
{
  "id": "resize",
  "use": "image-resizer",
  "type": "runtime.python3",
  "inputs": { "image": "$.req.body.image" }
}
```

<Note>
**`runtimeNode` is the typed TS form for a cross-runtime step.** `@blokjs/core`
exports `runtimeNode<In, Out>(name, "runtime.<lang>")` as a real value (issue
#424) вЂ” it gives you typed inputs/outputs for a cross-runtime node, and `step()`
lowers it to the same `{ use, type: "runtime.<lang>" }` step the JSON form above
produces, so the runner routes it through the gRPC runtime adapter. The
`blokctl nodes sync` stubs are these `runtimeNode(...)` values; import and use
them directly:

```ts
import { workflow, step, runtimeNode, http } from "@blokjs/core";

// typed cross-runtime stub вЂ” executes via the runtime adapter
const resizeImage = runtimeNode<{ image: string }, { url: string }>(
  "image-resizer",
  "runtime.python3",
);

export default workflow("Resize Image", { version: "1.0.0", trigger: http.post("/resize") }, (req) => {
  step("resize", resizeImage, { image: req.body.image });
});
```

The second argument is the runtime kind (`"runtime.python3"`); the
`blokctl nodes sync` stubs pass the catalog ref (`"runtime.python3:image-resizer"`)
in the same slot вЂ” `step()` accepts either and derives the kind from it.
</Note>

Then start everything together:

```bash
blokctl dev
```

`blokctl dev` reads `.blok/config.json`, spawns each sidecar, waits for its gRPC port to come up, and then starts your triggers вЂ” exactly as if you'd scaffolded the runtime at project creation.

### Options

| Flag | Purpose |
| --- | --- |
| `-d, --directory <path>` | Project directory (default: current directory) |
| `--local <path>` | Use a local Blok checkout for the SDK source instead of fetching by version |
| `--grpc-port <port>` | Use a nonвЂ‘default gRPC port for this runtime |
| `--force` | Reinstall even if the runtime is already present (does a clean reвЂ‘copy) |
| `--skip-toolchain-check` | Add even if the language toolchain isn't detected |
| `-y, --yes` | Skip prompts (nonвЂ‘interactive вЂ” for CI/scripts) |

<Warning>
**Toolchain not installed?** `add` stops with the exact tool you're missing and a link to install it вЂ” for example, *"Java toolchain not detected (need **Maven**). Install Maven: вЂ¦"*. Install it and reвЂ‘run, or pass `--skip-toolchain-check` if you'll set it up later.
</Warning>

## `runtime remove` вЂ” remove a language

Cleanly undoes everything `add` did вЂ” and it's careful with your code.

```bash
blokctl runtime remove go
```

<Steps>
  <Step title="Warns you about dependent workflows">
    First it scans your project for steps that use `runtime.<lang>` and lists the files. Those steps will fail once the sidecar is gone, so you know exactly what to update.
  </Step>
  <Step title="Asks before deleting your nodes">
    Your runtime nodes in `runtimes/<lang>/nodes/` are **kept by default**. `remove` asks before deleting them; pass `--purge-nodes` to delete them without a prompt.
  </Step>
  <Step title="Undoes the wiring">
    Removes the entry from `.blok/config.json`, the `RUNTIME_<LANG>_*` vars from `.env.local`, the `supervisord.conf` block, and the `.blok/runtimes/<lang>/` SDK directory.
  </Step>
</Steps>

<Warning>
Any workflow step with `type: "runtime.<lang>"` will fail at run time after you remove that language. Update or delete those steps вЂ” or just `blokctl runtime add <lang>` to put it back. Removal is fully reversible.
</Warning>

### Options

| Flag | Purpose |
| --- | --- |
| `-d, --directory <path>` | Project directory (default: current directory) |
| `--purge-nodes` | Also delete your custom nodes in `runtimes/<lang>/nodes/` |
| `-y, --yes` | Skip prompts (keeps your custom nodes) |

## How runtimes are wired (under the hood)

If you're curious what these commands actually change, a runtime lives in **four places** вЂ” and that's the whole story. There's no hidden state, and nothing in your application code:

| Where | What's there |
| --- | --- |
| `.blok/config.json` в†’ `runtimes.<lang>` | The runtime's command, ports, version, and working directory |
| `.env.local` в†’ `RUNTIME_<LANG>_*` | Host + gRPC port the runner connects to |
| `supervisord.conf` в†’ `[program:<lang>_runtime]` | How the sidecar boots in a container deploy |
| `.blok/runtimes/<lang>/` + `runtimes/<lang>/nodes/` | The SDK server (managed) + **your** runtime nodes (yours) |

Each language has a fixed default gRPC port:

| Runtime | gRPC port |
| --- | --- |
| Go | `10001` |
| Rust | `10002` |
| Java | `10003` |
| C# / .NET | `10004` |
| PHP | `10005` |
| Ruby | `10006` |
| Python 3 | `10007` |

Because it's all configuration, a runtime added later is **indistinguishable** from one you picked at `create project` вЂ” same files, same shape, started the same way by `blokctl dev`.

## FAQ

<AccordionGroup>
  <Accordion title="I scaffolded my project with only Node. How do I add Go?">
    `cd` into the project and run `blokctl runtime add go` (you'll need the Go toolchain installed). That's it вЂ” write your Go nodes in `runtimes/go/nodes/`, reference them with `type: "runtime.go"` steps, and `blokctl dev` will start the Go sidecar alongside your triggers.
  </Accordion>
  <Accordion title="Will removing a runtime delete my code?">
    No вЂ” not unless you ask. Your nodes in `runtimes/<lang>/nodes/` are kept by default; `remove` prompts before touching them and only deletes them if you confirm (or pass `--purge-nodes`). It does delete the managed SDK directory under `.blok/runtimes/<lang>/`, which is regenerated by `add`.
  </Accordion>
  <Accordion title="The gRPC port is already in use.">
    `add` refuses if the runtime's default port is taken (by another runtime in the project, or a live process on your machine) and tells you to pass `--grpc-port <n>`. The override is threaded everywhere consistently вЂ” config, `.env.local`, and the boot command.
  </Accordion>
  <Accordion title="Does this match `create project --runtimes`?">
    Yes, exactly. `runtime add` reuses the same setup code and writes the same configuration, so a runtime added later behaves identically to one scaffolded at creation. `blokctl dev` and the runner can't tell the difference.
  </Accordion>
  <Accordion title="What if my project is on an unreleased / local Blok version?">
    `add` fetches the SDK that matches your project's pinned `@blokjs/runner` version, so the gRPC contract always lines up. If that version isn't published (e.g. you're developing Blok itself), point at a local checkout with `--local /path/to/blok`.
  </Accordion>
</AccordionGroup>
