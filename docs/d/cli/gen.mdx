`blokctl gen app-types` generates the `BlokApp` type — a types-only
`blok-app.d.ts` that indexes every TypeScript workflow by its dotted name. The
[typed client](/d/client/overview) imports it to infer every call.

## `gen app-types`

```bash
blokctl gen app-types [--dir <path>] [--out <file>] [--dry-run]
```

| Flag | Description |
|---|---|
| `-d, --dir <path>` | Directory of TypeScript workflow files to scan. Defaults to the first of `triggers/http/src/workflows`, `src/workflows`, `workflows/ts`, or `workflows` that exists. |
| `-o, --out <file>` | Output file. Defaults to `./blok-app.d.ts`. |
| `--dry-run` | Print the generated file instead of writing it. |

## What it scans

The generator reads each TypeScript workflow file and extracts its `name`
statically — it does not run your code. Only workflows with a literal `name:`
string are included.

JSON-authored workflows are skipped: there is no TypeScript type to import. A
codegen path for JSON, non-TypeScript, and cross-repo clients is
[planned](/d/client/overview#what-ships-today).

## The generated file

For workflows named `users.list`, `users.create`, and `health`:

```ts blok-app.d.ts
// AUTO-GENERATED by `blokctl gen app-types`. Do not edit by hand.
// Regenerate after adding, removing, or renaming a workflow.
// Consumed by `@blokjs/client`: `createBlokClient<BlokApp>()`.

import type usersList from "./triggers/http/src/workflows/users/list";
import type usersCreate from "./triggers/http/src/workflows/users/create";
import type health from "./triggers/http/src/workflows/health";

export type BlokApp = {
  users: {
    list: usersList;
    create: usersCreate;
  };
  health: health;
};
```

The dotted name drives the nesting: `users.list` becomes `BlokApp.users.list`;
a plain `health` stays at the top level.

## Name collisions

Names must be unique — see the
[naming convention](/d/introduction/workflows#naming-convention). When two
workflows resolve to the same path, or a name is both a leaf and a group (for
example `users` and `users.list`), the generator drops the conflicting one and
warns:

```
⚠️  name collision — "users" overlaps another workflow's path and was dropped.
```

Generation still completes, but the dropped workflow is missing from `BlokApp`,
so the client can't reach it. Rename one of the workflows and regenerate.

## When to run it

The type is not regenerated automatically. Run it whenever you add, remove, or
rename a workflow — and wire it into your prebuild so a committed `blok-app.d.ts`
never goes stale:

```json package.json
{
  "scripts": {
    "prebuild": "blokctl gen app-types"
  }
}
```

A stale `BlokApp` is a type-versus-runtime mismatch: the client's types won't
match what the server actually runs.
