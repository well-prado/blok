This walks through the full path: declare a workflow's input and output, generate
the `BlokApp` type, install the client, and call the workflow with end-to-end
type safety.

## Prerequisites

- A Blok project with the [HTTP trigger](/d/triggers/http) running
- A TypeScript frontend that can import types from the server (a monorepo is the
  common case)

<Steps>
  <Step title="Declare a typed workflow">
    Add `input` and `output` Zod schemas to a workflow. They are the typed
    contract — `input` types the call, `output` types the return.

    ```ts triggers/http/src/workflows/users/list.ts
    import { workflow, $ } from "@blokjs/helper";
    import { z } from "zod";

    const User = z.object({ id: z.string(), name: z.string() });

    export default workflow({
      name: "users.list",
      version: "1.0.0",
      description: "List users, optionally filtered by a query.",
      input: z.object({ q: z.string().optional() }),
      output: z.object({ users: z.array(User), total: z.number().int() }),
      trigger: { http: { method: "GET", path: "/api/users" } },
      steps: [
        {
          id: "fetch",
          use: "@blokjs/api-call",
          inputs: { url: "https://example.com/api/users", method: "GET" },
        },
        {
          id: "out",
          use: "@blokjs/respond",
          inputs: { body: { users: $.state.fetch.data, total: 0 } },
          ephemeral: true,
        },
      ],
    });
    ```

    The terminal output must match the declared `output` shape. The name uses
    the [dotted convention](/d/introduction/workflows#naming-convention) so it
    becomes `blok.users.list`.
  </Step>

  <Step title="Generate the BlokApp type">
    Run the generator from your project root. It scans your TypeScript workflow
    files and writes a types-only `blok-app.d.ts`.

    ```bash
    npx blokctl gen app-types
    ```

    The generated file points at your workflow files — it carries no new
    information:

    ```ts blok-app.d.ts
    // AUTO-GENERATED by `blokctl gen app-types`. Do not edit by hand.
    // Regenerate after adding, removing, or renaming a workflow.
    // Consumed by `@blokjs/client`: `createBlokClient<BlokApp>()`.

    import type usersList from "./triggers/http/src/workflows/users/list";

    export type BlokApp = {
      users: { list: usersList };
    };
    ```

    Commit this file, or regenerate it as a prebuild step. See
    [`blokctl gen app-types`](/d/cli/gen) for flags and defaults.
  </Step>

  <Step title="Install the client in your frontend">
    ```bash
    npm install @blokjs/client
    ```
  </Step>

  <Step title="Create the client and call the workflow">
    Pass the generated `BlokApp` as the type argument. The returned object's
    shape is inferred entirely from it.

    ```ts
    import { createBlokClient } from "@blokjs/client";
    import type { BlokApp } from "../server/blok-app";   // types only, erased at build

    const blok = createBlokClient<BlokApp>({
      baseUrl: "https://api.example.com",
    });

    const { users, total } = await blok.users.list({ q: "ada" });
    //      ^ string[]-typed    ^ number
    ```

    `blok.users.list` is `(input: { q?: string }) => Promise<{ users: User[]; total: number }>`.
    A wrong field name, a missing required input, or reading a property that
    isn't in `output` is a compile error.
  </Step>
</Steps>

## What just happened

The client POSTed your input to `https://api.example.com/__blok/rpc/users.list`,
the HTTP trigger looked the workflow up by name and ran it through the normal
runner (middleware included), and the terminal output came back as JSON — typed
the whole way because `BlokApp` carried `users.list`'s `input` and `output`
types into the call.
