Set `debounce` on a trigger and rapid same-key dispatches collapse
into one run instead of N. Trailing mode waits for silence; leading
mode fires once and suppresses follow-ups. The pattern matches Lodash
`_.debounce` semantics with a few additions for tail-latency
guarantees.

## Quickstart â€” document autosave

A document save endpoint that collapses rapid keystrokes into one save
500ms after the user stops typing, but no later than 5s after the
first keystroke:

<CodeGroup>
```ts TypeScript
import { workflow, step, http } from "@blokjs/core";
import postgresWrite from "@blokjs/postgres-write";

export default workflow(
  "save-doc",
  {
    version: "1.0.0",
    trigger: http.post("/save/:docId", {
      debounce: {
        key: "$.req.params.docId",
        mode: "trailing",
        delay: "500ms",
        maxDelay: "5s",
      },
    }),
  },
  (req) => {
    step("save", postgresWrite, { row: req.body });
  },
);
```

```json JSON
{
  "name": "save-doc",
  "version": "1.0.0",
  "trigger": {
    "http": {
      "method": "POST",
      "path": "/save/:docId",
      "debounce": {
        "key": "$.req.params.docId",
        "mode": "trailing",
        "delay": "500ms",
        "maxDelay": "5s"
      }
    }
  },
  "steps": [
    { "id": "save", "use": "@blokjs/postgres-write", "inputs": { "row": "$.req.body" } }
  ]
}
```
</CodeGroup>

The first POST creates a `delayed` run with `pingCount: 1`. Each
subsequent POST within 500ms returns 202 Accepted with `intoRunId`
pointing at the active run, and bumps the active run's `pingCount`.
The save fires once when 500ms of silence pass, OR when 5s have
elapsed since the first ping (whichever comes first).

## Field reference

Defined in [core/workflow-helper/src/types/TriggerOpts.ts:128-163](https://github.com/deskree-inc/blok/blob/main/core/workflow-helper/src/types/TriggerOpts.ts#L128-L163).

<ParamField path="key" type="string" required>
  Debounce key â€” a literal string or a `$.<path>` path string the
  runner resolves against the live `ctx`. Pings sharing the resolved key
  collapse. Typically derived from the request payload
  (`"$.req.params.docId"`, `"$.req.body.userId"`). It's a config string,
  not a handle â€” the debounce gate runs at run-entry, before the
  workflow body.
</ParamField>

<ParamField path="mode" type='"leading" | "trailing"' default='"trailing"'>
  `trailing` (default) waits for `delay` ms of silence then fires once
  with the latest payload. `leading` fires immediately and suppresses
  follow-ups within `delay` ms.
</ParamField>

<ParamField path="delay" type="number ms or duration string" required>
  Debounce window. Number (ms) or
  [duration string](/d/scheduling/overview#duration-grammar)
  (`"500ms"`, `"5s"`, `"2m"`). In trailing mode, this is the silence
  window. In leading mode, this is the suppression window after the
  first fire.
</ParamField>

<ParamField path="maxDelay" type="number ms or duration string">
  Force a fire after this many ms even if pings keep coming. Bounds
  tail latency. Must be `>= delay` when set. Ignored in leading mode
  (the run already fired).
</ParamField>

## Modes

### Trailing (default) â€” wait for silence

Each ping resets a `delay` ms timer. The run fires after `delay` ms
of silence. The first ping creates a `debounced` run (transient);
subsequent pings within the window are `debounced` terminal with
`intoRunId` pointing at the active run.

```
T=0ms    POST â†’ run_a created, pingCount=1, status="debounced", timer=500ms
T=200ms  POST â†’ run_b created, status="debounced" terminal, intoRunId=run_a, run_a.pingCount=2, timer reset to 700ms
T=400ms  POST â†’ run_c created, status="debounced" terminal, intoRunId=run_a, run_a.pingCount=3, timer reset to 900ms
T=900ms  silence â†’ timer fires â†’ run_a flips to "running", executes the save, completes
```

### Trailing with `maxDelay` â€” bounded tail latency

Continuous pings extend the silence window indefinitely. `maxDelay`
caps the wait â€” the active run fires at `firstPingAt + maxDelay`
regardless of subsequent pings.

```
T=0ms     POST â†’ run_a created, timer=500ms, maxDelayDeadline=5000ms
T=400ms   POST â†’ timer reset to 900ms
T=900ms   POST â†’ timer reset to 1400ms
... continuous pings ...
T=5000ms  maxDelay hit â†’ run_a fires regardless of pings still arriving
```

This is the right pattern for autosave: collapse keystrokes for typing
sessions (trailing), but guarantee a save within 5s even if the user
keeps typing (`maxDelay`).

### Leading â€” fire then suppress

The first ping fires synchronously through the normal pipeline.
Subsequent pings within `delay` ms are dropped with `debounced`
terminal status. The window closes after `delay` ms of silence.

```
T=0ms    POST â†’ run_a fires immediately, completes at T=20ms
T=100ms  POST â†’ run_b created, "debounced" terminal, intoRunId=run_a
T=300ms  POST â†’ run_c created, "debounced" terminal, intoRunId=run_a
T=500ms  silence â†’ window closes
T=600ms  POST â†’ run_d fires immediately (new window)
```

Useful for click suppression: protect a like button or favorite
endpoint from double-firing on rapid clicks.

```ts
trigger: http.post("/like/:postId", {
  debounce: {
    key: "$.req.params.postId",
    mode: "leading",
    delay: "1s",
  },
}),
```

## One run record per ping

Each ping creates its own run record via `tracker.startRun()`. The
trade-off: a 1000-ping burst writes 1000 rows (1 active + 999
coalesce-loser terminals). Storage grows linearly with ping volume.

The `evictOldRuns` cap on the runs table bounds total storage; tune
`BLOK_RUNS_CAP` (or the equivalent config) if you expect heavy
debounce traffic.

`pingCount` on the active run reflects the number of pings absorbed.
Studio renders it on the run row as **Pings: N**.

## Behavior

The debounce gate runs first, before [delay](/d/scheduling/delay-and-ttl)
and [concurrency](/d/reliability/concurrency-keys). It owns its own
scheduling â€” a debounce config effectively shadows any `delay` on the
same trigger.

<Steps>
  <Step title="Resolve key">
    The runner evaluates `debounce.key` against the live `ctx`. Failure
    to resolve (throw or `null`) is fail-open â€” the gate is skipped
    and the run proceeds synchronously.
  </Step>
  <Step title="Register on coordinator">
    `DebounceCoordinator.register(...)` returns one of three outcomes:
    `fire-immediate` (leading + fresh window), `schedule-trailing`
    (trailing + fresh window), or `coalesce` (existing window).
  </Step>
  <Step title="On fire-immediate â€” run synchronously">
    Caller proceeds through the normal pipeline. The window now
    suppresses follow-ups for `delay` ms.
  </Step>
  <Step title="On schedule-trailing â€” defer">
    Mark run `debounced` (transient), register the timer, throw
    `DeferredDispatchSignal`. HTTP returns 202; worker ACKs.
  </Step>
  <Step title="On coalesce â€” return 202 with `intoRunId`">
    Mark THIS ping's run `debounced` (terminal) with `intoRunId` set
    to the active run. Bump the active run's `pingCount` via
    `tracker.recordDebouncePing` (no event emitted to avoid timeline
    bloat under burst). HTTP returns 202; worker ACKs.
  </Step>
  <Step title="Fire â€” when the timer expires">
    `DebounceCoordinator.onFire` calls `dispatchDeferred` on the
    active run. Status flips `debounced â†’ running` and the steps
    execute.
  </Step>
</Steps>

## Durability

Debounce coalescing crosses two state planes: the silence-window
timer (in-memory) and the persisted dispatch row (sqlite-backed when
the trigger overrides `extractDispatchPayload`).

| State | Layer | Crash behavior |
|---|---|---|
| Active run record + `pingCount` + latest payload | sqlite `workflow_runs` (migration v8) | Survives restart |
| Silence-window timer (`setTimeout`) | in-memory `DebounceCoordinator` | Lost â€” recovered timers fire immediately on boot instead of waiting for additional silence |
| Persisted dispatch row (when opted in) | sqlite `scheduled_dispatches` (migration v9) | Survives restart |

The HTTP trigger overrides `extractDispatchPayload` by default, so
debounce dispatches go through the durability layer described on the
[delay & TTL page](/d/scheduling/delay-and-ttl#durability). On boot,
`HttpTrigger.recoverDispatches()` re-registers timers for any
`debounced` rows it finds:

- **Past TTL** â†’ marked `expired`, row deleted.
- **Active dispatch** â†’ re-registered to fire on next tick (silence-
  window state is gone; the recovered timer doesn't wait for fresh
  silence). New live pings post-restart open a fresh debounce window
  using the in-memory coordinator as usual.

Latest-payload-wins still applies â€” coalesce upserts the row keyed on
the active run id with the new `scheduledAt` and new payload. The
recovered timer fires with the most recently captured payload.

For workloads where mid-window crash recovery matters (autosave with
strict no-loss guarantees), the persisted dispatch row plus
recover-and-fire-immediately is the v1 contract. Cross-process
debounce coordination remains single-process only â€” see the warning
in the next section.

## Recipes

<AccordionGroup>
  <Accordion title="Document autosave (trailing + maxDelay)">
    `mode: "trailing", delay: "500ms", maxDelay: "5s"` keyed on
    `"$.req.params.docId"`. Rapid keystrokes collapse into one save
    after silence; continuous typing guarantees a save within 5
    seconds of the first keystroke.
  </Accordion>

  <Accordion title="Click suppression (leading)">
    `mode: "leading", delay: "1s"` keyed on `"$.req.params.postId"`. The
    first like fires; rapid double-clicks within a second drop
    silently.
  </Accordion>

  <Accordion title="Search-typeahead (trailing)">
    `mode: "trailing", delay: "300ms"` keyed on `"$.req.body.sessionId"`
    (one debounce window per search session). Fires once when the user
    stops typing for 300ms.
  </Accordion>

  <Accordion title="Workflow-wide singleton lock">
    `mode: "leading", delay: "100ms"` keyed on a **literal string**
    (e.g. `"global-rebuild-lock"`). The first request fires; concurrent
    requests within 100ms drop. Useful for self-protecting endpoints
    that should never run twice in quick succession.
  </Accordion>

  <Accordion title="Latest-wins payload">
    Trailing mode with continuous pings â€” each ping overwrites the
    captured payload via the closure on the active run. The fire
    consumes the latest payload, not the first. Document this if your
    payloads aren't append-only.
  </Accordion>
</AccordionGroup>

## Common pitfalls

<AccordionGroup>
  <Accordion title="Setting maxDelay < delay">
    Rejected at workflow load (when both are numeric). Make sure the
    floor (delay) doesn't exceed the ceiling (maxDelay).
  </Accordion>

  <Accordion title="Expecting maxDelay to apply to leading mode">
    It doesn't. Leading mode fires immediately, so there's no tail to
    bound. The schema accepts `maxDelay` on leading-mode debounce for
    forward-compatibility but it's ignored at runtime.
  </Accordion>

  <Accordion title="Coalesce losers polluting the runs list">
    1000 pings under heavy burst = 1000 run records. Tune
    `evictOldRuns` to bound storage, or filter by `status != debounced`
    in dashboards. Keep `debounced` records short-lived if they're
    cluttering the list.
  </Accordion>

  <Accordion title="Debounce key resolves to null">
    Fail-open â€” the gate is skipped and the run proceeds synchronously.
    Use `BLOK_MAPPER_MODE=strict` if you want fail-fast on missing
    payload fields in production.
  </Accordion>
</AccordionGroup>

## Not yet shipped

- **`mode: "throttle"`** â€” rate-cap (fire every N ms regardless of
  pings). Different invariant from debounce; could compose with
  [concurrency keys](/d/reliability/concurrency-keys) once it lands.
- **Dispatch-time payload merging** â€” each ping CONTRIBUTES to the
  final payload (instead of OVERWRITES). v1 ships latest-wins;
  contribute-style merging is on the roadmap.
- **Cross-process debounce** â€” coordinating debounce windows across
  processes (today the `DebounceCoordinator` is in-memory and
  single-process, so each process debounces its own traffic). Not yet
  built.
