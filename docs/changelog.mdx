<Update label="v1.1.0" description="2026-06-28">
## `subworkflow()` — the handle-DSL sub-workflow primitive

**New**
- [`subworkflow(id, name, inputs, opts?)`](/d/composition/sub-workflows) — invoke another workflow like a function from the handle DSL, the callback mirror of `step()`. Returns a typed handle to the child's response. `name` is a literal **or** a handle for polymorphic dispatch (one trigger → N handlers, paired with `allowList`). Opts: `{ wait, dispatch, allowList, idempotencyKey, retry, as, ephemeral }`. Emits the object-style `{ id, subworkflow, inputs }` shape under the hood, so it stays wire-compatible with JSON and the object DSL.

**Docs**
- [Context & State](/d/fundamentals/context-and-state) reframed handle-first; the security [cookbook](/d/security/cookbook) and [threat model](/d/security/threat-model) corrected.

Lockstep bump of all public packages to `1.1.0`.
</Update>

<Update label="v1.0.0" description="2026-06-28">
## The typed-handle authoring DSL

Workflows are now authored in TypeScript with **typed handles** — no `$`, no `js/`, no raw `ctx` strings. `step()` returns a handle to a step's output; pass it straight into later steps and the runner resolves it before the node runs.

**New**
- [`@blokjs/core`](/d/introduction/workflows) — the handle-DSL barrel: `workflow`, `step`, `branch`, `forEach`, `switchOn`, `tryCatch`, plus `tpl`, `http`, `defineNode`, `runtimeNode` and the typed comparators (`eq`/`ne`/`gt`/`gte`/`lt`/`lte`/`not`). Subpath split — `.` is the light DSL (tree-shaken, no grpc/sqlite), `./runtime` the engine, `./testing` the harnesses; `sideEffects: false`.
- Local nodes **auto-discover** from `src/nodes/<name>/index.ts` by their `defineNode` `name` — no hand-edited registry. `node.name` is the canonical `use:` reference; duplicate names throw at load.

**Changed**
- The legacy `Nodes.ts` hand-editing friction is gone. Object-style (`$.state`) and JSON workflows still compile to the same IR and remain fully supported.
- CLI scaffolds modernized to the handle DSL.

Lockstep bump of all public packages to `1.0.0`.
</Update>

<Update label="v0.7.0" description="2026-06-26">
## Modular observability

Observability is now **opt-in and modular**. Pick what you want at `create project` time or retrofit it any time with the new [`blokctl observability`](/d/cli/observability) command group.

**New**
- [`blokctl observability add | remove | list | status`](/d/cli/observability) — manage 7 modules in an existing project: `metrics`, `tracing`, `trace-store`, `logging`, `alerting`, `error-sink`, and the `obs-stack` dev stack.
- `create project --observability <list>` and `--obs-stack none|lite|full` to choose modules up front.
- Studio renders the terminal/transient run states added across recent releases — `crashed`, `timedOut`, `throttled`, `cancelled`, `expired`, `queued`, `delayed` — as status badges and toasts.

**Changed**
- `create project` no longer copies the full Prometheus/Grafana/Loki/Tempo stack unconditionally. The dev stack now defaults to **`none`**; restore the old behavior with `--obs-stack full` (or add it later with `blokctl observability add obs-stack`).
- Metrics remain **on by default** for backward compatibility, but are now opt-out via `BLOK_METRICS_DISABLED=1` — no import-time side effect, and the `/metrics` exporter is only registered when enabled. (There is no `BLOK_METRICS_ENABLED`.)
- Tracing and the error sink are **inert by default** — they do nothing until you set `OTEL_EXPORTER_OTLP_ENDPOINT` / `SENTRY_DSN`.

Lockstep bump of all public packages to `0.7.0`.
</Update>
