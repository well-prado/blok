### v0.4.1 release plan

1. **Pre-flight check** (~1 min)

   ```bash
   bun run release --dry-run --skip-build --skip-tests
   ```

   Will fail on lockstep version mismatch (everything is at `0.4.0` on disk; we need to bump to `0.4.1`).

2. **Manual lockstep bump** (~5 min) — edit the 8 public `package.json` files from `0.4.0` to `0.4.1`. Cross-package dep ranges are already at `^0.4.0` which includes `0.4.1`, so no other range edits needed. Bump `BLOKJS_DEP_RANGE` and `GITHUB_REPO_RELEASE_TAG` in [packages/cli/src/commands/create/project.ts](https://github.com/deskree-inc/blok/blob/main/packages/cli/src/commands/create/project.ts) — `BLOKJS_DEP_RANGE` stays `^0.4.0` (any 0.4.x is fine), but `GITHUB_REPO_RELEASE_TAG` becomes `v0.4.1`.

3. **Tag** (~1 min)

   ```bash
   git tag -a v0.4.1 -m "0.4.1 — defineNode empty-array fix, AuthMiddleware deprecation, setAuthorizeFn, ctx.env allowlist"
   ```

4. **Local-pack validation** (~10 min) — see [release-runbook](/c/devtools/release-runbook#local-pack-validation).

5. **Publish with batched OTP** (~1 min)

   ```bash
   bun run release --otp <code> --yes --skip-build --skip-tests
   ```

   `--skip-build/--skip-tests` is fine because pre-commit gated everything in main; rebuild is unnecessary if the workspace is clean.

6. **Push tag**

   ```bash
   git push --tags
   ```

7. **Post-publish smoke test** — see [release-runbook · Post-publish smoke test](/c/devtools/release-runbook#post-publish-smoke-test). For v0.4.1 specifically, also test:
   - `blokctl create project` clones the v0.4.1 tag (verify scaffolded `package.json` cross-deps at `^0.4.0` resolve to `0.4.1` from npm)
   - The empty-arms branch in scaffolded `triggers/http/src/workflows/empty.ts` no longer crashes when hit (`curl http://localhost:4000/empty-helper`)
   - Instantiating `AuthMiddleware` emits the deprecation warning to stderr

### What's deliberately NOT in v0.4.1

- **Removal of `BLOK_ROUTING_LEGACY` escape hatch** — that's a v0.5 breaking change. Grace period started 2026-05-07; conventional 1-2 months.
- **Removal of AuthMiddleware** — v0.5. v0.4.1 only deprecates.
- **Tier C/D/E/F/G feature work** — demand-driven, ships when a user pulls. Not bundled into the patch.

**Estimated v0.4.1 ship effort: ~30 minutes end-to-end.** It's the single highest-leverage thing remaining for the framework right now.

## v0.5 — accumulated breaking changes

v0.5 is a coordinated cleanup release that drops every escape hatch and deprecated surface accumulated through 0.4.x. The version bump signals "if you depended on a v0.3 or v0.4 transition flag, read this first."

### Breaking changes shipped in v0.5

#### 1. Drop `BLOK_ROUTING_LEGACY=1` escape hatch

**Affected files:**
- [triggers/http/src/runner/WorkflowRouter.ts](https://github.com/deskree-inc/blok/blob/main/triggers/http/src/runner/WorkflowRouter.ts) — remove the `legacyMode` branch, the `isLegacyRoutingEnabled()` helper, and the deprecation warning. Strict-mode becomes the only mode.
- [triggers/http/__tests__/unit/WorkflowRouter.test.ts](https://github.com/deskree-inc/blok/blob/main/triggers/http/__tests__/unit/WorkflowRouter.test.ts) — remove the legacy-mode test cases.
- [docs/c/migration-guides/v0.4-explicit-paths.mdx](https://github.com/deskree-inc/blok/blob/main/docs/c/migration-guides/v0.4-explicit-paths.mdx) — update the "v0.5 removal timeline" to "removed in this version."

**Effort:** ~30 minutes mechanical work + docs + 1 changeset.

**Pre-conditions:** ≥1 month after v0.4.0 published on npm (started 2026-05-07). Earliest reasonable v0.5: **2026-06-07**. More conservative: 2 months → **2026-07-07**.

#### 2. Remove `AuthMiddleware`, `JWTAuthProvider`, `APIKeyAuthProvider`

**Affected files:**
- [core/runner/src/security/AuthMiddleware.ts](https://github.com/deskree-inc/blok/blob/main/core/runner/src/security/AuthMiddleware.ts) — delete entire file.
- [core/runner/src/security/index.ts](https://github.com/deskree-inc/blok/blob/main/core/runner/src/security/index.ts) — drop the three classes + their config types from the barrel exports.
- [core/runner/src/index.ts](https://github.com/deskree-inc/blok/blob/main/core/runner/src/index.ts) — drop the same from the runner's top-level barrel.
- [core/runner/src/__tests__/security/AuthMiddleware.test.ts](https://github.com/deskree-inc/blok/blob/main/core/runner/src/__tests__/security/AuthMiddleware.test.ts) — delete (23 tests).
- [docs/d/security/cookbook.mdx](https://github.com/deskree-inc/blok/blob/main/docs/d/security/cookbook.mdx) — drop the "AuthMiddleware is deprecated" section entirely; the recommended pattern (Pattern A workflow-level gate with `jose`/`hono/jwt`) stays as the canonical recipe.
- [documentation/modules/security.md](https://github.com/deskree-inc/blok/blob/main/documentation/modules/security.md) — drop the AuthMiddleware section.
- [documentation/structure/AI_INDEX.md](https://github.com/deskree-inc/blok/blob/main/documentation/structure/AI_INDEX.md) — drop the AuthMiddleware row.
- [documentation/structure/SITE_MAP.md](https://github.com/deskree-inc/blok/blob/main/documentation/structure/SITE_MAP.md) — drop the AuthMiddleware row (item 47).

Note: the rest of `core/runner/src/security/` (OAuthOIDCProvider, RBAC, ABAC, AuditLogger, SecretManager, EncryptionAtRest, PIIDetector, TLSConfig) **stays**. Those are production-grade composable primitives — only AuthMiddleware was deprecated as example-grade.

**Effort:** ~30 minutes mechanical removal + docs + 1 changeset.

**Pre-conditions:** v0.4.1 has been on npm long enough that any user importing AuthMiddleware has seen the deprecation warning. Same window as the routing legacy drop.

#### 3. Drop legacy `set_var` field (candidate, not committed)

**Status:** Open question for v0.5.

The v2 step shape default-stores every step's output to `ctx.state[id]`. The legacy `set_var: false` is normalized to `ephemeral: true` at workflow load. Since v2 was the default surface in v0.3, anyone writing new workflows already uses `ephemeral: true`. The `set_var` shim costs nothing at runtime (load-time normalization only).

**Decision criteria for v0.5:**

- **If** there are still v1 JSON workflows actively maintained by the community, keep the shim through v0.5.
- **If** the v1 surface is effectively dead (no GitHub issues, no npm telemetry showing legacy-shape workflows), drop in v0.5.

**Recommendation:** keep through v0.5, drop in v1.0. The runtime cost is zero; the migration burden of forcing `set_var` removal on top of routing-legacy removal is high.

#### 4. Drop legacy `name`/`node` v1 step shape (candidate, not committed)

**Status:** Open question. Same rationale as `set_var`. The `WorkflowNormalizer` accepts both shapes; cost is load-time only.

**Recommendation:** keep through v0.5, revisit at v1.0.

### v0.5 release plan

Same flow as v0.4.1 but with a minor bump:

1. Pre-flight + manual lockstep bump to `0.5.0` across all 8 packages.
2. Bump `BLOKJS_DEP_RANGE` → `^0.5.0` (this WILL break v0.4.x scaffolds — that's the point of a minor bump).
3. Bump `GITHUB_REPO_RELEASE_TAG` → `v0.5.0`.
4. Update cross-package dep ranges across the monorepo from `^0.4.0` → `^0.5.0` (the `bun run release` pre-flight will catch any miss).
5. Run the v0.5 cleanup commits (drop routing legacy + remove AuthMiddleware).
6. Tag, validate, publish.
7. Update the `v0.4 explicit-paths migration guide` to reflect "removed in v0.5."

**Estimated v0.5 ship effort: ~1-2 hours including breaking changes + ~half day of coordinated migration-guide and cookbook updates.**

## Open BACKLOG.md items, by tier

Status as of 2026-05-07 — every item below is **open**, demand-driven, with file:line refs in [BACKLOG.md](https://github.com/deskree-inc/blok/blob/main/BACKLOG.md). All Tier A bugs, all H3+H4 integration tests, all I1+I2 docs are SHIPPED.

### Tier C — Cross-process scaling (4 items, ~7-10 days total)

These add cross-process coordination to existing single-process primitives. None block production single-process deployments; they unlock horizontal scaling.

#### C1 · Cross-process debounce keys (NATS KV / Redis)

**Effort:** ~1-2 days.
**Why now or later:** Currently `DebounceCoordinator` is in-process. Two pods handling the same `(workflow, debounceKey)` won't coalesce — both fire. Affects multi-pod deployments using `debounce: { trailing }` for autosave-style endpoints.

**Sketch:** mirror `NatsKvConcurrencyBackend`'s OCC pattern. One JSON document per `(workflow, debounceKey)` holding `{firstPingAt, lastPingAt, pingCount, activeRunId, scheduledAt, mode, delayMs, maxDelayMs}`. CAS-update on each ping. Active process owns the timer; coalesce losers update the doc and ACK.

**Trade-off:** cross-process debounce coordination via NATS KV adds a network round-trip per ping. For high-throughput debounce (autosave at 100Hz per doc), this can be the bottleneck. Document the latency and offer a per-key-fanout escape hatch.

**Reference:** [BACKLOG.md C1](https://github.com/deskree-inc/blok/blob/main/BACKLOG.md#c1--cross-process-debounce-keys-nats-kv--redis), [core/runner/src/scheduling/DebounceCoordinator.ts](https://github.com/deskree-inc/blok/blob/main/core/runner/src/scheduling/DebounceCoordinator.ts)

#### C2 · Cross-process scheduler coordination

**Effort:** ~2-3 days.
**Why now or later:** With separate sqlite files per process (typical setup), each process owns its own dispatches — no double-fire. With PG-backed deployments sharing one DB, two processes both running `recoverDispatches()` could double-fire the same row.

**Sketch:** add `claimed_by` + `claimed_at` columns to the `scheduled_dispatches` table (PG migration v10). On boot, only claim rows that are unclaimed OR claimed by a stale process (lease expired). Heartbeat the claim while the timer is registered.

**Reference:** [BACKLOG.md C2](https://github.com/deskree-inc/blok/blob/main/BACKLOG.md#c2--cross-process-scheduler-coordination), [core/runner/src/tracing/PostgresRunStore.ts](https://github.com/deskree-inc/blok/blob/main/core/runner/src/tracing/PostgresRunStore.ts)

#### C3 · Per-lease key model for high-cardinality NATS KV buckets

**Effort:** ~1-2 days.
**Why now or later:** Today's NATS KV uses one document per `(workflow, key)` bucket holding all leases. With >50 active leases per bucket, the JSON doc grows + CAS contention escalates. Per-lease keys would distribute contention.

**Recommendation:** wait for a real workload. Document the 1–50 leases assumption prominently in [`docs/d/reliability/concurrency-keys.mdx`](https://github.com/deskree-inc/blok/blob/main/docs/d/reliability/concurrency-keys.mdx) until then. The H3 integration tests we just shipped exercise the document model under contention; they'll surface the limit.

**Reference:** [BACKLOG.md C3](https://github.com/deskree-inc/blok/blob/main/BACKLOG.md#c3--per-lease-key-model-for-high-cardinality-nats-kv-buckets)

#### C4 · Redis backend (alternative to NATS KV) — ✅ SHIPPED

**Status:** SHIPPED on `feat/c4-redis-concurrency-backend`. Mirrors `NatsKvConcurrencyBackend` 1:1 modulo atomicity primitive — Redis uses server-side **Lua scripts** for acquire / release / per-bucket purge, so each operation is a single `EVAL` with no OCC retry loop (Lua runs single-threaded against the keyspace). Same `{leases:[…]}` JSON-document storage shape per `(workflow, key)` bucket, same lazy-purge-on-acquire semantics, same FW-5 production refusal for the default key prefix (`blok-concurrency`). Connection defaults `connectTimeout: 5s`, `maxRetriesPerRequest: 0`, `enableOfflineQueue: false`, `lazyConnect: true` — trigger startup never hangs on broker reachability.

**Env vars:** `BLOK_CONCURRENCY_BACKEND=redis` to opt in, plus `BLOK_CONCURRENCY_REDIS_URL` (or discrete `_HOST`/`_PORT`/`_USERNAME`/`_PASSWORD`/`_DB`/`_TLS`), and `BLOK_CONCURRENCY_REDIS_KEY_PREFIX` (production refuses the default). Requires `ioredis` peer dep.

**Source:**
[`core/runner/src/concurrency/RedisConcurrencyBackend.ts`](https://github.com/deskree-inc/blok/blob/main/core/runner/src/concurrency/RedisConcurrencyBackend.ts)
+ [`core/runner/__tests__/unit/concurrency/RedisConcurrencyBackend.test.ts`](https://github.com/deskree-inc/blok/blob/main/core/runner/__tests__/unit/concurrency/RedisConcurrencyBackend.test.ts).

**Reference:** [BACKLOG.md C4](https://github.com/deskree-inc/blok/blob/main/BACKLOG.md#c4--redis-backend-alternative-to-nats-kv)

### `docs/d/operations/env-vars.mdx` claims sqlite default for `BLOK_TRACE_STORE`

The env-vars page at line 24 says:

```
| BLOK_TRACE_STORE | sqlite | Run store backend: sqlite, inmemory, or postgres. |
```

But [core/runner/src/tracing/createStore.ts:38](https://github.com/deskree-inc/blok/blob/main/core/runner/src/tracing/createStore.ts#L38) reads:

```ts
const type = opts?.type || (process.env.BLOK_TRACE_STORE as StoreType) || "memory";
```

So when no env var is set, the default is **memory**, not **sqlite**. The doc is wrong, OR the code default should change to sqlite (which is what most users probably expect for production).

**Recommendation:** decide which is correct, then align. For production durability, `sqlite` is the better default; for tests, `memory` is right. A pragmatic fix: keep `memory` for `NODE_ENV=test`, default to `sqlite` otherwise.

**Effort:** ~1 hour (decision + code change + doc fix + test).

## Infrastructure investments

Tracked but not blocking. Each is a multi-day project; pick when capacity is available.

### CI/CD bootstrap

**Status:** None. `.github/workflows/` does not exist.

**Why it matters:**

- Today's gate is the local `pre-commit` hook running `biome check` + `nx run-many -t test` (~3.5 seconds for the runner suite, longer for the full nx run). Catches most regressions, but only on the developer's machine.
- A passing pre-commit on macOS doesn't guarantee the same on Linux (the multi-language SDKs ship Docker containers; cross-platform issues sneak through).
- Per-PR test runs would catch what local pre-commits miss, especially for changes that touch the SDK side.
- The release-pipeline-hardening recommendation in the [release-runbook](/c/devtools/release-runbook) — option (1) "changesets bot in CI" — depends on CI existing.

**Why not yet:**

- Bootstrapping CI on a multi-runtime monorepo (Node + Go + Rust + Java + C# + PHP + Ruby + Python + NATS + sqlite + Postgres + Redis) is non-trivial. Each SDK needs a job; cross-runtime parity tests need orchestration; the build matrix is real.
- Currently no contributor PRs to gate against. As long as commits flow through one developer with a working local environment, CI is a quality investment, not a correctness gate.

**Recommendation:** stage in three increments:

1. **Lightweight `pre-commit` mirror** (~half day) — single GitHub Action that runs on PRs to `main`: `bun install && bunx biome check && bunx nx run-many -t test`. No SDK matrix. Catches ~80% of regressions.
2. **SDK matrix** (~1-2 days) — one job per SDK with the appropriate toolchain (`actions/setup-go@v5`, `actions/setup-java@v4`, etc.). Run that SDK's parity tests. Slow but parallel.
3. **Release automation** (~half day) — wire the existing `bun run release` script into a release-tag-triggered workflow.

### Performance benchmarking harness

**Status:** [core/runner/src/__tests__/tier2-followups/Tier2Benchmarks.test.ts](https://github.com/deskree-inc/blok/blob/main/core/runner/src/__tests__/tier2-followups/Tier2Benchmarks.test.ts) exists and runs as part of the unit suite (~3 seconds). Single-snapshot benchmarks; no historical regression detection.

**What's missing:**

- A bench harness that runs benchmarks separately from unit tests (so unit suites stay fast).
- Persistent storage of benchmark results (sqlite or JSON over git history).
- A regression detection step that fails CI when a benchmark drops by >X%.

**Effort:** ~2-3 days for a real harness; ~1 day for a stub that just stores baselines.

**Priority:** lower than CI bootstrap, but the existing benchmarks are good seeds.

### Security audit

**Status:** Threat model documented at [docs/d/security/threat-model.mdx](https://github.com/deskree-inc/blok/blob/main/docs/d/security/threat-model.mdx); cookbook at [docs/d/security/cookbook.mdx](https://github.com/deskree-inc/blok/blob/main/docs/d/security/cookbook.mdx). **Effort:** ~$15-30K for a 2-week external review covering supply chain (dependencies), node trust model, sub-workflow boundaries, env-var leakage, NATS KV multi-tenancy, the Studio API. Output: a CVSS-scored finding report + remediation plan.

**Timing:** schedule for the v1.0 release window (12-24 months out per ROADMAP.md).

## v1.0 considerations

Far-term — this is a horizon section, not a near-term plan.

### What v1.0 means for Blok

The pre-1.0 era:

- Lockstep major bumps across 8 packages.
- Breaking changes in minor releases (0.4 → 0.5).
- "It works on the developer's laptop" is the stability commitment.

v1.0 inverts those:

- Independent semver per package (so `@blokjs/runner@1.5.0` doesn't force `@blokjs/helper` to bump).
- Breaking changes only in major releases.
- Stability commitment backed by API freeze + deprecation cycles.
- LTS support window declared.

### Blockers between 0.5 and 1.0

Roughly in priority order:

| Blocker | Effort | Notes |
|---|---|---|
| Independent semver migration | ~1 week | Migrate from lockstep to per-package versions; update changesets config; update release runbook |
| All 7 SDKs at parity | varies | Currently the gRPC contract is shared but feature parity (e.g., wait.for, sub-workflows, idempotency cache writes) needs per-SDK validation |
| Cross-runtime test matrix in CI | ~1 week | The 34/34 parity tests run locally; need to run on every PR |
| Production deployment patterns | ~1-2 weeks | Documented Kubernetes deployment, AWS Lambda, Vercel, Fly.io, Render. Each with operator guidance |
| Formal security audit | ~3-4 weeks lead time | External review (see above) |
| Migration guide v0.x → v1.0 | ~half day | Single page documenting the breaking surface |
| Stable npm dist-tag strategy | ~1 hour | `latest`, `next`, `lts` semantics |

**Total:** ~6-8 weeks of focused work plus calendar time for the audit. A reasonable v1.0 target is **Q4 2026 or Q1 2027** if v0.5 ships in summer 2026.

### What v1.0 explicitly will NOT include

- CRIU process-snapshot checkpointing (covered by composable building blocks via `wait.for`).
- Step-level concurrency keys (different invariant set; deferred to a later major).
- Long delays >24h via internal scheduler (recommend cron + external scheduler).
- Debounce `mode: "throttle"` (different from coalesce; deferred unless user pulls).

These are tracked as "out of scope" in [BACKLOG.md Tier J](https://github.com/deskree-inc/blok/blob/main/BACKLOG.md#tier-j--out-of-scope-deliberately).

## Recommended order

Given the goals "ship reliably + unblock production users":

1. **v0.4.1 patch** (~30 min) — push the 5 stacked commits to npm. Gives users the empty-arms branch fix, the AuthMiddleware deprecation warning, the setAuthorizeFn API, and the env allowlist. Highest leverage per minute spent. **Use the new `bun run release` script.**

2. **Doc hygiene pass** (~1 day) — fill the empty `index.mdx` files; fix the env-vars store-default drift; write `c/migration-guides/v1-to-v2-reliability.mdx` (Tier I3); decide on `documentation/` vs `docs/` consolidation. Low effort, high signal-to-noise.

3. **CI bootstrap (lightweight)** (~half day) — single GitHub Action mirroring pre-commit on every PR. Doesn't need SDK matrix. Lays groundwork for release automation.

4. **v0.5 release** (~6-8 weeks after v0.4.1 published) — drop `BLOK_ROUTING_LEGACY` + remove AuthMiddleware. Coordinated breaking-change release with migration guide updates.

5. **Demand-driven Tier C/D/E/F/G work** — pull when a user reports they need it. Don't speculate.

6. **CI matrix expansion** + **performance harness** — when contributor velocity makes it worthwhile.

7. **v1.0 lead-up** — security audit + cross-runtime CI matrix + production patterns. Calendar-dominated, not effort-dominated.

This is the playbook. Each step's effort fits a single working session; cumulative timeline spans ~6 months from v0.4.1 to v1.0 if the demand-driven work is balanced with the planned releases.
