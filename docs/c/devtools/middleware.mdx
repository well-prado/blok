In v0.5, **middleware is just a workflow**. You declare a workflow with `"middleware": true` in its definition, and any other workflow can opt into it by listing its name in one of two places.

## Two declaration sites — workflow-level vs trigger-level

**Workflow-level** (v0.5.2+) — declare once at the top of the workflow body. Applies to **every** trigger of the workflow. Use this when the chain (auth, rate-limit) is uniform regardless of which trigger fired:

```json
{
  "name": "v05-jwt-protected",
  "middleware": ["jwt-auth"],
  "trigger": {
    "http": { "method": "POST", "path": "/" }
  },
  "steps": [/* ... */]
}
```

**Trigger-level** — declare on a specific trigger config. Applies only when that particular trigger fires. Use this when the chain varies by trigger (e.g. only the public HTTP endpoint needs CAPTCHA, the worker queue doesn't):

```json
"trigger": {
  "http": {
    "method": "POST",
    "path": "/orders",
    "middleware": ["captcha", "rate-limit"]
  }
}
```

When **both** are set, workflow-level runs **first**, then trigger-level — so a workflow-level `auth` chain stashes `ctx.state.identity` before a trigger-level `rate-limit` consumes it.

Middleware run **before** the main workflow, in the order declared, sharing the same `ctx`. A middleware can stash data on `ctx.state` for downstream middleware/steps (`identity`, `rateLimitBucket`, etc.). A middleware can short-circuit by throwing — typically via `@blokjs/throw` with an HTTP status code (`401` for auth, `429` for rate-limit).

> **Don't put `middleware: true` AND `middleware: [...]` on the same workflow.** The two are mutually exclusive (one means "I AM a middleware", the other means "I USE these middleware"). The normalizer rejects the conflict at load time.

## Authoring a middleware

Middleware workflows go in `triggers/http/workflows/json/middleware/`. They look like normal workflows, except:

1. The top-level `"middleware": true` flag marks them as middleware-only (they don't get their own HTTP route).
2. They typically end either by throwing (short-circuit) or by publishing context for the main workflow to read (`@blokjs/ctx-publish`).

### JWT auth (production)

The shipped [`jwt-auth.json`](https://github.com/deskree-inc/blok/blob/main/triggers/http/workflows/json/middleware/jwt-auth.json) middleware uses `@blokjs/jwt-verify` (powered by [`jose`](https://github.com/panva/jose)) to verify the bearer token's signature, issuer, audience, and expiry, then stashes the decoded claims at `ctx.state.identity`. Configuration comes from environment variables, so deploy-time secrets never live in workflow JSON:

| Env var | Purpose |
|---|---|
| `JWT_SECRET` | HMAC shared secret for HS256/HS384/HS512 (dev/test) |
| `JWT_JWKS_URI` | Remote JWKS endpoint URL for RS256/ES256/EdDSA (production case — Auth0 / Okta / Cognito / Clerk all publish one). Takes precedence over `JWT_SECRET`. |
| `JWT_ISSUER` | Expected `iss` claim — strongly recommended |
| `JWT_AUDIENCE` | Expected `aud` claim — strongly recommended |
| `JWT_ALGORITHMS` | Comma-separated allowlist (default: `["HS256"]` symmetric / `["RS256"]` asymmetric). The `none` algorithm is never accepted regardless of allowlist. |
| `JWT_CLOCK_TOLERANCE_SECONDS` | Seconds of clock skew to tolerate when checking `exp`/`nbf`. Default 0. |

Use it on any trigger:

```json
"trigger": {
  "http": {
    "method": "POST",
    "path": "/orders",
    "middleware": ["jwt-auth"]
  }
}
```

After `jwt-auth` runs, downstream steps read the identity from `ctx.state.identity`:

```json
{
  "expression": "({ greeting: `hello, ${ctx.state.identity.sub}`, role: ctx.state.identity.claims.role })"
}
```

The `identity` shape is:

```ts
{
  sub: string | undefined,         // claims.sub
  iss: string | undefined,         // claims.iss
  aud: string | string[] | undefined,  // claims.aud
  exp: number | undefined,         // claims.exp (UNIX seconds)
  claims: Record<string, unknown>, // full decoded payload — read custom claims here
  via: "jwt-auth"                  // origin tag, useful in logs
}
```

#### Verification failures

Every failure mode produces a structured `401` response with a stable `reason` discriminator authors can branch on (or just emit verbatim):

| `reason` | Meaning |
|---|---|
| `missing_token` | No `Authorization: Bearer <…>` header |
| `malformed_token` | Token isn't a parseable JWT |
| `invalid_signature` | Signature verification failed |
| `token_expired` | `exp` is in the past (modulo clock tolerance) |
| `token_not_yet_valid` | `nbf` is in the future |
| `issuer_mismatch` | `iss` doesn't match `JWT_ISSUER` |
| `audience_mismatch` | `aud` doesn't match `JWT_AUDIENCE` |
| `algorithm_not_allowed` | Token uses an algorithm not in `JWT_ALGORITHMS` |
| `claim_validation_failed` | Generic claim validation failure (other than the specific cases above) |
| `verification_failed` | Catch-all for unrecognized failures |

The HTTP response body always carries `{ error: "Unauthorized", reason: "<reason>", message: "<human readable>" }`. A `tryCatch.catch` arm wrapping a downstream step can also read these via `$.error.code === 401` and `$.error.message`.

### Auth check (header-token equality, demo only)

The shipped [`auth-check.json`](https://github.com/deskree-inc/blok/blob/main/triggers/http/workflows/json/middleware/auth-check.json) is a tiny demo for end-to-end testing the middleware machinery — it just compares the `Authorization` token to an `X-Expected-Token` header. **Don't ship this to production** — copy `jwt-auth` instead.

### Rate limit (in-memory, single-process)

The shipped [`rate-limit.json`](https://github.com/deskree-inc/blok/blob/main/triggers/http/workflows/json/middleware/rate-limit.json) middleware uses `@blokjs/in-memory-kv` to track per-identity buckets. Fixed-window, 5 req/60s default. The bucket key falls back to `X-Forwarded-For` when no `identity.token` is present, so it works whether or not an auth middleware ran first.

**Use this for:** local dev, single-process deployments, internal tools where rate-limit precision across replicas doesn't matter.

**Don't use this for:** production behind a load balancer (counters are per-process, so 3 replicas effectively triple the limit), long-running services (counters reset on restart).

### Rate limit (Redis-backed, production)

The shipped [`redis-rate-limit.json`](https://github.com/deskree-inc/blok/blob/main/triggers/http/workflows/json/middleware/redis-rate-limit.json) middleware mirrors `rate-limit.json` step-for-step but swaps `@blokjs/in-memory-kv` for `@blokjs/redis-kv`. Same algorithm, same defaults, durable storage:

- Counters survive process restarts.
- Multiple replicas share state — the limit holds across the fleet.
- Per-key TTL set via Redis `SET ... PX <windowMs>`, so expired buckets evict automatically (no background reaper).
- Keys are namespaced `blok:rate-limit:<identity>` so cleanup / inspection is grep-friendly (`KEYS blok:rate-limit:*` from redis-cli).

Configuration:

| Env var | Default | Purpose |
|---|---|---|
| `REDIS_URL` | `redis://127.0.0.1:6379` | Where the `ioredis` client connects |

Use it on any trigger:

```json
"trigger": {
  "http": {
    "method": "POST",
    "path": "/api",
    "middleware": ["jwt-auth", "redis-rate-limit"]
  }
}
```

The full demo workflow [`v05-redis-protected.json`](https://github.com/deskree-inc/blok/blob/main/triggers/http/workflows/json/v05-redis-protected.json) chains both `jwt-auth` and `redis-rate-limit` at the workflow level (v0.5.2 declaration), and is exercised end-to-end by the smoke gate (skipped when `REDIS_URL` is unset). The smoke run validates the bucket increments correctly across requests and that the 6th request inside the window returns `429` with a `retryAfterSec` header value.

The underlying `@blokjs/redis-kv` helper is general-purpose — `set` / `get` / `delete` / `list` actions, JSON-encoded values, optional per-key `ttlMs`. Use it directly anywhere your workflow needs durable KV (idempotency keys, distributed locks, session storage). The `ioredis` dependency is loaded dynamically on first invocation, so workflows that don't use the helper pay nothing at boot.

## Runtime contract

- **Order matters.** Middleware run in the order declared on the trigger. Earlier middleware can publish state that later middleware read.
- **Single ctx.** All middleware share the same `ctx` as the main workflow — there's no per-middleware isolation. A `ctx.state.identity` written by `auth-check` is read by `rate-limit` and the main workflow alike.
- **Short-circuit by throwing.** A middleware that throws aborts the chain. The main workflow does **not** run. The error's `code` and `body` (when set via `@blokjs/throw`) become the HTTP response.
- **Pass-through is implicit.** A middleware that runs to completion without throwing passes control to the next middleware. There is no explicit `next()` call.
- **Middleware are tagged in traces.** Steps that ran as part of a middleware chain are tagged with origin `mw:<middleware-name>` in Blok Studio's StepRail, so you can tell at a glance which steps came from auth, rate-limit, vs the main workflow body.

## Real-world examples

[`triggers/http/workflows/json/v05-jwt-protected.json`](https://github.com/deskree-inc/blok/blob/main/triggers/http/workflows/json/v05-jwt-protected.json) is the production-grade demo — a single-step main workflow guarded by `jwt-auth`. The smoke gate exercises seven cases against it (missing token, malformed, bad signature, expired, wrong issuer, wrong audience, valid) and asserts the structured `reason` field on each failure.

[`triggers/http/workflows/json/v05-protected.json`](https://github.com/deskree-inc/blok/blob/main/triggers/http/workflows/json/v05-protected.json) is the simpler demo — `auth-check` (header-token equality) + `rate-limit`. Curl with `Authorization: Bearer <token>` matching `X-Expected-Token: <token>` → 200; sixth request inside 60s → 429.

[`triggers/http/workflows/json/v05-order-fulfillment.json`](https://github.com/deskree-inc/blok/blob/main/triggers/http/workflows/json/v05-order-fulfillment.json) shows a multi-middleware chain on a real workflow — `auth-check` + `rate-limit` run before a `switchOn` that branches to per-order-type pipelines, with the rate-limit bucket keyed off `ctx.state.identity.token` written by auth-check.

All three are curl-validated by `bun run v05:smoke`.

## Caveats

- **`jwt-auth` is production-shippable; `auth-check` is not.** Use `jwt-auth` in real deployments. `auth-check` is a header-token-equality demo retained for tutorials and the rate-limit interop test — it has no signature, no expiry, no issuer/audience checks, and is trivially bypassed by anyone who can guess the value of `X-Expected-Token`.
- **JWKS keys are cached per process, not per request.** The first request to a JWKS-backed endpoint triggers a fetch; subsequent requests reuse the cached JWKS. `jose` handles rotation automatically by refetching when a token's `kid` isn't in the cache. Cold-start latency on the first request after process boot is one round-trip to the JWKS endpoint.
- **`rate-limit` and `redis-rate-limit` ship side-by-side.** Pick `rate-limit` for local dev / single-process; pick `redis-rate-limit` for any production deployment behind a load balancer. They have identical configuration knobs and identical 429 response shape, so swapping is a one-line change in the trigger's `middleware: [...]` list.
- **No per-step middleware.** Today middleware applies at the workflow or trigger level only — there's no way to declare middleware on a single step. If you need step-scoped logic, wrap the step in a `tryCatch` or sub-workflow.
- **Middleware participate in trace origin tagging, not in concurrency-key resolution.** A trigger's `concurrencyKey` is resolved from ctx **before** middleware run, so an `identity.sub` written by `jwt-auth` cannot be used as the concurrency key. Resolve concurrency keys from `ctx.request.headers` or `ctx.request.body` instead, or hold off on middleware-driven concurrency until that resolution order is reworked.
