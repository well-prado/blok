This guide is the executable spec for turning the [v0.5 primitives roadmap](/c/devtools/workflow-primitives-roadmap) and the [15 mock examples](https://github.com/deskree-inc/blok/tree/main/examples/v05-primitives) into shipped code. Every example becomes a real, curl-able workflow against public APIs (no API keys required) plus a few in-memory helper nodes. Three Docker services (Redis, Postgres, MailHog) cover the cases that genuinely need state.

The doc is intentionally specific — every function signature, every Zod schema, every test, every real-API substitution is named here so a future agent can execute without re-deriving design.

---

## Architecture overview

Each new primitive ships in three layers:

1. **TS DSL helper** in [`core/workflow-helper/src/components/`](https://github.com/deskree-inc/blok/tree/main/core/workflow-helper/src/components) — author-facing function, returns a typed step object that JSON-stringifies to the v0.5 JSON shape.
2. **Workflow normalizer discriminator** in [`core/runner/src/workflow/WorkflowNormalizer.ts`](https://github.com/deskree-inc/blok/blob/main/core/runner/src/workflow/WorkflowNormalizer.ts) — recognizes the new step shape and converts it to an internal step + config map entry.
3. **Internal flow node** in `core/runner/src/nodes/` — the runtime implementation. Either a `defineNode({ flow: true, ... })` (for `forEach`, `loop`, `switch`) or runner-intercepted (for `tryCatch`, which needs error-handling control the flow-node pattern doesn't give).

Plus, for middleware:

4. **`WorkflowRegistry` extensions** — global middleware + middleware lookup.
5. **`TriggerBase.run()` extensions** — middleware chain resolution and execution.

The key reuse: every primitive uses `defineNode` with the v2 `this.flow` discriminator we already shipped. The runner's `RunnerSteps` already handles flow-node recursion. Inner-step config bubbling already shipped (the `innerNodes` map from the recent `branch` fix).

## TS DSL contracts

The TS DSL function for each primitive returns a typed step object. The compiled output **exactly mirrors** the JSON shape — TS authors write `forEach({...})` and what lands in the workflow is `{ id, forEach: {...} }`. JSON authors write the same shape directly.

Core principle: **TS gives you autocomplete + Zod validation at definition time. JSON gives you the same shape with no type checking. The runner accepts both identically.**

### `forEach`

```ts
import type { ForEachStep, V2Step } from "@blokjs/helper";

export interface ForEachOpts {
  /** Stable identifier — visible in traces, referenced as `$.state[id]`. */
  id: string;
  /** Array source. Literal expression string or `$` proxy. */
  in: string | unknown;
  /**
   * Per-iteration variable name. Each iteration sets `ctx.state[as]` to the
   * current item and `ctx.state[as + "Index"]` to the 0-based position.
   * Cleared after the forEach completes.
   */
  as: string;
  /** Sub-pipeline run for each item. */
  do: V2Step[];
  /** Default `"sequential"` (one at a time, awaited). */
  mode?: "sequential" | "parallel";
  /** When `mode: "parallel"`, max concurrent inner pipelines. Default 10. */
  concurrency?: number;
  /** Skip the entire forEach at runtime. Default true. */
  active?: boolean;
  /** Halt the workflow after this step completes. */
  stop?: boolean;
}

export function forEach(opts: ForEachOpts): ForEachStep;
```

**Compiled output:**

```ts
{
  id: opts.id,
  forEach: {
    in: typeof opts.in === "string" ? opts.in : unwrapProxies(opts.in),
    as: opts.as,
    mode: opts.mode ?? "sequential",
    concurrency: opts.concurrency ?? 10,
    do: unwrapProxies(opts.do),
  },
  // active / stop hoisted to top level if set
}
```

**Zod validation at definition time:**

```ts
const ForEachSchema = z.object({
  id: z.string().min(1),
  in: z.union([z.string().min(1), z.unknown()]),
  as: z.string().min(1).regex(/^[a-zA-Z_][a-zA-Z0-9_]*$/, "as must be a valid identifier"),
  do: z.array(V2StepSchema).min(1),
  mode: z.enum(["sequential", "parallel"]).default("sequential"),
  concurrency: z.number().int().min(1).max(1000).default(10),
  active: z.boolean().optional(),
  stop: z.boolean().optional(),
});
```

### `loop`

```ts
export interface LoopOpts {
  id: string;
  /**
   * JS expression evaluated against ctx BEFORE each iteration.
   * Loop continues while truthy. Falsy → exit.
   */
  while: string | unknown;
  /** Sub-pipeline run each iteration. */
  do: V2Step[];
  /**
   * Hard cap on iterations. Default 1000 (override via env `BLOK_LOOP_MAX_ITERATIONS`).
   * Hitting the cap throws `LoopMaxIterationsError`.
   */
  maxIterations?: number;
  active?: boolean;
  stop?: boolean;
}

export function loop(opts: LoopOpts): LoopStep;
```

**Iteration variable:** `ctx.state[id + "Index"]` is the 0-based loop counter, available in `while` expression and inner steps.

### `switch` (helper named `switchOn`)

`switch` is a JS reserved word, so the helper function is named `switchOn`. The JSON shape stays `switch:` (where it's not a keyword).

```ts
export interface SwitchCase {
  /** Match value. Strict equality (`===`). Array means "any of these". */
  when: string | number | boolean | (string | number | boolean)[];
  do: V2Step[];
}

export interface SwitchOpts {
  id: string;
  /** Value to switch on. Literal or `$` proxy. */
  on: string | unknown;
  cases: SwitchCase[];
  /** Run when no case matches. Omitted = no-op (skip the switch step). */
  default?: V2Step[];
  active?: boolean;
  stop?: boolean;
}

export function switchOn(opts: SwitchOpts): SwitchStep;
```

**Compiled output:** `{ id, switch: { on, cases, default } }` — JSON shape uses `switch:` key (acceptable since JSON keys aren't reserved-word constrained).

### `tryCatch`

```ts
export interface TryCatchOpts {
  id: string;
  /** Run first. If any step throws or returns an error, switch to catch[]. */
  try: V2Step[];
  /** Runs when try[] errors. Has access to `$.error`. */
  catch: V2Step[];
  /** Always runs, after try[] succeeds OR after catch[] finishes (even on catch errors). */
  finally?: V2Step[];
  active?: boolean;
  stop?: boolean;
}

export function tryCatch(opts: TryCatchOpts): TryCatchStep;
```

**`$.error` proxy:** new proxy variable accessible only inside `catch[]`. Shape: `{ message: string, code?: number, name?: string, stack?: string, cause?: unknown }`. Resolves at expression-evaluation time inside catch steps.

### Middleware shape

Two flavors of middleware:

#### Middleware as a workflow

```ts
import { middleware } from "@blokjs/helper";

export default middleware({
  name: "auth-check",
  version: "1.0.0",
  description: "JWT auth middleware",
  steps: [
    // Same step shape as a regular workflow.
    // Use `stop: true` on a step to short-circuit the chain.
  ],
});
```

`middleware()` is a thin wrapper around `workflow({...})` that sets `middleware: true` and skips the `trigger:` requirement. Compiled output:

```ts
{
  name: "auth-check",
  version: "1.0.0",
  description: "...",
  middleware: true,
  steps: [...],
}
```

#### Middleware references on triggers / workflows

```ts
trigger: {
  http: {
    method: "POST",
    path: "/api/process",
    middleware: ["auth-check", "rate-limit"],   // string[] of middleware workflow names
  },
}

// OR on the workflow:
workflow({
  name: "Admin Delete",
  trigger: { http: { ... } },
  middleware: ["auth-check", "admin-only", "audit-log"],
  steps: [...],
});

// OR process-global:
import { WorkflowRegistry } from "@blokjs/runner";
WorkflowRegistry.getInstance().useGlobalMiddleware(["request-id", "audit-log"]);
```

Resolution order at run-entry: process-global → trigger-level → workflow-level → workflow steps. Stop semantics: any middleware step with `stop: true` short-circuits the chain (response sent, main steps skipped).

## Workflow normalizer extensions

The discriminator chain in `normalizeWorkflow()` gains four new branches, in this order:

```ts
// Existing v0.4:
if (isPlainObject(step.branch))               { /* normalize branch */ }
if (typeof step.subworkflow === "string")     { /* normalize subworkflow */ }
if (isPlainObject(step.wait))                 { /* normalize wait */ }

// New v0.5:
if (isPlainObject(step.forEach))              { /* normalize forEach */ }
if (isPlainObject(step.loop))                 { /* normalize loop */ }
if (isPlainObject(step.switch))               { /* normalize switch */ }
if (isPlainObject(step.tryCatch))             { /* normalize tryCatch */ }

// Fallback:
const { internalStep, nodeConfig } = normalizeRegularStep(step, ...);
```

Each `normalizeXStep()` function follows the same pattern as `normalizeBranchStep()` (already shipped):

1. Validate the shape (throw `MissingFieldError` on missing required fields).
2. Recursively normalize inner steps via `normalizeStepList(steps, sourcePath)` (a new shared helper that runs the discriminator chain).
3. Return `{ internalStep, nodeConfig, innerNodes }` where `innerNodes` is the bubbled-up config map (so inner step inputs can be looked up via `ctx.config[innerStepName]`).

The top-level `normalizeWorkflow()` merges every primitive's `innerNodes` into `internalNodes` via `Object.assign()` — same pattern we shipped for `branch` in the recent fix.

## Internal node implementations

Each primitive has a runtime node in `core/runner/src/nodes/`. They use `defineNode({ flow: true, ... })` so they integrate with the existing flow-node machinery (`processFlow()` → returns `NodeBase[]` of sub-steps to recurse into).

### `ForEachNode`

```ts
// core/runner/src/nodes/ForEachNode.ts
import { defineNode } from "../defineNode";
import { z } from "zod";
import type { Context, NodeBase } from "@blokjs/shared";
import { Runner } from "../Runner";

export const ForEachNode = defineNode({
  name: "@blokjs/forEach",
  description: "Iterate over a collection, sequential or parallel.",
  flow: true,
  input: z.object({
    in: z.string(),
    as: z.string(),
    mode: z.enum(["sequential", "parallel"]),
    concurrency: z.number().int().min(1).max(1000),
    steps: z.array(z.any()),  // pre-resolved to NodeBase[] by the runner
  }),
  output: z.array(z.any()),

  async execute(ctx, input) {
    const items = resolveExpr(input.in, ctx);
    if (!Array.isArray(items)) {
      throw new Error(`forEach.in must resolve to an array, got ${typeof items}`);
    }
    if (items.length === 0) return [];

    const runner = new Runner(input.steps as NodeBase[]);
    const results: unknown[] = [];

    if (input.mode === "sequential") {
      for (let i = 0; i < items.length; i++) {
        const childCtx = createIterationContext(ctx, input.as, items[i], i);
        await runner.run(childCtx);
        results.push(childCtx.response.data);
      }
    } else {
      // Parallel with bounded concurrency (p-limit pattern, in-tree to avoid dep)
      const sema = new Semaphore(input.concurrency);
      await Promise.all(items.map(async (item, i) => {
        await sema.acquire();
        try {
          const childCtx = createIterationContext(ctx, input.as, item, i);
          await runner.run(childCtx);
          results[i] = childCtx.response.data;  // index-preserving
        } finally {
          sema.release();
        }
      }));
    }

    return results;
  },
});
```

**Implementation notes:**

- `createIterationContext()` clones `ctx`, sets `state[as] = item`, sets `state[as + "Index"] = i`. Doesn't share mutations with parent (per-iteration isolation).
- Per-iteration step output lands in the iteration's `state` scope; the iteration's `ctx.response.data` becomes the array element in `results`.
- The forEach step's own output (in parent ctx.state[forEachId]) is the full results array.
- Errors in any iteration propagate up — the parent's error handling (`tryCatch` wrapping the forEach, or parent-level retry) decides what happens.

### `LoopNode`

```ts
// core/runner/src/nodes/LoopNode.ts
export const LoopNode = defineNode({
  name: "@blokjs/loop",
  flow: true,
  input: z.object({
    while: z.string(),
    maxIterations: z.number().int().min(1),
    steps: z.array(z.any()),
  }),
  output: z.unknown(),

  async execute(ctx, input) {
    const runner = new Runner(input.steps as NodeBase[]);
    let iteration = 0;
    let lastResult: unknown = undefined;

    while (true) {
      // Set the loop index BEFORE evaluating while so the condition can use it
      const loopIndexKey = `${ctx.currentStepId}Index`;
      (ctx.state as Record<string, unknown>)[loopIndexKey] = iteration;

      const shouldContinue = evalExpression(input.while, ctx);
      if (!shouldContinue) break;

      if (iteration >= input.maxIterations) {
        throw new LoopMaxIterationsError(
          `Loop ${ctx.currentStepId} exceeded maxIterations=${input.maxIterations}`,
        );
      }

      const childCtx = cloneContextForIteration(ctx);
      await runner.run(childCtx);
      ctx.state = mergeState(ctx.state, childCtx.state);  // carry state forward
      lastResult = childCtx.response.data;
      iteration++;
    }

    return lastResult;
  },
});
```

**Implementation notes:**

- The while-expression evaluation is the same `Mapper` machinery used for step inputs (`Mapper.replaceObjectStrings` etc.).
- State is merged forward so condition variables survive across iterations.
- `LoopMaxIterationsError` extends `BlokError` so callers can `instanceof` discriminate (vs operator-cancel or step timeout).
- Exported from `@blokjs/runner` for consumer use.

### `SwitchNode`

```ts
// core/runner/src/nodes/SwitchNode.ts
export const SwitchNode = defineNode({
  name: "@blokjs/switch",
  flow: true,
  input: z.object({
    on: z.string(),
    cases: z.array(z.object({
      when: z.union([z.string(), z.number(), z.boolean(), z.array(z.union([z.string(), z.number(), z.boolean()]))]),
      steps: z.array(z.any()),
    })),
    default: z.array(z.any()).optional(),
  }),
  output: z.array(z.any()),  // returns the matching arm's steps

  async execute(ctx, input) {
    const value = evalExpression(input.on, ctx);

    for (const c of input.cases) {
      const matches = Array.isArray(c.when) ? c.when.includes(value) : c.when === value;
      if (matches) return c.steps as NodeBase[];
    }

    return (input.default ?? []) as NodeBase[];
  },
});
```

**Implementation notes:**

- Strict equality (`===`) — no coercion.
- First match wins (no fall-through).
- Empty `default` (or omitted) returns `[]` — the runner treats empty flow result as "skip step, advance" (already supported via the empty-array fix in `defineNode`).

### `TryCatchNode` — runner-intercepted

`tryCatch` is the only primitive that's runner-intercepted (not a flow node) because it needs to install error-handling around a step list, which the flow-node return-an-array pattern doesn't give cleanly.

```ts
// core/runner/src/RunnerSteps.ts (extension)
// In the step iteration loop:
if (step.tryCatch) {
  return await this.runTryCatchStep(ctx, step);
}

private async runTryCatchStep(ctx: Context, step: TryCatchInternalStep) {
  const tryRunner = new Runner(step.tryCatch.try);
  const catchRunner = new Runner(step.tryCatch.catch);
  const finallyRunner = step.tryCatch.finally ? new Runner(step.tryCatch.finally) : null;

  let caughtError: Error | null = null;
  let finalResponse: ResponseContext;

  try {
    await tryRunner.run(ctx);
    finalResponse = ctx.response;
  } catch (err: unknown) {
    caughtError = err instanceof Error ? err : new Error(String(err));
    // Expose $.error to catch[] via ctx
    (ctx as unknown as Record<string, unknown>).error = serializeError(caughtError);
    try {
      await catchRunner.run(ctx);
      finalResponse = ctx.response;
    } catch (catchErr: unknown) {
      // Catch block itself threw — this propagates up after finally runs.
      caughtError = catchErr instanceof Error ? catchErr : new Error(String(catchErr));
      finalResponse = { success: false, data: null, error: caughtError };
    }
  } finally {
    if (finallyRunner) {
      try {
        await finallyRunner.run(ctx);
      } catch (finallyErr) {
        // Finally errors are logged but don't override the prior error (matches JS semantics).
        ctx.logger.error(`tryCatch.finally threw: ${finallyErr}`);
      }
    }
  }

  if (caughtError && !ctx.response.success) throw caughtError;
  return finalResponse;
}
```

**Implementation notes:**

- `$.error` proxy resolves to `ctx.error` set inside the catch path.
- Cleared after `finally` completes so downstream steps don't see stale error state.
- `finally` ALWAYS runs, even if catch throws (matches JS `try/catch/finally` semantics).
- If catch succeeds (no rethrow), the tryCatch step succeeds — error is consumed.
- If catch rethrows, the tryCatch step fails, error propagates to the parent step's error handling.

## Helper nodes needed

The 15 examples reference fictional nodes. For real workflows we need a small library of generic helpers. Each is a thin `defineNode` wrapper. Naming: `@blokjs/<verb-noun>`.

| Node | Purpose | Inputs | Outputs |
|---|---|---|---|
| `@blokjs/expr` | Evaluate a JS expression in ctx, return result | `{ expression: string }` | `unknown` |
| `@blokjs/ctx-publish` | Set `ctx.state[name] = value` (alternative to step-default-store for explicit naming) | `{ name: string, value: unknown }` | `{ name, value }` |
| `@blokjs/ctx-publish-many` | Set multiple `ctx.state[k] = v` in one step | `{ values: Record<string, unknown> }` | `{ count: number }` |
| `@blokjs/throw` | Force-throw an error with a custom message | `{ message: string, code?: number }` | (never returns) |
| `@blokjs/log` | Emit a log line at a level | `{ level: "info" \| "warn" \| "error", message: string }` | `{ level, message }` |
| `@blokjs/audit-log` | Append an event to an in-memory audit ring (or a real store via `BLOK_AUDIT_STORE` env) | `{ event: string, ...attrs }` | `{ event, attrs, timestamp }` |
| `@blokjs/metrics-emit` | Emit a metric (no-op without OTel exporter) | `{ event: string, value?: number, attrs?: Record<string, unknown> }` | `{ event, attrs }` |
| `@blokjs/in-memory-kv` | Simple in-memory key-value store for testing | `{ action: "get" \| "set" \| "delete" \| "list", key?: string, value?: unknown }` | depends on action |
| `@blokjs/json-schema` | Validate data against a JSON schema | `{ schema: object, data: unknown }` | `{ valid: boolean, errors: string[] }` |
| `@blokjs/respond` | Already exists — terminal HTTP response | `{ status, body, headers? }` | `{ status, body, headers }` |
| `@blokjs/api-call` | Already exists — generic HTTP fetch | `{ url, method, headers, body, ... }` | response-shaped |

The new ones (`expr`, `ctx-publish`, `ctx-publish-many`, `throw`, `log`, `audit-log`, `metrics-emit`, `in-memory-kv`, `json-schema`) ship in a new package: `@blokjs/helpers` (or fold into `@blokjs/runner` as defineNode constants — TBD).

**Recommendation:** new monorepo package `nodes/utility/helpers@1.0.0/` that exports all of them. Mirrors the existing pattern (`@blokjs/api-call`, `@blokjs/if-else`, `@blokjs/react`).

## Open design questions (decisions before code)

These need a call from the user before we start coding so we don't backtrack.

### Q1. `switch` helper function name

JS reserved word means we can't use `switch()`. Options:

| Name | Pros | Cons |
|---|---|---|
| `switchOn({...})` | Preserves "switch" word, intent clear | Verb form is unusual |
| `match({...})` | Pattern-matching language convention | Different word from JSON shape |
| Just use object form `{ id, switch: {...} }` | Zero new word | No autocomplete on the helper call |

**Recommendation:** `switchOn`. Same word the JSON uses, no collision.

### Q2. `forEach` variable scoping

When `as: "item"` is set, where does `ctx.state.item` live? Options:

| Approach | Detail |
|---|---|
| **Per-iteration scope** | Inner steps see `ctx.state.item`; clears after the forEach. Inner step IDs that match `as` collide. |
| **Namespaced** | `ctx.iter.item` — separate from `ctx.state`. No collision but adds a new namespace. |
| **Documented convention** | Per-iteration scope with rule: "don't use the same name as a step `id`." |

**Recommendation:** documented convention (the third option). `ctx.state.item` is the most natural mapping. Document the no-collision rule. Lint rule in v0.6 if it bites users.

### Q3. `loop` counter sugar

Should we add `count: 10` for "do this 10 times"? Or only `while`?

**Recommendation:** `while` only for v0.5. `count` is `loop({ while: '$.state.iIndex < 10', do: [...] })` — slightly verbose but unambiguous. Add `count` sugar in v0.6 if users ask.

### Q4. `tryCatch` — does an error in `catch` propagate after `finally`?

JS semantics: yes. The error propagates from `catch` after `finally` finishes. We match.

If `finally` itself throws, JS replaces the original error. We diverge: log the finally error, propagate the original. Reason: original errors are usually more actionable for debugging.

**Recommendation:** as stated above.

### Q5. Middleware vs sub-workflow — crisp distinction?

Both are workflows. Both are looked up in `WorkflowRegistry`. Both run during request handling. What's the difference?

| Aspect | Sub-workflow | Middleware |
|---|---|---|
| **Invocation** | Explicit step in parent (`{subworkflow: "name"}`) | Implicit chain at run-entry |
| **Context** | Isolated child ctx (own `state`, own `request.body` from inputs) | Same ctx as main workflow (state shared) |
| **Order** | Inline at the step's position in `steps[]` | Always before main steps |
| **Short-circuit** | Returns to parent step | `stop: true` aborts the whole request |
| **Use case** | Encapsulate composable business logic (signup → email → profile) | Cross-cutting concerns (auth, rate-limit, audit) |

**Recommendation:** crisply documented in `docs/d/composition/middleware.mdx`. The `middleware: true` flag in the workflow envelope is the formal distinguishing marker — middleware-marked workflows can't be invoked as sub-workflows and vice versa.

### Q6. `$.error` proxy availability

Should `$.error` be available everywhere or only inside `catch[]`?

**Recommendation:** only inside `catch[]`. Outside, it's `undefined`. Resolved at expression-evaluation time — the runner sets `ctx.error` when entering a catch block and clears it on exit. Documented.

Alternative considered: `$.tryError` (more explicit, doesn't shadow potentially-named `error` keys in user data). Slightly safer but unfamiliar. Going with `$.error` because it's the obvious choice.

### Q7. Helper nodes package — bundled with `@blokjs/runner` or separate?

The new helpers (`expr`, `ctx-publish`, etc.) — should they be:

| Option | Pros | Cons |
|---|---|---|
| Separate package `@blokjs/helpers` | Clean dependency boundary, can version independently | Another package to install |
| Bundled in `@blokjs/runner` exports | Always available, no extra install | Bloats runner exports |
| Bundled in scaffold | Only present in scaffolded projects | Not available to library consumers |

**Recommendation:** separate package `nodes/utility/helpers@1.0.0/` published as `@blokjs/helpers`. Mirrors the existing `@blokjs/api-call`, `@blokjs/if-else`, `@blokjs/react` pattern. Add to scaffold dependencies by default.
