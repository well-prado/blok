This page documents how to ship a new Blok release end-to-end. Every release publishes 15 packages to npm in a coordinated bump and validates the published surface against a real `npx blokctl create project` smoke test.

The release script at [scripts/release.ts](https://github.com/deskree-inc/blok/blob/main/scripts/release.ts) automates the publish step and runs pre-flight checks for the most common drift modes. The surrounding context — when to bump, what to tag, how to validate — is documented here.

## The 15 publishable packages

Pre-1.0 Blok publishes 15 packages to npm. Every other package in the monorepo (the `templates/*` SDKs, `@blokjs/studio`, `@blokjs/syntax`, `@blokjs/lsp-server`, `@blokjs/trigger-http`, the scaffold private packages) is `private: true` and stays off npm.

The list expanded from 8 to 15 in v0.6.0. `trigger-http`'s source (which the CLI scaffolds into every new project) imports `@blokjs/{trigger-sse, trigger-webhook, trigger-websocket, helpers}` directly — scaffolded projects cannot `bun install` unless all four are published. `@blokjs/trigger-{cron, grpc, pubsub}` ship alongside for v0.6 surface parity even though they're not transitively required by the HTTP scaffold.

## Lockstep versioning

Pre-1.0 Blok bumps all 15 packages **in lockstep**. There is one Blok version — `v0.6.0` means every public package is at `0.6.0`. We do not ship `@blokjs/runner@0.7.0` while `@blokjs/helper` stays at `0.6.0`.

Why: lockstep keeps the surface coherent for users (one tag, one mental model) and avoids the cross-package-dep-range trap that bit the v0.4 release. When `@blokjs/runner` moves from `0.4.0` to `0.6.0`, every dep range across the monorepo also has to move from `^0.4.0` to `^0.6.0` — otherwise bun resolves cross-package deps to the npm registry's stale version instead of the workspace, and `nx run-many -t test` starts failing with extensionless-import errors.

The release script enforces this — it refuses to publish if any cross-package `@blokjs/*` dep range across the workspace doesn't include the target version, OR if the dep isn't using the `workspace:*` protocol (v0.6.0+).

### `@blokjs/core` floats independently (#380)

`@blokjs/core` is the **exception** to lockstep. It is the leaf author barrel — nothing internal depends on it, and it re-exports the engine — so it can ship a new (minor) version on its own without dragging the trigger/node/CLI surface. The pre-flight is a **hybrid**:

- `checkLockstepVersion` validates one shared version for every package **except** `@blokjs/core`.
- `checkCrossDepRanges` resolves a dep on `@blokjs/core` against **core's own** version, everything else against the lockstep version — using real caret/tilde semantics, so a trigger pinning `@blokjs/core@^1.1.0` accepts a floated `1.2.0` but **fails** a `2.0.0` major drift.
- `checkCliConstants` requires the scaffold's `BLOKJS_DEP_RANGE` to admit **both** the lockstep version and core's version (a fresh project pins both).

To float core: bump only `core/core/package.json`, keep the rest lockstepped, and re-run `bun run release --dry-run` — it prints `Floating: @blokjs/core @ <version>`. The pure pre-flight functions live in `scripts/release-preflight.ts` and are unit-tested in `packages/cli/tests/release-preflight.test.ts` (no npm/git needed). Floating core across a **major** requires bumping the scaffold's `GITHUB_REPO_RELEASE_TAG`/`BLOKJS_DEP_RANGE` too, or pre-flight blocks the publish.

## Normal flow — per-PR changesets

Most releases follow the changesets workflow:

1. **Per PR**, run `bunx changeset` from any PR that touches a public package. Authors classify as `patch` / `minor` / `major` per package.
2. **At release time**, on a release branch, run `bunx changeset version`. This consumes the `.changeset/*.md` entries, applies the version bumps, and updates cross-package dep ranges automatically.
3. **Bump the CLI scaffold constants** in [packages/cli/src/commands/create/project.ts](https://github.com/deskree-inc/blok/blob/main/packages/cli/src/commands/create/project.ts):
   - `GITHUB_REPO_RELEASE_TAG` — the git tag `blokctl create project` clones from
   - `BLOKJS_DEP_RANGE` — the dep range hardcoded into scaffolded `package.json`s
4. **Tag**: `git tag -a v<version> -m "<release notes>"`.
5. **Validate**: see "Local-pack validation" below.
6. **Publish**: `bun run release --otp <code> --yes`.
7. **Push**: `git push && git push --tags`.

Per-PR changesets are the discipline that keeps the registry in sync with `main`. The v0.4 release accumulated ~80 commits without any changesets — the catch-up bump below was the price.

## Catch-up flow — when changesets math doesn't apply

If the registry has fallen far behind `main` (e.g. `@blokjs/helper@0.2.1` on npm but the workspace is at the API surface for `0.4.0`), `bunx changeset version` won't bridge the gap cleanly. Skip it and bump manually:

1. **Manual lockstep bump**: edit each of the 15 public `package.json`s to the target version directly.
2. **Cross-package dep alignment**: every `@blokjs/<published-package>` dep range across the monorepo (including private trigger packages) must include the new version. **Preferred since v0.6.0**: switch internal deps to `"workspace:*"` so bumps don't propagate to every consumer manually — the release script accepts both. For literal ranges: if `@blokjs/runner` was at `^0.2.0`, every consumer becomes `^0.4.0` (then `^0.6.0` for v0.6). Without this, bun resolves cross-package deps to the npm registry's stale version instead of the workspace.
3. **CLI scaffold constants**: bump both `BLOKJS_DEP_RANGE` and `GITHUB_REPO_RELEASE_TAG` in [packages/cli/src/commands/create/project.ts](https://github.com/deskree-inc/blok/blob/main/packages/cli/src/commands/create/project.ts).
4. **Delete consumed changesets**: any `.changeset/*.md` accumulated during the gap is now redundant — the manual bump replaces them. Delete to avoid double-bumping on the next changesets run.
5. **Lockfile**: `bun install` regenerates `bun.lock` with the new ranges.
6. **Tag**: `git tag -a v<version> -m "<release notes>"`.
7. **Pre-flight**: `bun run release --dry-run` verifies lockstep, cross-dep ranges, CLI constants, git tag, and a clean tree.
8. **Publish**: `bun run release --otp <code> --yes`.

## Local-pack validation

Before invoking `npm publish` (which is irreversible after 72h), validate the publish surface against a fresh scaffold:

```bash
mkdir -p /tmp/blok-tgz
for dir in core/shared core/workflow-helper core/runner \
           nodes/web/api-call@1.0.0 nodes/control-flow/if-else@1.0.0 \
           nodes/web/react@1.0.0 nodes/utility/helpers@1.0.0 \
           triggers/worker triggers/sse triggers/websocket \
           triggers/webhook triggers/pubsub triggers/cron triggers/grpc \
           packages/cli; do
  (cd "$dir" && npm pack --pack-destination /tmp/blok-tgz)
done
```

Then scaffold a project via the local CLI build, override the scaffold's `@blokjs/*` deps to point at the `.tgz` files via `package.json` `overrides`, and `npm install && npm run dev`. If the trigger boots and `GET /__blok/health` returns structured JSON, the publish surface is valid.

This catches missing files in `package.json:files`, missing exports in `package.json:exports`, broken type declarations, and stale dep ranges — without committing irreversible npm publishes.

## Running the release script

```bash
bun run release --dry-run              # pre-flight + plan, no publish
bun run release --otp 123456 --yes     # publish all 15 with OTP
bun run release --otp 123456 --yes --skip-tests --skip-build  # if validated separately
```

What it does:

- Reads all 15 publishable `package.json`s and asserts versions match (lockstep).
- Walks every `package.json` in the workspace and asserts each `@blokjs/<published-package>` dep range either is `workspace:*` or includes the workspace version literally.
- Reads `packages/cli/src/commands/create/project.ts` and asserts `BLOKJS_DEP_RANGE` and `GITHUB_REPO_RELEASE_TAG` match the workspace version.
- Asserts the local git tag `v<version>` exists.
- Asserts the working tree is clean.
- Optionally runs `bun run build` and `bunx nx run-many -t test` (skip with flags).
- Publishes the 15 in dependency order using a single OTP.

What it does **not** do:

- It does not bump versions, push tags, or commit. Those are deliberate human steps — the script's role is to turn an already-bumped, already-tagged repository into a published release with a single command.
- It does not publish under a non-default dist-tag (e.g. `next`). Edit the script if you need that.
- It does not publish private packages.

## OTP batching

`npm publish` requires a TOTP code per call. The TOTP window is ~30 seconds, which is enough to publish all 15 packages in a single batch with one OTP. Pass the same code via `--otp` and the script runs all 15 publishes within the window.

If a publish fails mid-batch (expired OTP, network blip), the script reports which packages succeeded. Re-run with a fresh OTP — already-published versions fail with `EPUBLISHCONFLICT` which is benign and the script will continue past them.

## Post-publish smoke test

After the script reports all 15 published, validate against real npm from a cold environment:

```bash
mkdir /tmp/blok-real && cd /tmp/blok-real
npx blokctl@latest create project --name myapp \
  --triggers http --runtimes node --package-manager npm
cd myapp && npm run dev
```

Expected: `blokctl@<new-version>` installs from real npm, scaffold clones the new tag, `npm install` resolves all `@blokjs/*` to the new version, and the trigger boots without `SyntaxError: Export named 'workflow' not found in module '@blokjs/helper'` or similar.

If the smoke test fails, the most common cause is a cross-package dep range still pointing at an older version — check the scaffolded project's `package.json` and the `BLOKJS_DEP_RANGE` constant in `project.ts`.

## Pre-publish checklist

The release script verifies the first four automatically:

- [ ] All 15 `package.json` files have the target version.
- [ ] Every `@blokjs/<published-package>` dep range across the monorepo includes the target version.
- [ ] `BLOKJS_DEP_RANGE` and `GITHUB_REPO_RELEASE_TAG` in `packages/cli/src/commands/create/project.ts` match.
- [ ] The git tag `v<version>` exists locally.
- [ ] `bunx nx run-many -t test` is green across all projects.
- [ ] `bun run build` produces a clean dist for every public package.
- [ ] A local-pack scaffold smoke test passes.

## Versioning conventions

- Pre-1.0: lockstep bumps across all 15 packages. `0.X.0` for breaking changes; `0.X.Y` for additive changes and bug fixes.
- Tags are annotated (`git tag -a`) with the full release notes embedded in the tag message. `git show v<version>` should produce a complete changelog.
- Tags are pushed manually — `git push --tags` after the post-publish smoke test confirms a successful release.

## When a release goes wrong

- **Wrong content published**: npm allows unpublishing within 72 hours via `npm unpublish <pkg>@<version>`. After 72h, deprecate (`npm deprecate`) and ship a fix in the next patch.
- **One package failed mid-batch**: re-run the script with a fresh OTP — succeeded packages fail benignly with `EPUBLISHCONFLICT`.
- **Tag points at the wrong commit**: delete locally (`git tag -d v<version>`), retag, force-push the tag (`git push --tags --force`). This is one of the few cases where force-push on tags is acceptable.
- **Workspace and registry diverge after partial publish**: bump the patch version, retag, run the script again. Don't try to "patch over" a partial publish by re-publishing the same version — npm refuses.
