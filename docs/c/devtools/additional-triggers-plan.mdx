## The problem

Blok ships nine trigger packages under `triggers/`. Today their state is uneven:

| Trigger | State | What works |
|---|---|---|
| `http` | Production-ready | Full HTTP server via Hono. Middleware, concurrency, scheduling, raw body, sub-workflows, all of it. |
| `worker` | Production-ready | Background jobs via NATS JetStream / BullMQ / InMemory adapters. Pluggable adapter pattern is correct. |
| `cron` | Production-ready | Scheduled execution via the `cron` npm package. Single-process. |
| `grpc` | Production-ready | But as a **transport** for cross-language nodes (Python/Go/Rust/etc.), not a workflow entry trigger. |
| `websocket` | Abstract class only | `WebSocketTrigger extends TriggerBase` exists, but no concrete subclass, no WS library wired in, no `bun run ws:dev`. The schema validates `trigger: { websocket: {...} }` but nothing picks it up. |
| `sse` | Abstract class only | Same shape. SSE is HTTP streaming so this should be trivially mountable on the existing Hono server — but isn't. |
| `webhook` | Abstract class only | Today, authors implement webhook intake by writing an HTTP workflow with a signature-verify middleware (e.g. `github-webhook-verify.json` shipped in v0.6). Works but lacks the "first-class trigger" ergonomics. |
| `queue` | Adapter scaffolds exist, **but → see Q1 resolution** | Files at `triggers/queue/src/adapters/{Kafka,RabbitMQ,SQS,Redis,NATS}Adapter.ts` but no `QueueTrigger.listen()` boot path. **Per Q1 below: this trigger name is dropped; the adapters move into the existing `worker` trigger.** |
| `pubsub` | Adapter scaffolds exist | GCP / AWS / Azure provider scaffolds but no integrated dev experience. |

The authoring surface (workflow JSON / TS DSL) accepts every trigger type today — the schema in [TriggerOpts.ts](https://github.com/deskree-inc/blok/blob/main/core/workflow-helper/src/types/TriggerOpts.ts) is comprehensive. The integration gap is at the trigger boot layer: nothing scans workflows with `trigger: { websocket: {...} }` and brings up a WebSocket server.

**The goal of this plan:** close the gap so authors can write the JSON line and have the workflow respond. No need to subclass anything, write infrastructure code, or run a separate dev server per trigger.

## What I learned from other frameworks

Before designing the Blok surface, three parallel research passes surveyed the existing ecosystem. The relevant takeaways:

### Established frameworks: Laravel, Rails, NestJS, AdonisJS

The strongest patterns:

**Laravel Broadcasting + Reverb (WebSocket).** Split between three artifacts: an event class declaring `broadcastOn(): Channel[]`, a `routes/channels.php` file holding `Broadcast::channel($pattern, $authClosure)` for authorization, and the Reverb server that handles the WS protocol. The author never writes WS handshake code — they `dispatch($event)` and the broadcaster fans out to authorized subscribers. Pattern strings (`orders.{orderId}`) use route-style placeholders with route-model-binding-equivalent semantics.

**Spatie WebhookClient (Laravel).** The cleanest webhook-as-config surface across all the frameworks I surveyed:
```php
'configs' => [[
    'name'                  => 'stripe',
    'signing_secret'        => env('STRIPE_WEBHOOK_SECRET'),
    'signature_header_name' => 'Stripe-Signature',
    'signature_validator'   => DefaultSignatureValidator::class,
    'webhook_profile'       => ProcessEverythingWebhookProfile::class,
    'process_webhook_job'   => ProcessStripeWebhookJob::class,
]]
```
Every step is a swappable class. Adding a new webhook source = one config entry + (if signature scheme differs) one `SignatureValidator` implementation.

**Rails ActionCable / Solid Cable.** Channels are classes under `app/channels/` with `subscribed`, `unsubscribed`, `received` callbacks. **Turbo Streams** layers an even higher-level declarative pattern on top: `broadcasts_to :room` on the model installs three lifecycle callbacks that auto-broadcast partials to a stream key. Rails 8's Solid stack runs the whole thing in-process (no separate worker).

**NestJS WebSocket Gateways + `@Sse()` decorator.** Pure decorator-driven authoring. A `@WebSocketGateway()` class collects `@SubscribeMessage('chat')` handler methods. The `@Sse('notifications')` decorator on a controller method returns `Observable<MessageEvent>` — *the same controller class mixes `@Get`, `@Post`, AND `@Sse`*, the SSE response is "just another route." This is the cleanest "SSE = HTTP streaming, don't reinvent it" model I saw.

**NestJS Microservices (Queue + Pub/Sub).** One abstraction (`@MessagePattern` for RPC, `@EventPattern` for fan-out) drives every transport — Redis, NATS, Kafka, MQTT, RabbitMQ, gRPC. Transport is selected at bootstrap. Routing patterns use wildcards (`'time.us.*'` for NATS, etc.). The pattern split is meaningful: **request-reply** (one consumer wins) vs **pub-sub** (all subscribers receive).

### Modern TS-first frameworks: Hono, Trigger.dev, Inngest, CF Workers, Vercel, tRPC

The strongest patterns:

**Hono's `upgradeWebSocket()`.** WebSocket routes look like HTTP routes:
```ts
app.get('/ws/:room', upgradeWebSocket((c) => ({
  onOpen:    (_, ws) => ws.send('hi'),
  onMessage: (e, ws) => ws.send(`echo: ${e.data}`),
  onClose:   ()      => console.log('bye'),
})))
```
Same `app`, same middleware chain (jwt-auth applies to the upgrade request), same path-routing tree. `@hono/node-ws` provides `injectWebSocket(server)` to hook into the existing `http.Server` from `@hono/node-server`'s `serve()`.

**Hono's `streamSSE()`.** Long-lived GET handler with structured event writes:
```ts
app.get('/events', (c) => streamSSE(c, async (stream) => {
  stream.onAbort(() => sub.close());
  for await (const evt of upstream) {
    await stream.writeSSE({ id: String(evt.seq), event: evt.type, data: JSON.stringify(evt) });
  }
}))
```
Zero extra deps. Honors `Last-Event-Id` for reconnect.

**Inngest's event-first model.** Function declares its triggers inline; ONE function can wake on MULTIPLE triggers (events + crons mixed):
```ts
export const onboard = inngest.createFunction(
  {
    id: 'onboard-user',
    triggers: [
      { event: 'user.signup' },
      { event: 'user.reinvited' },
      { cron: 'TZ=Europe/Paris 0 12 * * 5' },
    ],
    throttle: { limit: 3, period: '1min' },
  },
  async ({ event, step }) => { ... },
)
```

**Inngest's webhook intake.** Author writes a small transform function (`(evt, headers) => ({name, data, id})`) in the Inngest dashboard. The transform runs in Inngest's cloud, normalizes the webhook payload into an event, and downstream functions trigger on the event name. Author never touches HTTP routing OR signature verification (Inngest handles signing for known providers).

**Cloudflare Workers Hibernation API.** Decouples connection identity from handler runtime:
```ts
this.state.acceptWebSocket(pair[1]);          // hibernation-mode WS
pair[1].serializeAttachment({ userId });      // 2KB per-conn scratchpad survives hibernation

async webSocketMessage(ws, msg) {             // invoked per-message, not per-connection
  const { userId } = ws.deserializeAttachment();
  // ...
}
```
Per-connection state survives without keeping the process hot. Billable duration stops between messages. The cleanest "stateless WS triggers" abstraction in production today.

**Cloudflare Workers default-export handler keys.** Single module with named keys for each trigger surface:
```ts
export default {
  async fetch(req, env, ctx)     { /* HTTP */ },
  async scheduled(c, env, ctx)   { /* Cron */ },
  async queue(batch, env, ctx)   { /* Queue */ },
  async email(message, env, ctx) { /* Email Workers */ },
};
```
The runtime dispatches based on what triggered. One module, multiple trigger surfaces, single source of typed bindings.

**tRPC `subscription()`.** Typed async-generator subscriptions over SSE or WebSocket transport. `tracked(id, data)` + the `Last-Event-Id` header give at-least-once resume from disconnect:
```ts
export const appRouter = router({
  onPostAdd: t.procedure
    .input(z.object({ topic: z.string() }))
    .subscription(async function* (opts) {
      for await (const [post] of on(ee, 'post:add', { signal: opts.signal })) {
        yield tracked(post.id, post);
      }
    }),
});
```

### Cross-cutting insights

Three observations recur across all the frameworks worth stealing:

1. **Pattern-matched channel/topic/queue strings are universal.** `orders.{orderId}` (Laravel), `time.us.*` (NestJS NATS), `users/:id` (Adonis Transmit), `chats/{roomId}/messages` (Reverb). Wildcards (`*`, `>`, `:name`) and brace placeholders are the lingua franca. Authors think in patterns, not concrete names.

2. **Authorization co-locates with the channel definition, not the transport.** Laravel's `Broadcast::channel($pattern, $authClosure)`, Adonis's `transmit.authorize($pattern, $cb)`, Rails's per-channel `subscribed` method. Auth is a CHANNEL property, not a SERVER property.

3. **Events, queues, and webhooks are distinct concepts, but normalize to "an event" once they're inside the runtime.** Inngest's webhook transform is the cleanest example — a Stripe POST becomes `stripe/invoice.paid` event, downstream functions trigger on the event name. The wire-level intake (signature verify, HMAC check, queue ack) is separated from the workflow's logical trigger.

## Design principles for Blok

These principles guide every per-trigger choice below:

### P1 — Workflow JSON / TS DSL is the source of truth

Authors should NEVER need to write a class subclass, register middleware imperatively, or boot a separate dev server. The workflow JSON line `trigger: { websocket: { path: "/ws/chat" } }` makes the workflow live as a WebSocket route. Period.

### P2 — Share the Hono port whenever possible

Same TCP port, same Hono app, same middleware chain, same Studio trace router. The HTTP trigger already owns port 4000 in dev. Triggers that CAN coexist on Hono should mount on the same `Hono` instance: WebSocket (via `@hono/node-ws`), SSE (via `hono/streaming`), and Webhook (just a POST route).

Triggers that CAN'T (Queue consumers, Pub/Sub subscribers — they hold long-lived broker connections, not HTTP) run as worker processes via the existing `WorkerTrigger` pattern. Producers (publishing to a queue or topic) CAN be Hono routes.

### P3 — Per-provider plugins, not one-size-fits-all

Webhooks have provider-specific signature schemes (Stripe `t=ts,v1=sig`, GitHub `sha256=hex`, Slack `v0:ts:body`, Shopify `base64(hmac)`). Don't try to unify these — provide a registry of named providers (`provider: "stripe"`), let each be a small plugin implementing a `WebhookVerifier` interface, fall back to Standard Webhooks ([standardwebhooks.com](https://www.standardwebhooks.com/)) for unknown providers.

Same pattern for queues / pubsub: ONE adapter interface (`subscribe`, `publish`, `disconnect`), MANY backend plugins (BullMQ, NATS, Kafka, Redis Streams, SQS, GCP Pub/Sub).

### P4 — Authentication via existing middleware chain

The middleware chain shipped in v0.5/v0.6 (process-global → workflow-level → trigger-level — see [middleware.mdx](./middleware.mdx)) applies UNIFORMLY across trigger kinds:

- **WebSocket upgrade**: Hono middleware sees the upgrade request (headers, cookies) — `jwt-auth` rejects unauthorized upgrades with 401.
- **SSE open**: middleware runs once at stream open.
- **Webhook POST**: signature-verify middleware reads `ctx.request.rawBody`, runs HMAC, sets `ctx.state.verifiedEvent` or returns 401.
- **Queue/Pub-Sub**: middleware can't be HTTP-shaped (no request); auth enforced at the broker level (NATS NKeys, Redis AUTH, Kafka SASL) AND/OR per-message via a workflow-level middleware that validates the message payload.

The JSON authoring surface is identical: `trigger.{kind}.middleware: ["jwt-auth"]` works the same for HTTP, WebSocket, SSE, Webhook.

### P5 — JS Web Standards (Hono philosophy)

Hono picked Fetch API + Web Streams as its foundation. Triggers built on Hono should follow the same philosophy: `Request`/`Response`, `ReadableStream`, `AbortSignal`. This keeps the framework portable to non-Node runtimes (Bun, Deno, Cloudflare Workers) over time.

For triggers that DON'T inherit from Hono (Queue, Pub/Sub consumers), use Node-native primitives but match the same conceptual shape — async iterators for streams, `AbortSignal` for cancellation, `Promise` for unit-of-work.

### P6 — Observability is non-optional

Every trigger emits OTel spans, NodeRun records, and Studio-visible traces. The existing `RunTracker.startRun` / `startNode` / `completeRun` flow already handles this for HTTP/Worker/Cron. New triggers extend the same pattern.

### P7 — Operational primitives (concurrency, scheduling, retries) are uniform

The existing v0.5/v0.6 ops primitives (`concurrencyKey`, `delay`, `ttl`, `debounce`, `retry`, `idempotencyKey`, `maxDuration`) should work on ALL triggers, not just HTTP. The `ConcurrencyOptsFields` / `SchedulingOptsFields` in [TriggerOpts.ts](https://github.com/deskree-inc/blok/blob/main/core/workflow-helper/src/types/TriggerOpts.ts) are designed for cross-trigger reuse.

## The unified workflow trigger model

Every Blok trigger fits one mental model, regardless of transport:

```
Inbound event → [Trigger Auth] → [Trigger Validation] → Workflow Run → [Outbound Response]
```

**Inbound event** is transport-specific:
- HTTP: HTTP request
- WebSocket: connect handshake OR a message on an open socket
- SSE: HTTP GET on the stream URL
- Webhook: signed HTTP POST from a foreign system
- Queue: a delivered message (BullMQ job / NATS message)
- Pub/Sub: a published message on a subscribed topic
- Cron: a timer firing

**Trigger Auth** is uniform — middleware chain runs against whatever request-shaped object the trigger built. For non-HTTP triggers, the trigger constructs a synthetic `ctx.request` from the incoming event.

**Trigger Validation** — schema validation, raw-body capture for webhooks, idempotency cache lookup, concurrency gate, scheduling gate. All shared via `TriggerBase`.

**Workflow Run** — the runner runs the steps as it always has. `ctx.request.body` is the inbound payload (regardless of source); `ctx.state` accumulates step outputs.

**Outbound Response** — transport-specific again:
- HTTP / Webhook: HTTP response from `ctx.response.data`
- WebSocket: zero or more messages written via `ctx.connection.send(...)` (new primitive)
- SSE: zero or more events yielded via `ctx.stream.writeSSE(...)` (new primitive)
- Queue / Pub/Sub: optional follow-up publish via `@blokjs/queue-publish` step (existing pattern)
- Cron: no response (run is logged for ops)

This is the picture an author should have. The differences between triggers are entirely in inbound event shape and outbound response capabilities. Authoring is uniform — same `steps: [...]` body, same middleware chain, same primitives (forEach, tryCatch, wait, etc.).

## Trigger-by-trigger detailed design

### WebSocket trigger

**Authoring surface (JSON):**

```json
{
  "name": "chat-message-handler",
  "trigger": {
    "websocket": {
      "path": "/ws/chat/:roomId",
      "middleware": ["jwt-auth"],
      "events": ["message", "typing", "leave"],
      "concurrencyKey": "$.req.params.roomId",
      "concurrencyLimit": 100,
      "heartbeatInterval": 30000,
      "maxConnections": 10000,
      "messageRateLimit": 100
    }
  },
  "steps": [
    {
      "id": "validate",
      "use": "@blokjs/expr",
      "inputs": {
        "expression": "ctx.request.body.event === 'message' ? { valid: true } : { valid: false }"
      }
    },
    {
      "id": "broadcast",
      "use": "@blokjs/ws-broadcast",
      "inputs": {
        "channel": "$.req.params.roomId",
        "event": "message",
        "payload": { "from": "$.state.identity.userId", "text": "$.req.body.text" }
      }
    }
  ]
}
```

**Authoring surface (TS DSL):**

```ts
export default workflow({
  name: "chat-message-handler",
  trigger: {
    websocket: {
      path: "/ws/chat/:roomId",
      middleware: ["jwt-auth"],
      events: ["message", "typing", "leave"],
      concurrencyKey: $.req.params.roomId,
    },
  },
  steps: [
    { id: "validate", use: "@blokjs/expr", inputs: { expression: "..." } },
    { id: "broadcast", use: "@blokjs/ws-broadcast", inputs: { ... } },
  ],
});
```

**Library choice:** [`@hono/node-ws`](https://github.com/honojs/middleware/tree/main/packages/node-ws) (depends on `ws`). Mounts on the same `http.Server` as HTTP. Path-routed via the same Hono app.

**Lifecycle hooks (`ctx.connection.on(...)` shape):**

The workflow runs at three lifecycle points, distinguished by `ctx.request.body.event` (Blok-synthetic field):

- `event: "connect"` — fires once on WS upgrade. `ctx.request.headers` has the upgrade request headers. Workflow can return data that gets sent as the first message (or throw to reject the upgrade).
- `event: "message"` — fires per incoming message. `ctx.request.body.data` is the message payload (parsed JSON if `Content-Type` negotiation indicated JSON, otherwise raw string/buffer). Workflow can broadcast / reply via `@blokjs/ws-broadcast` and `@blokjs/ws-reply` helpers.
- `event: "disconnect"` — fires once on WS close. Workflow can do cleanup (publish "user-left" event, free per-room state).

The `events: ["message", "typing", "leave"]` field on the trigger config is an **allowlist** — only matching `event` fields in incoming messages dispatch the workflow. Unmatched messages are dropped silently (or logged at trace level). This matches the Inngest event-matching pattern.

**Per-connection state (`ctx.connection.attachment`):**

WebSocket connections need a small scratchpad — userId derived from the JWT, the room the user joined, etc. — that's available to every workflow run on that connection. The pattern (inspired by Cloudflare's `serializeAttachment`):

```ts
// First workflow run (on "connect"): set the attachment
ctx.connection.setAttachment({ userId: ctx.state.identity.userId, joinedAt: Date.now() });

// Later runs (on "message" or "disconnect"): read it
const { userId, joinedAt } = ctx.connection.attachment;
```

Implementation: an in-memory `WeakMap<WebSocket, unknown>` keyed by the `ws` instance, set/get via `ctx.connection.setAttachment` / `ctx.connection.attachment`. Persistent across the connection's lifetime, GC'd on close.

**Helpers (new nodes to ship):**

- `@blokjs/ws-broadcast` — `inputs: { channel: string, event: string, payload: unknown, exceptSelf?: boolean }`. Sends to all connections matching the channel (room) name. The `exceptSelf` flag is the canonical "send to everyone except the sender" pattern.
- `@blokjs/ws-reply` — `inputs: { event: string, payload: unknown }`. Sends back to the connection that sent the message. Shorthand for `ws-broadcast` with `channel: $.connection.id`.
- `@blokjs/ws-close` — `inputs: { code?: number, reason?: string }`. Closes the connection cleanly. Useful in `connect` workflows for auth failures.

**Heartbeat:** built into the trigger, not the author's concern. Configured via `heartbeatInterval` (default 30s). The trigger sends WS ping frames on the interval; on missed pong (2 × interval), the trigger closes the connection with code 1011 and fires the `disconnect` workflow.

**Rate limiting:** `messageRateLimit` (default 100/sec per connection). Implemented as a per-connection token bucket. Excess messages dropped with a `error: rate_limit` reply.

**Auth at upgrade:** the middleware chain (`jwt-auth`) runs on the HTTP upgrade request. Standard 401 response refuses the upgrade.

**Cross-process broadcast (`@blokjs/ws-broadcast` across instances):**

In-process broadcast is straightforward — a `Map<roomId, Set<WebSocket>>` indexed at upgrade time. For multi-instance deployments, the broadcast helper publishes to a Pub/Sub backplane (NATS subject `blok.ws.broadcast.{roomId}`) which other instances forward to their local connections. Same pattern Adonis Transmit and Laravel Reverb use.

This requires a `BLOK_WS_BACKPLANE` env var pointing at a NATS / Redis instance. Single-instance dev: unset, broadcast stays in-process.

**Observability:**

- One `WorkflowRun` per workflow invocation (per message processed). Multiple workflows = multiple runs.
- Connect / disconnect runs are tagged with `triggerType: "websocket.connect"` / `"websocket.disconnect"` for trace filtering.
- New OTel counters:
  - `blok_websocket_connections_active` (gauge)
  - `blok_websocket_messages_received_total{workflow}`
  - `blok_websocket_messages_dropped_total{workflow, reason}` (`reason ∈ {rate_limit, no_handler, auth_failed}`)
- Studio's run list filter learns a `triggerType: "websocket.*"` shortcut.

**Connection cap:** `maxConnections` (default 10000) is a hard cap on concurrent open WS connections per instance. Exceeded → upgrade refused with `503 Service Unavailable`.

### SSE trigger

**Authoring surface (JSON):**

```json
{
  "name": "live-order-updates",
  "trigger": {
    "sse": {
      "path": "/events/orders/:orderId",
      "middleware": ["jwt-auth"],
      "channels": ["order:{orderId}", "user:{userId}"],
      "heartbeatInterval": 15000,
      "retryInterval": 3000
    }
  },
  "steps": [
    {
      "id": "subscribe",
      "use": "@blokjs/sse-subscribe",
      "inputs": {
        "channels": "$.trigger.channels",
        "lastEventId": "$.req.headers['last-event-id']"
      }
    },
    {
      "id": "stream",
      "use": "@blokjs/sse-stream",
      "inputs": {
        "source": "$.state.subscribe",
        "eventName": "order-update",
        "retryMs": 3000
      }
    }
  ]
}
```

The shape is interesting because SSE is fundamentally "open a connection, then emit events for as long as the connection's alive." It doesn't fit the "request → response" model of HTTP. Two patterns are viable:

**Pattern A — Workflow returns an event source.** The workflow runs once at stream open, returns a stream / async iterator, and the trigger pumps events from it until the connection closes. This is the simpler model — most workflows just dispatch to an existing event bus.

**Pattern B — Workflow re-runs per event.** The trigger subscribes to a source (per `channels` config), and every event on the source triggers a fresh workflow run that decides what to do with the event (filter, transform, send). More flexible but heavier.

**Recommendation: Pattern A as the default, with Pattern B as opt-in.** Pattern A maps cleanly onto Hono's `streamSSE` and async iterators. Author writes a workflow that yields events via `@blokjs/sse-stream` (helper below), the trigger pumps them. Pattern B (per-event workflow runs) is more powerful but should be opt-in via `mode: "per-event"` on the trigger config — out of scope for the initial PR.

**Library choice:** Hono's built-in [`streamSSE`](https://hono.dev/docs/helpers/streaming#streamsse) from `hono/streaming`. Zero extra deps. Honors `Last-Event-Id` for reconnect.

**Helpers (new nodes):**

- `@blokjs/sse-subscribe` — subscribes to one or more channels on the in-process event bus (or NATS / Redis Streams backplane). Returns an async iterator of events. The channels config supports pattern interpolation: `"order:{orderId}"` with the trigger's path params substituted.
- `@blokjs/sse-stream` — consumes the iterator from `@blokjs/sse-subscribe` and writes SSE-framed events via `c.writeSSE(...)`. Each event has `event:`, `data:`, `id:`, and `retry:` fields per the SSE spec.
- `@blokjs/sse-publish` — counterpart of `@blokjs/ws-broadcast`. Publishes an event to a channel that SSE subscribers consume.

**Reconnect handling:** `Last-Event-Id` HTTP header on the inbound request indicates the last event the client received. `@blokjs/sse-subscribe`'s `since: lastEventId` option replays from that point forward (or from the start if the buffer doesn't have it). For long histories, back the event bus with a Redis Stream or NATS JetStream.

**Heartbeat:** `heartbeatInterval` (default 15s) sends an SSE comment line (`:keep-alive\n\n`) on the interval. Prevents proxy idle timeouts (nginx default 60s, ALB 60s).

**Backpressure:** Hono's `streamSSE` awaits `node:stream`'s `drain` event automatically. Slow clients slow the producer naturally.

**Observability:**

- One `WorkflowRun` per stream open (Pattern A). The run lives as long as the stream — status `running` until the connection closes, then `completed`.
- `blok_sse_connections_active{workflow}` gauge.
- `blok_sse_events_sent_total{workflow, channel}` counter.
- Studio's run detail shows the connection's age + events-sent count.

**Auth:** identical to HTTP — `middleware: ["jwt-auth"]` runs on the GET request at stream open. Per-event reauthorization (token expires mid-stream) is out of scope for v1; document.

### Webhook trigger

**Authoring surface (JSON):**

```json
{
  "name": "stripe-invoice-paid",
  "trigger": {
    "webhook": {
      "provider": "stripe",
      "path": "/webhooks/stripe",
      "events": ["invoice.paid", "invoice.payment_failed"],
      "secretEnv": "STRIPE_WEBHOOK_SECRET",
      "tolerance": 300,
      "idempotencyKey": "$.req.headers['stripe-signature']"
    }
  },
  "steps": [
    {
      "id": "log",
      "use": "@blokjs/audit-log",
      "inputs": {
        "event": "$.req.body.type",
        "id": "$.req.body.id"
      }
    },
    {
      "id": "dispatch",
      "subworkflow": "$.req.body.type",
      "inputs": { "stripeEvent": "$.req.body" }
    }
  ]
}
```

**Authoring surface — custom signature scheme (workflow whose provider isn't a built-in):**

```json
{
  "trigger": {
    "webhook": {
      "path": "/webhooks/my-saas",
      "signature": {
        "scheme": "hmac-sha256",
        "header": "X-Acme-Signature",
        "format": "sha256={hex}",
        "secretEnv": "ACME_WEBHOOK_SECRET",
        "tolerance": 600,
        "timestampHeader": "X-Acme-Timestamp"
      }
    }
  }
}
```

For unknown providers, fall back to Standard Webhooks ([standardwebhooks.com](https://www.standardwebhooks.com/)) — uniform HMAC-SHA256 with `webhook-id` / `webhook-timestamp` / `webhook-signature` headers.

**Library choice for built-in providers:**

- Stripe: [`stripe@^17`](https://www.npmjs.com/package/stripe) — `stripe.webhooks.constructEvent(rawBody, sig, secret, tolerance)`.
- GitHub: [`@octokit/webhooks@^13`](https://www.npmjs.com/package/@octokit/webhooks) — `Webhooks#verifyAndReceive({ id, name, signature, payload })`.
- Slack: DIY HMAC SHA-256 of `v0:{ts}:{rawBody}` against `X-Slack-Signature`. No SDK needed.
- Shopify: DIY HMAC SHA-256 base64 against `X-Shopify-Hmac-Sha256`.
- Generic: [`svix@^1`](https://www.npmjs.com/package/svix) for Standard Webhooks.

**Built-in `events: [...]` allowlist:** like WebSocket's events allowlist, this filters which webhook event types reach the workflow. Stripe sends ~150 event types; the workflow only cares about a few. Unmatched events are stored in the audit log (for replay) but don't trigger workflow runs.

**Replay protection:** webhook verifier caches the `eventId` (Stripe's `id`, GitHub's `X-GitHub-Delivery`, Svix's `webhook-id`) in the existing `idempotency_cache` table for a TTL matching the signature tolerance window (default 5 min). On hit: return 200 with `{ status: "duplicate", eventId }` and don't run the workflow.

**Sub-workflow dispatch pattern (the canonical pattern):**

Webhooks usually carry an event type that dispatches to one of many handlers. The cleanest authoring shape:

```json
{
  "trigger": { "webhook": { "provider": "stripe", "path": "/webhooks/stripe" } },
  "steps": [
    {
      "id": "route",
      "subworkflow": "$.req.body.type",
      "inputs": { "stripeEvent": "$.req.body" }
    }
  ]
}
```

The `subworkflow: "$.req.body.type"` field is **polymorphic** — the destination workflow name is resolved at runtime from the event type. This requires `WorkflowRegistry` to support polymorphic name resolution; today it only does static names. Adding this is in scope for the webhook trigger PR.

Each event type gets its own workflow: `invoice.paid`, `customer.subscription.created`, etc. Each handles ONE event type, no switch statement needed. The flagship demo of "this is what Blok is for" — declarative routing all the way down.

**Migration from existing v0.6 pattern:** The shipped `v05-github-webhook-router.json` uses HTTP + middleware for HMAC verify. After the webhook trigger lands, that workflow simplifies to:

```json
// before (v0.6)
{
  "trigger": { "http": { "method": "POST", "middleware": ["github-webhook-verify"] } },
  "steps": [...]
}

// after (v0.7)
{
  "trigger": { "webhook": { "provider": "github", "secretEnv": "GITHUB_WEBHOOK_SECRET" } },
  "steps": [...]
}
```

The verify middleware becomes built-in for the named provider. Same security guarantees (byte-exact `rawBody` HMAC, replay protection, timestamp window).

**Adapter interface (refining the existing one):**

```ts
interface QueueAdapter {
  readonly name: string;
  readonly provider: string;

  connect(opts: ConnectOpts): Promise<void>;
  disconnect(): Promise<void>;
  healthCheck(): Promise<boolean>;

  // Consumer side
  subscribe(opts: SubscribeOpts, handler: (msg: QueueMessage) => Promise<void>): Promise<Subscription>;
  stopProcessing(queue: string): Promise<void>;

  // Producer side
  publish(opts: PublishOpts, payload: unknown): Promise<{ messageId: string }>;
  publishBatch?(opts: PublishOpts, payloads: unknown[]): Promise<{ messageIds: string[] }>;
}

interface SubscribeOpts {
  queue: string;
  consumerGroup?: string;
  concurrency?: number;
  prefetch?: number;
  fromBeginning?: boolean;
}

interface PublishOpts {
  queue: string;
  delayMs?: number;
  priority?: number;
  dedupId?: string;        // BullMQ jobId; NATS Nats-Msg-Id; SQS FIFO MessageDeduplicationId
  ttlMs?: number;
}
```

This matches the existing `triggers/worker/src/adapters/{NATSAdapter,BullMQAdapter,InMemoryAdapter}.ts` shape. The `queue` trigger's adapter files (`KafkaAdapter`, `RabbitMQAdapter`, etc.) fill in the rest.

**Operational:**

- Concurrency: per-adapter (BullMQ's `concurrency` option, NATS's `max_ack_pending`).
- Retries: per-adapter (BullMQ's `attempts` + exponential backoff, NATS's redelivery).
- Dead-letter: per-adapter (BullMQ's `removeOnFail` + DLQ, RabbitMQ's `x-dead-letter-exchange`).
- Idempotency: per-message via the provider's `dedupId` mechanism. Falls back to `idempotencyKey` on the workflow's first step if the adapter doesn't support broker-level dedup.

### Pub/Sub trigger

Pub/Sub is distinct from Queue: pub/sub is **1:N fan-out by default** (every subscriber sees every message), and topic patterns use wildcards (`orders.*.created`). Queue is **1:1 within consumer group** (competing consumers).

The provider matrix is intentionally different from Queue (although some providers — NATS, Kafka, Redis — appear in both because they support both semantics).

**Authoring surface (JSON, NATS subject hierarchy):**

```json
{
  "name": "audit-all-order-events",
  "trigger": {
    "pubsub": {
      "provider": "nats",
      "topic": "orders.>",
      "durable": true,
      "startFrom": "earliest"
    }
  },
  "steps": [
    {
      "id": "log",
      "use": "@blokjs/audit-log",
      "inputs": { "event": "$.trigger.topic", "data": "$.req.body" }
    }
  ]
}
```

**GCP Pub/Sub:**

```json
{
  "trigger": {
    "pubsub": {
      "provider": "gcp",
      "topic": "projects/my-project/topics/orders",
      "subscription": "projects/my-project/subscriptions/blok-orders-audit",
      "ack": true,
      "maxMessages": 10
    }
  }
}
```

**Redis Streams:**

```json
{
  "trigger": {
    "pubsub": {
      "provider": "redis-streams",
      "topic": "events:orders",
      "consumerGroup": "blok-audit",
      "startFrom": "$"
    }
  }
}
```

**Library picks per provider:**

| Provider | Library | Notes |
|---|---|---|
| nats | `nats` (Core + JetStream) | Default. Cheapest infra. Subject wildcards. |
| redis-streams | `ioredis` | Durable Redis streams (vs transient Redis pub/sub). |
| kafka | `kafkajs` | High-throughput. Consumer groups. |
| gcp | `@google-cloud/pubsub` | Cloud-locked. |
| aws | `@aws-sdk/client-sns` + `@aws-sdk/client-sqs` | SNS for fan-out, SQS for queueing. |
| azure | `@azure/service-bus` | Azure Service Bus. |

**Adapter interface:**

```ts
interface PubSubAdapter {
  readonly provider: string;
  connect(): Promise<void>;
  disconnect(): Promise<void>;

  publish(topic: string, payload: unknown, opts?: { partitionKey?: string; orderingKey?: string }): Promise<void>;

  subscribe(
    topicPattern: string,
    handler: (msg: PubSubMessage) => Promise<void>,
    opts?: {
      durable?: boolean;
      consumerGroup?: string;     // present → competing-consumer; absent → fan-out
      startFrom?: "earliest" | "latest" | { seq: number } | { timestamp: number };
    },
  ): Promise<Subscription>;
}
```

Note: omitting `consumerGroup` is fan-out (every subscriber sees every message); providing one is competing-consumer (1 of N within group). This single field disambiguates the two semantics — matches NestJS's `@MessagePattern` vs `@EventPattern` distinction.

**Producer (publishing from a workflow):**

`@blokjs/pubsub-publish` step:

```json
{
  "id": "broadcast",
  "use": "@blokjs/pubsub-publish",
  "inputs": {
    "provider": "nats",
    "topic": "orders.created.us-east",
    "payload": { "orderId": "$.state.created.id" }
  }
}
```

## Server architecture

**Same Hono port (the HTTP server):**

The HTTP trigger's existing `serve({ fetch: app.fetch, port })` call hosts:

- HTTP routes (the main trigger today)
- WebSocket routes via `@hono/node-ws`'s `injectWebSocket(server)`
- SSE routes via Hono's `streamSSE` helper (just GET handlers)
- Webhook routes (POST handlers with provider-specific verifier middleware)
- Studio trace router (`/__blok/*`)
- Cron jobs (in-process timers — no port)
- Queue/Pub-Sub **producers** (workflow steps that publish — no separate port)

**Separate worker processes:**

- Queue **consumers** — long-lived broker connections, decoupled from HTTP lifecycle. Started via `bun run queue:dev` (renamed from `worker:dev`).
- Pub/Sub **subscribers** — same shape.

The orchestrator (a new `bun run dev` that boots everything, or the existing `blokctl dev` from CLAUDE.md) constructs ONE Hono app, threads it through every "same-port" trigger via constructor injection, then calls `serve(...)`. Worker-process triggers are started in parallel child processes. All share the same `WorkflowRegistry` (workflow name → workflow definition) which is loaded from `WORKFLOWS_PATH`.

```
┌─ HTTP server process (port 4000) ─────────────────────┐
│  Hono app                                              │
│    ├─ HTTP trigger routes                              │
│    ├─ WebSocket trigger routes (via @hono/node-ws)     │
│    ├─ SSE trigger routes (via hono/streaming)          │
│    ├─ Webhook trigger routes (provider-specific verify)│
│    ├─ Studio /__blok/* trace router                    │
│    └─ Queue / Pub-Sub PRODUCERS (workflow steps)       │
│  Cron trigger (in-process timers)                      │
└────────────────────────────────────────────────────────┘

┌─ Queue worker process ────────────────────────────────┐
│  Adapter (BullMQ / NATS / Kafka / RabbitMQ / SQS)      │
│  WorkflowRegistry (shared via filesystem scan)         │
│  RunTracker → same sqlite as HTTP process              │
└────────────────────────────────────────────────────────┘

┌─ Pub/Sub worker process ──────────────────────────────┐
│  Adapter (NATS / Redis Streams / Kafka / GCP / AWS)    │
│  Same WorkflowRegistry / RunTracker                    │
└────────────────────────────────────────────────────────┘
```

The HTTP process is the most complex — it serves 5 trigger types on one port. Each trigger registers its Hono routes during boot, then `serve()` starts the server. The trigger's `listen()` method (already defined on `TriggerBase`) takes the shared Hono app as a constructor argument:

```ts
const app = new Hono<AppBindings>();
const http = new HttpTrigger(app);
const ws = new WebSocketTrigger(app);
const sse = new SSETrigger(app);
const webhook = new WebhookTrigger(app);

await Promise.all([http.listen(), ws.listen(), sse.listen(), webhook.listen()]);
// `listen()` now means "register my routes on the shared app", not "start a server".
// The one server-starter is `serve(app)` at the end of HttpTrigger.listen().
```

This refactor of `TriggerBase.listen()` semantics is one of the implementation PRs.

## Authentication / middleware integration

The existing 3-tier middleware chain (process-global → workflow-level → trigger-level) from v0.5/v0.6 extends UNIFORMLY:

```json
// Same workflow, different trigger — same middleware chain works.

// As an HTTP trigger
{
  "trigger": {
    "http": { "method": "POST", "middleware": ["jwt-auth", "rate-limit"] }
  }
}

// As a WebSocket trigger
{
  "trigger": {
    "websocket": { "path": "/ws/chat", "middleware": ["jwt-auth", "rate-limit"] }
  }
}

// As an SSE trigger
{
  "trigger": {
    "sse": { "path": "/events", "middleware": ["jwt-auth"] }
  }
}

// As a Webhook trigger
{
  "trigger": {
    "webhook": { "provider": "stripe", "middleware": ["audit-log"] }
  }
}
```

The middleware definition (`middleware/jwt-auth.json`) is identical across all four — it reads `ctx.request.headers.authorization`, verifies the JWT, sets `ctx.state.identity`. The trigger's `applyMiddlewareChain` method (shipped in v0.6 PR #75) runs the chain before the main workflow regardless of trigger kind.

**Queue / Pub-Sub middleware** runs against the synthesized `ctx.request` the trigger builds from the inbound message. Most production use cases will skip middleware on queue triggers (broker-level auth via NATS NKeys / Redis ACL / Kafka SASL is the right layer), but the chain is available if needed for per-message validation.

## Observability

Every new trigger participates in the existing tracing infrastructure:

**`WorkflowRun` records:**

- HTTP / Webhook: one run per request (existing behavior).
- WebSocket: one run per workflow invocation per message. `connect`, `disconnect`, and `message` events each create their own runs.
- SSE: one run per stream open (the run lives as long as the connection in Pattern A).
- Queue: one run per delivered message (existing `WorkerTrigger` behavior).
- Pub/Sub: one run per subscribed message.

**Trigger type tagging:**

`WorkflowRun.triggerType` gets new values:

- `"websocket.connect"`, `"websocket.message"`, `"websocket.disconnect"`
- `"sse.open"` (and `"sse.event"` if Pattern B opt-in)
- `"webhook"`
- `"worker"` (existing — gains adapter tag in `triggerSummary`, e.g. `worker[bullmq]/email-jobs`)
- `"pubsub"`

Studio's run list filter learns these new values.

**OTel counters (added across all new triggers):**

```
blok_websocket_connections_active{workflow} (gauge)
blok_websocket_messages_received_total{workflow}
blok_websocket_messages_dropped_total{workflow, reason}
blok_sse_connections_active{workflow} (gauge)
blok_sse_events_sent_total{workflow, channel}
blok_webhook_received_total{provider, event_type}
blok_webhook_rejected_total{provider, reason}  // reasons: invalid_signature, replay, timestamp_outside_window
blok_worker_messages_consumed_total{provider, queue}
blok_worker_messages_failed_total{provider, queue}
blok_pubsub_messages_received_total{provider, topic}
```

**Studio rendering:**

- Run list shows trigger icon per type (existing pattern).
- WebSocket run detail shows the connection age, message count, and a link to peer messages on the same connection.
- SSE run detail shows events-sent timeline.
- Webhook run detail shows the verified signature, event type, and replay-cache lookup outcome.

## Implementation phasing

5 sequenced PRs. Each independently mergeable; each ships a usable trigger end-to-end (workflow + smoke + tests + docs).

### PR 1 — Hono server architecture refactor (~3 days, LOW risk)

**Why first:** all the same-port triggers depend on the Hono app being injectable rather than constructed inside `HttpTrigger`. Today `HttpTrigger` owns the `Hono` instance. Refactor so:

- `TriggerBase` accepts an optional `app: Hono` constructor argument.
- `HttpTrigger.listen()` becomes "register HTTP routes on the shared app and call `serve()` if no other trigger has started the server".
- New top-level orchestrator (e.g. `blokctl dev` or a `bun run dev`) constructs the Hono app, instantiates every same-port trigger, awaits all `listen()`s.

Backward-compatible: `new HttpTrigger()` without an arg keeps working (constructs its own Hono).

No new triggers in this PR. Just the plumbing.

- **gRPC as a workflow trigger** — gRPC remains a transport for cross-language nodes. If we want to expose Blok workflows as gRPC services (which would make sense for some users), that's a separate v0.7 ask.
- **Email Workers (inbound email triggers)** — Cloudflare-specific pattern. Not on the roadmap.
- **GraphQL subscriptions** — would naturally layer on the SSE trigger if implemented. v0.7 follow-up.
- **WebSocket protocol negotiation (MQTT-over-WS, GraphQL-over-WS)** — keep WebSocket as raw text/binary. Higher-level protocols are application concerns.
- **Webhook event archival / retrospective replay UI** — store verified events in a `webhook_events` table for replay via Studio. Useful but separate concern.
- **Queue ordering / partition keys** — FIFO SQS, Kafka partition keys, NATS subjects with ordering — defer to demonstrated demand.

## Recommended resolutions

The 7 questions raised in the initial design pass, with concrete recommendations + reasoning. Six confirm the initial position; **Q1 flips** to a simpler model after further thought.

### Resolutions at a glance

| Q | Recommendation | Confidence | Change to spec |
|---|---|---|---|
| Q1 — Merge `worker` / `queue`? | **Drop the `queue` trigger name. Keep `worker` only. Fold all queue adapters into the existing worker trigger.** | High | **FLIP** from initial spec |
| Q2 — WS per-message vs per-connection? | **Per-message workflow runs.** | High | Confirms spec |
| Q3 — SSE Pattern A vs B default? | **Pattern A default. Pattern B opt-in via `mode: "per-event"`.** | High | Confirms spec |
| Q4 — Webhook polymorphic dispatch canonical? | **Yes. Document as canonical pattern; ship polymorphic name resolution in PR 4. Authors who prefer single-workflow + `switchOn` still works.** | High | Confirms spec |
| Q5 — Per-trigger `concurrencyKey` on WS? | **Same as HTTP — max N concurrent workflow runs per resolved key (i.e. per-message slots, since Q2 picks per-message runs). Connection cap and per-conn rate are separate fields.** | High | Confirms spec |
| Q6 — WS binary frames? | **Text default. Binary opt-in via `mode: "binary"` on the trigger config.** | High | Confirms spec |
| Q7 — Adapter discovery? | **Per-workflow `provider` field. Env-var (`BLOK_WORKER_ADAPTER`) sets the default when `provider` is omitted (preserves v0.6 behavior).** | High | Confirms spec with clarification |

The detailed reasoning for each is below.

### Q1 — Drop the `queue` trigger name; keep `worker` only

**Recommendation:** flip the initial spec's position. Do NOT introduce a `queue` trigger name. Instead, **extend the existing `worker` trigger's adapter pool from 3 backends (BullMQ, NATS JetStream, InMemory) to 7 (add Kafka, RabbitMQ, SQS, Redis Streams, pg-boss).** The `triggers/queue/` package directory becomes dead code that gets deleted; its adapter scaffolds (`KafkaAdapter.ts`, `RabbitMQAdapter.ts`, etc.) move into `triggers/worker/src/adapters/`.

**Reasoning:**

The initial spec proposed merging `worker → queue` because "queue is what's technically happening." Reconsidering: that's the IMPLEMENTATION view, not the AUTHOR view. The Node ecosystem's user-facing primitive is uniformly called "worker" or "task":

- **BullMQ**: `new Worker(queue, handler)` — the workers consume from queues.
- **Sidekiq** (Ruby, the dominant queue lib): "Sidekiq workers."
- **Trigger.dev**: `task()` — "tasks" are the unit of work, queues are infrastructure.
- **Inngest**: `inngest.createFunction()` — "functions" are the unit.
- **AdonisJS adonis-jobs / adonis-bull-queue**: `Job` classes, dispatched via `queue.dispatch(JobClass)`.

NONE of these expose "queue" as the user-facing trigger name. "Queue" is the infrastructure underneath; "worker" or "task" or "function" is what the author writes. The friendly mental model is: "I'm writing a worker that runs whenever a job arrives" — not "I'm subscribing to a queue."

The provider field handles the "which backend" question without needing two trigger names. `trigger: { worker: { provider: "kafka", topic: "user-events" } }` is just as expressive as `trigger: { queue: { provider: "kafka", topic: "user-events" } }`, and it's 100% backward-compatible with v0.6 worker workflows (which use `provider: "bullmq" | "nats" | "in-memory"` today).

**Trade-offs:** the `worker` name might feel slightly opinionated for cases that are clearly "topic consumer" rather than "background job processor" (e.g., consuming a Kafka analytics stream). But this is a documentation problem, not an API problem — the trigger does the right thing regardless of how authors think about it.

**Implementation impact on PR 5:** smaller, simpler scope. No new `queue` trigger directory, no alias mechanism, no migration path. Just extend the existing `WorkerTrigger`'s adapter list and add the 5 new adapter classes. PR title becomes "feat(v0.7): extend Worker trigger with Kafka/RabbitMQ/SQS/Redis/pg-boss adapters". Estimated effort drops slightly (4-5 days → 3-4 days) because there's no new trigger plumbing to write.

### Q2 — Per-message WebSocket workflow runs

**Recommendation:** confirm the initial spec. Each WebSocket message creates its own `WorkflowRun`. The `connect` and `disconnect` events also create their own runs. Connection-level state (auth context, room membership, session counter) lives in `ctx.connection.attachment` — a per-connection scratchpad set during the `connect` workflow and read by subsequent `message` workflows.

**Reasoning:**

The per-message model has clear, decisive advantages on all systems-design dimensions:

- **Trace coherence.** Studio's existing trace UI renders one workflow run as one tracked execution. A 1-hour chat session as ONE workflow run would mean a 1-hour-running NodeRun that pegs the UI's "Running" indicator, breaks duration metrics, and produces no useful trace timeline.
- **Failure isolation.** A bad message that throws shouldn't kill the connection — per-message runs let the failure bubble up to a single message's response, leaving the connection alive for the next message.
- **Standard ops primitives apply.** `concurrencyKey`, `idempotencyKey`, `retry`, `maxDuration`, `delay` all assume one-workflow-run = one-unit-of-work. Per-message preserves this. Per-connection would force redefining each primitive's semantics for the WS case (does `retry: 3` mean retry the whole 1-hour connection?).
- **Run model match.** Blok's whole architecture assumes one trigger event → one run. Per-message keeps that invariant; per-connection breaks it.

The per-connection model's appeal — "matches the chat session mental model" — is served just as well by the `ctx.connection.attachment` API (Cloudflare-style scratchpad). Session-scoped state is a property of the CONNECTION, not the RUN. Each message reads the attachment, processes, optionally mutates the attachment, returns.

**Trade-offs:** an extremely chatty connection (say, 1000 msgs/sec) produces 1000 runs/sec in tracing, which could be expensive at scale. Mitigation: existing `BLOK_TRACE_ENABLED=false` kill-switch shipped in v0.5, or a future per-trigger sampling knob (e.g., trace 1 in 100 messages). Document as a known scaling concern.

**Implementation impact:** none beyond what's already in PR 2's scope. The `ctx.connection.attachment` API is the load-bearing primitive.

---

### Q3 — SSE Pattern A default, Pattern B opt-in

**Recommendation:** confirm the initial spec. Pattern A (workflow runs once at stream open, yields events for the connection's lifetime) is the default. Pattern B (per-event workflow run with full step pipeline) is opt-in via `trigger.sse.mode: "per-event"`.

**Reasoning:**

SSE's overwhelmingly common use case is **"subscribe to an upstream event source and forward filtered/transformed events to the connected client."** Examples: live order updates, dashboard tickers, log tails, AI chat completions streaming character-by-character. In every case, the workflow's job is to OWN the event stream — pull from a source (NATS subject, Redis Stream, in-process bus, AI provider) and yield via `streamSSE`.

Pattern A maps cleanly onto this:

```json
{
  "steps": [
    { "id": "subscribe", "use": "@blokjs/sse-subscribe", "inputs": { "channels": ["order:{orderId}"] } },
    { "id": "stream", "use": "@blokjs/sse-stream", "inputs": { "source": "$.state.subscribe" } }
  ]
}
```

One run at open. The workflow lives as long as the connection. Subscribe + forward via async iterators. Done.

Pattern B (per-event workflow runs) is overpowered for this. For a 100 events/sec stream, it would produce 100 workflow runs/sec — observability noise, run-storage bloat, and significant overhead relative to the actual work (forwarding bytes).

**When Pattern B is right:** the rare workflow that wants to RUN FULL STEP CHAINS per event — for example, an SSE feed where each event must:
- Pass through `@blokjs/jwt-verify` (per-event reauthorization)
- Be transformed by 5 different steps with retry/idempotency semantics
- Conditionally branch into a sub-workflow

That's a real use case but uncommon enough that opt-in is correct.

**Trade-offs:** Pattern A's "yield from a step" requires a specific step kind (`@blokjs/sse-stream`). Authors who don't use that step get nothing — the workflow just runs to completion at stream open without emitting events. Mitigation: workflow normalizer warns if `trigger: { sse: {...} }` workflow has no `@blokjs/sse-stream` step.

**Implementation impact on PR 3:** Pattern A only. Pattern B deferred to a future PR (or v0.7.x patch release if demand emerges).

---

### Q4 — Webhook polymorphic sub-workflow dispatch as canonical

**Recommendation:** confirm the initial spec. Polymorphic sub-workflow dispatch (`subworkflow: "$.req.body.type"`) is the **recommended canonical pattern**. Document loudly. Authors who prefer single-workflow with `switchOn` for tight-coupling event types are also supported — both shapes work.

**Reasoning:**

Webhook providers fall into two cardinality classes:

- **High cardinality** (Stripe: 150+ events, GitHub: 60+ events, Slack: 40+ events). Polymorphic dispatch wins decisively here. One workflow file per event type (`stripe.invoice.paid.json`, `stripe.customer.subscription.created.json`, etc.) means:
  - Each event handler is small, focused, easy to test/review.
  - Adding a new event type = adding a new file (no editing of a giant switch).
  - Studio shows one trace per event type — clean observability.
  - Sub-workflow lineage in tracing follows naturally.
- **Low cardinality** (Slack Slash Commands: 1 event type, Custom webhooks: often 1-3 types). Single-workflow with `switchOn` is fine and avoids file fan-out.

The polymorphic case is the more demanding implementation requirement (today's `subworkflow:` field only supports static string names; we need mapper-evaluated expressions like `$.req.body.type` or `js/ctx.req.body.type`). Ship that feature in PR 4 (webhook trigger).

**Trade-offs:**

- **Polymorphic resolution failure modes need clear errors.** If `$.req.body.type` resolves to a workflow name that doesn't exist, the runner needs a clean "unknown workflow" error with a list of registered workflows in the same provider's namespace.
- **Workflow naming convention.** For Stripe, conventional names are `stripe.invoice.paid` — but Blok's `WorkflowRegistry` doesn't enforce namespacing today. PR 4 should add a `namespace` field (`trigger.webhook.namespace: "stripe"`) that prefixes the resolved subworkflow name, so `$.req.body.type === "invoice.paid"` resolves to workflow `stripe.invoice.paid` automatically.

**Implementation impact on PR 4:** scope grows slightly — add polymorphic sub-workflow resolution + the namespace field to `WorkflowRegistry`. Estimated effort goes from 5-6 days to 6-7 days. Worth it: polymorphic dispatch is the flagship demo of "Blok-as-declarative-router."

### Q5 — Per-trigger `concurrencyKey` matches HTTP semantics

**Recommendation:** confirm the initial spec. `concurrencyKey` on any trigger means "max N concurrent workflow runs per resolved key." Since Q2 picks per-message runs on WebSocket, this naturally becomes per-message slots on WS. Per-connection limits and per-connection rates use separate fields (`maxConnections` and `messageRateLimit` respectively).

**Reasoning:**

Three different concerns each get their own knob:

| Concern | Knob | Semantics |
|---|---|---|
| Max concurrent processing per business key | `concurrencyKey: "$.req.params.roomId"` | "max 100 message-processing slots per room" (same key, same slots, just on WS as on HTTP) |
| Max total open connections (DoS protection) | `maxConnections: 10000` | "trigger refuses upgrade beyond 10K total connections" |
| Max message rate per connection | `messageRateLimit: 100` | "drop messages above 100/sec per individual connection" |

Overloading `concurrencyKey` to ALSO mean per-connection limits would create ambiguity. The HTTP precedent is correct and uniform.

**Trade-offs:** authors who want "max 100 chatters in a room" need to express it via `maxConnections` + a workflow-level check during the `connect` workflow (count room members, throw if exceeded). That's a tiny bit of boilerplate but separates the connection-count concern from the run-concurrency concern. Future improvement: a `maxConnectionsPerKey` field for first-class connection capping per key.

**Implementation impact:** none — the existing `ConcurrencyOptsFields` from `TriggerOpts.ts` already works on all triggers. Just add `maxConnections` + `messageRateLimit` to the `WebSocketTriggerOptsSchema` per the spec.

### Q6 — Text-mode default, binary opt-in

**Recommendation:** confirm the initial spec. WebSocket triggers default to text mode (UTF-8 framing, JSON parse on payload). Binary mode opt-in via `trigger.websocket.mode: "binary"` — in binary mode, `ctx.request.body` is a `Buffer`, no JSON parse.

**Reasoning:**

The 95/5 rule. 95% of WS use cases (chat, notifications, real-time data, dashboards, AI streaming) are JSON over text frames. Defaulting to text:
- Lets the workflow author use `$.req.body.foo` naturally (mapper expressions assume parsed objects).
- Matches the HTTP trigger's body handling (which also parses JSON by default).
- Browser clients send text by default.

Binary mode (the 5%) is for media streaming (audio chunks, video frames), IoT (compact binary protocols), or custom framed protocols. These need:
- No JSON parse on incoming buffers.
- The ability to send Buffer responses via `@blokjs/ws-broadcast` / `@blokjs/ws-reply`.
- Author-managed framing (length prefixes, custom headers).

Opt-in via `mode: "binary"` keeps the common case clean while making the unusual case explicit.

**Trade-offs:** "mixed mode" workflows (some text frames, some binary on the same connection) aren't supported by this flag — the mode is per-trigger. Workarounds: separate WS endpoints per mode, or future enhancement for per-message mode auto-detection. Document as a known limitation for now.

**Implementation impact:** ~30 lines in PR 2 — a branch in the inbound-message handler that skips JSON parse and passes the Buffer through to `ctx.request.body`.

---

### Q7 — Per-workflow `provider` field with env-var default

**Recommendation:** confirm the initial spec with explicit clarification on the env-var fallback. The workflow's `trigger.worker.provider` field picks the adapter (`bullmq` | `nats` | `kafka` | `rabbitmq` | `sqs` | `redis-streams` | `pg-boss` | `in-memory`). When `provider` is omitted, fall back to the `BLOK_WORKER_ADAPTER` env var (preserves v0.6 behavior). When neither is set, error at workflow load time.

**Reasoning:**

Heterogeneous deployments are a real need. Production setups often have:
- A high-priority queue on BullMQ (Redis-backed, supports priority + delays)
- An event spine on Kafka (high-throughput, partitioned, replayable)
- A long-tail notifications queue on SQS (cloud-native, no infra to manage)

A single env-var-per-process model would force operators to run THREE separate worker processes, one per backend, each with its own subset of workflows. That's operational overhead for what's fundamentally a configuration choice.

Per-workflow `provider` solves this with one worker process and one adapter pool — the pool lazily creates a connection per unique provider config (one BullMQ Redis connection for all bullmq workflows, one Kafka client for all kafka workflows, etc.).

The env-var fallback preserves v0.6 backward compatibility: existing `trigger.worker` configs without `provider` continue working under `BLOK_WORKER_ADAPTER=bullmq` (or whatever was set).

**Trade-offs:** the adapter pool layer needs careful connection-lifecycle management (one connection per provider config, reused across workflows, cleaned up on shutdown). Mitigation: this is exactly the pattern the existing `WorkerTrigger` already implements for its 3 adapters — extend the pool's pluggable interface, no new architecture needed.

**Implementation impact on PR 5:** moderate complexity. Each new adapter (Kafka, RabbitMQ, SQS, Redis Streams, pg-boss) lives in `triggers/worker/src/adapters/` alongside the existing 3. The `Configuration` layer resolves `provider` against the registered adapter list at workflow load. Existing tests should be augmented with per-adapter integration tests (one per provider, real broker via docker-compose for CI).
