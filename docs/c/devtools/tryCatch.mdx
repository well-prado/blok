---
title: "tryCatch"
description: "JS-like exception handling for sub-pipelines — try/catch/finally with an `$.error` proxy in the catch arm and finally semantics that match JavaScript exactly."
icon: "shield-halved"
---

`tryCatch` wraps a sub-pipeline in JavaScript-style exception handling. If any step inside `try` throws, control jumps to `catch`. The optional `finally` arm runs unconditionally — on success, on caught error, and even on an uncaught throw from inside `catch`.

Use it for:
- **Sagas** — multi-step work where you need to roll back on failure.
- **Best-effort steps** — non-critical work you'd rather log than crash on.
- **Always-run cleanup** — audit logs, metrics, lock release that must fire whether the work succeeded or not.

---

## Authoring (TS DSL)

```ts
import { workflow, tryCatch, $ } from "@blokjs/helper";

export default workflow({
  name: "saga",
  trigger: { http: { method: "POST" } },
  steps: [
    tryCatch({
      id: "transactional-write",
      try: [
        { id: "create",   use: "@blokjs/api-call", inputs: { url: "...", method: "POST" } },
        { id: "risky",    use: "@blokjs/api-call", inputs: { url: $.req.body.url, method: "POST" } },
        { id: "finalize", use: "@blokjs/api-call", inputs: { url: "...", method: "POST" } },
      ],
      catch: [
        {
          id: "rollback",
          use: "@blokjs/api-call",
          inputs: {
            url: "...",
            method: "POST",
            body: { reason: $.error.message, errorName: $.error.name },
          },
        },
      ],
      finally: [
        { id: "metric", use: "@blokjs/audit-log", inputs: { event: "saga-completed" } },
      ],
    }),
  ],
});
```

## Runtime contract

- **`try` runs first.** If every step succeeds, `catch` is skipped and `finally` runs.
- **On any throw**, `ctx.error` is populated and `catch` runs. `$.error` is **only** defined inside `catch` — it's `undefined` in `try` and `finally`.
- **`finally` always runs**, even when `catch` itself throws. JavaScript-style: a throw from `finally` supersedes a pending throw from `catch`.
- **State mutations from any block** — try, catch, or finally — are visible to subsequent top-level steps. That's how saga rollback flags propagate.
- **Catch can swallow OR rethrow.** A catch that runs to completion without throwing makes the tryCatch step "succeed" — execution continues with the next top-level step. To fail the workflow on a caught error, throw explicitly via `@blokjs/throw` (lets you set the HTTP status code and response body).
- **An errored step does NOT write `ctx.state`.** When a step inside `try` throws, `ctx.state[<step.id>]` is left **`undefined`** — not an empty object, not a placeholder. That means `ctx.state['create-account'] !== undefined` is a truthful "did this step actually succeed?" check inside the catch arm. (Pre-v0.5.1 the runner persisted an empty envelope on error; that was a bug, since fixed.)

### The `$.error` envelope

| Field | Type | Source |
|---|---|---|
| `$.error.message` | `string` | The original error message — unwrapped through the `.cause` chain so framework wrap text doesn't leak |
| `$.error.name` | `string` | Error class name (e.g. `"Error"`, `"UnauthorizedError"`, `"ZodError"`) |
| `$.error.stack` | `string \| undefined` | Stack trace when the error carries one |
| `$.error.code` | `number \| undefined` | HTTP-style status carried by `GlobalError` — set via `@blokjs/throw inputs.code`, or by the framework's ZodError → 400 mapping. Lets a catch arm re-throw with the same status, branch on 4xx vs 5xx, or include it in audit-log payloads |
| `$.error.stepId` | `string \| undefined` | The id of the try-arm step that threw. Lets a catch arm route by failure point ("payment failed → notify billing", "inventory failed → notify warehouse") without regex-parsing the message |

### Lifecycle — when `$.error` is in scope

`$.error` (and the underlying `ctx.error`) is scoped to the `catch` arm only. The runner snapshots whatever `ctx.error` held when the tryCatch step entered, populates it with the caught envelope just before `catch` runs, and **restores the snapshot before `finally` runs and before downstream top-level steps execute**. This mirrors JavaScript's binding rules exactly — the catch variable falls out of scope when the catch block exits.

| Location | `$.error` |
|---|---|
| Step before the tryCatch | unchanged (typically `undefined`) |
| Inside `try` | `undefined` (no error yet) |
| Inside `catch` | populated with the envelope above |
| Inside `finally` | restored to the pre-tryCatch value (typically `undefined`) |
| Step after the tryCatch (catch swallowed) | restored to the pre-tryCatch value |
| Step after the tryCatch (catch rethrew) | downstream steps don't run — the throw propagates |

If you need the caught error visible downstream (audit log, response shaping), copy it into `ctx.state` from inside `catch`:

```json
{
  "id": "stash-error",
  "use": "@blokjs/expr",
  "inputs": {
    "expression": "({ failedAt: ctx.error.stepId, reason: ctx.error.message, code: ctx.error.code })"
  }
}
```

Downstream steps reference it as `$.state['stash-error'].failedAt` etc. — values stored in `ctx.state` from inside the catch arm survive past the tryCatch boundary even though `ctx.error` does not. This is the same pattern the [conditional rollback example](#conditional-rollback-—-the-canonical-saga-pattern) below uses for its `outcome` / `failedAt` envelope.

Nested tryCatch nests the snapshots correctly: an inner catch sees the inner error, and when the inner tryCatch exits, the outer catch's `$.error` is restored (not clobbered).

## Conditional rollback — the canonical saga pattern

A common saga pattern: only roll back work that actually completed. Pre-v0.5.1 this required a workaround marker step; post-v0.5.1 the natural existence check works directly.

```json
{
  "id": "signup-saga",
  "tryCatch": {
    "try": [
      { "id": "create-account", "use": "@blokjs/api-call", "inputs": { /* ... */ } },
      { "id": "send-welcome",   "use": "@blokjs/api-call", "inputs": { /* ... */ } },
      { "id": "create-profile", "use": "@blokjs/api-call", "inputs": { /* ... */ } }
    ],
    "catch": [
      {
        "id": "decide-rollback",
        "branch": {
          "when": "ctx.state['create-account'] !== undefined",
          "then": [
            { "id": "rollback-account", "use": "@blokjs/api-call", "inputs": { /* ... */ } }
          ]
        }
      },
      {
        "id": "respond-failed",
        "use": "@blokjs/throw",
        "inputs": {
          "message": "Signup failed",
          "code": 500,
          "body": {
            "outcome": "failed",
            "failedAt": "js/ctx.error.stepId",
            "reason": "js/ctx.error.message",
            "upstreamCode": "js/ctx.error.code || null"
          }
        }
      }
    ]
  }
}
```

If `create-account` itself fails: `ctx.state['create-account']` is `undefined`, the branch's `then` doesn't fire, no rollback is needed. The throw step still runs and emits the structured 500.

If a later step (`send-welcome` or `create-profile`) fails: `ctx.state['create-account']` is set to the actual response, the branch's `then` fires, the rollback request goes out. `$.error.stepId` tells the response handler exactly which step blew up.

## Caveats

- **Errors thrown inside `catch` are not re-caught.** They propagate after `finally` runs, just like JavaScript. Don't nest a tryCatch inside a catch arm if you only want to retry — use `retry: { maxAttempts }` on the inner step instead.
- **Errors thrown inside `finally` supersede everything.** If `try` succeeded but `finally` throws, the workflow sees the `finally` error. If `catch` is currently propagating an error and `finally` also throws, the `finally` error wins.
- **Use `@blokjs/throw` for HTTP-status-shaping.** Plain `throw new Error(...)` from a `defineNode` step surfaces as HTTP 500 by default. `@blokjs/throw` with `{ code: 401, body: {...} }` lets you control both the status code and the JSON response — important when the catch arm is the user-facing failure path.
- **No mid-step durable checkpoint.** A `wait` step inside `try` will not resume the catch arm correctly across process restarts. Tracked in the v0.5 backlog as an open architectural gap.
