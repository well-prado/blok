## Overview

### What is ISO 27001

ISO/IEC 27001:2022 is the international standard for information security management systems (ISMS). It provides a systematic approach to managing sensitive information through:

- **Risk-based approach** -- Identify, assess, and treat information security risks
- **Process framework** -- Plan-Do-Check-Act (PDCA) cycle
- **Control catalogue** -- 93 controls in 4 themes (Annex A)
- **Continuous improvement** -- Regular audits and management reviews

### ISMS Structure

The ISMS for a Blok-based application consists of:

| ISMS Component | Blok Implementation |
|---|---|
| **Context of the organization (Clause 4)** | Architecture documentation, stakeholder analysis |
| **Leadership (Clause 5)** | Security policies, role definitions |
| **Planning (Clause 6)** | Risk assessment, Statement of Applicability |
| **Support (Clause 7)** | Documentation, training, communication |
| **Operation (Clause 8)** | Security controls, operational procedures |
| **Performance evaluation (Clause 9)** | Monitoring, internal audits, management review |
| **Improvement (Clause 10)** | Corrective actions, continual improvement |

### Step 2: Threat and Vulnerability Identification

| Threat | Vulnerability | Asset Affected | Likelihood | Impact |
|---|---|---|---|---|
| Brute force attack | Weak password policy | API Service | Medium | High |
| SQL injection | Insufficient input validation | Customer Database | Low | Critical |
| Insider threat | Excessive permissions | All data assets | Low | High |
| Key compromise | Insecure key storage | Encryption Keys | Low | Critical |
| Data exfiltration | Missing DLP controls | Customer Database | Medium | Critical |
| Service outage | Single point of failure | API Service | Medium | High |
| Log tampering | Mutable log storage | Audit Logs | Low | High |

### Step 3: Risk Analysis

```
Risk Level = Likelihood x Impact

           | Low Impact | Medium Impact | High Impact | Critical Impact |
High       |   Medium   |     High      |    High     |    Critical     |
Medium     |    Low     |    Medium     |    High     |      High       |
Low        |    Low     |     Low       |   Medium    |      High       |
```

### Step 4: Risk Treatment

| Treatment | Description | Example in Blok |
|---|---|---|
| **Mitigate** | Reduce risk through controls | Enable AuthMiddleware, RBAC, EncryptionAtRest |
| **Transfer** | Transfer risk to third party | Use managed Vault/AWS/GCP for key management |
| **Accept** | Accept residual risk | Document acceptance with management approval |
| **Avoid** | Eliminate the risk source | Do not process unnecessary sensitive data |

## Security Policy Templates

### Information Security Policy

**Purpose:** Establish the organization's commitment to information security and define the scope of the ISMS.

**Scope:** All Blok-based applications and supporting infrastructure.

**Policy Statement:**
1. Information assets shall be classified and protected according to their sensitivity
2. Access to information shall be granted on a need-to-know, least-privilege basis
3. All information security incidents shall be reported and investigated
4. Security controls shall be regularly reviewed and improved
5. All personnel shall receive security awareness training

### Access Control Policy

```typescript
import { RBAC, createDefaultRBAC } from "@blok/runner";

// Access Control Policy implementation
const rbac = createDefaultRBAC();

// Policy: Default roles enforce least privilege
// admin       -- Full access (restricted to security team)
// developer   -- Read, create, update, execute workflows and nodes
// operator    -- Execute and monitor workflows
// viewer      -- Read-only access
// service     -- Machine-to-machine execution only

// Policy: Access review procedure
async function conductAccessReview(rbac: RBAC, audit: AuditLogger): Promise<AccessReviewReport> {
  const roles = rbac.getRoles();
  const report: AccessReviewReport = {
    reviewDate: new Date().toISOString(),
    reviewer: "security-team",
    roles: roles.map(role => ({
      name: role.name,
      description: role.description || "",
      permissionCount: role.permissions.length,
      inheritsFrom: role.inherits || [],
      effectivePermissions: rbac.getEffectivePermissions(role.name).length,
    })),
    findings: [],
    nextReviewDate: getNextQuarterDate(),
  };

  // Check for overly permissive roles
  for (const role of roles) {
    const perms = rbac.getEffectivePermissions(role.name);
    for (const perm of perms) {
      if (perm.resource === "*" && perm.actions.includes("*") && role.name !== "admin") {
        report.findings.push({
          severity: "high",
          finding: `Role "${role.name}" has wildcard permissions -- review required`,
          recommendation: "Apply least privilege; restrict to specific resources and actions",
        });
      }
    }
  }

  audit.log({
    category: "system",
    severity: "info",
    action: "access_review_completed",
    success: true,
    details: {
      rolesReviewed: report.roles.length,
      findingsCount: report.findings.length,
    },
  });

  return report;
}

interface AccessReviewReport {
  reviewDate: string;
  reviewer: string;
  roles: Array<{
    name: string;
    description: string;
    permissionCount: number;
    inheritsFrom: string[];
    effectivePermissions: number;
  }>;
  findings: Array<{
    severity: string;
    finding: string;
    recommendation: string;
  }>;
  nextReviewDate: string;
}
```

### A.5.24/25/26 Incident Management

```typescript
import { AuditLogger, StructuredLogger, SentryIntegration } from "@blok/runner";

// Incident management workflow
interface SecurityIncident {
  id: string;
  title: string;
  severity: "low" | "medium" | "high" | "critical";
  status: "new" | "triaging" | "investigating" | "containing" | "recovering" | "closed";
  detectedAt: string;
  detectedBy: string;
  assignedTo: string;
  affectedAssets: string[];
  description: string;
  timeline: Array<{ timestamp: string; action: string; actor: string }>;
  rootCause?: string;
  corrective_actions?: string[];
  lessonsLearned?: string;
}

async function createIncident(
  incident: Omit<SecurityIncident, "id" | "timeline">,
  audit: AuditLogger,
): Promise<SecurityIncident> {
  const fullIncident: SecurityIncident = {
    ...incident,
    id: `INC-${Date.now()}`,
    timeline: [{
      timestamp: new Date().toISOString(),
      action: "Incident created",
      actor: incident.detectedBy,
    }],
  };

  audit.logSecurityEvent({
    action: "incident_created",
    severity: incident.severity === "critical" ? "critical" : "error",
    details: {
      incidentId: fullIncident.id,
      title: incident.title,
      severity: incident.severity,
      affectedAssets: incident.affectedAssets,
      isoReference: "A.5.24",
    },
    actor: { sub: incident.detectedBy },
  });

  return fullIncident;
}

async function updateIncidentStatus(
  incidentId: string,
  newStatus: SecurityIncident["status"],
  actor: string,
  notes: string,
  audit: AuditLogger,
): Promise<void> {
  audit.logSecurityEvent({
    action: "incident_status_update",
    severity: "warn",
    details: {
      incidentId,
      newStatus,
      notes,
      isoReference: "A.5.25",
    },
    actor: { sub: actor },
  });
}
```

### Automated Compliance Checks

```typescript
import { RBAC, AuditLogger, SecretManager } from "@blok/runner";

// Automated ISO 27001 compliance checks
async function runComplianceChecks(
  rbac: RBAC,
  audit: AuditLogger,
  secrets: SecretManager,
): Promise<ComplianceReport> {
  const checks: ComplianceCheck[] = [];

  // Check 1: RBAC roles defined (A.5.2)
  const roles = rbac.getRoles();
  checks.push({
    control: "A.5.2",
    name: "Information security roles defined",
    passed: roles.length >= 3, // At minimum: admin, user, viewer
    details: `${roles.length} roles defined: ${roles.map(r => r.name).join(", ")}`,
  });

  // Check 2: Admin role is restricted (A.8.2)
  const adminRole = rbac.getRole("admin");
  checks.push({
    control: "A.8.2",
    name: "Privileged access is defined",
    passed: adminRole !== undefined,
    details: adminRole ? "Admin role exists with documented permissions" : "Admin role not found",
  });

  // Check 3: Audit logging is active (A.8.15)
  checks.push({
    control: "A.8.15",
    name: "Audit logging is active",
    passed: audit.getEntryCount() >= 0, // Logger is initialized
    details: `AuditLogger initialized with ${audit.getEntryCount()} entries logged`,
  });

  // Check 4: Secret management configured (A.8.24)
  const providers = secrets.getProviders();
  checks.push({
    control: "A.8.24",
    name: "Secret management configured",
    passed: providers.length > 0,
    details: `${providers.length} secret providers configured: ${providers.map(p => p.name).join(", ")}`,
  });

  // Check 5: Cache security (A.8.9)
  const cacheStats = secrets.getCacheStats();
  checks.push({
    control: "A.8.9",
    name: "Secret cache has reasonable TTL",
    passed: cacheStats.enabled && cacheStats.maxSize <= 1000,
    details: `Cache enabled: ${cacheStats.enabled}, max size: ${cacheStats.maxSize}`,
  });

  const report: ComplianceReport = {
    timestamp: new Date().toISOString(),
    checksRun: checks.length,
    checksPassed: checks.filter(c => c.passed).length,
    checksFailed: checks.filter(c => !c.passed).length,
    checks,
  };

  // Log the compliance check
  audit.log({
    category: "system",
    severity: report.checksFailed > 0 ? "warn" : "info",
    action: "compliance_check",
    success: report.checksFailed === 0,
    details: {
      standard: "ISO 27001:2022",
      checksRun: report.checksRun,
      checksPassed: report.checksPassed,
      checksFailed: report.checksFailed,
    },
  });

  return report;
}

interface ComplianceCheck {
  control: string;
  name: string;
  passed: boolean;
  details: string;
}

interface ComplianceReport {
  timestamp: string;
  checksRun: number;
  checksPassed: number;
  checksFailed: number;
  checks: ComplianceCheck[];
}
```

### Continuous Improvement

ISO 27001 requires continual improvement (Clause 10). Track improvements through:

1. **Corrective Actions** -- Address non-conformities found during audits
2. **Preventive Actions** -- Address potential weaknesses before they become issues
3. **Management Review** -- Annual review of ISMS effectiveness
4. **Metrics Trending** -- Monitor security KPIs over time via Prometheus/Grafana
5. **Lessons Learned** -- Document and apply learnings from incidents
